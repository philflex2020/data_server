Source: source/generator/generator.py

The generator publishes synthetic cell telemetry to an MQTT broker. It supports battery, solar, wind, and localgen source types. The WebSocket API provides live control and sample data previews.

Client → Server

start→ server
Begin publishing MQTT messages at the configured interval for all enabled sources.
{"type": "start"}
stop→ server
Stop publishing. Already-published messages are not recalled.
{"type": "stop"}
get_status→ server
Request current configuration and running state. Server replies with a status message.
{"type": "get_status"}
update_config→ server
Hot-update generator configuration. Only supplied fields are changed; omitted fields retain their current values. Takes effect on the next publish loop.
{
  "type": "update_config",
  "config": {
    "sample_interval_seconds": 1,
    "topic_mode":              "per_cell",
    "payload_format":          "flat",
    "sources": [
      {
        "id":               "battery",
        "type":             "battery",
        "enabled":          true,
        "topic_prefix":     "batteries",
        "sites":            2,
        "racks_per_site":   3,
        "modules_per_rack": 4,
        "cells_per_module": 6
      }
    ]
  }
}
FieldTypeRequiredDescription
config.sample_interval_secondsintoptionalSeconds between publish loops
config.topic_modestringoptional"per_cell" — one message per cell; "per_cell_item" — one per measurement
config.payload_formatstringoptional"flat" — {"voltage": 3.71}; "nested" — {"voltage": {"value": 3.71, "unit": "V"}}
config.sourcesarrayoptionalSource definitions (see below)
sources[].idstringrequiredUnique source identifier
sources[].typestringrequired"battery" | "solar" | "wind" | "localgen"
sources[].enabledboolrequiredWhether this source publishes
sources[].topic_prefixstringrequiredMQTT topic root (e.g. "batteries")
sources[].sitesintrequiredNumber of sites
sources[].racks_per_siteintrequiredRacks per site
sources[].modules_per_rackintrequiredModules per rack
sources[].cells_per_moduleintrequiredCells per module

Server → Client

status← server
Sent on connect and after any config change or start/stop. Reflects current state.
{
  "type":       "status",
  "running":    true,
  "cell_count": 144,
  "config": {
    "sample_interval_seconds": 1,
    "topic_mode":              "per_cell",
    "payload_format":          "flat",
    "sources": [...]
  }
}
stats← server (broadcast ~1/s)
Publishing performance metrics, broadcast every second while running.
{
  "type":            "stats",
  "total_published": 86400,
  "mps":             144,
  "last_loop_ms":    2.3
}
FieldTypeDescription
total_publishedintTotal messages published since start
mpsintMessages per second in last loop
last_loop_msfloatWall-clock time of last publish loop in ms
samples← server (broadcast ~1/s)
One sample payload per source per loop, for previewing data in a dashboard without subscribing to MQTT.
{
  "type": "samples",
  "samples": [
    {
      "cell_path":    "batteries/site=0/rack=1/module=2/cell=3",
      "source_id":    "battery",
      "timestamp":    1741234567.891,
      "measurements": {
        "voltage":     {"value": 3.714, "unit": "V"},
        "current":     {"value": -0.42, "unit": "A"},
        "temperature": {"value": 28.3,  "unit": "°C"},
        "soc":         {"value": 71.2,  "unit": "%"},
        "soh":         {"value": 97.8,  "unit": "%"}
      }
    }
  ]
}

MQTT Output Format

Topic format

{topic_prefix}/site={site_id}/rack={rack_id}/module={module_id}/cell={cell_id}
# Example:
batteries/site=0/rack=1/module=2/cell=3
The generator does not include project= in the topic. The stress runner does. Both formats are handled correctly by writer.cpp via the topic_kv_map config.

Payload — flat format ("payload_format": "flat")

{
  "timestamp":   1741234567.891,
  "site_id":     0,
  "rack_id":     1,
  "module_id":   2,
  "cell_id":     3,
  "voltage":     3.714,
  "current":    -0.42,
  "temperature": 28.3,
  "soc":         71.2,
  "soh":         97.8
}

Payload — nested format ("payload_format": "nested")

{
  "timestamp": 1741234567.891,
  "voltage":   {"value": 3.714, "unit": "V"},
  "current":   {"value": -0.42, "unit": "A"},
  "temperature":{"value": 28.3, "unit": "°C"}
}
writer.cpp handles both formats — parse_payload() flattens nested {value, unit} objects automatically. The Telegraf/InfluxDB path requires flat format (see docs/system_faults.md fault #11).
