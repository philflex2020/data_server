1 Remove DuckDB from parquet_writer parquet_writer shipped

Current state

The live C++ writer pod (parquet-writer-long on .46) currently embeds a DuckDB connection for two purposes:

  • COV (Change-of-Value) suppression: a last_written in-process DuckDB table tracks the last-emitted value per series so unchanged values are not flushed.
  • Ad-hoc diagnostic queries fired against the local rg3 files while the writer is running.

This couples a heavy analytical database into a write-critical hot path. DuckDB initialisation adds startup latency, holds memory, and complicates the Docker image.

Target state

  • COV state moves to a lightweight in-memory C++ hash map (std::unordered_map<SeriesKey, LastValue>) — no database, no schema, no file I/O.
  • The same hash map becomes the source for the Arrow IPC push stream (item 2): once per second the writer serialises its current-value map and pushes it to connected subscribers.
  • Diagnostic query capability moves entirely to the query pod (see item 4).
  • The writer image shrinks; the writer binary no longer links DuckDB.

Affected files

FileChange
src/parquet_writer.cppRemove DuckDB include, connection setup, last_written table. Replace COV check with in-process hash map lookup. Add 1 Hz push timer that serialises map to Arrow IPC and sends to subscribers.
src/DockerfileRemove DuckDB dev library from build stage.
src/MakefileRemove -lduckdb linker flag.
The COV map lives only in RAM. A writer restart clears it — the first post-restart value for every series is always emitted, which is correct since values may have drifted during the outage.
2 Arrow IPC push stream: writer → query pod parquet_writer flux_duck shipped revised

Design principle: push, not poll

The writer owns the data and knows when it is ready. There is no benefit to flux_duck polling at some interval and hoping data has arrived. Instead the writer pushes a snapshot to all connected subscribers once per second — matching the 1 Hz delivery requirement — and subscribers receive it the moment it is sent.

Transport: Arrow IPC stream

The writer already builds Arrow Tables for every flush. Serialising the current-value snapshot as an Arrow IPC stream costs almost nothing:

  • No JSON encoding/decoding — the writer converts its C++ hash map to an Arrow RecordBatch in the same layout as the rg3 schema and writes the IPC frame directly to the socket.
  • flux_duck receives it via pyarrow.ipc.open_stream — zero extra conversion, already a pandas DataFrame in one call.
  • Both sides already link Arrow (C++ writer: arrow/api.h; flux_duck: pyarrow). No new dependency on either side.

Overhead target: < 1%

Estimated cost per 1 Hz push for a typical site (~3,000 active series):

OperationEstimated cost
Serialise 3,000-row RecordBatch to Arrow IPC~0.3 ms
Write frame to Unix socket (co-located pod, same node)~0.05 ms
Flush interval30,000 ms
Overhead fraction~0.35 ms / 1000 ms = 0.035%

Well within the < 1% budget even over TCP between pods on the same node.

Configuration (in site.yaml)

live_stream:
  enabled:  true
  address:  "unix:///tmp/pw_live.sock"   # or "tcp://0.0.0.0:8773"
  interval_ms: 1000                      # push cadence
Unix socket is preferred when writer and query pod are co-located on the same k3s node (shared hostPath volume for the socket file). TCP is used when they run on separate nodes.

Single frame type — sequence number drives buffer management

There is one frame type. Every SNAPSHOT carries a seq field: a monotonically increasing integer that the writer increments once per parquet write, immediately after the file is renamed into place.

FieldTypeMeaning
sequint32Flush cycle counter. Constant for all 1 Hz frames within one 30 s cycle. Increments when parquet lands.
ts, unit_id, device, instance, point_name, value, value_str, qualitysame as rg3 schemaLatest value per active series (COV map snapshot).

flux_duck state machine

Buffer management is a single rule: on seq change, discard the previous cycle's frames. No separate flush notification needed.

current_seq  = None
live_frames  = []       # Arrow batches for the current (in-progress) flush cycle
live_max_ts  = 0.0      # used as join boundary against parquet

def on_snapshot(seq, batch):
    if seq != current_seq:
        # seq changed → parquet has landed for previous cycle; drop it
        live_frames.clear()
        current_seq = seq
    live_frames.append(batch)
    live_max_ts = max(live_max_ts, batch['ts'].max())
Resilience: if any individual SNAPSHOT is dropped (non-blocking send, transient socket stall), the seq on the next received frame still triggers the trim. With a separate FLUSH frame a dropped notification would require special-case handling. The sequence number approach is self-healing — no edge cases.

What the IPC stream is NOT

Each SNAPSHOT carries the COV map: one row per series, most recent value. If a series changes 5 times in one second, only the last value appears — intermediate values are dropped. This is correct for live display. It is not a WAL. See the WAL section for how a delta stream could be added later without changing this frame layout.

Back pressure

The 1 Hz push runs on a dedicated timer thread. Socket writes are non-blocking: if the send buffer is full (slow or disconnected consumer), the frame is silently dropped and the writer continues. A slow query pod never impacts writer throughput.

This is an option, not a must-have. When live_stream.enabled: false the writer behaves exactly as today; flux_duck serves only parquet history.
3 Live data path in flux_duck flux_duck shipped revised

Goal

flux_duck currently serves only historical data from parquet files (hot rg3 tier and cold S3 compact-wide tier). The live receiver (item 2) fills a short in-memory buffer of recent Arrow IPC frames. When a Flux query range includes "now", flux_duck appends rows from the live buffer to the parquet result before returning annotated CSV.

Architecture

Grafana  ──── POST /api/v2/query ───▶  flux_duck  (port 8086)
                                              │
              ┌───────────────────────────────┼───────────────────────────┐
              │                               │                           │
         hot tier                        cold tier                   live buffer
     read_parquet(rg3)            read_parquet(S3 compact)        deque of Arrow frames
     DuckDB  (≤ hot_days)         DuckDB  (> hot_days)           received from writer
              │                               │                      via IPC socket
              └───────────────────────────────┼───────────────────────────┘
                                              │
                                   pandas concat + time-sort
                                              │
                                   annotated CSV response

Live buffer design

  • A background asyncio task (or thread) in executor.py receives Arrow IPC frames from the writer socket and appends them to a collections.deque(maxlen=120) — 120 seconds of 1 Hz frames.
  • Each frame is a pandas DataFrame in the same schema as the hot-tier parquet (ts, unit_id, device, instance, point_name, value, value_str, quality).
  • When a query range extends to now, the executor concatenates the relevant frames (filtered by time range and any field/unit_id filters from the Flux query) with the parquet result and re-sorts by ts.
  • The live buffer is entirely optional. If the socket is not connected or the writer does not have live_stream.enabled: true, the deque stays empty and queries return parquet-only results.

New config fields (in site.yaml or env var override)

Key / Env varDefaultEffect
live_stream.address / FLUX_LIVE_ADDRunsetIPC socket address to connect to. When unset the live tier is disabled.
live_stream.buffer_seconds / FLUX_LIVE_BUFFER120How many seconds of live frames to retain in the deque.

Join boundary — automatic via sequence number

When a Flux query range extends to now, the executor splits cleanly between parquet history and the live buffer using live_max_ts — the highest ts seen across all frames for the current sequence:

  • Parquet query: WHERE ts ≤ live_max_ts of the previous completed cycle (i.e. any cycle with seq < current_seq).
  • Live buffer rows appended to the result: ts > that boundary.
  • The two sets are disjoint by construction — no deduplication needed.
  • On seq change the previous cycle's live_max_ts becomes the new parquet boundary, the live buffer clears, and the new cycle starts accumulating.

5 flux_duck as data-location source of truth architecture shipped

Goal

Storage tier boundaries are currently scattered across env vars in the writer pod, the compactor script, and flux_duck config. With shared site.yaml (see below), flux_duck reads the canonical config and exposes it as a GET /locations endpoint. The compactor becomes a thin client that asks flux_duck where to read hot files and where to write cold files.

Proposed GET /locations response

{
  "hot": {
    "base":      "/pool4/rg3",
    "site_dirs": true,
    "days":      14,
    "format":    "rg3-long"
  },
  "cold": {
    "s3_bucket": "fractal-ems",
    "s3_prefix": "compact",
    "format":    "compact-wide"
  },
  "boundary_days": 14
}

Compactor integration

  • The compactor (convert_wide_v2_2_args.py) calls GET /locations at startup.
  • It reads from hot.base, writes to cold.s3_bucket / cold.s3_prefix.
  • The boundary date is derived from boundary_days.
  • No CLI arg duplication — one config source (the shared site.yaml) drives writer, query, and compaction.

■ Shared site.yaml config architecture shipped

Motivation

The writer and flux_duck are tightly coupled: they share the same data stream (IPC), the same hot-tier path, the same site ID, and (in the live path) the same flush timing. Having two separate config mechanisms (writer reads site.yaml; flux_duck reads env vars) means the same values must be set twice, and they can drift.

Proposal

Extend site.yaml with a flux_duck: section and have both components read the same file. The query pod mounts the same ConfigMap that the writer pod already mounts.

## site.yaml additions

capture:
  host_path: /media/phil/.../site-capture   # writer output (existing)

live_stream:
  enabled:      true
  address:      "unix:///tmp/pw_live.sock"
  interval_ms:  1000

flux_duck:
  hot_base:     /pool4/rg3                  # same as capture.host_path parent
  site_dirs:    true
  hot_days:     14
  quality:      true
  cold:
    s3_bucket:  fractal-ems
    s3_prefix:  compact
  live_buffer_seconds: 120

Config loading priority

When a site.yaml path is given, it wins. Env vars remain valid as overrides for standalone / ad-hoc use without a yaml (e.g. running flux_duck directly against .34 data). Priority: env vars > site.yaml > built-in defaults.

■ site.yaml editor tooling shipped

Motivation

The shared site.yaml is growing: mqtt, capture, rg3file, eks, live_stream, flux_duck sections, each with typed fields, enums, and constraints. A plain text editor offers no guard against typos, wrong types, or unknown keys. Before handing site config to anyone other than a dev, a schema-aware editor is needed.

What it does

  • Inline error squiggles — red for errors (unknown fields, wrong type), yellow for warnings (value out of expected range).
  • Hover documentation — each field shows its description and allowed values on hover.
  • Autocomplete — known field names suggested as you type.
  • Problem panel — click the error count in the toolbar to expand a list of all issues with line numbers; click a row to jump to the line.
  • Schema source — JSON Schema defined inline; covers all known site.yaml sections including the new live_stream and flux_duck blocks.
  • Download — saves the current editor content as site.yaml.

Technology

  • Monaco Editor (VS Code's engine) via CDN — no build step, no npm.
  • js-yaml for YAML parsing (syntax error detection with line/column).
  • Ajv (JSON Schema validator) for schema validation against the JSON Schema.
  • Served as a standalone HTML file — site_yaml_editor.html on the .46 web server (open editor).

Current limitations

  • No server-side file read/write — paste YAML in or use Load default; download to save. Full read/write from the filesystem is the next step (small FastAPI endpoint).
  • YAML mode in Monaco requires the monaco-yaml plugin for native YAML language service; the current version uses plaintext mode + external js-yaml + Ajv. Hover docs and autocomplete are not yet wired into Monaco's language service — that upgrade can be added when needed.

■ WAL substitute — future direction future design

The Arrow IPC push stream (item 2) is structurally similar to a WAL segment:

  • Frames arrive in flush order, monotone in ts.
  • They carry the exact same schema as the rg3 parquet row groups.
  • The live buffer frames (keyed by seq) can be appended to a persistent file (.arrow / .ipc), giving a durable recovery point.

Critical distinction: snapshot vs. delta

The SNAPSHOT frame carries the COV map — one row per series, most recent value. If a series changes 5 times in a second, only the last value is in the snapshot; the intermediate values are gone. This is intentional and correct for live display, but it means:

  • The IPC stream cannot reconstruct a missing 30 s parquet flush — the intermediate changes are not there.
  • A true WAL would require a separate delta stream: every received MQTT row, in arrival order. This is a higher-bandwidth channel and a separate design.

If WAL recovery ever becomes a requirement, the path is:

  1. Add a second frame type DELTA: every received row buffered at the writer and flushed after each parquet write, covering the same rows that went into the file.
  2. Persist DELTA frames on the query pod as an append file.
  3. On recovery: detect the missing parquet file; replay the DELTA file to reconstruct it.
Design thought only — no current implementation plan. The SNAPSHOT/FLUSH frame layout in items 2 and 3 is chosen to leave this path open (DELTA is just a third frame type in the same IPC channel).
