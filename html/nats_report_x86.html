Migration Strategy — Parallel Operation

Because FlashMQ delivers messages independently to each subscriber, the new writer.cpp pipeline can run alongside the existing Telegraf → InfluxDB system with zero changes to the existing service. Cutover risk is minimal.

PhaseActionRisk
1 · Deploy alongside Start writer.cpp on the host, subscribed to FlashMQ :1883 with a distinct client_id. Telegraf continues running unchanged. Both receive every message independently — MQTT fan-out, no interference. None — Telegraf / InfluxDB untouched
2 · Validate data Run both systems in parallel. Compare parquet output to InfluxDB via spot-check queries: row counts, field values, timestamps. Run the DuckDB query suite alongside the InfluxDB equivalent queries and confirm results agree. None — read-only comparison
3 · Deploy AWS writer Start writer.cpp on AWS in current_state mode, subscribed to the local FlashMQ (bridge from host). Validate current_state.parquet matches InfluxDB last() values. None — additive
4 · Migrate consumers Switch dashboard panels and alert queries from InfluxQL → DuckDB one at a time. Old queries remain available during transition. Extend host flush_interval_seconds to 900 (15 min) at this point. Incremental — per-panel rollback available
5a · Historical export (optional) Export pre-cutover InfluxDB history to parquet so DuckDB has a single continuous store. Not required — the two stores can coexist and a DuckDB query can union S3 parquet with an InfluxDB query at the application layer if long-tail history is needed. Simplest approach: leave InfluxDB in read-only mode as a cold archive and only query it for data older than the parquet start date. Optional — skip if historical continuity is not required
5b · Decommission Once all consumers are on DuckDB and the parallel run has been stable for an agreed period, stop Telegraf and InfluxDB. Remove bridge.py, NATS, MinIO if still present. Irreversible — confirm data retention policy first
No flag day. Phases 1–3 are fully additive — nothing is changed or removed. The existing Telegraf → InfluxDB path continues to run and serve production traffic throughout. Phase 5b is the only destructive step and can be deferred indefinitely.
Why parallel operation works: FlashMQ maintains a separate QoS-1 session per client_id. Telegraf's MQTT input and writer.cpp each have their own session and receive independent copies of every published message. Neither can starve or block the other. The only shared resource is the FlashMQ broker itself, which is already handling the full fleet load.

Component Old role Replaced by

bridge.py bottleneck: the old pipeline issued one await js.publish() per message, gating throughput to a single NATS round-trip at a time. Measured ceiling was ~4k msg/s at 24× scale. writer.cpp decouples MQTT receive from disk I/O entirely via a mutex-swap buffer pattern.

Architecture Notes

QoS-1 persistent session: mosquitto_new(id, clean_session=false) with exponential reconnect (2 s → 30 s). FlashMQ queues messages while writer.cpp is offline and drains them automatically on reconnect — the same durability guarantee NATS JetStream provided, without the infrastructure overhead.
FlashMQ bridge — topic group selection: source/bridge/bridge.yaml defines named topic groups (batteries_all, solar_all, per-site groups, etc.) and one or more bridge connections, each selecting which groups to forward. Regenerate and reload live: 
python3 source/bridge/gen_bridge_conf.py bridge.yaml \
        --output /etc/flashmq/bridge-conf.d/bridge.conf
flashmq --reload-config
rsync parquet → S3: a cron or systemd timer rsyncs the local parquet store to S3 every 5 minutes. Only completed files are transferred (writer.cpp atomically creates each file). DuckDB on AWS queries S3 for all historical data. 
# host cron — every 5 minutes
*/5 * * * * aws s3 sync /srv/data/parquet/ s3://bucket/parquet/ --storage-class STANDARD_IA
AWS writer eliminates the hot.parquet gap. Deploy writer.cpp on the AWS instance subscribed to the local FlashMQ (which receives the bridge from the host). It writes current_state.parquet — one row per sensor, always the latest reading — directly to local disk on AWS. DuckDB queries it without any rsync delay. The hot.parquet rsync is no longer needed on AWS.
Extend the host flush interval to 10–15 minutes. With current_state.parquet covering real-time queries on AWS, the partitioned S3 store is only needed for historical data — not for current state. A 15-minute flush interval produces ~96 files/site/day instead of 1 440 (at 1 min), reducing DuckDB scan time for date-scoped historical queries proportionally, and improving per-file compression and row-group statistics. rsync can also run on the same extended cadence. Set flush_interval_seconds: 900 (and match rsync cron) once the AWS writer is live. 
# AWS writer config (current_state mode — no partitioned store needed)
output:
  base_path: /data/parquet-aws        # local to AWS instance
  current_state_path: /data/parquet-aws/current_state.parquet
  hot_file_path: ""                   # not needed — current_state covers the gap
  flush_interval_seconds: 60
Lock-release before I/O: the mutex is held only during the buffer swap (std::exchange(g_buffers, {})), never during parquet writes. The MQTT receive thread never blocks on disk I/O regardless of flush duration.

DuckDB query on AWS — S3 history + local current_state (no rsync gap):

-- Current state: directly from AWS writer (local file, flush-interval lag only)
SELECT site_id, rack_id, module_id, cell_id, voltage, temperature, soc
FROM read_parquet('/data/parquet-aws/current_state.parquet', union_by_name=true)
WHERE voltage IS NOT NULL

-- Combined: full history + current state
SELECT * FROM read_parquet([
    's3://bucket/parquet/batteries/**/*.parquet',
    '/data/parquet-aws/current_state.parquet'
], union_by_name=true, hive_partitioning=true)
WHERE timestamp > epoch(now()) - 3600
ORDER BY timestamp
