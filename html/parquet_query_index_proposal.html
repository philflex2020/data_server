Baseline — Current Query Performance

Live comparison run 2026-05-02 on .34 against the same FlashMQ MQTT stream. InfluxDB holds a persistent TSM B-tree index; flux_duck scans parquet row groups from disk. The gap is architectural — O(log n) index lookup vs O(n) columnar scan — but parquet provides hooks to approach index-like behaviour without changing the file format.

Why the gap is largest for single-series: InfluxDB's TSM index stores one sorted run per (measurement, tag-set, field) — a point query seeks directly to the right pages in O(log n). Proposed Techniques — Overview

TECHNIQUE 1
Bloom Filters on point_name + unit_id
Add per-row-group bloom filters to every parquet file written by the compaction pipeline. DuckDB reads the filter header (a few KB) before deciding whether to fetch any row group data. A point_name = 'AvgCellT' query skips >99% of row groups with a single hash check.
Expected: 10–20× speedup · ~50–100 ms for single series 1 h
TECHNIQUE 2
Hive Partition by unit_id at Daily Level
Restructure the daily compacted files into per-unit subdirectories. DuckDB's hive partitioning eliminates entire files for other units at the filesystem level — no bloom filter lookup needed, no row group scan at all.
Expected: 10× file-level skip · composable with bloom filters
TECHNIQUE 3
Pre-aggregated 5 min Summary Tier
Write a units_5m.parquet alongside each daily compaction containing mean/min/max for every signal at 5-minute granularity. Route aggregateWindow(every: 5m) Flux queries to this file instead of raw data. The summary file is ~1 MB vs 1.1 GB raw — a Grafana panel query becomes a trivial read.
Expected: < 20 ms for any windowed dashboard query
TECHNIQUE 4
Persistent DuckDB Row-Group Index
Maintain a small rg_index.duckdb file that maps (unit_id, point_name, date) to exact row groups within the parquet files. Queries consult the index first, then call read_parquet(path, row_groups=[N,M]) to read only the 2–4 relevant groups. This is the closest equivalent to TSM's B-tree — O(log n) lookup, minimal I/O.
Expected: 20–50× speedup · approaches TSM for any query type
TECHNIQUE 5
Real-time All-Points via /live
The hardest query for parquet — all current signal values for a unit — is already solved by the /live endpoint. The parquet_writer pushes a one-row-per-signal Arrow IPC snapshot on each 30 s flush; live_receiver holds it in memory. Response is sub-millisecond with zero parquet I/O. A 2-minute hot-tier fallback covers disconnected periods.
Already shipped · < 1 ms · beats InfluxDB for this pattern

Technique 1 — Bloom Filters LOW EFFORT

Parquet stores optional bloom filter bitsets per column per row group in the file footer. DuckDB reads them before fetching row group data — a probe costs a few microseconds and one network/disk round-trip for the footer block. No query-path changes needed; this is purely a write-time change.

Technique 2 — Hive Partition by unit_id LOW EFFORT

Change the daily compacted file layout from one shared file to one file per unit. DuckDB's hive partitioning reads the directory name as a column value and prunes files before opening them — no I/O at all for non-matching units.

Technique 3 — Pre-aggregated 5 min Summary Tier MEDIUM EFFORT

90% of Grafana dashboard panels display a windowed aggregate (5 min mean, 15 min mean) rather than raw rows. InfluxDB handles these fast because it indexes on the already-computed data. We can match this by writing a tiny pre-aggregated parquet alongside each daily compaction. A 5-minute aggregate of one day's data for one unit is ~800 rows — a trivial DuckDB query.

Summary file generation — add to compaction step

import duckdb

def write_summary(raw_parquet_path, summary_path, intervals=(5, 15, 60)):
    """Write pre-aggregated summary parquet files alongside the raw daily file."""
    con = duckdb.connect()
    for minutes in intervals:
        out = summary_path.replace('.parquet', f'_{minutes}m.parquet')
        con.execute(f"""
            COPY (
                SELECT
                    time_bucket(INTERVAL '{minutes} minutes', ts)  AS ts,
                    unit_id,
                    device, instance, point_name,
                    AVG(value)   AS value,
                    MIN(value)   AS value_min,
                    MAX(value)   AS value_max,
                    COUNT(*)     AS n
                FROM read_parquet('{raw_parquet_path}',
                                  union_by_name=true, hive_partitioning=false)
                WHERE value IS NOT NULL
                GROUP BY 1, unit_id, device, instance, point_name
                ORDER BY ts
            ) TO '{out}' (FORMAT PARQUET, COMPRESSION ZSTD,
                          STATISTICS true,
                          BLOOM_FILTER_COLUMNS ['point_name', 'unit_id'])
        """)

# File size estimate for one unit, one day, 5m granularity:
#   ~1,437 signals × 288 buckets × ~40 bytes/row ≈ 16 MB (vs 92 MB raw)
#   After ZSTD: ~1–2 MB per unit per day

def _summary_sql(p, glob, now):
    """Query pre-aggregated summary file — no GROUP BY needed, just SELECT."""
    agg = p.window.fn   # 'mean' | 'min' | 'max' | 'count'
    col = {'mean': 'value', 'min': 'value_min',
           'max': 'value_max', 'count': 'n'}.get(agg, 'value')

       device||'/'||instance||'/'||point_name AS _field,
       {col} AS _value, unit_id

{where}
ORDER BY _time"""

Technique 4 — Persistent DuckDB Row-Group Index HIGHER EFFORT

The most complete solution. Maintain a small rg_index.duckdb file alongside the hot tier that maps (unit_id, point_name, date) to the exact row groups in each parquet file. A query consults the index first (microseconds), then reads only the 2–4 relevant row groups. This is structurally equivalent to InfluxDB's TSM index.

New file: python/flux_duck/rg_index.py

"""
rg_index.py — Row-group index for hot-tier parquet files.

Index schema:
  (unit_id, point_name, parquet_path, rg_id, ts_min, ts_max)

Build: called after each compaction step.
Query: called from executor.py before building SQL.
"""
import duckdb, os
from pathlib import Path

_INDEX_FILE = "rg_index.duckdb"
_SCHEMA = """
    CREATE TABLE IF NOT EXISTS rg_index (
        unit_id     VARCHAR NOT NULL,
        point_name  VARCHAR NOT NULL,
        path        VARCHAR NOT NULL,
        rg_id       INTEGER NOT NULL,
        ts_min      TIMESTAMPTZ NOT NULL,
        ts_max      TIMESTAMPTZ NOT NULL,
        PRIMARY KEY (path, rg_id, unit_id, point_name)
    );
    CREATE INDEX IF NOT EXISTS idx_lookup
        ON rg_index (unit_id, point_name, ts_min, ts_max);
"""

def open_index(hot_base: str) -> duckdb.DuckDBPyConnection:
    path = os.path.join(hot_base, _INDEX_FILE)
    con = duckdb.connect(path)
    con.execute(_SCHEMA)
    return con

def index_file(con, parquet_path: str) -> None:
    """Scan a newly-written parquet file and insert its row-group stats."""
    con.execute("""
        INSERT OR REPLACE INTO rg_index
        SELECT unit_id, point_name, ? AS path, row_group_id AS rg_id,
               MIN(ts) AS ts_min, MAX(ts) AS ts_max
        FROM parquet_scan(?, row_group_id=true)
        WHERE unit_id IS NOT NULL
        GROUP BY unit_id, point_name, row_group_id
    """, [parquet_path, parquet_path])

def lookup(con, unit_id: str, point_name: str,
           ts_start: str, ts_stop: str) -> dict[str, list[int]]:
    """Return {path: [rg_id, ...]} for row groups covering the time range."""
    rows = con.execute("""
        SELECT path, rg_id FROM rg_index
        WHERE unit_id   = ?
          AND point_name = ?
          AND ts_max    >= ?::TIMESTAMPTZ
          AND ts_min    <= ?::TIMESTAMPTZ
        ORDER BY path, rg_id
    """, [unit_id, point_name, ts_start, ts_stop]).fetchall()
    result = {}
    for path, rg_id in rows:
        result.setdefault(path, []).append(rg_id)
    return result

def _try_indexed_query(p, cfg, now):
    """Return indexed SQL if the query targets a single unit + single signal."""
    unit_ids    = [c.value for s in p.filters for c in s.conditions
                   if c.column == 'unit_id' and c.op == '==']
    point_names = [c.value for s in p.filters for c in s.conditions
                   if c.column == 'point_name' and c.op == '==']
    if len(unit_ids) != 1 or len(point_names) != 1:
        return None   # fan-out query — fall back to scan

    con = rgi.open_index(cfg.hot_base)
    rgs = rgi.lookup(con,
        unit_id=unit_ids[0], point_name=point_names[0],
        ts_start=_resolve_ts(p.range_spec.start, now),
        ts_stop=_resolve_ts(p.range_spec.stop, now) if p.range_spec.stop else str(now),
    )
    if not rgs:
        return None

    # Build read_parquet() with explicit row_groups per file
    # DuckDB syntax: read_parquet(['f1','f2'], row_groups=[[0,1],[2]])
    files = list(rgs.keys())
    groups = [rgs[f] for f in files]
    file_list  = ', '.join(ff"'{f}'" for f in files)
    group_list = ', '.join(str(g) for g in groups)
    return f"""
SELECT ts AS _time, 'units' AS _measurement,
       device||'/'||instance||'/'||point_name AS _field,
       value AS _value, unit_id
FROM read_parquet([{file_list}],
     row_groups=[{group_list}],
     union_by_name=true, hive_partitioning=false)
WHERE {_range_where(p.range_spec, now)}
  AND unit_id = '{unit_ids[0]}'
  AND point_name = '{point_names[0]}'
ORDER BY _time"""

Technique 5 — Real-time All-Points Query ALREADY SHIPPED

The hardest query pattern for a parquet system is "give me the latest value of every signal for unit X right now". On InfluxDB this is instant — a single index lookup on unit_controller_id returns the last record per series directly from the TSM B-tree. On flux_duck a raw scan returns 5.1 M rows (hits max_rows); a last() aggregation across 1 h of data takes ~14 s. 30-second staleness is acceptable — the parquet_writer flushes every 30 s.

Primary path — /live endpoint (already implemented)

The flux_duck server already maintains a live_receiver module that subscribes to the parquet_writer's Arrow IPC push stream. On each 30-second flush the writer pushes a snapshot table — one row per active series, current COV value — directly into memory. GET /live?site=0215F6BD serves it with zero parquet I/O.

# Query all current values for a unit — sub-millisecond response
curl http://localhost:8772/live?site=0215F6BD

# Response:
{
  "seq": 4821,          # increments each 30s flush
  "connected": true,
  "count": 1437,         # one row per active signal
  "rows": [
    {"unit_id": "0215F6BD", "point_name": "AvgCellT", "value": 23.4, "ts": "..."},
    ...
  ]
}

This is faster than InfluxDB for this query pattern — it is a push model (writer sends data) rather than a poll model (query hits disk). No bloom filters, no row groups, no SQL planner. A Grafana "current values" panel polling /live every 5 s gets data that is at most 30 s stale (the flush interval) regardless of how many signals the unit has.

In the flux_duck Flux API: there is no Flux syntax for "latest value per series". Route this query type to /live at the application layer — not through /api/v2/query. The live snapshot is the correct data source for real-time dashboard panels showing current state.

When /live is unavailable — range fallback

If live_receiver is disconnected (writer pod restart, network gap), the /live response includes "connected": false. The tempting fallback is a tight range query, but there is a correctness trap:

# Narrow window — fast but dangerous
FD_LIVE_FALLBACK="from(bucket:\"ems\")
  |> range(start: -2m)
  |> filter(fn: (r) => r[\"unit_id\"] == \"$UNIT\")
  |> last()"

# Hits ~4 files × 200 KB = ~800 KB I/O → ~200–500 ms
# BUT: any signal that hasn't fired in the last 2 minutes will be absent.
# Slow-moving points (alarm states, mode registers, binary inputs) are silently dropped.

# Wider window — correct but slow
FD_LIVE_1H="from(bucket:\"ems\")
  |> range(start: -1h)
  |> filter(fn: (r) => r[\"unit_id\"] == \"$UNIT\")
  |> last()"
# Scans full hot-tier slice → ~8–15 s. Not viable for a live panel.
The -2m range is not a safe fallback unless the site guarantees all signals publish at least once per minute. In real EMS deployments many alarm/state points fire only on change — they may not appear for hours. A live panel using range(-2m) will silently show stale or missing values for those signals.

Recommended fallback — "latest values" sidecar file

The correct solution when /live is unavailable: have the writer produce a tiny units_latest.parquet alongside each 30 s flush containing one row per signal (the last value seen in that window). At ~1,437 rows × ~40 bytes this is a ~60 KB file — trivially fast to query with guaranteed full signal coverage:

# In C++ writer flush — after flushing the main parquet file, write a sidecar:
// parquet_writer.cpp — add to flush_3file_rg() after main file write
//
// 1. Collect last-seen row per (unit_id, point_name) from the in-memory COV table
// 2. Write to  <flush_dir>/<timestamp>Z_latest.parquet
// 3. Atomically replace a fixed symlink:  latest.parquet → <timestamp>Z_latest.parquet
//    so query layer always reads one stable path.

# Python equivalent (for restructure_rg / test harness):
import duckdb, pyarrow.parquet as pq

def write_latest_sidecar(arrow_table, flush_dir: str, ts_tag: str) -> str:
    """One row per signal — last value seen in this flush window."""
    con = duckdb.connect()
    con.register('tbl', arrow_table)
    latest = con.execute("""
        SELECT DISTINCT ON (unit_id, point_name)
               unit_id, device, instance, point_name, ts, value
        FROM   tbl
        ORDER BY unit_id, point_name, ts DESC
    """).arrow()
    out = f"{flush_dir}/{ts_tag}_latest.parquet"
    pq.write_table(latest, out, compression='zstd', write_statistics=True)
    # update stable symlink
    link = f"{flush_dir}/latest.parquet"
    import os
    if os.path.islink(link): os.unlink(link)
    os.symlink(out, link)
    return out

# Query — always read the symlink, never needs a glob:
