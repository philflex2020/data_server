Two independent paths into the aws-sim Parquet Store, one query path out. The host writer.cpp creates full partitioned Parquet files and rsync-sim delivers them as history. The gap writer (optional) fills the most recent ≤60 s by writing current_state.parquet into the same directory. DuckDB globs **/*.parquet and sees everything — the subscriber API has no knowledge of which files came from rsync and which from the gap writer.

Gap Writer — Filling the rsync Lag

The rsync-sim loop copies completed Parquet files every 5 seconds, but writer.cpp flushes a new file only every 60 seconds. This means the most recent 60 seconds of data is never in /srv/data/parquet-aws-sim/ — it is still buffered in writer.cpp on the host.

The gap writer is a second instance of writer.cpp running on the aws-sim side, subscribed to FlashMQ :1884. It runs in current_state mode: instead of writing partitioned history files, it maintains a single current_state.parquet that always holds the latest reading for every sensor. DuckDB finds it automatically when globbing **/*.parquet.

DataSourceLagPath in /srv/data/parquet-aws-sim/
History (> 60 s old) rsync-sim from host writer.cpp 5 s rsync cadence project=*/site=*/YYYY/MM/DD/*.parquet
Current state (gap) gap writer — writer.cpp subscribed to :1884 ≤ 60 s flush interval current_state.parquet
Gap writer is optional at startup. The rsync-sim path works immediately without it — history queries just have a < 60-second blind spot at the leading edge. Start the gap writer when you want real-time completeness without changing any subscriber code.

Gap writer config — aws/gap_fill/config.yaml

mqtt:
  host: localhost
  port: 1884                    # FlashMQ aws-sim
  client_id: gap-fill-writer    # distinct from subscriber-api
  topic: "#"
  qos: 1

output:
  base_path: /srv/data/parquet-aws-sim   # same dir as rsync destination
  current_state_path: /srv/data/parquet-aws-sim/current_state.parquet
  hot_file_path: ""                      # not needed — current_state covers the gap
  flush_interval_seconds: 60
No subscriber changes required. The subscriber's local.path: /srv/data/parquet-aws-sim covers everything. current_state.parquet sits at the root of that directory and is found automatically by DuckDB's **/*.parquet glob alongside all historical files.

DuckDB Query Paths — Date-Scoped

Parquet files are named with the flush timestamp (20260317T120000Z.parquet) and organised into date directories (YYYY/MM/DD/). Rather than globbing **/*.parquet across the entire store, the subscriber API constructs date-scoped paths from the query time window — limiting DuckDB to only the date directories that overlap the requested range. This is the single biggest query performance lever: at a 60 s flush rate a one-day query opens ~60 files instead of the full store.

-- History query — date-scoped (server.py _date_paths() generates this)
-- Query window 2026-03-17 maps to exactly one date directory per site:
SELECT *
FROM read_parquet(
  ['/srv/data/parquet-aws-sim/project=0/site=3/2026/03/17/*.parquet',
   '/srv/data/parquet-aws-sim/current_state.parquet'],   -- gap writer (if present)
  hive_partitioning=true, union_by_name=true
)
WHERE timestamp >= 1742000000 AND timestamp <= 1742003600
ORDER BY timestamp DESC
LIMIT 1000

-- Query spanning midnight — two date directories passed as a list:
-- ['.../2026/03/16/*.parquet', '.../2026/03/17/*.parquet', '.../current_state.parquet']

-- DO NOT use **/*.parquet for history queries — opens every file in the store.
Why date-scoped paths? The Parquet filename encodes the flush timestamp, not individual message timestamps. A file flushed at 12:00:00 contains messages from ~11:59:00–12:00:00. DuckDB uses the date directory to prune which files it opens (file-level pruning), then the WHERE timestamp >= X predicate for row-level filtering within those files. Without date scoping, DuckDB would open every file and rely solely on row-level predicates — far more I/O for the same result.
current_state.parquet is added explicitly, not via glob. The gap writer writes /srv/data/parquet-aws-sim/current_state.parquet at the store root — outside all date directories. server.py checks for this file and appends it to every history query path list when present. union_by_name=true handles any column differences automatically. No other gap-fill logic exists in the subscriber.

Host Flush Interval — Extend Once Gap Writer Is Running

The host writer.cpp currently flushes every 60 seconds. With the gap writer on aws-sim writing current_state.parquet directly from live MQTT data, the rsync'd partitioned files are only needed for historical queries — not for current state.

Flush intervalFiles/site/dayDuckDB Q2/Q3 (est.)rsync cadence
60 s (current) 1 440 ~60 ms local every 5 s
5 min 288 ~12 ms local every 60 s
15 min 96 ~4 ms local every 5 min
Gap writer makes extended flush safe. current_state.parquet always holds the latest reading per sensor with at most one 60-second flush lag. Historical queries use the date-scoped partitioned files; current-state queries use current_state.parquet. The host flush interval and rsync cadence can be extended together once the gap writer is confirmed stable. 
# host writer.cpp — extend flush interval (config.fractal.yaml)
output:
  flush_interval_seconds: 300   # 5 min — 288 files/site/day

# rsync-sim — match cadence (no point rsyncing faster than flush rate)
while true; do
  rsync -a /srv/data/parquet/ /srv/data/parquet-aws-sim/
  sleep 60
done
