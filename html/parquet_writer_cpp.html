Motivation

The previous pipeline had a serial async bottleneck in bridge.py. Each message required a round-trip NATS JetStream acknowledgment before the next could be processed, capping throughput at ~4 k msg/s under 24× fleet load.

Durability preserved. writer.cpp uses a QoS-1 persistent MQTT session (clean_session=false). FlashMQ queues messages while the writer is offline and drains them on reconnect — the same guarantee NATS JetStream provided, without the extra process.

Internal Architecture

FlashMQ:1883  (QoS-1, persistent session, auto-reconnect)
    │
    │  mosquitto background thread → on_message callback
    ▼
parse_topic(topic)
    "batteries/site=0/rack=1/module=2/cell=3"
    → TopicInfo { source_type="batteries", kv{site=0,rack=1,module=2,cell=3} }

parse_payload(json, topic_info, project_id, cfg)
    {"timestamp":…, "voltage":{"value":3.7,"unit":"V"}, …}
    → Row { ints{project_id,site_id,rack_id,module_id,cell_id},
            floats{timestamp,voltage,current,temperature,…} }
    │
    │  lock g_mutex  →  g_buffers[{source_type, partition_value}].push_back(row)
    │  unlock
    │  notify flush thread if partition reaches max_messages_per_part
    │
    ▼  flush thread  (wakes on timer OR full partition)
    std::exchange(g_buffers, {})    ← swap under lock, release BEFORE I/O
    │
    ├─▶ flush_partition(path, rows, compression)   ← one file per (source, site)
    │       → Int64Builder  per int column   (nulls for missing)
    │       → DoubleBuilder per float column (nulls for missing)
    │       → parquet::arrow::WriteTable → FileOutputStream
    │       → /srv/data/parquet/batteries/project=0/site=0/2026/03/17/….parquet
    │
    └─▶ flush_partition(hot.parquet.tmp, all_rows, compression)  ← flat merge
            → fs::rename(hot.parquet.tmp, hot.parquet)           ← atomic swap
            → /srv/data/parquet/hot.parquet

Key design decisions

DecisionRationale
Lock-release before I/O Mutex held only during buffer swap (std::exchange). MQTT receive thread never blocks on disk I/O regardless of flush duration.
Dynamic schema Columns discovered per batch — batteries and solar have different field sets. Missing columns null-padded. Consistent with previous Python writer output.
Config-driven topic mapping topic_kv_map in config maps topic kv keys → row column names (e.g. site → site_id). partition_field selects which column drives buffer partitioning. No code changes needed for different sensor topologies.
Hot file for near-real-time DuckDB On every flush, all rows are flat-merged into hot.parquet via atomic temp+rename. Rsynced to S3 every ~90 s to cover the gap between partitioned-file rsync cycles. Disabled by leaving hot_file_path empty.
Current-state file (AWS writer mode) When current_state_path is set, writer maintains one row per unique sensor (keyed by source_type + all int identity fields) and writes it atomically on every flush. Intended for deployment on AWS: subscribe to the FlashMQ bridge, write current_state.parquet locally — DuckDB on AWS has sub-flush-interval current state with no rsync lag. Eliminates the need for hot.parquet rsync on the AWS side.
Partition key: (source_type, partition_value) One parquet file per source×site per flush interval. At 24× scale: 96 files/flush (48 sites × 2 source types).
QoS-1 persistent session FlashMQ queues for offline writer. Exponential reconnect backoff (2s–30s). No separate durability layer needed.
No retry on write failure Logged as ERROR, interval data dropped. Acceptable for telemetry. Add retry if SLA requires it.

Code Review Notes

AreaObservation
parse_topic Splits on /, O(n) in topic depth. Non-numeric kv values silently dropped (correct). Unknown keys kept — forward compatible with new sensor types.
parse_payload JSON safety thread_local simdjson::dom::parser — safe on bad input (returns row with topic metadata only on parse failure), no crash, no allocations across calls.
Arrow array building Two-pass: collect all column names, then fill per-column builders. Null-fill is explicit per row. Correct but O(rows × columns) — fine at 50 k row batches.
Hot file atomicity flush_partition(hot.tmp, …) then fs::rename(tmp, hot.parquet). DuckDB never reads a partial file. If the write fails, hot.tmp is removed and the previous hot.parquet is preserved.
g_cfg global pointer Set before MQTT thread starts, never mutated — safe. Could be const Config& captured reference in a future refactor.
Shutdown sequence SIGINT/SIGTERM → g_shutdown=true → g_flush_cv.notify_all() → flush thread exits → final flush → mosquitto_loop_stop. Clean shutdown with no data loss for in-progress buffer.
make_output_path time Uses std::chrono::system_clock::now() at flush time, not message receive time. All rows in a flush batch get the same timestamp in the filename — correct behaviour for interval-based partitioning.

Config Reference

mqtt:
  host: localhost
  port: 1883           # FlashMQ native MQTT port
  client_id: parquet-writer
  topic: "#"           # subscribe to all sources
  qos: 1

  # Maps topic kv key → row column name (all stored as int64).
  # Omit to use the defaults below.
  topic_kv_map:
    site:   site_id
    rack:   rack_id
    module: module_id
    cell:   cell_id

  # Which column from topic_kv_map to use as the partition key.
  partition_field: site_id

output:
  base_path: /srv/data/parquet
  project_id: 0
  partitions:          # one directory level per entry
    - "project={project_id}"
    - "site={site_id}"
    - "{year}"
    - "{month}"
    - "{day}"
  compression: snappy  # snappy | zstd | gzip | none
  flush_interval_seconds: 900  # 15 min recommended once AWS writer is live;
                               # current_state.parquet covers real-time gap on AWS
  max_messages_per_part: 50000

  # Rolling hot file — flat parquet rewritten on every flush.
  # Rsynced to S3 at higher frequency than the partitioned store to
  # cover the DuckDB gap.  Leave empty to disable.
  hot_file_path: /srv/data/parquet/hot.parquet

  # Current-state file — one row per unique sensor, always the latest reading.
  # Intended for AWS writer mode: deploy writer.cpp on AWS, subscribe to
  # the FlashMQ bridge.  current_state.parquet is written locally so DuckDB
  # on AWS has sub-flush-interval current state with no rsync lag.
  # With this in place, hot.parquet rsync is no longer needed on AWS.
  # Leave empty to disable.
  current_state_path: ""

DuckDB query — S3 partitioned store + local current_state on AWS:

-- on AWS: combine historical S3 parquet with local current_state
SELECT source, site_id, cell_id,
       arg_max(value, timestamp) AS latest_value,
       max(timestamp)            AS latest_ts
FROM read_parquet([
    's3://bucket/parquet/batteries/**/*.parquet',
    '/data/parquet-aws/current_state.parquet'    -- no rsync lag
], union_by_name=true, hive_partitioning=true)
GROUP BY 1,2,3
AWS writer eliminates the hot.parquet gap. With writer.cpp running on AWS (subscribed to the FlashMQ bridge), current_state.parquet is written locally — no rsync needed. DuckDB on AWS reads it directly with zero network lag. The gap between the partitioned S3 files and "now" is covered by current_state.parquet, updated every flush interval (~60 s).
