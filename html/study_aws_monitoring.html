What Needs Monitoring

Three distinct monitoring categories, each requiring different checks and thresholds.

1 — Infrastructure Health

Is the process running and connected?

  • NATS server reachable — GET :8222/healthz
  • S3/MinIO bucket accessible — /diag s3.bucket
  • Subscriber API responding — GET :8768/ping
  • NATS bridge connected — consumer count > 0 in /jsz
  • Parquet writer alive — consumer_pending not growing unboundedly
  • Process manager status — WS :8762 component map
2 — Data Flow Health

Is data moving at the expected rate?

  • NATS in_msgs_rate — matches expected site load
  • NATS consumer_pending — writer keeping up; should stay near 0
  • S3 write_interval_sec — should be ≤ flush_interval_seconds (60 s)
  • Subscriber API live_per_sec — matches expected msg rate
  • NATS stream fill % — should stay well below 100%
  • Slow consumer count — must be 0 (/varz slow_consumers)
3 — Data Completeness

Is all the expected data present in S3?

  • Distinct cell count in last 60 s — must equal expected total
  • Per-cell last-seen age — no cell silent for > expected interval
  • S3 row count per time window — matches expected volume
  • Gap detection — time spans with zero rows for a given cell
  • Value range validation — voltage/temp outside physical bounds
  • Timestamp skew — cells reporting future or very stale timestamps

The 60-second gap

Data published in the last flush_interval_seconds (default 60 s) lives only in the NATS JetStream stream — the Parquet Writer has not yet flushed it to S3. Any DuckDB / history query will miss this window. The only way to retrieve sub-60s data is the live WebSocket subscribe path (:8767). Monitoring systems must account for this gap: do not raise a "data missing" alert for events less than 90 s old.

Data Completeness Checks

Completeness checks verify that all expected cells are present in S3 within a given time window and that no cell has gone silent. These queries run via the query_history WebSocket command and can be issued from any monitoring process.

Check 1 — Distinct cell count in last 5 minutes

Run a query_history with limit: 10000 over the last 5 min, then count distinct (site_id, rack_id, module_id, cell_id) tuples in the result. Compare to expected cell count. A shortfall means one or more cells stopped publishing. 

// WebSocket request — count distinct cells in last 5 min
{
  "type":     "query_history",
  "query_id": "check-cell-count",
  "from_ts":  /* now - 300 */,
  "to_ts":    /* now */,
  "limit":    10000
}

// In the response, count unique combos:
const rows = result.rows;
const idx = Object.fromEntries(result.columns.map((c,i) => [c,i]));
const seen = new Set(rows.map(r =>
  `${r[idx.site_id]}.${r[idx.rack_id]}.${r[idx.module_id]}.${r[idx.cell_id]}`
));
// seen.size should equal expected cell count (e.g. 1152 for stress runner)
console.log("Distinct cells:", seen.size, "/ expected:", EXPECTED_CELLS);

Check 2 — Per-cell last-seen age

Find the most recent timestamp per cell and flag any cell whose last message is older than max_age_seconds (e.g. 2× the expected sample interval). 

// After receiving a query_history response:
const MAX_AGE_SECS = 10;   // 2× sample_interval_seconds (5s for stress runner)
const now = Date.now() / 1000;
const latestByCell = {};

for (const row of result.rows) {
  const key = `${row[idx.site_id]}.${row[idx.rack_id]}.${row[idx.module_id]}.${row[idx.cell_id]}`;
  const ts  = row[idx.timestamp];
  if (!latestByCell[key] || ts > latestByCell[key]) latestByCell[key] = ts;
}

const stale = Object.entries(latestByCell)
  .filter(([, ts]) => now - ts > MAX_AGE_SECS)
  .map(([key, ts]) => `${key}  age=${(now-ts).toFixed(1)}s`);

console.log("Stale cells:", stale.length);
stale.forEach(s => console.warn("STALE:", s));

Check 3 — S3 write interval health (from /diag)

The /diag response includes s3.write_interval_sec — the average time between the last 10 Parquet file uploads. If this exceeds flush_interval_seconds × 1.5 (90 s) the Parquet Writer may be slow or stalled. 

// Fetch /diag and check write interval
const diag = await fetch(`http://${host}:8768/diag`).then(r => r.json());
const wint = diag.s3.write_interval_sec;
const lag  = diag.nats.consumer_pending;

if (wint === null)             console.warn("No S3 files yet");
else if (wint > 90)            console.error(`SLOW WRITER: ${wint}s between files`);
else                           console.log(`Write interval OK: ${wint}s`);

if (lag > 10000)               console.error(`HIGH LAG: ${lag} messages unwritten`);
else if (lag > 1000)           console.warn(`LAG: ${lag} messages pending`);
else                           console.log(`Consumer lag OK: ${lag}`);

Fleet Monitoring Scale

At 80+ sites the question is not just what to monitor but how many monitoring processes are needed and what resources they consume.

Subscriber APIs
1 per site
Each site runs its own Subscriber API. The /diag endpoint on each is the primary health source.
NATS servers
1 per site
Each site has its own NATS JetStream instance. NATS monitoring :8222/varz per site.
Central monitor
1 per fleet
One fleet monitor polls /diag on all sites every 30s. Aggregates and alerts on thresholds.

Per-site monitoring (existing)

  • GET :8768/diag — returns full pipeline snapshot in one call
  • GET :8222/varz — NATS server health, msg rates, slow consumers
  • GET :8222/jsz — JetStream stream + consumer detail
  • WS :8761/:8762 — process manager, component up/down
  • Resource cost: 1 HTTP GET per site per 30s = negligible

Fleet-level aggregation (to design)

  • One central process polls each site's /diag in parallel
  • Aggregates: sites up/down, total msg rate, total S3 size, stale-cell counts
  • Alerting thresholds: consumer lag > N, write interval > 90s, S3 fill > 80%
  • Completeness check frequency: run cell-count check once per 5 min per site
  • 80 sites × 1 WS connection × 5min = 80 queries/5min = low load

InfluxDB2 Health Endpoints

InfluxDB2 exposes two HTTP health probes. No auth token required.

Liveness

curl http://HOST:8086/ping
# → HTTP 204 (no body) if alive

Returns 204 immediately. Use as the systemd healthcheck or load balancer probe. Returns 503 if InfluxDB2 is starting up or overloaded.

Readiness

Returns JSON with status: "pass" when InfluxDB2 is fully initialised and ready to accept writes. Use this before starting Telegraf.

30-Day Rolling Delete

The battery-data bucket was created with --retention 30d. InfluxDB2 automatically compacts and deletes data older than 30 days. There is no archive, no cold-storage tier, and no recovery option once data passes the retention window. If long-term trend analysis is required, a Flux downsampling task must be set up to copy minute-averaged data to a long-term bucket before the 30-day window expires.

Sample Flux Queries

Last reading per cell (heatmap / current state)

from(bucket: "battery-data")
  |> range(start: -5m)
  |> filter(fn: (r) => r._measurement == "battery_cell"
                    and r._field == "voltage")
  |> last()

Voltage trend — 1-minute averages over last hour

from(bucket: "battery-data")
  |> range(start: -1h)
  |> filter(fn: (r) => r._measurement == "battery_cell"
                    and r._field == "voltage"
                    and r.site_id == "0")
  |> aggregateWindow(every: 1m, fn: mean, createEmpty: false)
  |> yield(name: "voltage_1m")

Data completeness — cells seen in last 60 seconds

// Returns one row per cell — missing cells will be absent from results
// Expected: 1152 cells (spark-22b6) or 144 cells (phil-dev)
from(bucket: "battery-data")
  |> range(start: -60s)
  |> filter(fn: (r) => r._measurement == "battery_cell"
                    and r._field == "voltage")
  |> group(columns: ["site_id", "rack_id", "module_id", "cell_id"])
  |> last()
  |> count()

Write rate — messages per second (last 5 min)

from(bucket: "battery-data")
  |> range(start: -5m)
  |> filter(fn: (r) => r._measurement == "battery_cell"
                    and r._field == "voltage")
  |> aggregateWindow(every: 1s, fn: count, createEmpty: false)
  |> yield(name: "msg_rate")
