Study 2 — Data Recall & Query Patterns

Assignee brief: define the three recall tiers for each asset type, specify retention requirements, and benchmark query response times against the running stack.

Architecture update — 2026-03-17. The NATS JetStream / MinIO pipeline described in this study has been replaced by writer.cpp (FlashMQ → C++ writer → local parquet → rsync → S3). The three-tier model remains valid; the implementation has changed:

Tier 1 (real-time) — FlashMQ bridge → Subscriber API push.
Tier 2 (current-state) — current_state.parquet written locally on AWS by writer.cpp in current_state mode; one row per sensor, updated every flush (~60 s), no rsync lag.
Tier 3 (long-term) — S3 parquet rsynced from host every 15 min; queried via DuckDB with date-scoped glob paths.

Benchmark results: see DuckDB query reference and C++ writer details.

What to Research

Deliverable

  • Define retention windows for each tier: what is actually stored, for how long, and what happens at the limit

Three-Tier Recall Overview

Tier 1 — Real-time
Latency: < 1 second

NATS push consumer delivers messages to the Subscriber API which forwards them over WebSocket as live events. Latency is publish-to-browser in under 100ms under normal load.

Storage: NATS JetStream in-memory + disk

API call:

{"type":"subscribe","subject":"batteries.>"}

Use case: live dashboards, real-time alerts, operator displays

Tier 2 — Medium-term
Window: 1 second – 48 hours

Data still resides in NATS JetStream (48h rolling window, 8 GB max). For sub-60s recall this is the only path — data has not yet been flushed to Parquet. For 60s–48h recall it overlaps with Tier 3.

Important: the Subscriber API query_history call reads from S3 Parquet — it will miss the last ~60 seconds of data. To recall data from the last 60 seconds use the live subscribe path or implement a direct NATS replay consumer.

Retention: 48h rolling / 8 GB cap

At cap: oldest messages dropped first

Tier 3 — Long-term
Window: > 60 seconds (S3 flush)

The Parquet writer flushes to MinIO/S3 every 60 seconds (or every 10,000 messages, whichever comes first). Once flushed, data is queryable via DuckDB through the Subscriber API query_history call.

Partition path:

s3://battery-data/project={p}/site={s}/{year}/{month}/{day}/

Retention: indefinite (no lifecycle policy)

API call: query_history over WS :8767

Use case: historical analysis, trend reports, anomaly investigation

Note: at 1,152 msgs/s the NATS 8 GB cap is reached in approximately ~7.8 hours (8 GB ÷ ~285 KB/s ≈ 28,070 seconds). The 48h time window is the binding constraint only at lower message rates. At 144 msgs/s (generator): 8 GB ÷ ~35 KB/s ≈ ~63 hours — time window binds first.

The 60-Second Gap

Data in the last ~60 seconds is NOT in S3

The Parquet writer buffers incoming messages and flushes to S3 every 60 seconds or every 10,000 messages, whichever occurs first. Until a flush completes, the buffered data exists only in the Parquet writer's in-memory buffer and in the NATS JetStream stream.

Consequence: a query_history call with to_ts = now and from_ts = now - 30 will return zero rows even though 34,560 messages have been published in that window. The data simply has not landed in S3 yet.

At 1,152 msgs/s the 10,000-message cap is reached in ~8.7 seconds — so the effective gap is closer to 9–10 seconds rather than 60. At 144 msgs/s the 60-second time limit fires first — gap is the full 60 seconds.

Correct approach for sub-60s recall: subscribe to the live NATS stream via {"type":"subscribe","subject":"batteries.>"} on WS :8767. This delivers messages as they arrive with sub-100ms latency.

Research task: confirm the actual effective gap for each deployment by running a query at T+5s, T+15s, T+30s, T+70s after a known publish event. Document the first query window that reliably returns data.

Research Tasks

  1. Confirm effective gap duration. With the stress runner running at 1,152 msgs/s, send query_history calls at T+5s, T+15s, T+30s, T+70s after stack start. Record the first query window that returns non-zero rows. Document the actual flush trigger (time vs message count).
  2. Benchmark query_history response times. Run queries for windows of 30s, 5min, 1hr, and 24hr against the stress runner data set. Record elapsed_ms from the response, Parquet file count, and row count. Run each query three times and take the median.
  3. Measure partition pruning effectiveness. Run a 1-hour query scoped to a single site vs a cross-site query. Compare elapsed_ms. Confirm that DuckDB is pruning by the project={p}/site={s} prefix.
  4. Test the NATS stream at the 8 GB cap. Calculate when the cap is hit (8 GB ÷ 285 KB/s ≈ 7.8hr at full stress load). After the cap is hit, confirm that old messages are purged and that query_history still returns data from S3 for those earlier periods.
  5. Validate the live subscribe path under load. Connect to WS :8767 and subscribe while the stress runner is at 1,152 msgs/s. Confirm message delivery latency (timestamp in payload vs browser clock). Record any message drops.
  6. Specify retention for new asset types. Once Solar, Wind, and Grid are implemented (Study 1), confirm that their data follows the same three-tier path. Determine if different retention requirements apply (e.g. lower msg/s means the time-based window binds instead of the size cap).
