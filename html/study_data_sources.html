What to Research

Deliverable

Create a YAML config file for each asset type that a modified generator can load. The generator needs only minor changes to accept different source configs — the task is defining what each source looks like precisely enough to drive that implementation. For each asset type (Battery, Solar, Wind, Grid) produce:

  • A config_<type>.yaml file: fields, units, value ranges, sample interval
  • MQTT topic pattern following the existing batteries/project=N/site=N/... convention
  • Expected message rate for a typical site (e.g. solar farm 500 panels, wind farm 20 turbines)
  • Notes on realistic signal behaviour (solar ramp with daylight, wind stochastic, grid near-constant)
  • Confirmation that the pipeline (bridge → writer → API) passes the new messages without schema changes

Asset Type Reference

Battery Done

Implemented. See payloads above.

  • Topic: batteries/project=N/site=N/rack=N/module=N/cell=N
  • Fields: voltage, current, temperature, soc, soh, resistance, capacity, power
  • Rate: 144–1,152 msgs/s depending on topology
  • Partition key: project_id + site_id

Generator Config Template

New asset types should follow the same YAML structure. Add a top-level key for the asset class, then define topology and field ranges.

Battery config (existing — reference)

# generator/config.yaml (battery section)
mqtt:
  broker: localhost
  port: 1883

battery:
  mode: per_cell          # per_cell | flat
  interval_ms: 1000       # publish interval per cell
  topology:
    sites: 2
    racks_per_site: 3
    modules_per_rack: 4
    cells_per_module: 6
  ranges:
    voltage:     [3.0, 4.2]   # V  (LFP cell)
    current:     [-2.0, 2.0]  # A
    temperature: [20.0, 45.0] # C
    soc:         [10.0, 100.0]
    soh:         [80.0, 100.0]
    resistance:  [0.008, 0.020] # ohm
    capacity:    [90.0, 100.0]  # Ah

How to Connect and Verify

  1. Start the stack on spark-22b6. Open the process manager at http://spark-22b6:8762 and confirm that FlashMQ, NATS, the NATS bridge, the Parquet writer, and the generator (stress runner) are all running.
  2. Open monitor.html at http://phil-dev:8761 (or the local file path). Select "spark-22b6" and confirm that the live message rate shows ≈1,152 msgs/s and the NATS stream lag is stable.
  3. Open this page and click "spark-22b6 :8767" in the live bar above. The counter should increment at the rate matching the generator topology.
  4. Wait 60 seconds for the first Parquet flush to land in MinIO. Open the MinIO console and verify a file appears under s3://battery-data/project=0/site=0/{year}/{month}/{day}/.
  5. Run a 30-second history query from the Subscriber API test page (api_test.html) or via WebSocket. Set from_ts = now - 30, to_ts = now. Confirm you receive rows with correct timestamps and fields.
  6. Check the diagnostic endpoint. Run: curl http://spark-22b6:8768/diag and confirm nats_connected: true, s3_connected: true.
  7. After implementing a new asset type, repeat steps 1–6 with the new MQTT topic. Verify the NATS bridge forwards the message, the Parquet writer creates a file in the correct partition, and history query returns data.
