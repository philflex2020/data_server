Zero drops now — but dangerously thin headroom. At 1,100 metrics/sec, Telegraf's 100,000-metric in-memory buffer fills in ~91 seconds of InfluxDB unavailability. EFS latency spikes, TSM compaction pauses, or a pod restart can easily exceed this window. When the buffer overflows, metrics are silently dropped — no error, no alert, invisible gaps in your data.

TEST 6 Schema Cardinality

Tag keys observed on the battery_cell measurement:

  • cell_id, module_id, rack_id, site_id — expected hierarchy
  • host — Telegraf hostname
  • topic — full MQTT path e.g. batteries/project=0/site=0/rack=0/module=0/cell=0
The topic tag is a cardinality multiplier. Storing the full MQTT topic as a tag creates one InfluxDB series per unique topic per field. Each new cell added to the fleet adds 7 new series. Each new measurement type (solar, wind, localgen) adds more.
Fleet scopeEst. cellsFieldsActive seriesRisk
Current test (1 namespace) 624 7 4,368 Safe
6 production namespaces 3,744 7 26,208 Safe
6 namespaces + 4 source types ~15,000 7–8 ~112,000 Monitor
80-site full fleet ~100,000 7–8 ~750,000 Degraded

InfluxDB performance degrades measurably above ~1M series. The 80-site full fleet projection approaches this limit, forcing vertical instance scaling (larger, more expensive EC2 instances) rather than horizontal scale-out.

Storage Cost Comparison

At 6 production namespaces, 30-day retention, battery sources only (~124 billion points):

Storage formatBytes / pointTotal sizeAWS EFS cost/moAWS S3 cost/mo
InfluxDB TSM ~40 bytes ~4.5 TB ~$1,350 N/A
Parquet + Snappy (NATS path) ~5–8 bytes ~600–900 GB N/A ~$14–21

Parquet achieves 5–10× better compression than TSM for time-series float data due to columnar storage — all voltage values for all cells in a time window are stored together and compress extremely well. TSM stores data per-series, limiting compression opportunities.

The Version Trap

InfluxDB v2 Flux is deprecated. InfluxDB 3.0 (IOx, Apache Arrow/DataFusion) is now GA and is a completely different system — incompatible storage engine, no Flux support, SQL-only query interface. There is no in-place upgrade path from v2 to v3.

Staying on telegraf/InfluxDB does not avoid a migration. It defers one.
Migration optionQuery rewrite?Storage costScaling costCardinality risk
InfluxDB 3.0 (IOx) Yes — Flux → SQL Similar (new pricing TBD) Vertical scaling remains Improved but present
NATS + Parquet (DuckDB) Yes — Flux → SQL ~$14–21/month (S3) Horizontal, cheap None (columnar schema)

Both paths require rewriting queries from Flux to SQL. Only one delivers a 60%+ compute cost reduction and eliminates storage costs.
