The Three Pillars

T  — Test
pytest tests/unit/ runs 161 unit tests with no network required — every Python package has full unit coverage exercising serialisation, routing logic, and API behaviour. KPI benchmarks in tests/kpi/ measure end-to-end throughput, latency, and Parquet write rates under realistic load. Demo pages connect live to each service so wire formats and edge cases can be verified interactively in the browser.
T  — Train
The demo pages walk through each pipeline stage in sequence: start the stack in dashboard.html, generate cell data with generator.html, watch messages flow in monitor.html, query stored history in subscriber.html, then inspect S3 objects in aws.html. Each page introduces one new concept so a newcomer can build intuition for the full MQTT → NATS → Parquet → S3 path without reading source code first.
D  — Document
docs/NATS_STACK_DESIGN.md and docs/SUBSCRIBER_API.md cover architecture and wire formats. Each Python package ships its own README. docs/aws_deploy.md covers the spark-22b6 production deployment. api_test.html doubles as a living API reference — request/response examples stay in sync with the running server by construction.

Pipeline Overview

NATS Stack (custom Python)

  • Generator publishes synthetic cell telemetry over MQTT
  • FlashMQ :1883 brokers MQTT; NATS Bridge converts to JetStream subjects
  • NATS JetStream :4222 provides durable, replayable message store
  • Parquet Writer (pull consumer) batches and writes columnar files to MinIO/S3 :9000
  • Subscriber API :8767 (WebSocket) and :8768 (Flux HTTP) serve live + historical data
  • Stress Runner runs 24/7 on spark-22b6 to validate pipeline under load

orig_aws Stack (off-the-shelf)

  • Same generator and FlashMQ broker as the NATS stack
  • Telegraf consumes from FlashMQ and writes to InfluxDB2 :8086
  • dashboard.html can connect to InfluxDB2 for comparison queries
  • Runs in parallel — both stacks receive the same MQTT messages

Two Machines

  • phil-dev — development machine, generator source, pipeline work
  • spark-22b6 — full NATS stack running 24/7, stress runner active
  • Process manager WebSocket :8761 (phil-dev) and :8762 (spark-22b6)
  • Host defaults stored in localStorage keys ds_host / aws_host

Subscriber API Surface

  • WebSocket :8767 — live cell stream, NATS subject subscribe, history queries
  • HTTP :8768 — Flux-compatible query endpoint for InfluxDB2 clients
  • api_test.html provides raw WS command panel and Flux HTTP tester
  • subscriber.html wraps the most common history queries in a UI

Design Principles

Self-contained pages. Each demo is a single HTML file with no build step, no framework, and no external CDN dependency. Open the file directly in a browser pointed at a running server.

Two stacks, one page set. monitor.html and dashboard.html support both the NATS stack and the orig_aws InfluxDB2 stack via the same localStorage host keys (ds_host and aws_host). Switching between stacks is a host toggle, not a separate page.

Transparency. api_test.html shows the exact bytes sent and received over WebSocket and HTTP. There is no abstraction hiding the wire format from the developer.

Progressive disclosure. The five-step workflow (Start → Generate → Monitor → Query → Inspect) introduces one pipeline stage per page. A newcomer can follow the sequence and observe data flowing through every stage without reading source code.

Always in sync. Demo pages connect to the live server so request/response examples always reflect the currently deployed API — documentation cannot drift from reality.
