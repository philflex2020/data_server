1. The Problem

What we know

  • 80+ sites — batteries, solar, wind, grid
  • ~100,000 data points per site
  • MQTT topic-per-measurement — one message per sensor value
  • All telemetry flows into AWS: IoT Core → Lambda → InfluxDB2 on EC2+EFS
  • $76,200/month and growing with every new site

Why the bill is this shape

  • EC2 $36k — large instances needed to handle InfluxDB2 write throughput at this cardinality
  • EFS $10k — InfluxDB2 requires fast network-attached storage
  • IoT Core $3k — per-message pricing at high volume
  • Lambda $2k — routing layer between IoT and InfluxDB2
  • The architecture is not wrong — it is the wrong architecture for this problem

What the team has already achieved

EC2, EFS, S3, and data transfer have all been reviewed and optimised — that work is real and the bill would be higher without it. The remaining $7.6k in CloudWatch, IoT, and Lambda are also worth addressing. The proposal here is not a criticism of that work — it is a different question: what if we changed the underlying architecture rather than continuing to optimise the current one?

3. The New Architecture

Core pipeline

  • Site MQTT broker (FlashMQ) — unchanged, already running
  • NATS Bridge — subscribes to MQTT, publishes to NATS JetStream
  • NATS JetStream — durable, replayable message store (replaces IoT + Lambda)
  • Parquet Writer — batches messages, writes columnar files to S3
  • Subscriber API — serves live stream and historical queries from S3

Why this works at scale

  • NATS handles millions of messages/sec on a single t3.medium
  • Parquet columnar format gives 5–10× storage compression vs InfluxDB2
  • DuckDB queries Parquet on S3 directly — no database server needed for history
  • Self-hosted — no per-message AWS charges anywhere in the pipeline
  • Working proof-of-concept running on spark-22b6 right now

7. How We Work

●
Prove it, then roll it. Nothing goes to a production site until it has passed the defined test criteria on spark-22b6. No exceptions.
●
Troubled sites first. Phase 1 targets sites where the current solution is already struggling. The risk of change is lower than the risk of staying.
●
Templates over one-offs. Everything built this week is a template the team can replicate at each subsequent site. No bespoke deployments.
●
Existing work is not thrown away. FlashMQ stays. MQTT topics stay. The generator, the orig_aws stack, the CloudWatch work — all of it is an asset, not a liability. We are adding a better data path, not dismantling the existing one.
