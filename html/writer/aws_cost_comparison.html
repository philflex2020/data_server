6 · S3 Lifecycle — Recommended Configuration

S3 lifecycle rules can move Parquet files to cheaper tiers automatically after a configurable age. DuckDB and Athena can query S3-IA files with identical syntax; the only difference is a small per-GB retrieval fee on cold data. 

# S3 lifecycle policy (JSON / Terraform)
{
  "Rules": [{
    "ID":     "parquet-tiering",
    "Status": "Enabled",
    "Filter": { "Prefix": "parquet/" },
    "Transitions": [
      { "Days": 30,  "StorageClass": "STANDARD_IA"  },   // -46% vs Standard
      { "Days": 365, "StorageClass": "GLACIER_IR"   }    // -82% vs Standard ($0.004/GB)
    ]
  }]
}

The $0.03/GB retrieval fee only matters if you scan the entire 10-year history frequently; partitioned (per-unit) queries typically scan <1% of the total dataset per query.

7 · Why the Cost Gap Is So Large

FactorInfluxDB/TelegrafParquet/DuckDB
String storage model Full topic string (avg 84 bytes) written in every line protocol message. Repeated 81,420 times/sec = 6.9 MB/s of string overhead before values. Dictionary encoding: 1,770 unique point names stored once per row group (~106 KB). Every row stores a 2-byte index. String length has zero per-row cost.
Storage medium EBS gp3: $0.08/GB-month. Necessary for InfluxDB's random-read TSM query engine — cannot use S3 directly (OSS edition). S3: $0.023/GB-month (3.5× cheaper). Parquet is a scan-optimised columnar format — works natively with object storage.
On-disk size ~123 GB/day after 8× TSM compression. TSM block indices embed the full series key (tag strings) in every 1,000-row block header. ~67 GB/day after Parquet snappy compression. No per-row metadata; column statistics compressed with RLE and bit-packing.
Query scalability Full table scan unless a matching tag index exists. Adding new tag combinations requires re-ingestion or separate continuous queries. DuckDB/Athena prune files by directory (per-unit partitioning). A single-unit query scans 1/47th of the data = 47× cheaper per query.
Backup cost EBS snapshots recommended: +$0.05/GB-month on top of primary storage. At month 120 with backup: +$17,700/month (~$53K/month total). S3 has built-in 11× redundancy (99.999999999% durability). No backup tier needed. Cross-region replication optional: +$0.023/GB.

The InfluxDB/Telegraf path requires additional always-on AWS compute that the Parquet path does not — because Telegraf and the MQTT relay must run on AWS to receive, parse, and ingest data continuously. 10 · Data Loss Risk

BMS telemetry is not just operational data — it is the evidence base for warranty claims, safety incident investigation, battery degradation analysis, and regulatory compliance for energy storage systems. Data loss in this context has a direct financial and legal cost beyond the cost of re-collection (which is impossible for historical events).

InfluxDB data loss vectors

ScenarioProbabilityData lostDetection
Silent write drop — cache overflow High at sustained 81k writes/s Any message arriving while cache-max-memory-size is exceeded is silently dropped. No counter, no log line by default. At 81,420 writes/s, a 30-second compaction stall fills a 2 GB cache in ~34 seconds. Not visible in InfluxDB metrics by default. Requires external message count reconciliation.
WAL gap on crash Any time the EC2 or process is restarted WAL is flushed to TSM on a configurable schedule (default: cache-snapshot-memory-size = 25 MB). On a sudden crash, the last 25 MB of writes — roughly 3 minutes of Longbow data — may not be persisted. WAL recovery on restart re-applies the WAL, but only if the WAL file is intact. EC2 instance-store or EBS I/O failure during the crash may corrupt it.
EBS volume failure ~0.1–0.2%/year → ~1–2% over 10 years Total loss of all InfluxDB data if no backup, or loss of up to 24 hours if daily snapshots are used. 10 years of BMS history gone. EBS health alerts via CloudWatch, but recovery requires full restore from snapshot.
OOM kill (compaction + ingest contention) Occasional at high sustained rates OS kills the InfluxDB process mid-compaction. TSM files may be left in a partially merged state. InfluxDB recovery tooling (influx_inspect) can usually recover, but there is a window of data inconsistency. Visible in OS logs; not always visible in application metrics before it occurs.
Series cardinality explosion Low for known EMS schema If a firmware update changes the topic structure (e.g., adds a new tag segment), cardinality can spike from 1,770 to millions of series. InfluxDB TSI index fills RAM; writes are dropped and the database may become unresponsive until restarted with a reduced retention policy. Hard to predict without schema enforcement at the broker. Topic format changes are common in production BMS deployments.
Telegraf pipeline stall Occasional (GC pause, OOM, network blip) Telegraf buffers in memory (default: metric_buffer_limit = 10,000 metrics). At 81,420 metrics/s, the buffer fills in 123 milliseconds. Any stall longer than that drops data. Buffer overrun is logged but the lost data cannot be recovered. Telegraf logs Metric buffer overflow but this is often missed in production log noise. Dropped metrics are not reported to InfluxDB.
The most dangerous scenario: silent drops at normal operation. At 81,420 writes/s, InfluxDB's default cache-max-memory-size = 1 GB fills in ~12 seconds of message backlog. Any compaction cycle that takes longer than 12 seconds without flushing the cache causes silent drops — no error, no counter, no alert unless you have external message-count reconciliation in place. This is not a theoretical concern: it is the expected behaviour under the documented default configuration at this write rate.

Business cost of data loss for BMS systems

Loss eventBusiness impact
Missing fault/alarm data during a thermal event Warranty claim denied — OEM requires continuous telemetry log to validate cell fault sequence. A 3-minute WAL gap at the critical period is sufficient to void the claim. Battery pack replacement cost: $50k–$500k per unit.
Regulatory compliance gap Energy storage systems in many jurisdictions require continuous data retention for 5–10 years (UL 9540A, IEC 62933, local grid codes). Silent InfluxDB drops create gaps that may not be discovered until a compliance audit — at which point they cannot be back-filled.

Parquet path: all drops are observable and bounded. The ring_drops counter in /health is incremented atomically for every dropped message — there are no silent losses. At 300k msg/s (3.7× Longbow rate) the counter has been confirmed at zero in sustained testing. The SHM ring extension will reduce the crash-gap window from ~60s to near-zero by preserving unread messages across writer restarts.

11 · InfluxDB 3.0 — Does It Change the Picture?

InfluxDB 3.0 (IOx engine, released 2024) is a substantial re-architecture of InfluxDB. It replaces the TSM storage engine with Apache Arrow columnar Parquet files on object storage, and replaces the Flux query language with SQL (via Apache DataFusion). These are both directions our own Parquet/DuckDB approach already takes. This section examines what InfluxDB 3.0 actually offers, whether it can consume our existing Parquet files, and what a migration from InfluxDB 2.x would cost.

What changed in InfluxDB 3.0

DimensionInfluxDB 2.xInfluxDB 3.0 (IOx)
Storage engine TSM (Time-Structured Merge Tree) — proprietary columnar format on local EBS. No object storage in OSS edition. Parquet files on S3 / GCS / Azure Blob. Object storage is the primary store, not an add-on.
Query language Flux (functional, purpose-built, complex). InfluxQL supported but limited. SQL (ANSI-compatible via DataFusion). InfluxQL also supported. Flux is completely removed.
Series cardinality Hard RAM limit — TSI index held in memory. Millions of series → OOM. No cardinality limit. Column-based storage handles high cardinality natively.
WAL durability WAL on local EBS — lost on instance failure if no snapshot. Crash window: ~3 min at default config. WAL in object storage. Survives instance failure; replayed on restart automatically.
Write protocol InfluxDB line protocol over HTTP. Telegraf still required for MQTT → line protocol conversion. Line protocol + Apache Arrow Flight. Telegraf still required for MQTT ingest. Flight API adds options but does not eliminate the line protocol hop for MQTT sources.
Data transfer API HTTP REST only. Apache Arrow Flight (gRPC, columnar, high-throughput) available alongside HTTP.

Can InfluxDB 3.0 read our Parquet files?

No. InfluxDB 3.0's query engine (DataFusion) only sees Parquet files that are registered in its internal Catalog — a metadata store (SQLite or PostgreSQL) that tracks every object, its schema, and its time bounds. The Catalog is populated exclusively through InfluxDB's own write path (line protocol or Flight API). There is no mechanism to point InfluxDB 3.0 at an S3 prefix of externally-written Parquet files and query them.

This is the critical architectural difference from DuckDB/Athena, which treat S3 as a generic file system: SELECT * FROM 's3://bucket/parquet/**/*.parquet' with no catalog, no registration, no import step. InfluxDB 3.0's Parquet files are indistinguishable from ours at the byte level, but they are invisible to each other's query engines.

Migration cost: InfluxDB 2.x → 3.0

There is no supported migration path. InfluxDB has not provided an official tool to convert TSM data to IOx Parquet format. Historical data in InfluxDB 2.x remains queryable only by InfluxDB 2.x. A dual-stack must be maintained during any transition period.
Migration concernDetailEffort
Flux → SQL query rewrite Every dashboard, alert, task, and script written in Flux must be rewritten. Flux and SQL are not syntactically related — this is a full rewrite, not a find-and-replace. Flux's pipe-forward operator, filter(), aggregateWindow(), and pivot() functions all require SQL equivalents. Typical Flux queries are 5–20 lines; SQL equivalents are shorter but the translation is manual. High — every query, dashboard panel, and alert condition
Historical data access All data written to InfluxDB 2.x stays in TSM format. To access it in InfluxDB 3.0 it must be re-exported via the InfluxDB 2.x query API and re-ingested into 3.0 via line protocol. At 81,420 rows/s × years of history, this is a multi-week operation requiring both systems to run in parallel. Very high — years of data, manual pipeline, weeks of compute
Telegraf configuration Telegraf output plugin changes from outputs.influxdb_v2 to outputs.influxdb_v3 (or Flight). Input plugin and processor chain unchanged. Minor config change. Low — one config file change
Client SDK updates InfluxDB client libraries have 3.0 versions (Python, Go, Java, etc.). API calls change but the write/query patterns are similar. Any code using Flux via the query API must be rewritten to SQL. Medium — API calls similar, but any Flux queries in code need rewriting
InfluxDB Cloud migration InfluxDB Cloud (hosted 2.x) → Cloud Serverless (3.0) requires a manual data export/import. Flux-based alerts and dashboards in the Cloud UI have no automatic migration path. Tasks (scheduled Flux scripts) must be converted to SQL tasks manually. High — Cloud UI dashboards, tasks, and alerts all require manual rebuild

Retention and query limits by tier — clarified

Correction from earlier drafts: InfluxDB 3.0 Core does not hard-delete data after 7 days. The actual limitations are more nuanced and depend on tier. The "7 days" figure refers to default expiry on certain internal system tables in some Enterprise configurations, not on user databases. The binding constraint in Core is a 72-hour maximum query range in early OSS Alpha releases, combined with severe performance degradation on long-range queries because the Compactor — the process that merges and re-sorts Parquet files for efficient time-range scans — is an Enterprise-only component.

The Compactor is the hidden dependency. Without it, every flush produces a new set of small Parquet files that are never merged. A 10-year dataset would accumulate millions of uncompacted files; scanning any long time range requires opening and reading every file individually. Query time for a 1-year scan grows from minutes to hours or days as the file count compounds. The Compactor reorganises files into time-sorted, size-optimised chunks that allow DataFusion to prune efficiently — but it is only available in Enterprise and Cloud tiers.

Does InfluxDB 3.0 fix the storage cost problem? Largely yes. Moving from EBS ($0.08/GB) to S3 ($0.023/GB) reduces storage cost by ~3.5×. At year 10, storage drops from $35,400 to ~$6,210/month.
Can it read our existing Parquet files? No. InfluxDB 3.0 requires data to be ingested through its own write path and registered in its Catalog. External Parquet files are invisible.
Does it eliminate Telegraf and the relay overhead? No. The MQTT → line protocol → InfluxDB 3.0 pipeline still requires Telegraf and an AWS MQTT relay. These cost $190/month fixed — the same as 2.x.
What happens to all Flux queries? Full rewrite required. Flux is removed. Every dashboard, alert, task, and script must be rewritten in SQL. There is no migration tool.
What happens to historical InfluxDB 2.x data? Inaccessible in 3.0 without re-ingestion. A dual-stack must be maintained, or years of data must be re-piped through the new write path at significant compute and time cost.
Does InfluxDB 3.0 Core work for 10-year BMS retention? Not without the Compactor. InfluxDB 3.0 Enterprise's dominant cost becomes the software license ($4,167–$12,500/month every month, regardless of data volume). Infrastructure costs drop significantly — but the license floor means total cost never falls below roughly $11k/month, compared to $5,754/month for Parquet/DuckDB with no license fees ever.
Strategic conclusion: InfluxDB 3.0 converges toward the same architecture we already have (Parquet on S3, SQL query engine) but adds three costs that our path does not incur: a mandatory commercial Enterprise license (the free home tier is non-commercial only), Telegraf + FlashMQ relay EC2 for MQTT ingest, and Catalog lock-in that prevents our files from being queried by InfluxDB and vice versa. The Flux → SQL migration is a further one-time cost with no equivalent in our path. At every time horizon the Parquet/DuckDB path is cheaper, more durable, and produces files that are queryable by any standard tool without a license.

12 · Sample Query Cost — Max Cell Voltage

What is AWS Athena?

Athena is Amazon's serverless SQL query service. You point it at Parquet files sitting on S3, run a standard SQL query, and Athena automatically spins up a temporary fleet of ~200 worker nodes that scan the files in parallel. When the query finishes the fleet shuts down. There is no server to provision, patch, or pay for when idle — you are billed only for the bytes scanned ($5 per TB).

Because it parallelises across hundreds of nodes simultaneously, Athena finishes scans that would take a single machine hours in a matter of minutes. The same SQL query that runs on a local DuckDB installation also runs unchanged on Athena — the only difference is where the compute happens.

Athena
  • No server — fully managed by AWS
  • ~200 parallel workers per query
  • 50 TB scanned in ~15 minutes
  • Pay per query: $5/TB scanned
  • Idle cost: $0
  • Same SQL as DuckDB — no relearning
DuckDB local (spark1 / phil-dev)
  • Runs on your own machine
  • One machine, all available cores
  • 50 TB scanned in ~3–5 hours
  • Pay per query: $0
  • Idle cost: $0 (already running)
  • Identical SQL syntax

Rule of thumb: use local DuckDB for day-to-day queries on recent data (fast, free, no internet required). Use Athena for occasional large historical scans where waiting hours is impractical — the $25–$250 cost is a one-off, not a monthly charge.

A concrete query is the clearest way to compare systems. The query below retrieves the daily maximum cell voltage for each of 47 BMS units — a standard degradation-monitoring query that a BMS engineer or automated report would run regularly. It requires scanning the full time range and aggregating across all units.

The query

Parquet / DuckDB / AWS Athena (SQL)
-- works unchanged on local DuckDB, Athena, or InfluxDB 3.0
SELECT
    regexp_extract(filename, 'unit/([^/]+)', 1) AS unit_id,
    strftime(to_timestamp(ts), '%Y-%m-%d')      AS day,
    MAX(value_f)                                AS max_cell_voltage
FROM read_parquet('s3://bucket/parquet/unit/**/*.parquet')
WHERE point_name = 'Batt1_BMS_Max_cell_voltage'
GROUP BY unit_id, day
ORDER BY unit_id, day;
InfluxDB 2.x (Flux — required for InfluxDB 2.x)
// Flux has no concept of column projection —
// entire series records are returned before filtering
from(bucket: "bms_data")
  |> range(start: -1y)
  |> filter(fn: (r) =>
       r._measurement == "unit" and
       r.topic =~ /Max_cell_voltage/)
  |> aggregateWindow(
       every: 1d, fn: max, createEmpty: false)
  |> group(columns: ["unit_id"])

Note: Flux's aggregateWindow materialises the full result set in memory before returning. For 10 years of data across 47 units this can exhaust the InfluxDB server's memory budget.

13 · Notes and Caveats

  • InfluxDB compression varies widely. Simple integer/boolean values compress better than floats; irregular timestamps compress worse. The 8× estimate is typical for mixed IoT time series. Real-world range: 5–15×. Even at 15×, InfluxDB stored/day would be 66 GB (similar to Parquet), but at $0.08/GB vs $0.023/GB — EBS vs S3 alone is 3.5× more expensive.
  • InfluxDB Cloud pricing differs. InfluxDB Cloud (IOx engine) uses Parquet on S3 internally and charges ~$0.20/GB-month for storage — roughly 8.7× more than raw S3. Cloud plans also meter writes by data point count, adding $0.002–0.006/MB written on top of storage.
  • Athena cost is query-dependent. The $150/month estimate assumes typical monitoring queries scanning ~30 TB/month. With aggressive partitioning (topic_partition_segments: 2), per-unit queries scan 1/47 of total data; Athena costs could drop to $10–50/month. Heavy analytics could push costs higher.
  • InfluxDB downsampling is common in practice. Operators often configure continuous queries to aggregate data older than 30 days (e.g., 1-minute intervals). This reduces long-term storage but loses the original resolution — a trade-off that Parquet avoids entirely: all raw rows are preserved cheaply.
  • EC2 for InfluxDB may need to be larger. At 81,420 writes/sec sustained, r5.2xlarge is a lower bound. Production deployments often use r5.4xlarge ($730/month) or run a two-node cluster for HA ($1,460/month), increasing the 10-year total by $175K–$1.05M beyond the figures shown here.
  • Parquet writer runs on the edge. writer.cpp runs on the same machine as FlashMQ (the edge host). No additional EC2 is needed for data collection. The only AWS EC2 in the Parquet path is the small DuckDB query server.
  • Fleet scale multiplier. These figures are for a single site (47 BMS units). At 10 sites the costs scale proportionally; at 100 sites the 10-year InfluxDB bill approaches $218M vs $36M for Parquet/S3 Standard.
