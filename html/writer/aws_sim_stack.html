Telegraf Configuration (Longbow-style)

The MQTT topic format published by the EMS simulator follows the bench topic scheme. Telegraf parses the topic segments into tags using [[inputs.mqtt_consumer]] with a topic parsing template matching the Longbow field layout.

telegraf.conf — mqtt_consumer + influxdb_v2 output 
# ── MQTT input ─────────────────────────────────────────────────────────────
[[inputs.mqtt_consumer]]
  servers          = ["tcp://flashmq-aws-sim:1884"]
  topics           = ["bench/#"]
  qos              = 0
  client_id        = "telegraf-aws-sim"
  data_format      = "value"
  data_type        = "float"

  # topic: bench/_/{unit_id}/{device_type}/{device}/{point_name}/{dtype_hint}
  topic_tag        = ""
  [inputs.mqtt_consumer.topic_parsing]
    topic          = "bench/+/+/+/+/+/+"
    measurement    = "_/_/_/_/_/_/_"
    tags           = "_/unit_id/device_type/device/point_name/dtype_hint"
    fields         = ""

# ── Filter A: drop string dtype rows ───────────────────────────────────────
[[processors.filter]]
  order = 1
  [processors.filter.tagdrop]
    dtype_hint = ["string"]

# ── InfluxDB2 output ────────────────────────────────────────────────────────
[[outputs.influxdb_v2]]
  urls         = ["http://influxdb-aws-sim:8086"]
  token        = "ems-token-secret"
  organization = "ems-org"
  bucket       = "ems"
docker-compose.yml skeleton 
services:

  flashmq-aws-sim:
    image: flashmq:latest  # or build from /tmp/FlashMQ/build/
    ports: ["1884:1883"]
    volumes: ["./flashmq.conf:/etc/flashmq/flashmq.conf"]

  telegraf:
    image: telegraf:1.30
    depends_on: [flashmq-aws-sim, influxdb]
    volumes: ["./telegraf.conf:/etc/telegraf/telegraf.conf:ro"]

  influxdb:
    image: influxdb:2.7
    ports: ["8086:8086"]
    environment:
      DOCKER_INFLUXDB_INIT_MODE:     setup
      DOCKER_INFLUXDB_INIT_USERNAME: admin
      DOCKER_INFLUXDB_INIT_PASSWORD: adminpass1
      DOCKER_INFLUXDB_INIT_ORG:      ems-org
      DOCKER_INFLUXDB_INIT_BUCKET:   ems
      DOCKER_INFLUXDB_INIT_ADMIN_TOKEN: ems-token-secret
    volumes: ["influx-data:/var/lib/influxdb2"]

  # parquet store — bind-mounted from host rsync destination
  # subscriber_api mounts same path
  parquet-store:
    image: alpine
    volumes: ["/data/parquet-aws-sim:/data/parquet:ro"]
    command: tail -f /dev/null  # sidecar, keeps mount alive

Implementation Phases

1 Simulated AWS container current

Stand up the Docker stack (FlashMQ + Telegraf + InfluxDB2). Bridge or re-publish the same bench/# stream that feeds the parquet writers. Confirm Telegraf is ingesting and InfluxDB2 is accumulating series at the expected rate.

  • Write docker/aws-sim/docker-compose.yml + telegraf.conf
  • Start script: scripts/aws-sim-start.sh
  • Verify via InfluxDB2 UI at :8086 and a simple Flux query

2 Parquet rsync into container

Set up an rsync job (or inotify watch) to copy freshly flushed parquet files from the edge writer output into a host directory that is bind-mounted into the container. The subscriber_api can then query both InfluxDB2 (recent) and DuckDB-on-parquet (history) from the same process.

  • rsync destination: /data/parquet-aws-sim/
  • Cadence: every 60 s (matching flush_interval_seconds)
  • DuckDB glob: parquet_scan('/data/parquet/**/*.parquet')

3 Real-time queries

Extend subscriber_api (or add a new endpoint) to serve the latest N seconds of data from InfluxDB2. Compare latency and schema against the same window from DuckDB parquet.

  • Route ?mode=realtime&window=30s → Flux query on InfluxDB2
  • Route ?mode=realtime&src=parquet → DuckDB on latest flush file
  • WebSocket push for live updates

4 History queries

Range queries spanning hours or days. DuckDB partition pruning on the site=S/YYYY/MM/DD/ directory layout makes this fast. InfluxDB2 provides the comparison baseline.

  • Route ?mode=history&start=2026-04-10&end=2026-04-11
  • Benchmark: query time, data scanned, memory usage
  • Output: rows in wide-pivot vs unpivoted long format

5 Cost & performance comparison

With identical data in both stores, measure the headline numbers that justify the parquet path for long-term storage.

  • Storage ratio: InfluxDB2 WAL+TSM vs snappy parquet
  • Query latency: Flux vs DuckDB for 1-hour and 24-hour range
  • Ingest CPU: Telegraf+InfluxDB2 vs parquet_writer
  • Schema evolution: adding a new signal
