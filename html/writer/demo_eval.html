Notes & Context

Generator modes

  • Burst (g_slow_interval_ms=20000) mirrors real Evelyn — slow signals every 20 s in a burst; fast signals every second
  • Smooth (g_slow_interval_ms=1000) publishes every signal every second — inflates dedup benefit, use only for baseline load tests
  • Use burst mode for dedup evaluation
  • Both modes use the same 46-unit Evelyn dataset (ems_topic_template.json)

Filter definitions

  • Unfiltered (A): processors.pivot only — mirrors Longbow production today
  • Reg dedup (B): processors.dedup dedup_interval=60s — int/bool only via tagpass
  • float_dedup (C): [[processors.execd]] — all types: float by ε-threshold (default 0.5), int/bool/string by exact-match, 60 s heartbeat

Parquet filter options (B & C)

  • Sim burst mode — run generator with g_slow_interval_ms=20000; slow signals naturally publish every 20s — no extra bridge needed; parquet writer subscribes same :11888 topic
  • Burst caveat C — sim burst alone gives cadence dedup but not ε-threshold on floats; float values still written on every publish even if unchanged
  • Telegraf bridge B — inputs.mqtt_consumer → processors.dedup 60s → outputs.mqtt to a second topic; parquet writer subscribes filtered topic
  • Telegraf bridge C — same but with processors.execd float_dedup; gives full ε-threshold dedup on all types
  • Parquet writer binary unmodified for either approach

Expected reductions

  • Unfiltered: baseline — 100%
  • Reg dedup: est. −23% rows (int/bool only)
  • float_dedup: est. −40–60% rows (all types)
  • Parquet compression independent — typically 10–15× vs raw MQTT
  • Run for ≥10 min in burst mode before capturing metrics
