Why a C++ Publisher?

The Python real_stress_runner.py publishes EMS-format messages using the same topic template, but it is limited by Python's GIL, asyncio overhead, and paho's per-message JSON serialisation. In practice it saturates around 20–40k msg/s before the paho outbound queue backs up and messages are dropped.

ems_site_simulator is a single-binary C++ publisher that reaches 350k msg/s sustained on gx10-d94c (20 cores, 119 GB). It uses the same topic template and the same payload format — so real_writer sees identical data either way — but it can drive the writer to its actual ceiling rather than Python's.

Build

# From repo root
cd source/stress_runner
make ems_site_simulator

# Or manually:
g++ -O2 -std=c++17 ems_site_simulator.cpp -o ems_site_simulator \
    -lmosquitto $(pkg-config --libs simdjson)

Dependencies: libmosquitto-dev (already required by writer) and libsimdjson-dev (already required by writer).

Command-Line Reference

./ems_site_simulator [options]
--host <host> MQTT broker hostname or IP (default: localhost)
--port <port> MQTT broker port (default: 1883)
--template <path> ems_topic_template.json file. Auto-detected from standard locations (script dir, repo root) if omitted. (default: auto)
--units <n> Number of synthetic units to simulate. Generates hex IDs 0215F5DD, 0215F5DE, … Ignored if --unit-id is given. (default: 4)
--unit-id <id> Explicit hex unit ID (e.g. 0215F662). Repeat to add multiple units. Overrides --units. (repeatable)
--rate <n> Target total messages/sec. Applied as a sweep-level delay — the publisher completes one full topic sweep then sleeps for the remainder of the target window. Use 0 for unlimited (max speed). (default: 0 = unlimited)
--qos <0|1> MQTT QoS level. QoS 0 for maximum throughput; QoS 1 for delivery confirmation (significant overhead at high rates). (default: 0)
--id <client_id> MQTT client ID base string. (default: ems-stress)
--ws-port <port> WebSocket control server port. Connect a browser or script to send mode commands, fault injections, and receive live status. (default: 8769)
--soc <pct> Initial state-of-charge for all simulated units (0–100). Physics model then drifts SOC in real time based on active mode and current. (default: 50)

Rate limiting behaviour

Rate limiting is applied per sweep, not per message. At the end of each full topic sweep the publisher calculates how long the sweep took and sleeps for the remaining budget. This means:

  • At --rate 81420 with 4 units (7,080 topics/sweep) the sweep completes in < 100 ms and sleeps ~900 ms → effective 1 s cycle, matching real Longbow cadence.
  • At --rate 81420 with 46 units (81,420 topics/sweep) one sweep fills the entire 1 s window → no sleep needed, continual publish.
  • Setting --rate below topics/sweep has no effect (one sweep already takes longer than the window). Use more units or fewer units to adjust density.
Typical invocations on gx10-d94c: four parallel instances each with a different --id and topic prefix subset (batteries_a/b/c/d), each at --rate 350000, giving a combined 1.4M msg/s to the local writer fleet.

Payload format

// float dtype {"ts":"2024-11-15T21:27:52.775Z","value":3.700000} // integer / boolean_integer dtype {"ts":"2024-11-15T21:27:52.775Z","value":42}

The timestamp is the actual UTC wall time at publish, formatted as ISO 8601 with millisecond precision — matching the real Longbow recording format exactly. real_writer parses this with timestamp_format: iso8601 and stores it as FLOAT64 Unix epoch seconds.

43 of 46 units showed 0 V / 0 A — contactors open (idle/standby state). The 6–8 A currents at 1,350+ V correspond to ≈ 8–11 kW per unit, consistent with a balancing or low-rate top-up current, not a full dispatch event.

What we don't see in this recording

  • A full charge or discharge cycle (would take 3–6 hours at rated power)
  • Active dispatch (PCS commanded to 0 kW during the window)
  • SOC drift within the window — Modbus polling is too slow and 7.65 min too short
  • Fault events — all warning/fault flags are 0 throughout
Charge/discharge cycle indicators to look for in longer recordings: Batt1_Batt_MBMS_SOC rising continuously (charging) or falling (discharging) · Batt1_Batt_MBMS_System_Current > 0 sustained (positive = charge, negative would = discharge if sign convention used) · PCS kW positive (absorbing from grid to charge) or negative (injecting to grid from discharge) · Batt1_Charge_kWh / Discharge_kWh counters incrementing. The Operation mode register moving from 6 (standby) to 1 or 2 (active dispatch) is the clearest indicator.

EMS Physics Simulation

A background physics thread runs every 100 ms. Each unit has independent state: SOC, current, temperature, mode, and per-rack sub-state.

State machine

┌──────────────┐ │ STANDBY │◄──── default, I = 0 A └──────┬───────┘ set_mode │charge|discharge ▼ ┌───────────────────────┐ │ CHARGING (I > 0 A) │──── auto-trip when SOC ≥ 99.5% └───────────────────────┘ ┌───────────────────────┐ │ DISCHARGING (I < 0 A) │──── auto-trip when SOC ≤ 5% └───────────────────────┘ ┌──────────────┐ │ OFFLINE │ contactor open, I = 0 A └──────────────┘

Physics update (100 ms tick)

QuantityUpdate rule
SOC driftdSOC/dt = I / capacity × 100 / 3600 %/s
System voltageV = 1320 + 0.74 × SOC (LFP approximation)
Rack temperaturedT/dt = |I| / capacity × 40 / 3600 °C/s toward 25°C when idle (τ ≈ 300 s)
Charge counterIntegrated kWh when charging
Discharge counterIntegrated kWh when discharging

Fault auto-detection

FaultAuto-set when
FAULT_OVERTEMPany rack temp > 50 °C
FAULT_LOW_SOCunit SOC < 10 %
FAULT_UNDERVOLTcell voltage < 2.8 V (SOC too low)
FAULT_OVERCURR|I| > 600 A

WebSocket Control API

A WebSocket server listens on --ws-port (default 8769) and broadcasts a full status message every second to all connected clients. Send JSON commands to control simulation state. The EMS Simulator Control page provides a browser UI.

Server → Client (every second and on connect)

{
  "type":            "stats",
  "total_published": 123456,
  "mps":             81420,
  "active_tasks":    1,
  "interval":        1.0,
  "session_id":      "a1b2c3d4",
  "sync_seq":        12,
  "units": [
    { "id": "0215F662", "mode": "CHARGING", "contactor": true,
      "soc": 52.4, "current": 150.0, "temp": 28.3,
      "charge_kwh": 2344.1, "discharge_kwh": 677.0,
      "faults": [],
      "racks": [
        { "idx": 0, "soc": 51.9, "temp": 28.1, "faults": [] },
        ...
      ]
    }
  ]
}

Client → Server commands

CommandFieldsDescription
set_mode units, mode, current_a Set operating mode for target units. mode: "charge", "discharge", "standby", "offline". current_a: magnitude in amps (default 0).
set_contactor units, closed Open (false) or close (true) the HV contactor. Opening sets mode to STANDBY and current to 0.
inject_fault units, fault, value Inject a fault condition. See fault types below.
clear_faults units Clear all injected faults on the target units.
set_rate rate Change the publish rate target (msg/s) at runtime.
get_status — Request an immediate status reply (reply sent on next broadcast tick).

The units field is an array of unit ID strings, or ["ALL"] to target every unit.

Fault injection types

faultvalueEffect
overtemptemperature °CForces rack[0] temperature to value; clears when clear_faults
low_soc—Forces unit SOC to 8% (below low-SOC threshold)
undervolt—Forces rack[0] SOC to 3% → cell V ≈ 2.6 V (below 2.8 V threshold)
overvolt—Sets overvolt flag override (reported in FAULT_OVERVOLT signal)
overcurrentampsSets unit current to value (e.g. 700 A → above 600 A limit)
soc_imbalancespread %Distributes rack SOCs across ±spread/2 around unit SOC

Example commands (wscat or browser)

# Start charging all units at 150 A
{"type":"set_mode","units":["ALL"],"mode":"charge","current_a":150}

# Start discharging a single unit at 300 A
{"type":"set_mode","units":["0215F662"],"mode":"discharge","current_a":300}

# Inject overtemp on one unit (force rack 0 to 55°C)
{"type":"inject_fault","units":["0215F662"],"fault":"overtemp","value":55}

# SOC imbalance of 40% spread across racks
{"type":"inject_fault","units":["ALL"],"fault":"soc_imbalance","value":40}

# Clear all faults
{"type":"clear_faults","units":["ALL"]}

# Change publish rate to 10k msg/s
{"type":"set_rate","rate":10000}
