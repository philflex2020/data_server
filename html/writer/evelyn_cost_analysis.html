Implementation Path

Phase 1 — Zero-risk (Conservative, −26.8%)

Telegraf config changes only. No EMS changes. No data loss risk.

  • Add fieldexclude in [[inputs.mqtt_consumer]] to drop FEMS_PCS_State, InvStatus, RunMode, Mode, Application, ChaSt, LocRemCtl, Status
  • Add [[processors.dedup]] with dedup_interval = "10m" to suppress repeated integer/boolean values (COV approximation)
  • Redeploy Telegraf — no InfluxDB schema change needed

Phase 2 — Moderate (−40.6%): add float COV

  • Extend [[processors.dedup]] to also cover float fields with dedup_interval of 60s for site/unit, 10s for pcs/rack
  • Or configure EMS change-on-value publishing at the source controller

Phase 3 — Aggressive (−85.5%): deadband per signal

  • Use [[processors.starlark]] or a custom Telegraf processor with per-signal deadband thresholds derived from the p95 delta table in the noise analysis report
  • Or configure deadbands at the EMS/BMS controller level (preferred — reduces network traffic before Telegraf)
  • Use [[aggregators.basicstats]] with 60s windows for meter/site signals that only need minute-level resolution
Important: The p95 deadband filter loses transient events (fault onsets, mode transitions) that may fall below the threshold. Recommend keeping a separate high-fidelity raw stream to a short-retention bucket (e.g. 24h) and writing the filtered stream to the long-retention bucket. This preserves event detection while cutting long-term storage cost.

Appendix A — Extracting CSV from InfluxDB2

InfluxDB2 exports data as annotated CSV — a superset of plain CSV that prefixes each table block with three annotation rows (#group, #datatype, #default) describing column types and grouping, followed by a blank line and then the column header. Multiple "tables" (one per unique tag-set + field combination) are concatenated in the same file.

Method 1 — InfluxDB UI export

In the InfluxDB Data Explorer, build a Flux query, run it, then click CSV in the download toolbar. Suitable for ad-hoc exports up to a few million rows.

Method 2 — Flux query via HTTP API (used by influx_to_parquet.py)

POST /api/v2/query?org=<org>
Authorization: Token <token>
Content-Type: application/json
Accept: application/csv

{
  "query": "from(bucket:\"battery-data\") |> range(start: -3h) |> filter(fn: (r) => r._measurement == \"site\")",
  "type": "flux"
}

The response is streamed annotated CSV. influx_to_parquet.py uses Python requests with stream=True and reads the response line-by-line to avoid holding the entire payload in memory.

Method 3 — influx CLI

influx query \
  --org battery-org \
  --token "$INFLUX_TOKEN" \
  --raw \
  'from(bucket:"battery-data") |> range(start:-3h) |> filter(fn:(r)=>r._measurement=="site")' \
  > site.csv

Method 4 — Flux to() for large exports (server-side)

// Run as an InfluxDB task — writes directly to another bucket or CSV task result
from(bucket: "battery-data")
  |> range(start: 2026-03-23T00:00:00Z, stop: 2026-03-24T00:00:00Z)
  |> filter(fn: (r) => r._measurement == "site")
  |> to(bucket: "export-bucket")

Appendix B — CSV → Parquet Conversion

The influx_to_parquet.py script (and the standalone evelyn_to_parquet.py batch converter) parse the annotated CSV into a Pandas DataFrame and write Parquet using PyArrow with Snappy compression.

Parsing annotated CSV

Because each file contains multiple table blocks with repeated header rows, a plain pd.read_csv() call fails. The parser works as a state machine reading one line at a time: 

for line in file:
    if line.startswith('#'):   flush buffer; reset header    # annotation row
    if line == '':             flush buffer; reset header    # block separator
    if ',result,table' in line[:30]:
        flush buffer; header = line                          # new column header
    else:
        buf.append(line)                                     # data row

# flush() → pd.read_csv(StringIO(header + "\n" + "\n".join(buf)))

Each flushed chunk becomes a DataFrame; all chunks are concatenated at the end. This handles arbitrarily many table blocks per file without loading the whole file into memory at once.

Schema auto-detection (EMS vs standard)

EMS_DTYPES = {'float', 'integer', 'boolean_integer', 'string'}
unique_fields = set(df['_field'].unique())
is_ems = unique_fields.issubset(EMS_DTYPES)

if is_ems:
    # _field is a dtype hint, not a signal name
    df.rename({'_field': 'dtype'})
    mask_int = df['dtype'].isin(['integer','boolean_integer'])
    df['value_f'] = pd.to_numeric(df.loc[~mask_int, '_value'])
    df['value_i'] = pd.to_numeric(df.loc[mask_int,  '_value']).astype('Int64')
else:
    # _field IS the signal name (standard Telegraf schema)
    df.rename({'_field': 'point_name'})
    df['value_f'] = pd.to_numeric(df['_value'])

Appendix C — Noise Analysis SQL Queries (DuckDB)

All five queries run against the Parquet glob using DuckDB's read_parquet('.../*.parquet', union_by_name=true). The union_by_name flag is required because different measurement files have different tag columns (e.g. unit_controller_id is present in pcs.parquet but not site.parquet); missing columns are filled with NULL. All queries operate on value_f (float rows only) and group by measurement + point_name.

Query 1 — Coefficient of Variation (CV)

What it measures: CV = σ / |μ|. A dimensionless ratio of how large the signal's typical fluctuation is relative to its mean. CV = 0.01 means typical noise is 1% of the signal value; CV = 6 means the signal swings ±600% relative to its mean — usually because the mean is near zero (e.g. current during standby) making the denominator tiny. Use CV to rank signals by noisiness and identify which need the most aggressive deadbanding. Always check the mean column alongside CV: a near-zero mean inflates CV artificially even for low-amplitude noise. 

SELECT measurement, point_name,
       ROUND(AVG(value_f), 4)    AS mean,
       ROUND(STDDEV(value_f), 4) AS stddev,
       ROUND(STDDEV(value_f) / NULLIF(ABS(AVG(value_f)), 0), 6) AS cv
FROM read_parquet('*.parquet', union_by_name=true)
WHERE value_f IS NOT NULL
GROUP BY measurement, point_name
HAVING STDDEV(value_f) > 0        -- exclude perfectly static signals
ORDER BY cv DESC
LIMIT 40;

Evelyn findings: Top signals were SysCurrent (CV=6.7, mean=−63 A — near-zero standby current), PCS_P--Frequency Response (CV=5.6, mean=110 kW), and RTAC meter watt signals (CV=4–5, sign-alternating power flow). These are all legitimately noisy signals rather than sensor faults.

Query 2 — Delta Percentiles

What it measures: For each signal, computes the absolute sample-to-sample change (Δ) using the SQL window function LAG(), then calculates the 50th, 95th, and 99th percentile of those deltas. p95 is the recommended deadband threshold: setting the deadband to p95 means 95% of updates will be suppressed (only the top 5% of changes get through). The p99/p95 ratio reveals "spikiness" — a ratio > 5 means there are rare very large jumps that a deadband should still pass. LAG() is a window function that returns the previous row's value in the partition ordered by timestamp, giving the most recent prior reading per signal. 

WITH deltas AS (
  SELECT measurement, point_name,
         ABS(value_f - LAG(value_f) OVER
           (PARTITION BY measurement, point_name ORDER BY ts)) AS delta
  FROM read_parquet('*.parquet', union_by_name=true)
  WHERE value_f IS NOT NULL
)
SELECT measurement, point_name,
       ROUND(PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY delta), 4) AS p50,
       ROUND(PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY delta), 4) AS p95,
       ROUND(PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY delta), 4) AS p99,
       ROUND(MAX(delta), 4) AS max_delta,
       COUNT(*) AS samples
FROM deltas
WHERE delta IS NOT NULL AND delta > 0   -- exclude zero-delta (stuck) rows
GROUP BY measurement, point_name
ORDER BY p95 DESC
LIMIT 40;

How to use the results: For a signal with p95=12.4 kW, configure a deadband of 12.4 kW in Telegraf or the EMS. That signal will only publish when power changes by more than 12.4 kW, suppressing 95% of updates. The p50 column tells you the typical change — if p50 ≈ 0, the signal is mostly static with occasional large jumps.

Query 3 — Frozen / Stuck Sensor Detection

What it measures: Uses the "row-number gap" trick to identify consecutive runs of identical values. The idea: if you number all rows globally and also number rows within each (signal, value) group, the difference between those two row numbers is constant within a consecutive run of the same value — so GROUP BY on that difference collapses each run to a single group whose COUNT(*) is the run length. Runs longer than 20 are reported. Use this to detect sensors that have stopped updating (stuck at a constant value), or to identify signals where an upstream deadband is already applied (many runs of exactly the same value). 

WITH runs AS (
  SELECT measurement, point_name, value_f,
         -- global row number within signal
         ROW_NUMBER() OVER (PARTITION BY measurement, point_name ORDER BY ts)
         -- row number within (signal, value) group
       - ROW_NUMBER() OVER (PARTITION BY measurement, point_name, value_f ORDER BY ts)
         AS grp      -- constant within each consecutive run of the same value
  FROM read_parquet('*.parquet', union_by_name=true)
  WHERE value_f IS NOT NULL
)
SELECT measurement, point_name,
       value_f AS stuck_value,
       COUNT(*) AS consecutive_count
FROM runs
GROUP BY measurement, point_name, value_f, grp
HAVING COUNT(*) > 20       -- only report runs longer than 20
ORDER BY consecutive_count DESC
LIMIT 40;

Evelyn findings: HS-CONTROL-AGC had 268,458 consecutive identical readings (98.7% of its 3h history at the same value). All site and unit integer signals were effectively static — never changing across the entire 3-hour window. These are prime COV (change-on-value) candidates.

Query 4 — Z-score Outlier Scan

What it measures: For each signal, computes the population mean (μ) and standard deviation (σ) in a CTE, then joins back to the raw data to compute a Z-score for every reading: Z = |value − μ| / σ. Readings with Z > 3 are more than 3 standard deviations from the mean — statistically rare (<0.3% of a normal distribution). High Z-score readings are candidates for investigation: they may be sensor glitches, mode transitions (e.g. charge↔discharge), control commands, or genuine physical events. Unlike the CV query which is per-signal, the Z-score query finds individual timestamps when a reading was anomalous. 

WITH stats AS (
  SELECT measurement, point_name,
         AVG(value_f) AS mu, STDDEV(value_f) AS sigma
  FROM read_parquet('*.parquet', union_by_name=true)
  WHERE value_f IS NOT NULL
  GROUP BY measurement, point_name
  HAVING STDDEV(value_f) > 0
),
scored AS (
  SELECT d.measurement, d.point_name,
         ROUND(d.value_f, 4) AS value,
         strftime(d.ts, '%Y-%m-%dT%H:%M:%SZ') AS time,
         ROUND(ABS(d.value_f - s.mu) / NULLIF(s.sigma, 0), 2) AS z_score
  FROM read_parquet('*.parquet', union_by_name=true) d
  JOIN stats s USING (measurement, point_name)
  WHERE d.value_f IS NOT NULL
)
SELECT * FROM scored
WHERE z_score > 3
ORDER BY z_score DESC
LIMIT 40;

When Z-scores are misleading: If a signal has a bimodal distribution (e.g. it runs at −100 kW half the time and +100 kW the other half), the mean μ ≈ 0 and nearly all readings will show high Z-scores. Use the CV and delta queries together to distinguish genuine outliers from bimodal or highly variable signals.

Query 5 — Physics Consistency Check

What it measures: Joins voltage readings and SOC readings that are within 5 seconds of each other, then tests a simple linear physics model: V ≈ V₀ + k × SOC. The slope k and intercept V₀ are estimated from the data's own means (not hardcoded), making the query portable across chemistries and pack sizes. The residual = measured voltage − model prediction. Large residuals indicate: (a) the linear model is too simple for this chemistry, (b) the voltage or SOC sensor is miscalibrated, or (c) there is a significant temperature or rate effect not captured by the model. 

WITH params AS (
  -- estimate V₀ and k from the data (intercept = v_mean - 0.5*soc_mean)
  SELECT
    AVG(CASE WHEN point_name IN ('RackVoltage','SysVoltage') THEN value_f END) AS v_mean,
    AVG(CASE WHEN point_name IN ('RackSOC','SOC')            THEN value_f END) AS soc_mean
  FROM read_parquet('*.parquet', union_by_name=true)
  WHERE value_f IS NOT NULL
),
volt AS (
  SELECT measurement, ts, value_f AS voltage
  FROM read_parquet('*.parquet', union_by_name=true)
  WHERE point_name IN ('RackVoltage','SysVoltage') AND value_f IS NOT NULL
),
soc AS (
  SELECT measurement, ts, value_f AS soc_pct
  FROM read_parquet('*.parquet', union_by_name=true)
  WHERE point_name IN ('RackSOC','SOC') AND value_f IS NOT NULL
)
SELECT v.measurement AS volt_src, s.measurement AS soc_src,
       strftime(v.ts, '%Y-%m-%dT%H:%M:%SZ') AS time,
       ROUND(v.voltage, 2) AS voltage, ROUND(s.soc_pct, 2) AS soc,
       -- residual: measured V minus model prediction
       ROUND(v.voltage - (p.v_mean - 0.5*p.soc_mean + 0.5*s.soc_pct), 2) AS residual
FROM volt v
JOIN soc s     ON ABS(epoch(v.ts) - epoch(s.ts)) < 5   -- within 5 seconds
CROSS JOIN params p
WHERE p.v_mean IS NOT NULL
ORDER BY ABS(residual) DESC
LIMIT 40;

Evelyn findings: All residuals were negative (mean ≈ −47 V), meaning the model consistently over-predicts voltage. The residual range was −55 V to −17 V — the 38 V spread suggests a SOC-dependent non-linearity at the extremes of the charge curve (common in LFP chemistry). The calibrated model for this site is approximately V ≈ 1098 + 0.5 × SOC rather than the data-mean estimate of V ≈ 1145 + 0.5 × SOC. A physicist would fit a higher-order polynomial (or use a lookup table) rather than a linear model for production use.
