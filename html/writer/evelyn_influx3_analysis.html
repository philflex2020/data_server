InfluxDB 3 — Technical Analysis

1. Executive Summary

The company is evaluating InfluxDB 3. The current stack writes directly to Parquet (real_writer C++) and queries via DuckDB. InfluxDB 3 Core is built on Apache Arrow DataFusion and stores data natively as Parquet — the same format we already write. This changes the analysis significantly: InfluxDB 3 would add an ingestion layer to produce the same storage format we already have.

InfluxDB 3 does bring real value in specific areas — Grafana integration, InfluxQL compatibility, managed cloud path, and Arrow Flight SQL for BI tools. But for the edge-deployed single-site case, those benefits don't outweigh the operational complexity added.

The most honest answer: InfluxDB 3 is compelling at the cloud analytics tier, less so at the edge ingest tier where we've already solved the storage problem.

2. What InfluxDB 3 Actually Is

InfluxDB 3 is a ground-up rewrite. It shares almost nothing with InfluxDB 2 architecturally:

  • Written in Rust (not Go like v2)
  • Query engine: Apache Arrow DataFusion — the same engine as DuckDB's cousin. Full SQL, columnar execution, vectorized.
  • Storage format: Apache Parquet — yes, the same format real_writer already produces
  • No TSM files — the cardinality limits and compaction pain of v2 are gone
  • No Flux query language (InfluxQL still works, SQL is primary)
  • Ingest: HTTP Line Protocol — same wire format as v2, so Telegraf configs still work
  • Query protocol: Apache Arrow Flight SQL — industry standard, many BI tools support it
  • Object storage: local disk, S3, GCS, Azure Blob
  • License: MIT (InfluxDB 3 Core, open-sourced 2025)

Key insight: InfluxDB 3's headline storage feature is Parquet. We already write Parquet. Adding InfluxDB 3 at the ingest layer is adding an intermediary to produce what we already produce directly.

5. Plusses of Moving to InfluxDB 3

5a. Grafana First-Class Support

InfluxDB 3 has a native Grafana datasource with full SQL support. The community DuckDB plugin for Grafana works but is less mature, less supported, and not in the default Grafana plugin catalogue. If Grafana is the mandated visualization layer, InfluxDB 3 is the cleaner path.

5b. InfluxQL Compatibility

If the company has existing InfluxDB 2 dashboards, queries, or tooling, InfluxDB 3 runs InfluxQL with minimal migration. Our DuckDB approach requires rewriting any InfluxQL from scratch.

5c. Arrow Flight SQL — BI Tool Ecosystem

Arrow Flight SQL is supported by Tableau, Superset, DBeaver, and many enterprise BI tools natively. DuckDB-WASM is powerful but browser-only. For a company wanting customers to connect their own BI tools, Flight SQL is a better standard interface than custom HTTP endpoints.

5d. Managed Cloud Path

InfluxDB Cloud Serverless = same SQL/InfluxQL interface, S3-backed, zero ops. If the company wants to move edge data to a managed cloud analytics service, InfluxDB 3 self-hosted → InfluxDB Cloud is a direct lift-and-shift. Our parquet approach requires building that cloud pipeline ourselves (viable with S3 + Athena/DuckDB, but requires more engineering).

5e. Telegraf Input Ecosystem

Telegraf has 200+ input plugins. Any hardware or protocol that isn't MQTT can be added to the ingest pipeline by dropping in a Telegraf config, no code changes. Our real_writer currently only handles MQTT with a custom parser — other ingest sources require C++ work.

6. Minuses of Moving to InfluxDB 3

6a. We Already Write Parquet

This is the core irony. InfluxDB 3's headline architectural feature — "stores data as Parquet" — is something we already do. We would be adding an ingestion layer (Telegraf + HTTP + InfluxDB process) to produce the same storage format we produce directly today.

6b. mqtt_topic Column Lost

Our Parquet schema stores the full MQTT topic string (unit/3/rack0/module2/cell41/soc) as a column. This enables exact MQTT replay: read a historical Parquet row and re-publish it to any broker. InfluxDB 3 uses Line Protocol's measurement,tag=value field=value timestamp format — the raw MQTT topic is not stored. Once data goes through Telegraf → InfluxDB 3, the original topic is gone.

6c. Parquet Files Not Directly Accessible

When real_writer produces Parquet, those files can be: rsynced, copied to S3, opened in pandas, loaded into Spark, queried with any DuckDB instance anywhere. InfluxDB 3 manages its Parquet internally under its own directory layout — you query via Flight SQL. You can export, but it's not a simple file copy. This breaks the rsync-based push_agent architecture.

6d. Live Streaming Still Needs MQTT Layer

InfluxDB 3 is a time-series database, not a message broker. It does not provide live streaming to browsers. FlashMQ WebSocket + mqtt.js is still required regardless of whether InfluxDB 3 is used. InfluxDB 3 only helps with the historical query layer — not the live display layer.

6e. Operational Complexity at the Edge

Current edge deployment: copy one binary + config, run it. InfluxDB 3 edge stack: install InfluxDB 3 (~200 MB), install Telegraf, configure both, manage WAL path, object storage config, retention policies, two firewall ports. More moving parts, more failure modes, harder to support in the field.

6f. InfluxDB 3 Core Is Very New

InfluxDB 3 Core was open-sourced in 2025. Production deployments at scale are limited. DuckDB on Parquet is proven at petabyte scale across many production deployments. This is a real risk for a customer-facing product.

7. Migration Options (if chosen)

A — Parallel ingest Run real_writer (Parquet) AND Telegraf→InfluxDB3 simultaneously from the same FlashMQ topic Parquet portability preserved; InfluxDB 3 for dashboards; gradual migration ~2× storage; two ingest pipelines to maintain
B — InfluxDB 3 as query layer only Point InfluxDB 3 at existing Parquet files as object store; investigate schema compatibility No ingest change; reuse existing files Schema mismatch (our columns ≠ Line Protocol); may not work cleanly
C — Full replace Replace real_writer with Telegraf→InfluxDB3; use InfluxDB 3 export for long-term Parquet Single pipeline; InfluxDB 3 ecosystem mqtt_topic column lost; rsync push_agent broken; sub-ms ingest lost; 3 services at edge

Recommendation if migrating: Option A (parallel) with a 6-month evaluation window. Keep real_writer for the primary Parquet archive. Add InfluxDB 3 as a dashboard-facing query layer. Evaluate whether the dashboard benefits justify the storage doubling cost before committing to Option C.

8. Query Capability: Side-by-Side SQL

Latest value per sensor

-- InfluxDB 3 (SQL)
SELECT unit_id, point_name, LAST(value)
FROM ems
GROUP BY unit_id, point_name;

-- DuckDB on current_state.parquet (already computed by real_writer)
SELECT unit_id, point_name, value, dtype
FROM parquet_scan('/data/parquet/current_state.parquet')
ORDER BY unit_id, point_name;

Time range for one sensor

-- InfluxDB 3
SELECT time, value FROM ems
WHERE unit_id = '3' AND point_name = 'soc'
  AND time >= '2026-03-01' AND time < '2026-03-02'
ORDER BY time;

-- DuckDB
SELECT ts, value FROM parquet_scan('/data/parquet/2026/03/01/*.parquet')
WHERE unit_id = '3' AND point_name = 'soc'
ORDER BY ts;

Daily average SOC per unit

-- InfluxDB 3
SELECT DATE_BIN('1 day', time, '1970-01-01') AS day,
       unit_id, AVG(value) AS avg_soc
FROM ems WHERE point_name = 'soc'
GROUP BY day, unit_id ORDER BY day;

-- DuckDB
SELECT DATE_TRUNC('day', epoch_ms(ts)) AS day,
       unit_id, AVG(value) AS avg_soc
FROM parquet_scan('/data/parquet/**/*.parquet')
WHERE point_name = 'soc'
GROUP BY day, unit_id ORDER BY day;

Anomaly detection — values exceeding threshold with context window

-- DuckDB (window functions — more powerful)
SELECT ts, unit_id, point_name, value,
       LAG(value,  1) OVER w AS prev_value,
       LEAD(value, 1) OVER w AS next_value,
       AVG(value) OVER (w ROWS BETWEEN 5 PRECEDING AND 5 FOLLOWING) AS rolling_avg
FROM parquet_scan('/data/parquet/**/*.parquet')
WHERE point_name = 'temp' AND value > 45.0
WINDOW w AS (PARTITION BY unit_id ORDER BY ts);

-- InfluxDB 3: window functions in DataFusion are limited;
-- complex analytics like this typically require export to a separate tool
DuckDB is a full OLAP engine. For complex analytical queries (window functions, JOINs across tables, ML feature extraction), DuckDB on Parquet is materially more capable than InfluxDB 3's DataFusion-based SQL.

10. Recommendation

Choose InfluxDB 3 if:

  • The company already has InfluxDB 2 dashboards / Flux queries to preserve (InfluxQL compat)
  • Grafana is the mandated visualization layer and the community DuckDB plugin is unacceptable
  • Arrow Flight SQL access for enterprise BI tools (Tableau, Superset) is required
  • InfluxDB Cloud Serverless is attractive for reducing cloud ops burden
  • Multiple non-MQTT ingest sources are needed (Telegraf plugin ecosystem)
  • The team has existing InfluxDB operational experience

Stay with DuckDB-on-Parquet if:

  • Parquet portability is important (rsync, S3, pandas, Spark — all work today)
  • Edge deployment simplicity is a priority (1 binary vs 3 services)
  • Full MQTT replay capability must be preserved (mqtt_topic column)
  • DuckDB-WASM in-browser queries are valuable (zero server infrastructure)
  • Complex analytics (window functions, JOINs) are needed — DuckDB is more powerful
  • InfluxDB 3 Core maturity risk is unacceptable for a customer-facing product

Hybrid (Option A — Parallel ingest) if:

  • Dashboard tooling requires InfluxDB 3 / Grafana native datasource
  • Long-term Parquet archive and portability must be preserved
  • Willing to accept ~1.5× storage cost during transition period
  • Team wants to evaluate InfluxDB 3 in production before full commitment

Given the current architecture — edge-focused, parquet-native, 1-binary deployment — the default recommendation is to stay with DuckDB-on-Parquet and add InfluxDB 3 only as an optional cloud-tier analytics layer for customers who specifically need it. The real_writer + DuckDB path already delivers better query capability, lower storage overhead, and full MQTT replay — and InfluxDB 3's headline storage innovation (Parquet) is something we already have.
