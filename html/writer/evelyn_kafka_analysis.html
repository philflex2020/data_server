1. Executive Summary

Kafka is a distributed streaming platform built for high-durability multi-consumer message replay, decoupled producers/consumers, and stream processing at scale. The current stack (MQTT → FlashMQ → parquet_writer → Parquet) is fast, simple, and edge-deployable.

The question is not "replace MQTT with Kafka" — it is "where does Kafka add value in the overall EMS platform, if anywhere?"

Short answer: Kafka is compelling at the cloud aggregation tier (10+ sites feeding a central platform) and for message durability when writer restarts cannot lose data. It is not suitable at the edge ingest tier where the 1-binary deployment goal matters.

2. What Kafka Actually Is

  • Persistent log: every message written durably to disk, retained for configurable period (days/weeks). Any consumer replays from any offset. QoS-0 MQTT has no persistence — if subscriber is down, messages are lost.
  • Multiple independent consumers: many services consume the same topic simultaneously, each with its own offset position, without affecting each other.
  • Partitioned parallelism: topics partitioned across brokers. Consumer groups each own partitions — horizontal scaling built-in.
  • Kafka Connect: ecosystem of connectors. Relevant: MQTT source connector, S3/GCS sink (writes Parquet to object storage), InfluxDB sink.
  • Kafka Streams / ksqlDB: stream processing on Kafka topics — filtering, windowed aggregation, SQL-like syntax.
  • Throughput: millions of msg/s on a single broker. Our 81,420 msg/s is trivial for Kafka.
  • JVM required (plain Kafka). KRaft mode eliminates ZooKeeper. Docker image: ~800 MB.

        ▼
FlashMQ :1883
        │
        ├── parquet_writer  →  /data/parquet/   (PRIMARY)
        │
        └── [FlashMQ bridge]  →  fractal-phil FlashMQ :1884
                                       │
                                 subscriber_api (DuckDB)

Single consumer model. parquet_writer is the only persistent consumer. If parquet_writer restarts, QoS-0 messages during that window are lost. The bridge to fractal-phil is a second consumer but also ephemeral — if it drops, messages are lost on that path too.

4. Four Placement Options

Option A — Kafka as the message bus (replace FlashMQ at edge)

EMS hardware → [MQTT broker] → Kafka topic: ems.raw
                                      │
                    ┌─────────────────┼──────────────┐
                    │                 │              │
               parquet_writer       alert_service   analytics
               (Kafka consumer)

Replace FlashMQ with Kafka as the central broker. EMS devices still speak MQTT, but an MQTT-Kafka bridge converts to Kafka topics (e.g., Confluent MQTT proxy or EMQX with Kafka bridge).

Gains: message durability, multiple independent consumers, built-in 7-day replay.

Costs: Kafka is not an MQTT broker — EMS hardware speaks MQTT, always needs a bridge layer. ~800 MB Docker image. Incompatible with 1-binary edge deployment. Adds 5–50 ms latency vs FlashMQ sub-ms.

Not suitable for edge deployment

Option B — Kafka at the cloud aggregation tier only

Edge Site A:  FlashMQ → parquet_writer → parquet → push_agent ──→ Cloud Kafka :9092
Edge Site B:  FlashMQ → parquet_writer → parquet → push_agent ──→   (topic: site-{id}.raw)
Edge Site C:  FlashMQ → parquet_writer → parquet → push_agent ──→

                                    Cloud Kafka
                                         │
                          ┌──────────────┼──────────────┐
                          │              │              │
                   cloud-writer     alert_service   ML pipeline
                   (parquet S3)

Edge stays unchanged (FlashMQ + parquet_writer). Cloud Kafka receives from multiple sites, fan-out to cloud consumers.

Gains: site isolation (one slow site doesn't affect others), multi-consumer cloud fan-out, replay for re-processing.

Costs: cloud Kafka cluster to operate (or Confluent/MSK/Redpanda Cloud); parquet files already provide durable storage so Kafka retention duplicates.

Best placement option for multi-site deployments

Option C — Kafka for stream processing (Filter A+B in ksqlDB)

Edge FlashMQ bridge → Cloud Kafka (raw stream: ems.raw)
                              │
                        ksqlDB / Kafka Streams
                         ├── Filter A: DROP WHERE dtype='string'
                         ├── Filter B: change-on-value int/bool
                         └── Filter C: float deadband (future)
                              │
                        ems.filtered → InfluxDB3 / DuckDB sink

Filter logic (currently in Telegraf dedup + tagdrop) moves to a named, versioned, replayable ksqlDB job. New filter versions can be tested by replaying the raw Kafka topic.

Costs: ksqlDB is another heavy service. Telegraf approach already works and is simpler. Only valuable if filter logic grows into complex state machines.

Premature today — revisit when filter complexity grows

Option D — Redpanda as a Kafka-compatible drop-in

Redpanda is a Kafka-compatible broker written in C++. No JVM. No ZooKeeper. Single binary. ~100 MB Docker image vs Kafka's ~800 MB. All Kafka clients work against it.

EMS hardware → MQTT → Redpanda (supports MQTT + Kafka protocol)
                          │
           ┌──────────────┴──────────────┐
           │                             │
      parquet_writer                  cloud replication
      (Kafka consumer)             (Redpanda MirrorMaker)

Gains vs plain Kafka: single C++ binary, no JVM, much lower memory (~500 MB RAM vs 2+ GB), ZooKeeper-free.

Gains vs FlashMQ: persistent log, multi-consumer, full Kafka ecosystem.

Loses vs FlashMQ: FlashMQ is pure-MQTT optimized. Redpanda's MQTT is a secondary protocol — FlashMQ throughput and latency are better for pure MQTT workloads.

Most interesting Kafka-family option if persistence is needed at edge

5. What Kafka Genuinely Solves That the Current Stack Doesn't

5a. Message Durability on Writer Restart

Current pain: parquet_writer restart = lost messages (QoS-0, no buffer between broker and writer). With Kafka as the transport, the writer is a Kafka consumer with a committed offset. Restart from exactly where it left off — zero message loss, regardless of downtime duration (within retention window).

This is the single biggest operational improvement Kafka would bring to this stack.

5b. Independent Multi-Consumer Fan-out

Current pain: to add a second consumer (e.g., real-time alerting service), you add another MQTT subscriber. At 81k msg/s, each additional MQTT subscriber puts load on FlashMQ. With Kafka, all consumers read from disk independently — broker load stays constant regardless of consumer count.

5c. Cloud Aggregation of Multiple Edge Sites

Current pain: each edge site rsyncs parquet files to fractal-phil via push_agent. Adding a 3rd, 4th, 10th site means more rsync cron jobs, more collision risk, more config management. Kafka at the cloud tier lets each site be a Kafka producer — unified fan-out to InfluxDB3, DuckDB, alerting, ML pipelines from a single cluster.

5d. Testable, Replayable Stream Processing

Current pain: filter logic (dedup, deadband) lives in Telegraf config. Hard to test in isolation. Hard to verify. With Kafka + ksqlDB, filter logic is a versioned job — replay the raw topic with a new filter version and compare outputs.

6. What Kafka Does NOT Solve Here

Live browser streaming: Kafka is not a WebSocket server. FlashMQ + mqtt.js is still required for live display regardless of Kafka adoption. Kafka does not help here at all.
Parquet schema: Kafka's S3 sink connector writes Kafka's own file layout, not our custom schema (mqtt_topic column, custom partitions). parquet_writer's Parquet cannot be replicated with a generic Kafka connector.
Edge simplicity: Kafka (even Redpanda) is heavier than FlashMQ. For the 1-binary edge deployment goal, any Kafka solution is a regression.
Sub-ms ingest latency: Kafka batches writes (linger.ms, typically 5ms). FlashMQ delivers MQTT to parquet_writer in sub-ms. For edge EMS, 5ms is usually acceptable — but it's a measurable trade.
MQTT protocol: Kafka is not natively MQTT. EMS hardware speaks MQTT. A bridge layer is always required between hardware and Kafka.

7. Kafka vs NATS JetStream vs Redpanda

NATS JetStream is a lighter alternative to Kafka with similar persistence + replay features. Worth considering as an intermediate option between "no persistence" (FlashMQ QoS-0) and "full Kafka".

8. Config Examples

Kafka producer (Python) — push_agent replacement

from confluent_kafka import Producer
import json

producer = Producer({'bootstrap.servers': 'kafka-cloud:9092'})

def on_mqtt_message(client, userdata, msg):
    # Forward MQTT message to Kafka, preserving topic as key
    producer.produce(
        topic='ems.raw',
        key=msg.topic.encode(),
        value=msg.payload,
        headers={'site_id': SITE_ID, 'mqtt_topic': msg.topic}
    )
    producer.poll(0)  # non-blocking flush

Kafka consumer (Python) — cloud-side parquet writer

from confluent_kafka import Consumer

consumer = Consumer({
    'bootstrap.servers': 'kafka-cloud:9092',
    'group.id': 'parquet-writer',
    'auto.offset.reset': 'earliest',   # replay from beginning on new group
    'enable.auto.commit': False          # manual commit after write
})
consumer.subscribe(['ems.raw'])

while True:
    msg = consumer.poll(0.1)
    if msg is None: continue
    write_to_parquet(
        mqtt_topic=msg.key().decode(),
        payload=msg.value()
    )
    consumer.commit(msg)   # commit offset only after successful write

ksqlDB filter stream (Filter A+B)

-- Create stream from raw Kafka topic
CREATE STREAM ems_raw (
  mqtt_topic VARCHAR KEY,
  payload    VARCHAR
) WITH (kafka_topic='ems.raw', value_format='JSON');

-- Filter A: drop string dtype rows
CREATE STREAM ems_no_strings AS
  SELECT * FROM ems_raw
  WHERE EXTRACTJSONFIELD(payload, '$.dtype') != 'string'
  EMIT CHANGES;

-- Filter B: change-on-value (ksqlDB suppress/dedup — simplified)
CREATE TABLE ems_latest AS
  SELECT mqtt_topic,
         LATEST_BY_OFFSET(EXTRACTJSONFIELD(payload, '$.value')) AS last_value
  FROM ems_no_strings
  GROUP BY mqtt_topic
  EMIT CHANGES;

9. Recommended Placement in the Planned Scheme

Short term — single-site edge box

Keep FlashMQ + parquet_writer. Add QoS-1 on the FlashMQ bridge connection to cloud (already supported in FlashMQ bridge config). QoS-1 ensures cloud receives all messages even if the bridge drops briefly, solving message durability without Kafka.

# flashmq.conf bridge — change QoS from 0 to 1
bridge {
    address fractal-phil.local
    port 1884
    topic unit/#  in  1   # QoS-1 ensures delivery
    clientid evelyn-bridge
}

Medium term — multi-site cloud aggregation (5+ sites)

Site A FlashMQ bridge ─→ Cloud Kafka :9092
Site B FlashMQ bridge ─→   topic: ems.{site_id}.raw
Site C FlashMQ bridge ─→
                                  │
                   ┌──────────────┼──────────────────┐
                   │              │                  │
            cloud-writer     alert_service      ML pipeline
            (Kafka consumer  (Kafka consumer)  (Kafka consumer)
             → parquet S3)

Each site's FlashMQ bridges directly to cloud Kafka. Cloud parquet_writer is a Kafka consumer instead of MQTT subscriber. Independent consumer groups for alerting and analytics.

Long term — stream processing at scale

Add ksqlDB or Kafka Streams for the Filter A+B logic as site count grows. Replace Telegraf dedup config with a proper versioned stream processing job. This is only justified when the number of filter rules and edge cases exceeds what Telegraf config can cleanly express.

10. Decision Matrix

Add Kafka when…
✓You have 5+ edge sites feeding a central cloud analytics platform
✓Message replay is required — re-process historical MQTT stream with new filter logic
✓You have 3+ independent consumers of the same MQTT stream (alerting, ML, parquet, analytics)
✓Message loss on writer restart is unacceptable and QoS-1 MQTT bridge is insufficient
✓Stream processing complexity (filtering, windowed aggregation) is growing beyond Telegraf config
Do NOT add Kafka when…
✗Single edge site deployment — QoS-1 FlashMQ bridge is sufficient and far simpler
✗Edge box weight budget matters — Kafka is 800 MB+ JVM, incompatible with 1-binary goal
✗Team has no Kafka operational experience — significant learning curve, complex failure modes
✗Primary goal is live browser streaming — Kafka does not help here
✗MQTT protocol at the broker is required — EMS hardware speaks MQTT natively, Kafka always needs a bridge
Bottom line: Kafka belongs at the cloud aggregation tier for multi-site deployments. The edge box stays as FlashMQ + parquet_writer. If durability at the edge is the only concern, try QoS-1 bridge first, then NATS JetStream, before reaching for Kafka.

12. Kafka + Parquet/S3 — What Kafka Actually Covers

The Parquet vs InfluxDB2 cost analysis shows parquet/S3 is 20× cheaper for long-term storage but leaves a real-time question open. Kafka solves some of that — specifically recent history and gap recovery — but not live display.

Kafka does not solve live browser display. Kafka is a message bus, not a WebSocket server. A browser dashboard still needs either a direct MQTT subscription (FlashMQ already provides this) or a separate WebSocket bridge process. Adding Kafka does not change this requirement at all.

What Kafka adds to the parquet/S3 path specifically

  • Gap recovery: parquet_writer commits its Kafka offset after each flush. Restart from exactly where it left off — no lost messages within the 7-day retention window. This is the single biggest operational gain.
  • Recent history replay: if a dashboard needs "last 3 days" without querying S3, a Kafka consumer can seek to that offset. Useful for backfill on reconnect or re-processing with new filter logic.
  • Multi-consumer without MQTT load: alerting and analytics consume from Kafka disk, not from FlashMQ live stream — broker load stays constant regardless of consumer count.
  • Live display is unchanged: browser still subscribes to FlashMQ directly via MQTT/WebSocket. Kafka is not involved in this path.
