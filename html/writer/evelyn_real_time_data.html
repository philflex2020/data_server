1. The Real-Time Problem

real_writer receives MQTT messages, buffers them in memory, and flushes to Parquet every N seconds (configurable, typically 5–60 s). This means there is no in-process streaming path to a browser — data only becomes queryable after a flush.

Today, the real-time display path is:

FlashMQ → Telegraf → InfluxDB2 → Grafana / custom dashboard (polling)

This works but adds three services, each with its own config, latency, and storage overhead. The goal of this document is to show that the existing parquet stack already has everything needed to satisfy real-time display requirements — without Telegraf or InfluxDB.

real_writer already writes two near-real-time outputs on each flush:

  • hot_file_path — recent-rows Parquet file, rewritten each flush cycle
  • current_state_path — one row per sensor (latest value), rewritten each flush
  • GET /health on port 8771 — JSON metrics (msgs_received, flush_count, buffer_rows, drops)

2. Three-Layer Architecture

  ┌─────────────────────────────────────────────────────────┐
  │               LAYER 1 — LIVE STREAM                     │
  │                                                         │
  │  EMS hardware → MQTT :1883 → FlashMQ                    │
  │                                  │                      │
  │                           WS :9001 (new listener)       │

  │                           <100ms latency, true push     │
  └─────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │          LAYER 2 — CURRENT STATE SNAPSHOT               │
  │                                                         │
  │  real_writer flushes → current_state.parquet            │
  │                              │                          │
  │  bridge_api GET /query/current → DuckDB → JSON          │
  │                              │                          │
  │  Browser polls every 2–5 s  ─┘                         │
  └─────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │            LAYER 3 — HISTORICAL QUERIES                 │
  │                                                         │
  │  Archived parquet files served via HTTP (nginx)         │
  │                              │                          │
  │  DuckDB-WASM in browser ─────┘                         │
  │  Full SQL, no server needed                             │
  │  parquet_scan('http://host/parquet/*.parquet')          │
  └─────────────────────────────────────────────────────────┘

Each layer is independent. Layer 1 requires only a config change to FlashMQ. Layer 2 requires ~20 lines added to bridge_api. Layer 3 requires nginx to serve parquet files with CORS headers.

3. FlashMQ WebSocket Configuration

FlashMQ has native MQTT-over-WebSocket support. Add a second listen block to flashmq.conf:

# Existing MQTT listener
listen {
    port 1883
    protocol mqtt
    tcp_nodelay true
}

# New WebSocket listener for browser clients
listen {
    port 9001
    protocol websockets
}

That is the only change. No recompile, no new service. Expose port 9001 from the container. Browsers connect to ws://host:9001 using the standard MQTT-over-WebSocket protocol that mqtt.js speaks natively.

7. DuckDB-WASM Library Overview

DuckDB-WASM runs the full DuckDB engine inside the browser via WebAssembly. It can load Parquet files over HTTP range requests and execute full SQL — no server-side DuckDB needed.

// Load from CDN
import * as duckdb from 'https://cdn.jsdelivr.net/npm/@duckdb/duckdb-wasm/+esm';

// Instantiate
const JSDELIVR_BUNDLES = duckdb.getJsDelivrBundles();
const bundle = await duckdb.selectBundle(JSDELIVR_BUNDLES);
const worker = new Worker(bundle.mainWorker);
const logger = new duckdb.ConsoleLogger();
const db     = new duckdb.AsyncDuckDB(logger, worker);
await db.instantiate(bundle.mainModule, bundle.pthreadWorker);
const conn   = await db.connect();

// Register a remote parquet file
await db.registerFileURL(
  'current_state.parquet',
  'http://host:8772/static/current_state.parquet',
  duckdb.DuckDBDataProtocol.HTTP,
  false
);

// Query it
const result = await conn.query(
  "SELECT point_name, value, dtype FROM parquet_scan('current_state.parquet') ORDER BY point_name"
);

// Arrow table → JS objects
const rows = result.toArray().map(r => r.toJSON());
DuckDB-WASM uses HTTP range requests to read only the Parquet row groups it needs. The server must return Accept-Ranges: bytes and CORS headers. nginx handles both by default.

10. bridge_api Extension: /query/current

Add approximately 20 lines to bridge_api_container.py to serve the current state as JSON. Requires pip install duckdb in the container.

# Add to imports at top of bridge_api_container.py
import duckdb

# Add constant
CURRENT_STATE_PATH = "/data/parquet/current_state.parquet"

# In the request handler (handle_request function), add this elif branch:
elif path == "/query/current" and method == "GET":
    try:
        if not os.path.exists(CURRENT_STATE_PATH):
            respond(writer, "404 Not Found", {"error": "current_state.parquet not found"})
        else:
            con = duckdb.connect()
            result = con.execute(
                f"SELECT * FROM parquet_scan('{CURRENT_STATE_PATH}')"
            )
            cols = [d[0] for d in result.description]
            rows = [dict(zip(cols, r)) for r in result.fetchall()]
            con.close()
            respond(writer, "200 OK", {"columns": cols, "rows": rows})
    except Exception as e:
        respond(writer, "500 Internal Server Error", {"error": str(e)})

Enable current_state_path in writer.yaml:

output:
  base_path: /data/parquet
  # Add this line:
  current_state_path: /data/parquet/current_state.parquet

11. FlashMQ WebSocket: What to Know

At full Evelyn load (81,420 msg/s, ~109 bytes/msg ≈ 8.9 MB/s), a browser tab cannot keep up. Subscribe narrow.
Subscriptionmsg/sKB/s to browserUse case
unit/#81,420~8,900DO NOT USE in browser
unit/0/#~1,770~193Single unit dashboard
unit/0/+/soc~34~4Single point across modules
unit/0/rack0/+/soc~8~1Single rack, single point

Topic filtering happens at FlashMQ — only matching messages are forwarded to the WebSocket client. The broker load stays constant regardless of subscription narrowness. Always subscribe to the narrowest topic that satisfies the display need.
