1. The Customer UI Problem

The company ships an EMS product. Customers want their own branded dashboards, not generic Grafana panels with Grafana branding. Three distinct customer personas drive different integration requirements:

  • OEM integrator — embeds the EMS dashboard inside their own product web app. Needs a JavaScript component or clean iframe embed, not a Grafana link.
  • Enterprise operator — wants to customize layouts, add company logos, write custom alert views. Needs white-label or deep theming.
  • Developer customer — wants to build bespoke data apps on top of EMS data using their own tech stack (React, Vue, Python). Needs clean data APIs, not a dashboard tool.

2. What the Current Stack Already Provides

The existing bridge_api and FlashMQ already form an API-first data layer. Any visualization platform that speaks REST JSON and WebSocket can build a full UI on top:

This is already an API-first architecture. The choice of visualization platform is a UI layer decision, not a data layer decision. Any platform that can consume REST + WebSocket works.

3. Platform Comparison

3a. Grafana

Industry-standard observability platform. Plugin-based. Strong time-series focus. The most familiar tool in this space.

Customer UI integration

  • Panel embedding: individual panels via iframe. Set allow_embedding = true in grafana.ini. URL: http://grafana:3000/d-solo/UID?panelId=N&from=now-1h&to=now&kiosk=1
  • Limitations: iframes only — no JS SDK, no React component. Grafana chrome is always present unless using kiosk mode.
  • White-label: Grafana Enterprise only (paid). Removes branding, adds custom themes, SAML SSO.
  • Plugin development: custom panel plugins in React/TypeScript via Grafana plugin SDK. Steep learning curve, requires signing for production.

Data layer connection

  • DuckDB datasource plugin (community) — can query parquet files via parquet_scan()
  • JSON datasource plugin — polls bridge_api REST endpoints
  • No native MQTT/WebSocket live streaming — needs InfluxDB or Prometheus as intermediary
Poor fit for customer-branded product Good for internal ops dashboards

3b. Apache Superset

SQL-first BI platform. Dashboards built from SQL queries. Proper embedding SDK unlike Grafana's iframe-only approach.

Customer UI integration

  • Embedded SDK: @superset-ui/embedded-sdk — guest tokens, React component or iframe with auth. More flexible than Grafana.
  • White-label: possible via forking the React frontend and theming it
import { embedDashboard } from "@superset-ui/embedded-sdk";
embedDashboard({
  id: "dashboard-uuid",
  supersetDomain: "http://superset:8088",
  mountPoint: document.getElementById("container"),
  fetchGuestToken: () => yourBackend.getToken(),
  dashboardUiConfig: { hideTitle: true, hideChartControls: true }
});

Data layer

DuckDB SQLAlchemy connector — queries parquet with parquet_scan(). No native MQTT streaming — polling only.

Better embedding story than Grafana Heavy ops (Flask + Redis + metadata DB)

3c. Observable Framework

Static site generator for data apps. Markdown + JavaScript + DuckDB-WASM built-in. Built by Mike Bostock (D3 author). The most architecturally aligned option for this project.

Customer UI integration

  • Customers write .md files with SQL queries and custom charts → build outputs a static site they host anywhere
  • No embedding needed — the output IS the customer's page, fully branded
  • DuckDB-WASM first-class: const db = await DuckDBClient.of({data: FileAttachment("parquet/*.parquet")})
  • Observable Plot library built-in (from D3 author); ECharts, uPlot also work
// In a .md cell — queries our parquet directly in the browser
const db = await DuckDBClient.of({ ems: FileAttachment("current_state.parquet") });
const soc = await db.query(`SELECT unit_id, value FROM ems WHERE point_name='soc'`);
Plot.plot({ marks: [Plot.barY(soc, {x: "unit_id", y: "value"})] })
Best fit for developer customers building custom data apps Requires build step (Node.js) — not no-code

3d. Retool / Appsmith (No-code app builders)

Drag-and-drop internal app builders. Connect to REST APIs and WebSockets directly. Good for operator-class customers.

Customer UI integration

  • Customers drag components onto a canvas, connect them to bridge_api REST queries
  • Retool WebSocket support: built-in WebSocket resource — connects to FlashMQ port 9001 directly, binds data to components in real time
  • Appsmith: open-source, self-hostable alternative to Retool
  • Custom components: both support embedding React/HTML widgets for cases built-ins don't cover
Good for operator customers who want custom views without coding Not designed for end-product embedding or deep white-label

3e. Custom HTML + uPlot (Extend the existing approach)

The most product-appropriate model for customer-branded UI. The existing html/*.html pages already demonstrate this pattern. uPlot is the charting library Grafana uses internally for its high-performance panels.

uPlot key properties

  • ~40 KB, zero dependencies
  • Renders 100,000+ points at 60fps — handles 81k msg/s Evelyn data without dropping frames
  • Canvas-based, fully customizable styling
  • Used by Grafana internally for its time-series panels
// ECharts — similar power, more chart types
const chart = echarts.init(document.getElementById('chart'));
const data  = [];
client.on('message', (topic, payload) => {
  const { value } = JSON.parse(payload);
  data.push([Date.now(), value]);
  if (data.length > 500) data.shift();
  chart.setOption({ series: [{ data }] });
});
Full customer branding — 100% control No extra services to operate Customers need to write code (or use reference HTML as templates)

3f. Grafana Scenes (React SDK)

Grafana's newer SDK for building custom React apps that embed Grafana's rendering engine without the full Grafana UI. No iframes — Grafana panels become React components in your own layout.

import { SceneApp, SceneTimeSeries, SceneFlexLayout } from '@grafana/scenes';
const app = new SceneApp({
  pages: [new SceneAppPage({
    scene: buildScene()
  })]
});

Requires Grafana backend running as data API proxy. Relatively new (2023). Tight coupling to Grafana version.

Escape hatch from iframe limitation — if Grafana is already committed Requires Grafana backend + React + Grafana SDK expertise

3g. Apache ECharts (direct embedding)

The charting library Grafana uses internally. Used directly — no Grafana dependency. Supports 30+ chart types including time-series, heatmaps, gauge, scatter, geo maps.

  • CDN: https://cdn.jsdelivr.net/npm/echarts/dist/echarts.min.js (~1 MB)
  • Fully customizable themes, dark/light built-in
  • Streaming data: call chart.setOption() on each MQTT message
  • Works in any HTML page alongside mqtt.js — zero framework required
Best performance/feature ratio for direct embedding No Grafana dependency, full styling control

evelyn-sdk.js — Proposed API Surface

// Connection + data access
class EvelynClient {
  constructor({ host, apiPort = 8772, mqttPort = 9001 })
  async getHealth()              // GET :8771/health → {msgs_received, drops, …}
  async getStatus()              // GET :8772/status → {broker, writer, …}
  async getCurrentState()        // GET :8772/query/current → {columns, rows}
  subscribeUnit(unitId, cb)     // mqtt.js subscribe to unit/${unitId}/#
  subscribePoint(unit, pt, cb)  // subscribe to unit/${unit}/+/${pt}
  unsubscribe(topic)
  disconnect()
}

// uPlot time-series chart, pre-configured for Evelyn data
class EvelynChart {
  constructor(element, { title, maxPoints = 500, unit = '' })
  push(timestamp, value)        // add a live data point
  setData(rows)                 // replace with historical rows [{ts, value}]
  destroy()
}

// Auto-updating table of latest readings
class EvelynStateTable {
  constructor(element, { refreshMs = 2000, client })
  start() / stop()
}

Layer model

  • Layer 1 — Data APIs (already exists): bridge_api REST + FlashMQ WS :9001 + DuckDB /query/current
  • Layer 2 — evelyn-sdk.js (to build): small ES module wrapping the APIs. Shipped with the product. Zero dependencies except uPlot (~40 KB).
  • Layer 3 — Reference dashboards (already exists): the html/*.html pages become working templates showing customers how to use the SDK.
  • Layer 4 — Optional Grafana overlay: for customers who specifically require Grafana — DuckDB datasource plugin + bridge_api JSON endpoint. An option, not the primary path.

6. What This Means for Grafana Specifically

  • Keep Grafana as an optional integration for customers who already use it, not as the primary UI layer
  • Use the DuckDB datasource plugin to expose parquet data to Grafana without requiring InfluxDB
  • Grafana panel embedding (iframe) is acceptable for simple portal integration where branding is not critical
  • For full customer branding: the SDK approach is the right answer — Grafana cannot deliver this without Enterprise licensing
  • Do not invest in Grafana plugin development as a primary effort — the existing HTML architecture is more flexible and already works. A Grafana plugin requires the Grafana plugin SDK, webpack build tooling, and Grafana signing — significant maintenance overhead for a second-class experience.
