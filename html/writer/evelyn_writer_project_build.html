1. Overview — Two-Phase Build

The build is split into two phases so that the compilation step has no dependency on the data_server git repo:

Phase 2 does all the heavy work:

2. Phase 1 — Source Extract Script

Run once on build-dev from the data_server repo root. Collects only the files needed by Phase 2 — no build tools, no Docker required.

scripts/evelyn-src-extract.sh
#!/usr/bin/env bash
# evelyn-src-extract.sh — package source files from the data_server repo
#
# Run from the data_server repo root on build-dev.
# Output: dist/evelyn-src-YYYYMMDD.tar.gz
#
# Transfer the tar to the build machine, then:
#   tar -xzf evelyn-src-YYYYMMDD.tar.gz
#   bash evelyn-src/scripts/evelyn-k8s-build.sh
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
DATE="$(date +%Y%m%d)"
OUT_NAME="evelyn-src-${DATE}.tar.gz"
OUT="$REPO_ROOT/dist/$OUT_NAME"
BUILD_DIR="$(mktemp -d /tmp/evelyn-src-XXXXXX)"
STAGE="$BUILD_DIR/evelyn-src"

trap 'rm -rf "$BUILD_DIR"' EXIT

echo "=== Evelyn source extract ==="
echo "    repo:   $REPO_ROOT"
echo "    output: $OUT"
echo

mkdir -p "$REPO_ROOT/dist"
mkdir -p \
    "$STAGE/source" \
    "$STAGE/html/writer" \
    "$STAGE/docker/evelyn-native/configs" \
    "$STAGE/scripts"

# ── C++ source ────────────────────────────────────────────────────────────────
cp -r "$REPO_ROOT/source/parquet_writer"  "$STAGE/source/"

# ── HTML pages ────────────────────────────────────────────────────────────────
cp "$REPO_ROOT/html/writer/evelyn_docker_34.html"           "$STAGE/html/writer/"
cp "$REPO_ROOT/html/parquet_monitor.html"                    "$STAGE/html/"
cp "$REPO_ROOT/html/writer/parquet_viewer.html"              "$STAGE/html/writer/"
cp "$REPO_ROOT/html/writer/writer_design.html"               "$STAGE/html/writer/"
cp "$REPO_ROOT/html/writer/evelyn_real_time_data.html"       "$STAGE/html/writer/"
cp "$REPO_ROOT/html/writer/evelyn_container_design.html"     "$STAGE/html/writer/"
cp "$REPO_ROOT/html/writer/evelyn_writer_project_build.html" "$STAGE/html/writer/"

# ── Native install files ──────────────────────────────────────────────────────
cp "$REPO_ROOT/docker/evelyn-native/bridge_api_native.py"          "$STAGE/docker/evelyn-native/"
cp "$REPO_ROOT/docker/evelyn-native/install.sh"                     "$STAGE/docker/evelyn-native/"
cp "$REPO_ROOT/docker/evelyn-native/configs/flashmq.conf"          "$STAGE/docker/evelyn-native/configs/"
cp "$REPO_ROOT/docker/evelyn-native/configs/writer-native.yaml"    "$STAGE/docker/evelyn-native/configs/"

# ── Build script (self-contained in the tar) ──────────────────────────────────
cp "$REPO_ROOT/scripts/evelyn-k8s-build.sh"  "$STAGE/scripts/"

chmod +x \
    "$STAGE/scripts/"*.sh \
    "$STAGE/docker/evelyn-native/install.sh"

tar -czf "$OUT" -C "$BUILD_DIR" evelyn-src/

echo "=== Done ==="
echo "    $OUT"
echo "    $(du -sh "$OUT" | cut -f1)  $(tar -tzf "$OUT" | wc -l) files"
echo
echo "  Transfer to build machine, then:"
echo "    tar -xzf $OUT_NAME"
echo "    bash evelyn-src/scripts/evelyn-k8s-build.sh [--tag evelyn:1.2.3]"

5. Phase 2 — Build Script: evelyn-k8s-build.sh

Runs on any machine with Docker. Does not need the data_server repo — reads everything from the extracted source tar (SRC_DIR). The script is included inside the source tar at evelyn-src/scripts/evelyn-k8s-build.sh so no separate checkout is needed.

scripts/evelyn-k8s-build.sh
#!/usr/bin/env bash
# evelyn-k8s-build.sh — compile binaries and produce the deployment tar
#
# Usage:  bash evelyn-src/scripts/evelyn-k8s-build.sh [--tag evelyn:1.2.3]
#
# SRC_DIR defaults to the parent of this script (works when run from extracted
# evelyn-src tar). Override with: SRC_DIR=/path/to/evelyn-src bash ...
#
# Requires: docker (running)
# Output:   dist/evelyn-deploy-YYYYMMDD.tar.gz  (in SRC_DIR/dist/)
set -euo pipefail

# SRC_DIR = extracted evelyn-src directory (contains source/, html/, docker/, scripts/)
SRC_DIR="${EVELYN_SRC:-$(cd "$(dirname "$0")/.." && pwd)}"
DATE="$(date +%Y%m%d)"
OUT_NAME="evelyn-deploy-${DATE}.tar.gz"
OUT="$SRC_DIR/dist/$OUT_NAME"
BUILD_DIR="$(mktemp -d /tmp/evelyn-build-XXXXXX)"
STAGE="$BUILD_DIR/evelyn-deploy"
FLASHMQ_TAG="v1.9.1"
FLASHMQ_CACHE="$SRC_DIR/.cache/flashmq-${FLASHMQ_TAG}"
UBUNTU_IMAGE="ubuntu:24.04"

trap 'echo "[build] cleaning up $BUILD_DIR"; rm -rf "$BUILD_DIR"' EXIT

echo "=== Evelyn build ==="
echo "    src:    $SRC_DIR"
echo "    output: $OUT"
echo "    stage:  $STAGE"
echo

# ── Prerequisite checks ───────────────────────────────────────────────────────
if ! command -v docker &>/dev/null; then
    echo "ERROR: docker not found — install Docker and ensure the daemon is running"
    exit 1
fi
if ! docker info &>/dev/null; then
    echo "ERROR: Docker daemon not running or permission denied"
    exit 1
fi
if [[ ! -d "$SRC_DIR/source/parquet_writer" ]]; then
    echo "ERROR: $SRC_DIR/source/parquet_writer not found"
    echo "       Run evelyn-src-extract.sh on build-dev first, then:"
    echo "       tar -xzf evelyn-src-YYYYMMDD.tar.gz"
    exit 1
fi

mkdir -p "$SRC_DIR/dist"
mkdir -p \
    "$STAGE/bin" \
    "$STAGE/lib" \
    "$STAGE/html/writer" \
    "$STAGE/configs" \
    "$STAGE/scripts" \
    "$STAGE/logrotate" \
    "$STAGE/k8s"

# ── Step 1: build parquet_writer ─────────────────────────────────────────────
# (defined in its own section below)
build_parquet_writer

# ── Step 2: build FlashMQ ────────────────────────────────────────────────────
build_flashmq

# ── Step 3: collect repo files ────────────────────────────────────────────────
collect_files

# ── Step 4: write Kubernetes manifests ────────────────────────────────────────
write_k8s_manifests

# ── Step 5: package ───────────────────────────────────────────────────────────
tar -czf "$OUT" -C "$BUILD_DIR" evelyn-deploy/
echo
echo "=== Done ==="
echo "    $OUT"
echo "    $(du -sh "$OUT" | cut -f1)  $(tar -tzf "$OUT" | wc -l) files"

The five build functions are defined below. They are all in the same script file — split here for clarity.

6. Step 1 — Build parquet_writer (Docker)

Mounts source/parquet_writer/ read-only into an Ubuntu 24.04 container, installs dev headers, runs make parquet_writer, and then extracts the runtime .so files needed for bare-metal installs. Both the binary and libs are written to the host via an /out mount.

build_parquet_writer() {
    echo "[step 1] building parquet_writer..."

    # /src is read-only so the container cannot dirty the repo.
    # /out is writable — binary + runtime .so files land in $BUILD_DIR.
    docker run --rm \
        -v "$SRC_DIR/source/parquet_writer":/src:ro \
        -v "$BUILD_DIR":/out \
        "$UBUNTU_IMAGE" bash -c "
            set -e
            export DEBIAN_FRONTEND=noninteractive
            apt-get update -q
            apt-get install -y -q --no-install-recommends \
                build-essential \
                libarrow-dev libparquet-dev \
                libmosquitto-dev libyaml-cpp-dev \
                libsnappy-dev libzstd-dev liblz4-dev \
                libssl-dev libsimdjson-dev
            cp -r /src /build && cd /build
            make parquet_writer
            cp parquet_writer /out/parquet_writer
            # Extract runtime .so files for native (non-Docker) deployment.
            # ldd gives the resolved path (e.g. /usr/lib/x86_64-linux-gnu/libarrow.so.1600).
            # We also copy all versioned siblings (libarrow.so, libarrow.so.1600, etc.)
            # so the binary's DT_NEEDED soname is satisfied without ldconfig on the target.
            mkdir -p /out/lib
            ldd /out/parquet_writer \
                | awk '/=> \// { print \$3 }' \
                | grep -E 'libarrow|libparquet|libmosquitto|libyaml-cpp|libsnappy|libzstd|liblz4|libsimdjson' \
                | sort -u \
                | while IFS= read -r sopath; do
                    libdir=\$(dirname "\$sopath")
                    libstem=\$(basename "\$sopath" | cut -d. -f1)  # e.g. libarrow
                    # copy .so symlinks and real versioned files
                    find "\$libdir" -maxdepth 1 -name "\${libstem}.so*" \
                        | xargs -I@ cp -P @ /out/lib/
                  done
        "

    cp "$BUILD_DIR/parquet_writer" "$STAGE/bin/parquet_writer"
    chmod +x "$STAGE/bin/parquet_writer"
    echo "    → bin/parquet_writer  ($(du -sh "$STAGE/bin/parquet_writer" | cut -f1))"

    mkdir -p "$STAGE/lib"
    cp "$BUILD_DIR/lib/"* "$STAGE/lib/"
    echo "    → lib/  ($(ls "$STAGE/lib/" | wc -l) runtime .so files)"
}
Why copy /src to /build? Arrow's pkg-config sometimes resolves relative paths. Copying to /build inside the container avoids any path issues with the read-only mount and keeps the host repo clean.
Runtime libs: The build copies the full symlink chain for each dependency (e.g. libarrow.so → libarrow.so.1600 → libarrow.so.1600.0.0) using cp -P (preserve symlinks) plus the real file. This ensures the binary's embedded DT_NEEDED soname resolves correctly without relying on the target's package manager. For native installs, install.sh copies lib/ to /opt/evelyn/lib/, writes /etc/ld.so.conf.d/evelyn.conf, and runs ldconfig. In the Docker container (Option C) apt-get install libarrow1600 libparquet1600 ... provides the libs instead.

7. Step 2 — Build FlashMQ (Docker, cached)

Clones the tagged FlashMQ release inside an Ubuntu 24.04 container and builds with CMake. The compiled binary is cached at .cache/flashmq-<tag> in the repo root — subsequent builds skip the Docker step entirely if the cached binary exists.

build_flashmq() {
    # Use cached binary if already built for this tag — saves ~2-3 min
    if [[ -f "$FLASHMQ_CACHE" ]]; then
        echo "[step 2] FlashMQ $FLASHMQ_TAG already cached — skipping Docker build"
        cp "$FLASHMQ_CACHE" "$STAGE/bin/flashmq"
        chmod +x "$STAGE/bin/flashmq"
        echo "    → bin/flashmq  ($(du -sh "$STAGE/bin/flashmq" | cut -f1))  [from cache]"
        return
    fi

    echo "[step 2] building FlashMQ $FLASHMQ_TAG (not cached)..."

    if [[ -d "$SRC_DIR/flashmq-src" ]]; then
        echo "    → using local source at $SRC_DIR/flashmq-src (air-gapped)"
        docker run --rm \
            -v "$BUILD_DIR":/out \
            -v "$SRC_DIR/flashmq-src":/flashmq:ro \
            "$UBUNTU_IMAGE" bash -c "
                set -e
                export DEBIAN_FRONTEND=noninteractive
                apt-get update -q
                apt-get install -y -q --no-install-recommends \
                    cmake build-essential libssl-dev
                cmake -B /flashmq/build -S /flashmq \
                      -DCMAKE_BUILD_TYPE=Release
                cmake --build /flashmq/build --parallel \$(nproc)
                cp /flashmq/build/flashmq /out/flashmq
            "
    else
        docker run --rm \
            -v "$BUILD_DIR":/out \
            "$UBUNTU_IMAGE" bash -c "
                set -e
                export DEBIAN_FRONTEND=noninteractive
                apt-get update -q
                apt-get install -y -q --no-install-recommends \
                    cmake build-essential git libssl-dev
                git clone --branch $FLASHMQ_TAG --depth 1 \
                    https://github.com/halfgaar/FlashMQ.git /flashmq
                cmake -B /flashmq/build -S /flashmq \
                      -DCMAKE_BUILD_TYPE=Release
                cmake --build /flashmq/build --parallel \$(nproc)
                cp /flashmq/build/flashmq /out/flashmq
            "
    fi

    # Save to cache so next build skips this step
    mkdir -p "$SRC_DIR/.cache"
    cp "$BUILD_DIR/flashmq" "$FLASHMQ_CACHE"

    cp "$BUILD_DIR/flashmq" "$STAGE/bin/flashmq"
    chmod +x "$STAGE/bin/flashmq"
    echo "    → bin/flashmq  ($(du -sh "$STAGE/bin/flashmq" | cut -f1))  [built + cached]"
}
Internet required on first build unless running air-gapped. Pre-clone FlashMQ into flashmq-src/ inside the extracted source tar (i.e. place it at $SRC_DIR/flashmq-src/). When that directory exists the build script mounts it read-only and skips the git clone entirely — no internet needed. Add .cache/ to .gitignore.
Cache invalidation: Changing FLASHMQ_TAG (e.g. to v2.0.0) automatically misses the cache because the path includes the tag. To force a rebuild without changing the tag: rm .cache/flashmq-v1.9.1.

8. Step 3 — Collect files

All HTML pages, Python scripts, shell scripts, and config files are copied from the extracted source tar (SRC_DIR) into the staging directory. No repo access needed.

collect_files() {
    echo "[step 3] collecting files from source tar..."
    R="$SRC_DIR"
    S="$STAGE"

    # ── HTML pages ────────────────────────────────────────────────────────────
    cp "$R/html/writer/evelyn_docker_34.html"         "$S/html/index.html"
    cp "$R/html/parquet_monitor.html"                  "$S/html/"
    cp "$R/html/writer/parquet_viewer.html"            "$S/html/writer/"
    cp "$R/html/writer/writer_design.html"             "$S/html/writer/"
    cp "$R/html/writer/evelyn_real_time_data.html"     "$S/html/writer/"
    cp "$R/html/writer/evelyn_container_design.html"   "$S/html/writer/"
    cp "$R/html/writer/evelyn_writer_project_build.html" "$S/html/writer/"

    # ── Python scripts + requirements ─────────────────────────────────────────
    # bridge_api_native.py    — systemd-aware management API for bare-metal installs
    cp "$R/docker/evelyn-native/bridge_api_native.py"        "$S/"

    # requirements.txt — installed by install.sh (native) or pip layer in Dockerfile
    cat > "$S/requirements.txt" <<'REQEOF'
duckdb>=0.10
PyYAML>=6.0
REQEOF

    # ── Shell scripts ─────────────────────────────────────────────────────────
    cp "$R/docker/evelyn-native/install.sh"           "$S/scripts/"
    cp "$R/scripts/evelyn-k8s-build.sh"               "$S/scripts/"  # include build script in deploy tar
    chmod +x "$S/scripts/"*.sh

    # ── Logrotate config (native install) ─────────────────────────────────────
    # install.sh copies this to /etc/logrotate.d/evelyn
    mkdir -p "$S/logrotate"
    cat > "$S/logrotate/evelyn" <<'LOGROTATEOF'
/var/log/supervisor/*.log {
    daily
    rotate 7
    compress
    delaycompress
    missingok
    notifempty
    copytruncate
}
LOGROTATEOF

    # ── Configs ───────────────────────────────────────────────────────────────
    # Default configs installed to /etc/evelyn/ by install.sh on first run.
    cp "$R/docker/evelyn-native/configs/flashmq.conf"        "$S/configs/flashmq.conf"
    cp "$R/docker/evelyn-native/configs/writer-native.yaml"   "$S/configs/writer-native.yaml"

    echo "    → $(find "$S" -type f | wc -l) files staged"
}
All shell scripts are chmod +x in the staging directory so they are executable when extracted on the target machine without needing a separate chmod step.

9. Step 4 — Write Kubernetes manifests

The K8s manifests are written as heredocs inside the build script — no separate template files to maintain. They are generated fresh each build so the image tag embedded in deployment.yaml is always current.

write_k8s_manifests() {
    echo "[step 4] writing Kubernetes manifests..."
    K="$STAGE/k8s"
    IMAGE_TAG="${EVELYN_IMAGE:-evelyn:latest}"

    # namespace.yaml
    cat > "$K/namespace.yaml" <<'EOF'
apiVersion: v1
kind: Namespace
metadata:
  name: evelyn
EOF

    # configmap.yaml  — default configs; override with site-specific values
    cat > "$K/configmap.yaml" <<EOF
apiVersion: v1
kind: ConfigMap
metadata:
  name: evelyn-config
  namespace: evelyn
data:
  flashmq.conf: |
$(sed 's/^/    /' "$STAGE/configs/flashmq.conf")
  writer.yaml: |
$(sed 's/^/    /' "$STAGE/configs/writer.yaml")
EOF

    # pvc.yaml
    cat > "$K/pvc.yaml" <<'EOF'
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: evelyn-parquet-pvc
  namespace: evelyn
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Gi
  # storageClassName: ""   # uncomment and set for a specific StorageClass
EOF

    # deployment.yaml
    cat > "$K/deployment.yaml" <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: evelyn
  namespace: evelyn
  labels:
    app: evelyn
spec:
  replicas: 1
  selector:
    matchLabels:
      app: evelyn
  template:
    metadata:
      labels:
        app: evelyn
    spec:
      containers:
        - name: evelyn
          image: ${IMAGE_TAG}
          imagePullPolicy: IfNotPresent
          ports:
            - name: mqtt
              containerPort: 1883
              protocol: TCP
            - name: mqtt-ws
              containerPort: 9001
              protocol: TCP
            - name: http-ui
              containerPort: 8080
              protocol: TCP
            - name: health
              containerPort: 8771
              protocol: TCP
            - name: bridge-api
              containerPort: 8772
              protocol: TCP
          volumeMounts:
            - name: parquet-data
              mountPath: /data/parquet
            - name: evelyn-config
              mountPath: /etc/evelyn
              readOnly: false
          readinessProbe:
            httpGet:
              path: /health
              port: 8771
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /health
              port: 8771
            initialDelaySeconds: 15
            periodSeconds: 30
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
            limits:
              cpu: "4"
              memory: "4Gi"
      initContainers:
        - name: init-config
          image: busybox:1.36
          command:
            - sh
            - -c
            - |
              if [ ! -f /etc/evelyn/writer.yaml ]; then
                echo "[init] installing default writer.yaml"
                cp /etc/evelyn-defaults/writer.yaml /etc/evelyn/writer.yaml
              fi
              if [ ! -f /etc/evelyn/flashmq.conf ]; then
                echo "[init] installing default flashmq.conf"
                cp /etc/evelyn-defaults/flashmq.conf /etc/evelyn/flashmq.conf
              fi
          volumeMounts:
            - name: evelyn-config
              mountPath: /etc/evelyn
            - name: evelyn-defaults
              mountPath: /etc/evelyn-defaults
              readOnly: true
      volumes:
        - name: parquet-data
          persistentVolumeClaim:
            claimName: evelyn-parquet-pvc
        - name: evelyn-config
          emptyDir: {}
        - name: evelyn-defaults
          configMap:
            name: evelyn-config
EOF

    # service.yaml
    cat > "$K/service.yaml" <<'EOF'
apiVersion: v1
kind: Service
metadata:
  name: evelyn
  namespace: evelyn
spec:
  type: NodePort
  selector:
    app: evelyn
  ports:
    - name: mqtt
      port: 1883
      targetPort: 1883
      nodePort: 31883
      protocol: TCP
    - name: mqtt-ws
      port: 9001
      targetPort: 9001
      nodePort: 31901
      protocol: TCP
    - name: http-ui
      port: 8080
      targetPort: 8080
      nodePort: 30880
      protocol: TCP
    - name: health
      port: 8771
      targetPort: 8771
      nodePort: 30871
      protocol: TCP
    - name: bridge-api
      port: 8772
      targetPort: 8772
      nodePort: 30872
      protocol: TCP
EOF

    echo "    → k8s/namespace.yaml"
    echo "    → k8s/configmap.yaml  (flashmq.conf + writer.yaml embedded)"
    echo "    → k8s/pvc.yaml"
    echo "    → k8s/deployment.yaml  (image: ${IMAGE_TAG})"
    echo "    → k8s/service.yaml    (NodePort 31883/31901/30880/30871/30872)"
}

# Final packaging — $BUILD_DIR is cleaned by the EXIT trap
tar -czf "$OUT" -C "$BUILD_DIR" evelyn-deploy/

echo
echo "=== Done ==="
echo "    $OUT"
echo "    $(du -sh "$OUT" | cut -f1)  $(tar -tzf "$OUT" | wc -l) files"

echo "  Deploy:   see evelyn-deploy/k8s/ and evelyn-deploy/html/writer/evelyn_writer_project_build.html"

The trap ... EXIT at the top of the script deletes $BUILD_DIR automatically on exit — whether the build succeeds or fails.

ConfigMap strategy

The evelyn-config ConfigMap is used two ways:

  • As the defaults source: the init-config init container copies from /etc/evelyn-defaults/ (mounted from ConfigMap) into /etc/evelyn/ (emptyDir) only when the files are absent.
  • As the bootstrap values: once a site-specific bridge has been configured via POST /bridge, the live config in /etc/evelyn/ diverges from the ConfigMap — this is intentional. The ConfigMap represents factory defaults.
Config persistence caveat: The evelyn-config volume is emptyDir — it is lost when the pod restarts. For production, replace the emptyDir with a second PVC or a ConfigMap + mutation webhook so bridge changes survive pod restarts. The parquet data PVC is persistent.

13. Notes & Caveats

TopicDetail
FlashMQ binary size ~8–12 MB stripped. Building from source takes ~2–3 min on a 4-core machine. To cache across builds, push the FlashMQ binary to a separate Docker layer or store it in bin/flashmq-cache in the repo and skip Step 2 if already present.
parquet_writer binary size ~15–20 MB. Arrow/Parquet libs are linked dynamically. The build step extracts the full .so symlink chain from the build container (e.g. libarrow.so → libarrow.so.1600 → libarrow.so.1600.0.0) into lib/. install.sh copies these to /opt/evelyn/lib/ and drops /etc/ld.so.conf.d/evelyn.conf + runs ldconfig. In the Docker container the runtime image's apt-get install libarrow1600 handles this instead.
Python dependencies bridge_api_container.py and bridge_api_native.py require duckdb and PyYAML. Both are in requirements.txt. In the container the Dockerfile runs pip install -r requirements.txt. For native installs, install.sh runs pip3 install -r requirements.txt (requires python3-pip on the target — Ubuntu: apt-get install python3-pip). All other imports (asyncio, json, subprocess, os, re) are Python stdlib.
K8s config persistence Config volume is emptyDir by default — lost on pod restart. For production, use a second PVC (evelyn-config-pvc) or bake the site config into the image at CI time.
Single-writer design This deployment pattern is for one parquet_writer instance per container / host. For multi-writer topologies, deploy separate containers with different site_id, MQTT topic prefix, and data path.
nginx — when needed nginx is only required if you want the HTML dashboards served from the container itself (Option C). For bare-metal installs where a host web server already exists, nginx can be omitted — remove the [program:nginx] block from supervisord.conf and skip the nginx apt install. The parquet_writer, FlashMQ, and bridge_api work independently of it.
Log files In the container, supervisord rotates logs via stdout_logfile_maxbytes/stdout_logfile_backups (nginx redirected to stdout). For native installs, install.sh copies logrotate/evelyn to /etc/logrotate.d/evelyn — rotates /var/log/supervisor/*.log daily, 7-day retention. Force a test with logrotate -d /etc/logrotate.d/evelyn.
Air-gapped build Step 2 (FlashMQ) requires internet access to clone from GitHub. For air-gapped environments, pre-clone FlashMQ into docker/evelyn-container/flashmq-src/ and modify the build step to mount and build from that path instead.
Arrow APT repo Ubuntu 24.04 ships Arrow 14 in the default apt repo. If a newer Arrow is required, add the Apache Arrow APT source inside the Docker build container before apt-get install.
bridge_api_container.py This file is generated / defined in evelyn_container_design.html and must be written to docker/evelyn-container/bridge_api_container.py in the repo before running the build script. The build script will fail with a clear error if it is absent.

  "
