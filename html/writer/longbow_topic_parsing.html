Longbow MQTT Topic Parsing

How real Longbow data maps to parquet_writer config — segment roles, partition design, and multi-site handling

TL;DR
  • Real Longbow topics follow ems/site/{site_id}/unit/{unit_id}/{device}/{instance}/{point_name}/{dtype_hint} — 9 segments, confirmed from a 1.3M row recording.
  • The recording contains 4 distinct topic depths (9/8/7/6 segments) for different signal classes — a single topic_segments array only handles depth-9 correctly.
  • Use topic_patterns (already implemented in parquet_writer.cpp) with one pattern per depth — first match wins.
  • Output file paths are always correct for all depths — site_id is at segment 2 in every variant. Only stored column values break without patterns.
  • unit_id must be a column, not a filename prefix — 46 units × per-flush files = small-file problem.
  • Set output.site_id to empty in config so the topic-extracted value drives partitioning. The config value is a fallback only.
  • String payloads (dtype_hint=string) are kept — IP addresses, version strings, ifconfig dumps stored as UTF-8.
  • Three small code changes needed: (1) don't overwrite topic-extracted site_id, (2) store string JSON values in row.strings, (3) remove redundant static site_id in path vars.

Segment 0 (ems) is always source_type — it is consumed automatically as parts[0] and becomes the first directory level in the output path. Do not add a skip for it in topic_segments — the array starts at segment 0, which means index 0 = "_" skips the source_type and index 1 maps to site. Start the segments array at index 0 = "_" to skip ems, index 1 = "_" to skip site, etc.

2. Segment Role Decisions

SegExample valueRoleRationale
0 ems source_type Always parts[0] — auto-consumed, feeds first output dir level. Use "_" in segments to skip storing it as a column.
1 site skip Constant literal — no information value.
2 0215D1D8 partition + column Site serial. Extracted as site_id column and used as the partition directory value. Enables one writer to handle multiple sites on a shared broker.
3 unit skip Constant literal — no information value.
4 0228168A column Unit serial (46 units per site). Stored as unit_id column — not a file prefix. Using it as a prefix would create 46 files per flush cycle (small-file problem).
5 pcs / bms / rack column Device class. Key filter for queries — "give me all PCS signals".
6 pcs_2 / bms_1 column Device instance. Differentiates multiple PCS or BMS units within the same hardware unit.
7 Hz / PF / Batt2_SBMU1_... column Signal name (point_name). Primary query dimension — "give me Hz over time".
8 float / integer / string dtype_hint Drives INT64 vs FLOAT64 cast; also signals string payloads (IP addresses, ifconfig dumps). String values are stored as UTF-8 in the value_str column.

Numeric float keep

Payload: {"ts":"2024-11-15T21:27:51.836Z","value":60.010000}

→ stored as FLOAT64 row
→ point_name = "Hz", device = "pcs"

Integer / fault flag keep

Payload: {"ts":"2024-11-15T21:27:52.775Z","value":0.000000}

→ stored as INT64 row (value cast: 0)
→ mostly zero — Filter B (dedup) eliminates these

4. parquet_writer Config

Standard Longbow — single or multi-site broker

mqtt:
  host: localhost
  port: 1883
  client_id: parquet-writer-longbow
  topic: "ems/#"
  qos: 0

  topic_parser: positional
  topic_segments:
    - "_"        # 0: "ems"       → source_type (auto), skip as column
    - "_"        # 1: "site"      → constant literal, skip
    - site_id    # 2: "0215D1D8"  → partition value + column
    - "_"        # 3: "unit"      → constant literal, skip
    - unit_id    # 4: "0228168A"  → column (46 values — not a file prefix)
    - device     # 5: "pcs"       → column
    - instance   # 6: "pcs_2"     → column
    - point_name # 7: "Hz"        → column
    - dtype_hint # 8: "float"     → INT64 vs FLOAT64 cast, not stored

  partition_field: site_id        # uses value extracted from topic segment 2
  timestamp_field:  ts
  timestamp_format: iso8601

output:
  base_path: /data/parquet-longbow
  partitions:
    - "site={site_id}"
    - "{year}"
    - "{month}"
    - "{day}"
  compression: snappy
  flush_interval_seconds: 60
  max_messages_per_part: 5000000

wal:
  enabled: true

health:
  enabled: true
  port: 8771

guard:
  min_free_gb: 2.0
Output path produced
/data/parquet-longbow/ems/site=0215D1D8/2026/03/30/20260330T120000Z.parquet
partition_field: site_id — the writer uses the value extracted from topic segment 2 as the partition directory value, rather than the static output.site_id config field. This allows a single writer instance to receive data from multiple sites on one broker and split them into separate directory trees automatically.

6. File Prefix Strategy

output.filename_prefix is a static config value intended for writer-process-level collision avoidance — not for per-message metadata. Site-level separation is handled by the partition directory; unit-level data lives as columns.

Use caseRecommended approachExample output path
Single writer, single site No prefix needed — partition dirs are sufficient /data/parquet/ems/site=0215D1D8/2026/03/30/{ts}.parquet
Multi-site broker partition_field: site_id from topic → automatic split /data/parquet/ems/site=0215D1D8/…/{ts}.parquet
/data/parquet/ems/site=SiteSN/…/{ts}.parquet
Two writer processes, shared dir filename_prefix: writer-a in config …/writer-a_20260330T120000Z.parquet
Multi-slice (gx10 style) Topic prefix (A/unit/#) → source_type = "A" → separate dir /data/parquet/A/site=… vs /data/parquet/B/site=…
The gx10 multi-slice approach (prefix letter in topic = first segment = source_type) is the cleanest design — no extra config needed, each writer gets its own output directory tree automatically just from the topic subscription filter.

7. Topic Depth Analysis — Real Recording

Analysis of ref_data/longbow_recording.csv.gz (1,304,167 rows) reveals four distinct topic depths. A single topic_segments array only correctly handles the dominant 9-segment form.

DepthCount%Example topicStructure
91,238,69895% ems/site/0215D1D8/unit/0215F562/bms/bms_1/AvgCellT/float unit signals with device + instance
827,9752% ems/site/0215D1D8/unit/0215F562/root/AllowedMaxP/float unit root signals — no instance level
712,8101% ems/site/0215D1D8/meter/meter_1/Current/float meter / site-device — no unit segment

What a single 9-entry topic_segments produces for each depth

apply_segments maps positionally — segs[i] → parts[i], stops at whichever is shorter. The output directory is always correct (site_id is segment 2 in all depths). The column values break when depth < 9. 

9-seg (correct):
ems / site / 0215D1D8 / unit / 0215F562 / bms  / bms_1    / AvgCellT   / float
 _      _    site_id    _     unit_id    device  instance   point_name   dtype_hint

→ device="bms"  instance="bms_1"  point_name="AvgCellT"  dtype_hint="float"
→ /data/parquet-longbow/ems/site=0215D1D8/2026/03/30/{ts}.parquet  ✓

8-seg (columns shifted — dtype_hint lost):
ems / site / 0215D1D8 / unit / 0215F562 / root / AllowedMaxP / float
 _      _    site_id    _     unit_id    device  instance      point_name
                                                               ↑ "float" stored as point_name
                                                               dtype_hint = (missing — seg 8)

→ device="root"  instance="AllowedMaxP"  point_name="float"  dtype_hint=""
→ value stored as float by default (correct by luck)
→ /data/parquet-longbow/ems/site=0215D1D8/2026/03/30/{ts}.parquet  (dir ✓, columns ✗)

7-seg (meter topics — no 'unit' in path):
ems / site / 0215D1D8 / meter / meter_1 / Current / float
 _      _    site_id     _     unit_id    device    instance
                              ↑ "meter_1" stored as unit_id
                                          ↑ "Current" stored as device
                                                     ↑ "float" stored as instance
                              point_name = ""  dtype_hint = ""

→ unit_id="meter_1"  device="Current"  instance="float"  point_name=""
→ /data/parquet-longbow/ems/site=0215D1D8/2026/03/30/{ts}.parquet  (dir ✓, columns ✗)

6-seg (site-level root — no unit):
ems / site / 0215D1D8 / root / AllowedMaxP / float
 _      _    site_id    _     unit_id        device
                              ↑ "AllowedMaxP" stored as unit_id
                                              ↑ "float" stored as device
                              instance, point_name, dtype_hint = all empty

→ unit_id="AllowedMaxP"  device="float"  everything else empty
→ /data/parquet-longbow/ems/site=0215D1D8/2026/03/30/{ts}.parquet  (dir ✓, columns ✗)
Key insight: the output file path is correct for all four depths because site_id is always at segment 2. Only the stored column values break for depths 6, 7, and 8. This is exactly the problem topic_patterns solves.

8. topic_patterns — Four-Pattern Config

topic_patterns is already implemented in parquet_writer.cpp — match_topic_pattern(), Config::TopicPattern, and the multi-pattern branch in parse_topic_positional() all exist. It just needs the right config. First matching pattern wins; + matches any single segment, # matches remainder. 

mqtt:
  topic: "ems/#"
  topic_parser: positional   # topic_patterns takes over when present (topic_segments ignored)
  partition_field: site_id
  timestamp_field:  ts
  timestamp_format: iso8601

  topic_patterns:

    # 9-seg: unit signals with device + instance (95% of traffic)
    # ems/site/{site}/unit/{unit}/{device}/{instance}/{point}/{dtype}
    - match: "ems/site/+/unit/+/+/+/+/+"
      segments: ["_","_","site_id","_","unit_id","device","instance","point_name","dtype_hint"]

    # 8-seg: unit root signals — no instance level
    # ems/site/{site}/unit/{unit}/root/{point}/{dtype}
    - match: "ems/site/+/unit/+/+/+/+"
      segments: ["_","_","site_id","_","unit_id","device","point_name","dtype_hint"]

    # 7-seg: meter / site-device signals — no unit
    # ems/site/{site}/{device}/{instance}/{point}/{dtype}
    - match: "ems/site/+/+/+/+/+"
      segments: ["_","_","site_id","device","instance","point_name","dtype_hint"]

    # 6-seg: site-level root signals — no unit or instance
    # ems/site/{site}/root/{point}/{dtype}
    - match: "ems/site/+/+/+/+"
      segments: ["_","_","site_id","device","point_name","dtype_hint"]

Resulting columns per pattern

Patternsite_idunit_iddeviceinstancepoint_namedtype_hint
9-seg 0215D1D8 0215F562 bms bms_1 AvgCellT float
8-seg 0215D1D8 0215F562 root — AllowedMaxP float
7-seg 0215D1D8 — meter meter_1 Current float
6-seg 0215D1D8 — root — AllowedMaxP float
DuckDB handles sparse columns across row groups — rows where unit_id or instance are absent simply have nulls in those columns. All four patterns write to the same partition directory and the same Parquet file per flush cycle.

9. String Payload Handling

String-typed messages (dtype_hint = string) carry system metadata: IP addresses, ifconfig output, software version strings. These are kept in full — they are stored as UTF-8 in the Parquet row alongside numeric signals. ~4% of Longbow message volume.

String signals (all kept)

ems/site/SiteSN/unit/0215D1D8/root/IP_address/string
→ value: "10.7.2.72"

ems/site/SiteSN/unit/0215D1D8/root/software_version/string
→ value: "v3.1.4"

ems/site/SiteSN/unit/0215D1D8/root/ifconfig/string
→ value: "eth0  inet addr:10.7.2.72 ..."  (multi-KB blob)

Parquet schema note

All three dtype_hint values map to columns:

  float   → value  FLOAT64
  integer → value  INT64
  string  → value  UTF-8 string

DuckDB handles the mixed column type across
row groups transparently when querying.

Ifconfig blobs are large but low-frequency
(~1/s vs 81k/s for numeric signals) so their
storage impact is negligible.

10. DuckDB Query Examples

-- All Hz readings for site 0215D1D8, last 24 hours
SELECT ts, unit_id, instance, value
FROM read_parquet('/data/parquet-longbow/ems/site=0215D1D8/2026/03/30/*.parquet')
WHERE point_name = 'Hz'
ORDER BY ts;

-- PCS power factor across all units
SELECT ts, unit_id, instance, value
FROM read_parquet('/data/parquet-longbow/ems/site=0215D1D8/*/*/*.parquet')
WHERE device = 'pcs' AND point_name = 'PF';

-- All sites — fault flags that fired (value != 0)
SELECT site_id, unit_id, device, point_name, value, ts
FROM read_parquet('/data/parquet-longbow/ems/site=*/2026/03/30/*.parquet')
WHERE value != 0 AND point_name LIKE 'Batt%warning%'
ORDER BY ts DESC
LIMIT 100;
DuckDB prunes partition directories automatically from the glob path — site=0215D1D8 in the path filters to a single site without scanning others. Columns (unit_id, device, point_name) are filtered within each file using Parquet column statistics.
