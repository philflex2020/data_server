AWS Cost Reduction — Management Proposal

Infrastructure optimisation: data archival + edge filtering + InfluxDB migration path

Prepared: March 2026  ·  Author: Engineering  ·  Status: Proposal — awaiting approval

Executive Summary

AWS spend peaked at $88k/month in September 2025 and has been reduced to $55k/month in March 2026 through IoT Core and Lambda migration work already completed. However, two cost lines are now growing: Elastic File System (InfluxDB storage) +60% and EC2 (compute) +15% over the same period, driven by new client onboarding. Without further action these will reverse the savings already made.

This proposal outlines two engineering initiatives that together are estimated to save $13,000–16,000/month ($156k–192k/year) and hold the bill roughly flat as new clients are added — effectively making future growth near cost-neutral on infrastructure.

Both initiatives build on work already in progress. Combined engineering effort is estimated at 6–8 weeks with no customer-facing risk — all changes are in the data backend, not the product UI or data collection pipeline.

The growth problem

EFS stores all InfluxDB2 time-series data. Every new client site adds series and write volume permanently. At the current growth rate of ~10%/month, EFS alone reaches $20k/month by end of 2026. EC2 follows a similar curve as InfluxDB ingest and query load grows. The two together will exceed the IoT/Lambda savings within 6 months without action.

Proposed Initiatives

Initiative 1 — Data Archival: move historical InfluxDB data to cheap cold storage

InfluxDB2 stores all data on EFS at ~$0.30/GB. Data older than 90 days is rarely queried in real time but must be retained for warranty, compliance, and engineering analysis. Parquet files on S3 store the same data at ~$0.023/GB — 13× cheaper, with better compression giving an effective ratio of ~130× vs InfluxDB on EFS.

The query tool (subscriber_api) already reads parquet files via DuckDB. A routing layer means callers — including Grafana and the UI — see a single API regardless of whether data is in InfluxDB or the parquet archive. No product change required.

Initiative 2 — Edge Filtering: reduce data volume before it reaches AWS

Today, every sensor reading from every site is sent to AWS whether or not the value changed. A Telegraf processor on the edge system (our hardware at each site) suppresses readings that are within a configurable tolerance of the previous value. Testing shows this reduces message volume by ~27–30% with no loss of meaningful data.

This reduces EC2 write load, EFS write volume, and data transfer simultaneously. The edge processor is a configuration change — no firmware update, no customer impact.

Execution Plan

1
2 weeks

Harden the query API (subscriber_api)

Extend the existing data query service to support both InfluxDB2 (live/recent data) and DuckDB/parquet (archive data) behind a single API. Queries crossing the boundary between the two are merged transparently. Add a health endpoint and HTTP interface alongside the existing WebSocket.

  • No changes to InfluxDB, no data moved yet
  • Validated against real data on fractal-phil
  • Full test suite: unit tests + integration tests against live InfluxDB
Risk: Low — additive only, no existing behaviour changed
2
1 week

Extract historical data to parquet — no deletion yet

Run the archive job against production InfluxDB2. All data older than 90 days is extracted, written to date-partitioned parquet files on S3, and verified (row counts must match exactly before we proceed). Nothing is deleted from InfluxDB at this stage.

  • Parquet files are the safety net — data exists in two places simultaneously
  • Archive job runs in the background; no service interruption
  • Row count verification is automated and logged
Risk: Low — read-only operation on InfluxDB, data is duplicated not moved
3
1–2 weeks

Validation — query both backends, compare results

With data in both InfluxDB and parquet, run identical queries against both backends and compare results. Grafana dashboards, the UI, and engineering spot-checks all run against the new unified API. Sign-off required before proceeding to deletion.

  • Automated diff tests: same query → same results from both backends
  • Manual spot-check: engineering team validates 3–5 representative sites
  • No time pressure — InfluxDB data remains intact throughout
Risk: Low — no data deleted, rollback is trivially "keep using InfluxDB"
4
1 day

Delete historical data from InfluxDB — EFS bill drops immediately

Once validation passes, delete data older than 90 days from InfluxDB2 buckets. InfluxDB releases the EFS space within hours. The bill reduction appears on the next AWS billing cycle.

  • Deletion is per-bucket, per-time-range — can be done incrementally by site if preferred
  • Parquet archive is the permanent record; InfluxDB is now a 90-day rolling window
  • Set InfluxDB2 retention policy to 90 days to automate going forward
Risk: Medium — irreversible, but only after Phase 3 sign-off. Parquet is the backup.
5
2–3 weeks

Edge filtering — deploy to production sites (rolling)

Roll out the Telegraf dedup/filter configuration to edge systems site by site. Each deployment is a config file update and Telegraf restart — no firmware change, no outage. Monitor InfluxDB write rate and EFS growth after each site is updated.

  • Start with 2–3 pilot sites, measure actual reduction, then roll out broadly
  • Configurable tolerance per signal type — conservative defaults, tunable per site
  • The subscriber_api is already in place from Phase 1, so historical queries are unaffected
Risk: Low — config change only, easily reverted per site
6
ongoing

InfluxDB2 → InfluxDB3 migration (parallel track, no deadline)

The subscriber_api routing layer built in Phase 1 is designed to support InfluxDB3 as a drop-in hot backend. When the team decides to migrate, a shadow mode runs both databases in parallel and compares query results until confidence is high, then a single config change cuts over. No product change required.

  • InfluxDB3 stores data as parquet on S3 natively — EFS cost disappears entirely
  • This initiative is not on the critical path for the cost savings above
  • The archival work in Phases 1–4 also serves as the migration dataset for v3
Risk: Low — shadow mode validates before cutover, rollback is one config line

Why Now

Low risk window

  • No code changes to the data collection pipeline or product UI
  • Parquet archive is read-only extraction — data is duplicated before anything is deleted
  • Edge filter is a Telegraf config update, reversible per site in minutes
  • subscriber_api changes are additive — existing WebSocket clients unaffected

Cost of waiting

  • Each month without archival: ~$5,370 in avoidable EFS spend
  • Each month without edge filtering: ~$6,700 in avoidable EC2/EFS/transfer spend
  • EFS growth compounds — delay of 6 months costs ~$50k extra before saving begins
  • InfluxDB3 migration becomes cheaper once v2 has a 90-day retention and parquet archive is in place

Ask

Engineering request: 6–8 weeks of focused engineering time to execute Phases 1–5. No additional infrastructure procurement required. No customer communication required. Estimated payback period: <3 weeks from Phase 4 completion (first reduced EFS bill covers the engineering cost).
  • Phase 1–3: engineering-only, no approvals needed to proceed
  • Phase 4 (delete from InfluxDB): requires sign-off from engineering lead after validation report
  • Phase 5 (edge filter rollout): requires site-by-site co-ordination with field/DevOps team
  • Phase 6 (InfluxDB3): separate decision, not on critical path for cost savings
Assumptions:
