MQTT Topic Formats

Longbow (raw MQTT) vs Evelyn (InfluxDB extract) — how each maps to the parquet schema

Origin of the two formats

Longbow format raw MQTT
Source: packet capture from live EMS hardware

Taken directly off the MQTT broker from a running Longbow system. Every message is exactly as the EMS firmware publishes it — topic structure and JSON payload are unmodified. This is the native wire format.

The parquet writer (real_writer.cpp) was designed around this format and subscribes directly to unit/#.

Evelyn format InfluxDB extract
Source: CSV export from InfluxDB2 database

Taken from the Telegraf → InfluxDB2 pipeline. Telegraf subscribes to the same MQTT broker, applies topic_parsing to split the topic into tags, optionally filters and deduplicates, then writes line-protocol to InfluxDB.

The CSV export reflects the InfluxDB measurement schema — tag names, field names, and timestamps may differ from the raw wire format.

Key point: both formats originate from the same MQTT broker and the same topic structure. The difference is entirely in how Telegraf processes the data before it reaches InfluxDB. The parquet writer bypasses Telegraf and reads raw MQTT directly, so its schema is anchored to the Longbow wire format.

Evelyn — Telegraf processing and InfluxDB schema

Pipeline

MQTT broker :1883 → Telegraf mqtt_consumer → topic_parsing → processors.dedup (opt) → InfluxDB2 → CSV export

Telegraf topic_parsing

Telegraf splits the raw MQTT topic into tags using topic_parsing:

[[inputs.mqtt_consumer]]
  name_override = "ems"
  topics        = ["unit/#"]
  data_format   = "json"
  json_time_key = "ts"
  json_time_format = "2006-01-02T15:04:05.999Z07:00"

  [[inputs.mqtt_consumer.topic_parsing]]
    topic = "unit/+/+/+/+/+"
    tags  = "_/unit_id/device/instance/point_name/dtype"

The _ placeholder discards the unit literal prefix. Each remaining segment becomes an InfluxDB tag. The JSON value field becomes the field.

InfluxDB2 measurement schema (Evelyn CSV columns)

InfluxDB columnKindSourceNotes
_time timestamp payload ts RFC3339, nanosecond precision in InfluxDB
_measurement measurement name_override ems (or ems_float / ems_int in filtered path)
unit_id tag topic[1] Hex serial, e.g. 02281660
device tag topic[2] bms, pcs, rack
instance tag topic[3] bms_1, pcs_2, rack_0
point_name tag topic[4] Signal name, e.g. Batt1_SBMU1_Current
dtype tag topic[5] float, integer, boolean_integer
value field payload Always FLOAT64 in InfluxDB regardless of dtype — no int/float split
In the InfluxDB schema dtype is preserved as a tag (indexed string). In the raw parquet schema it is consumed and discarded — the type information is encoded structurally as value_i vs value_f column presence.
Baseline config only. The Telegraf configuration collected from the live system subscribes to unit/# with no filtering processors. Filter A (string drop) and Filter A+B (+ int/bool dedup) exist only in the Evelyn cost-comparison demo stack and are not present in standard collected configs.

topic_parsing variants and their effect

The topic_parsing stanza is the single most impactful config choice — it determines which topic segments become queryable InfluxDB tags and which are lost. Several variants are in use across different environments.

Variant 1 — Full 6-segment (standard collected config)

All six topic segments extracted, including dtype as a tag. This is the baseline config found in the live system.

[[inputs.mqtt_consumer.topic_parsing]]
  topic = "unit/+/+/+/+/+"
  tags  = "_/unit_id/device/instance/point_name/dtype"
Tag producedExample valueNote
unit_id02281660hex serial
devicebms
instancebms_1
point_nameBatt1_SBMU1_Current
dtypefloatretained as queryable tag
Best fidelity. All structural metadata is preserved. dtype can be used to filter or group queries by signal type.

Variant 2 — 5-segment (no dtype tag)

Omits the sixth segment — dtype is silently dropped. All values land in a single value field as FLOAT64 regardless of type.

[[inputs.mqtt_consumer.topic_parsing]]
  topic = "unit/+/+/+/+"
  tags  = "_/unit_id/device/instance/point_name"
Tag producedNote
unit_id, device, instance, point_nameSame as Variant 1
dtypeNot captured — lost
Type information lost. Integer and boolean signals cannot be distinguished from float signals in InfluxDB queries. The parquet writer, which parses dtype from the topic directly, is unaffected.

Variant 3 — Split measurement by dtype

Uses two separate mqtt_consumer inputs with different name_override values to route float and integer signals into separate InfluxDB measurements. This avoids a mixed-type field but requires two Telegraf input blocks.

[[inputs.mqtt_consumer]]
  name_override = "ems_float"
  topics        = ["unit/+/+/+/+/float"]
  [[inputs.mqtt_consumer.topic_parsing]]
    topic = "unit/+/+/+/+/float"
    tags  = "_/unit_id/device/instance/point_name/_"

[[inputs.mqtt_consumer]]
  name_override = "ems_int"
  topics        = ["unit/+/+/+/+/integer", "unit/+/+/+/+/boolean_integer"]
  [[inputs.mqtt_consumer.topic_parsing]]
    topic = "unit/+/+/+/+/+"
    tags  = "_/unit_id/device/instance/point_name/_"
MeasurementSignalsEffect
ems_floatdtype=floatFloats isolated — cleaner aggregation
ems_intdtype=integer, boolean_integerIntegers isolated — exact values preserved
Used in the cost-comparison demo stack (Evelyn filtered path) to enable per-type dedup processors. This separation mirrors the value_f / value_i split in the parquet schema.

Variant 4 — Prefixed topic (multi-sim test environments)

On gx10 with 4 simultaneous simulators, each slice uses a namespace prefix (A/unit/#, B/unit/#, …) to avoid collision. Telegraf must account for the extra leading segment.

[[inputs.mqtt_consumer]]
  topics = ["A/unit/#"]
  [[inputs.mqtt_consumer.topic_parsing]]
    topic = "A/unit/+/+/+/+/+"
    tags  = "_/_/unit_id/device/instance/point_name/dtype"

Both leading segments are discarded with _. The resulting tag set is identical to Variant 1 — the prefix is never stored.

Test environments only. Production topics do not carry a prefix. The parquet writer handles this via a depth_offset config option.

Multi-simulator prefix (test environments only)

On gx10 with 4 simultaneous simulators, each slice uses a topic prefix (A/unit/#, B/unit/#, etc.) to avoid namespace collision. The writer config uses a depth offset to skip the prefix — it is never stored in parquet and does not appear in production data.

# config.gx10-evelyn-a.yaml
mqtt:
  topic: "A/unit/#"

topic_segments:
  depth_offset: 1    # skip the "A" prefix, treat "unit" as segment 0
  - unit_id
  - device
  - instance
  - point_name
  - dtype_hint
