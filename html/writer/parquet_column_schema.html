Parquet Writer — Column Schema Reference

How MQTT topic segments and payload fields become parquet columns — and how config options reshape the schema.

Overview

Every incoming MQTT message produces one parquet row. Columns come from three sources:

MQTT topic
→ string/int columns
(segment mapping)
+
JSON payload
→ ts, value/value_int
+
Config injection
→ site_id, project_id,
mqtt_topic, sample_count

The writer supports three storage layouts:

  • Long — one row per message; point_name is a string value column; ID columns stored separately. Standard time-series format.
  • Long + compound point_name — same one-row-per-message, but ID columns collapsed into a single path string in point_name.
  • Wide — each unique signal path becomes its own parquet column; one sparse row per message. Pivot/spreadsheet style. Configured via compound_field_name.

Additional options: drop_columns removes unwanted columns; wide_point_name suppresses site_id and project_id after the partition key is captured.

Fixed Columns

ts and value are always present. The four metadata columns are present by default but can be suppressed individually via output: flags.

Topic-Derived Columns (Long Schema)

The writer supports two topic parsers selected by mqtt.topic_parser. Both produce the long layout by default: one row per message, signal name as a string value in point_name.

Option A — Positional parser default for Evelyn/Longbow

Topic segments are mapped to column names by position using topic_segments.

Topic: A/unit/0215D1D8/bms/bms_1/Batt1_SBMU1_Current/float

mqtt:
  topic_parser: positional
  topic_segments:
    - "_"          # 0: slice prefix — skip
    - "_"          # 1: literal "unit" — skip
    - unit_id      # 2: → row.strings["unit_id"]   = "0215D1D8"
    - device       # 3: → row.strings["device"]     = "bms"
    - instance     # 4: → row.strings["instance"]   = "bms_1"
    - point_name   # 5: → row.strings["point_name"] = "Batt1_SBMU1_Current"
    - dtype_hint   # 6: consumed (selects float/int column type) — not stored

Reserved segment names: dtype_hint (consumed, not stored), field_name (renames the value column to the segment value — wide-sparse schema), _ (skip this position).

ColumnTypeValue (example)
unit_idSTRING0215D1D8
deviceSTRINGbms
instanceSTRINGbms_1
point_nameSTRINGBatt1_SBMU1_Current

Option B — KV parser default for stress_runner / BMS

Topic segments are key=value pairs. topic_kv_map maps key names to column names (all stored as INT64). Also produces the long layout.

Topic: batteries/site=2/rack=1/module=3/cell=4/voltage

mqtt:
  topic_parser: kv          # (or omit — kv is the default)
  topic_kv_map:
    site:   site_id         # → row.ints["site_id"]   = 2
    rack:   rack_id         # → row.ints["rack_id"]   = 1
    module: module_id       # → row.ints["module_id"] = 3
    cell:   cell_id         # → row.ints["cell_id"]   = 4
    # last bare segment ("voltage") → row.strings["point_name"] = "voltage"
ColumnTypeValue (example)
site_idINT642
rack_idINT641

compound_point_name optional

Collapses several segment columns into a single point_name string using / as separator. The named columns are removed from the row as they are consumed — they do not appear as separate columns in parquet.

Works with both parsers. Columns are looked up in row.strings first, then row.ints, so it handles both string and int segment types.

Evelyn / Longbow example

mqtt:
  topic_parser: positional
  topic_segments:
    - "_"          # slice prefix — skip
    - "_"          # "unit" — skip
    - unit_id      # "0215D1D8"
    - device       # "bms"
    - instance     # "bms_1"
    - point_name   # "Batt1_SBMU1_Current"
    - dtype_hint   # consumed

  compound_point_name: [unit_id, device, instance, point_name]

Result:

ColumnTypeValueNote
point_nameSTRING0215D1D8/bms/bms_1/Batt1_SBMU1_CurrentAll four segments joined
unit_id——Removed (consumed by compound)
device——Removed
instance——Removed

BMS / KV example

mqtt:
  topic_kv_map:
    site:   site_id
    rack:   rack_id
    module: module_id
    cell:   cell_id

  compound_point_name: [site_id, rack_id, module_id, cell_id, point_name]

Result: point_name = "2/1/3/4/voltage" — all individual ID columns removed.

Do not include site_id in compound_point_name if it is your partition_field — it must remain in the row for the partition key lookup. Use wide_point_name: true instead to suppress it from parquet output after the key is captured.

drop_columns optional

Removes named columns from every row after all other processing (including compound_point_name). Checks row.strings, row.ints, and row.floats.

Use when segment columns are redundant — for example, already encoded in the compound point name, or already captured in the partition directory path.

mqtt:
  topic_parser: positional
  topic_segments:
    - "_"
    - "_"
    - unit_id
    - device
    - instance
    - point_name
    - dtype_hint

  # Keep point_name as a standalone column; drop the others
  drop_columns: [unit_id, device, instance]

drop_columns is applied after compound_point_name, so listing a column in both is harmless — compound_point_name already removes it.

Do not drop the column named in partition_field — the writer uses it to route rows to the correct partition buffer.

Wide Schema — compound_field_name pivot layout

In the wide layout every unique signal path becomes its own parquet column rather than a string value in point_name. Each message still produces one row, but that row has only one non-null value column — the column named after its signal path. Arrow/Parquet stores the nulls efficiently so the file is not bloated.

compound_field_name lists the segment columns to join (same join logic as compound_point_name), but the result becomes the column name for the value, not a string cell. Both float and integer signals are stored as FLOAT64 for schema uniformity across the pivot table — dtype_hint is still consumed (to avoid being written as a column) but type promotion is suppressed.

compound_field_name and compound_point_name are mutually exclusive — do not set both.

Config — Evelyn wide schema

mqtt:
  topic_parser: positional
  topic_segments:
    - "_"          # "A" slice prefix — skip
    - "_"          # "unit" literal — skip
    - unit_id      # "0215D1D8"
    - device       # "bms"
    - instance     # "bms_1"
    - point_name   # "Pack_Voltage"
    - dtype_hint   # consumed — not stored

  # Wide: join these into the value column name.
  # site_id is NOT included — it comes from config, not the topic.
  compound_field_name: [unit_id, device, instance, point_name]

output:
  site_id: "0215D1D8"
  wide_point_name: true      # suppresses site_id / project_id columns
  store_mqtt_topic: false
  store_sample_count: false

Resulting row for topic A/unit/0215D1D8/bms/bms_1/Pack_Voltage/float

After a full sweep — DuckDB pivot view

Because each row carries one signal, a full sweep of 34 signals × 12 units produces 408 sparse rows. DuckDB can pivot them into a dense view at query time:

-- All signals for one unit at a given second (wide pivot)
SELECT ts,
       "0215D1D8/bms/bms_1/Pack_Voltage"          AS voltage,
       "0215D1D8/bms/bms_1/Pack_Current"           AS current,
       "0215D1D8/bms/bms_1/Pack_SOC"               AS soc
FROM read_parquet('/data/parquet-evelyn/a/**/*.parquet')
WHERE ts BETWEEN 1743494400 AND 1743494460
  AND ("0215D1D8/bms/bms_1/Pack_Voltage" IS NOT NULL
    OR "0215D1D8/bms/bms_1/Pack_Current" IS NOT NULL)
ORDER BY ts;
Column names containing / must be double-quoted in DuckDB SQL. If you prefer unquoted names, use _ as the separator by post-processing, or restrict compound_field_name to just [point_name] and partition by unit.

Optional Metadata Columns optional

Three metadata columns are written by default but can be individually suppressed under output::

output:
  store_mqtt_topic: false    # omit mqtt_topic string column
  store_project_id: false    # omit project_id int column
  store_sample_count: false  # omit sample_count int column

All three default to true so existing configs are unaffected.

wide_point_name convenience flag

When using compound_point_name, site_id and project_id are already encoded in the point path and are redundant as standalone columns. wide_point_name: true removes both in one step:

output:
  wide_point_name: true   # drops site_id and project_id from parquet output
The partition key is captured before wide_point_name erases site_id, so directory partitioning by site still works correctly. You do not need to change partition_field.

Recommended config for full wide schema (Evelyn)

mqtt:
  topic_parser: positional
  topic_segments:
    - "_"          # slice prefix
    - "_"          # "unit"
    - unit_id
    - device
    - instance
    - point_name
    - dtype_hint
  compound_point_name: [unit_id, device, instance, point_name]

output:
  site_id: "0215D1D8"
  partition_field: site_id   # still works — key captured before wide_point_name drops it
  wide_point_name: true
  store_mqtt_topic: false
  store_sample_count: false

Resulting columns: ts · value or value_int · point_name (compound path)

DuckDB Query Examples

Long schema (individual columns)

-- All voltage readings for unit 0215D1D8, rack 1
SELECT ts, value
FROM read_parquet('/data/parquet-evelyn/a/**/*.parquet')
WHERE unit_id = '0215D1D8'
  AND point_name = 'Batt1_SBMU1_Current'
ORDER BY ts;

Long+compound schema (point_name path)

-- Same query with compound_point_name: [unit_id, device, instance, point_name]
SELECT ts, value
FROM read_parquet('/data/parquet-evelyn/a/**/*.parquet')
WHERE point_name = '0215D1D8/bms/bms_1/Batt1_SBMU1_Current'
ORDER BY ts;

-- Wildcard: all signals for one unit (use LIKE)
WHERE point_name LIKE '0215D1D8/%'

-- Wildcard: a specific signal across all units
WHERE point_name LIKE '%/Batt1_SBMU1_Current'
Parquet dictionary encoding compresses repeated point_name strings efficiently — the compound path costs no more storage than separate columns.

Analysis

Long vs Long+compound (0.9% size difference): Parquet dictionary encoding eliminates repeated strings for low-cardinality columns (unit_id, device, mqtt_topic) almost completely — each dictionary is tiny and rows reference it by a small index. The actual measurement bytes (ts, value) dominate and are identical in both schemas. The real gain from compound is flush speed: 4× faster (542 ms vs 2,412 ms per flush) because Arrow builds and encodes only 5 array builders instead of 12.

Wide (pivot) — twice the storage, 4× slower flush: With 408 signal paths across 12 units, every flush must build 411 Arrow array builders, most of them nearly all-null. Null encoding overhead drives storage to 26 MB vs 13 MB (long), and flush times to 19 s for 1.6M rows. The MQTT reconnection storm visible in the logs is a symptom of the writer falling behind — the buffer fills during slow flushes, causing backpressure. Wide layout works well when signal cardinality per partition is small (<50 columns) or data is pre-aggregated.

Recommendation: Long+compound for high-rate EMS ingest. 5 columns, 4× faster flush than Long, storage equivalent, DuckDB queries straightforward with LIKE patterns on point_name.
Wide (pivot) is practical only at low column counts. Consider partitioning by unit_id and using only the signal name (not full path) as the column, reducing from 408 columns to 34 per partition file.
