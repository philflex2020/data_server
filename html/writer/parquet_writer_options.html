Summary — Parquet/S3 vs InfluxDB2 for Long-Term Site Storage
  • Wide Pivot Parquet on S3 is 12–25× cheaper than InfluxDB2 on EBS for long-term storage at an Evelyn site (12 units, 34 signals, 1 Hz).
  • At 10 years, InfluxDB2 costs ~$7,250/year in storage alone vs ~$285/year for tuned Wide Parquet on S3.
  • InfluxDB2 also requires an always-on EC2 instance (~$730–1,500/year) — Parquet writer can run on-premises at zero AWS compute cost.
  • Wide Pivot chosen over Norm-Long for this comparison — dashboard-friendly column format, ~25% larger on S3 but native Grafana/BI compatibility.
  • Hybrid option: add 1-week InfluxDB retention alongside Parquet/S3 — keeps all existing Grafana dashboards, drops storage cost by 98%, 10-year total ~$4,000–6,000 (87–91% cheaper than InfluxDB2 long-term).

Hybrid Architecture — InfluxDB as Short-Term Real-Time Tier

Instead of choosing between InfluxDB (expensive long-term) and Parquet/S3 (no real-time dashboard), use both: InfluxDB handles the last 7 days for live dashboards; Parquet/S3 handles the archive. The two write paths run in parallel from the same MQTT stream.

Key insight: InfluxDB's cost problem is almost entirely caused by years of accumulating data on EBS. Cut retention to 1 week and storage drops from $7,248/yr to under $15/yr flat — it never grows. EC2 compute is what remains, or eliminate that too by running InfluxDB on the edge server alongside the parquet writer.

subscriber_api placement: subscriber_api is a lightweight Python/DuckDB service — it can run on the same EC2 instance as InfluxDB (no extra cost), on-prem on the edge server ($0 AWS), or via AWS Athena (serverless, ~$5/TB scanned, no always-on server needed). DuckDB queries S3 parquet files directly, so there is no separate database to provision or maintain.
