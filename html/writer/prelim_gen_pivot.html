Wide-device per-row COV = 0% (not an estimate — confirmed by generator). Per-row COV on a wide-device table drops a row only when ALL data columns are unchanged vs the previous row for that unit. Every rack device contains RackCurrent (and similar fast signals) that update on ~95% of scan cycles. Therefore no rack row is ever fully static, and COV never fires. This is a structural property of mixed fast/slow device tables — not a simulator artefact.

Narrow per-signal COV = 82% reduction (measured): 204,720 → 36,741 rows. Each slow signal stays flat for ~12 cycles on average; COV fires on ~92% of its readings.

Fast/slow split solves the wide-device COV problem. Use --split to separate fast signals (always written, no COV) from slow signals (COV fires frequently on the slow-only table). The slow-only rack file has ~41 cols; per-signal COV on those columns achieves savings comparable to narrow per-signal COV on the same signal subset.

If COV-filtered narrow format is used: 36,741 rows × 6.9 B/row ≈ ~254 KB — a dramatic reduction from the uncovered 1,414 KB. Wide-device + split + slow-file COV would be roughly comparable on the slow signal portion.

Key finding vs the 17-cycle real file: The 17-cycle file showed 24×–28× penalty for wide formats. At 60 cycles the penalty drops to 1.08×–1.11× — a dramatic improvement. Wide-device with zstd is effectively at parity today. However, plain wide-device per-row COV delivers 0% savings due to always-changing fast signals. The correct approach for wide-device COV savings is the fast/slow split (--split): fast signals written every cycle without COV; slow signals written to a separate file where COV fires frequently. For files covering ≥ 10-minute windows, wide-device + split + slow-file COV + zstd is the optimal storage layout.
