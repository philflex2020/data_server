prelim_generator.py — Reference & Design Notes

Script: scripts/prelim_generator.py  ·  Schema: 9 devices, 451 points  ·  Formats: narrow, wide-device, wide-point-name

Purpose

Synthetic BESS parquet generator using the real prelim schema (9 devices, 451 points). Generates narrow, wide-device, and wide-point-name formats with optional time jitter, timestamp bucketing, and COV filtering. Used to test storage format tradeoffs at controlled data volumes — unlike the 2m50s real snapshot, the generator can produce hours of data with hundreds of scan cycles to reach the wide-format break-even point.

CLI Reference

Option Default Description

--cov off Apply COV filtering (per-signal on narrow; per-row on wide)
--split off Generate separate fast/slow wide-device files per device

Signal Speed Classes

Each point is assigned a change-probability per scan cycle based on keywords in the point name.

Class Change prob/cycle Keywords matched Examples

slow 8% soc, soh, temp, _t, cellv, voltage, avgcell RackSOC, MaxCellT, RackVoltage

COV Behaviour

Key finding from 10-min run (60 cycles, 8 units):

Narrow per-signal COV: 204,720 → 36,741 rows — 82% reduction. Each slow signal (temp, SOC, voltage) stays flat for ~12 cycles on average, firing COV on ~92% of its readings.

Wide-device per-row COV: 0% reduction confirmed by generator. Wide-device per-row COV drops a row only when ALL columns are unchanged. Since every device has at least one fast signal (RackCurrent, PCS_P, Hz) that changes on ~95% of cycles, no row is ever fully static — COV never fires. This is a structural property of mixed fast/slow device tables, not a simulation artifact.

Conclusion: Per-signal COV belongs in narrow format. For wide format, the device must be split into fast-only and slow-only sub-tables before per-row COV can deliver meaningful savings.

Why the Split Works

The --split flag separates each device's signals into a fast file (no COV needed — always changing) and a slow file (COV fires frequently). For rack devices with ~49 points, ~8 are fast-class and ~41 are slow/medium/bool. The probability that all 41 slow-file columns stay flat in a single cycle is approximately 0.9241 ≈ 3%, so roughly 3% of slow-file rows get dropped by per-row COV. Combined with the fact that per-signal slow signals fire COV on ~92% of their readings, the slow file achieves savings comparable to narrow per-signal COV on the slow signal subset.

File Signals Cols COV type Savings
rack_1_fast.parquet RackCurrent, PCS_P, Hz… ~8 none — always changing, every row meaningful
rack_1_slow.parquet RackSOC, temps, voltages… ~41 per-row + per-signal significant — slow signals stable for ~12 cycles

Time Jitter + Bucketing

Without jitter, all 49 points in a rack scan arrive at exactly the same millisecond — unrealistic. Real hardware scanners emit points over a 5–50 ms window per device.

--time-jitter 15 adds ±15 ms uniform scatter per point. This breaks pivoting — different timestamps produce a sparse wide table or multiple rows per cycle.

--bucket 20 snaps all timestamps to a 20 ms grid (floor(ts_ms / 20) * 20) before pivoting, collapsing the jitter back to one row per scan cycle. This is equivalent to DuckDB's time_bucket('20ms', timestamp) at query time.

Without bucketing, jittered data produces ~49 rows per device per cycle instead of 1 — the wide tables explode. With bucketing, dense wide rows are restored and COV can work correctly.

DuckDB Query Patterns — Fast/Slow Split Files

DuckDB handles split parquet files natively. The COV-filtered slow file has gaps (missing rows = unchanged value) — the ASOF JOIN pattern is purpose-built for this.

Key insight: Don't fill COV gaps in the slow file. Use ASOF JOIN to carry the last known value forward on demand at query time. This keeps the stored file small while giving queries a complete view.

1 — Query each file independently (common case)

-- Real-time dashboard: fast file only, last 5 minutes
SELECT timestamp, unit, RackCurrent, PCS_P, Hz
FROM 'wide_device_bms_1_rack_1_fast.parquet'
WHERE unit = '01C4F65E'
  AND timestamp > now() - INTERVAL '5 minutes';

-- SOC / temperature trend: slow file only
SELECT timestamp, unit, RackSOC, MaxCellT, RackVoltage
FROM 'wide_device_bms_1_rack_1_slow.parquet'
WHERE timestamp > '2026-01-15 00:00:00';

2 — ASOF JOIN: align slow (COV-gapped) to fast timestamps

-- Carries last known slow value forward to every fast-file row.
-- No need to fill gaps in the slow file — DuckDB does it on the fly.
SELECT f.timestamp, f.unit,
       f.RackCurrent, f.PCS_P,
       s.RackSOC, s.MaxCellT, s.RackVoltage
FROM 'wide_device_bms_1_rack_1_fast.parquet'  f
ASOF JOIN 'wide_device_bms_1_rack_1_slow.parquet' s
  ON f.unit = s.unit
 AND f.timestamp >= s.timestamp;

3 — Glob across all racks

-- All 6 rack slow files in one query — DuckDB globs natively
-- and pushes the predicate into each file's column stats.
SELECT rack_file, unit, timestamp, RackSOC, MaxCellT
FROM read_parquet('wide_device_bms_1_rack_*_slow.parquet',
                  filename = true) AS t(rack_file)
WHERE RackSOC < 20;   -- low-SOC alert across all racks

4 — UNPIVOT back to narrow for mixed-signal analytics

-- Bring wide back to narrow when you need all signals in one result set.
SELECT timestamp, unit, signal, value FROM (
  SELECT * FROM 'wide_device_bms_1_rack_1_fast.parquet'
) UNPIVOT (value FOR signal IN (RackCurrent, PCS_P, Hz))
UNION ALL
SELECT timestamp, unit, signal, value FROM (
  SELECT * FROM 'wide_device_bms_1_rack_1_slow.parquet'
) UNPIVOT (value FOR signal IN (RackSOC, MaxCellT, RackVoltage))
ORDER BY timestamp, unit, signal;

5 — time_bucket() for query-time timestamp alignment (no --bucket needed)

-- If files were written with jitter and no --bucket, align at query time.
-- Equivalent to --bucket 20 applied at write time.
SELECT time_bucket(INTERVAL '20ms', timestamp) AS ts_bucket,
       unit, avg(RackCurrent) AS avg_current
FROM 'wide_device_bms_1_rack_1_fast.parquet'
GROUP BY ts_bucket, unit
ORDER BY ts_bucket, unit;

6 — Full cross-device join: rack + BMS + PCS in one query

-- Join slow files across device types on (timestamp, unit).
-- Works cleanly because bucketed timestamps are identical.
SELECT r.timestamp, r.unit,
       r.RackSOC, r.MaxCellT,
       b.MaxCellT  AS bms_MaxCellT,
       p.Hz
FROM 'wide_device_bms_1_rack_1_slow.parquet'  r
JOIN 'wide_device_bms_1_slow.parquet'         b USING (timestamp, unit)
JOIN 'wide_device_pcs_1_fast.parquet'         p USING (timestamp, unit)
WHERE r.unit = '01C4F65E'
ORDER BY r.timestamp;
