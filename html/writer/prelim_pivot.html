Prelim Parquet — Wide vs Narrow Layout Experiment

Source: prelim_0000011876.parquet  ·  Site: 0215D1EC  ·  3,057 rows · 7 cols · 8 units · 2m 50s window  ·  2025-11-19

Result: narrow wins decisively. The original IOx narrow format (3.9 B/row) is already highly efficient. Wide-unit pivot balloons to 28× larger because parquet column metadata (~20 KB/col overhead) dominates when there are only 17 rows across 453 columns. Wide-device (one table per device type) is still 24× larger in aggregate. COV filtering had zero effect — only 17 scan events in the 170 s window, all unique.

Wide format only wins when scan-cycle count >> column count. With 437 points and only 17 snapshots, the crossover is far out of reach for this file.

Wide-Unit Layout — All Codecs

Pivot: one row per (unit, timestamp). 453 columns = unit + time + 451 device/point pairs. 17 rows total. Column metadata overhead dominates — compression has almost no effect.

Why wide-unit fails here: Parquet stores schema metadata for every column name as a UTF-8 string in the file footer. With 453 columns averaging ~25 chars each ≈ ~11 KB of column name strings alone, on top of row group and page headers. At only 17 data rows, that metadata accounts for the vast majority of the file. The actual float64 data is tiny — 17 rows × 451 cols × 8 bytes = ~61 KB uncompressed, which compresses well, but the footer overhead is inescapable.

Break-even estimate: wide-unit starts winning around ~30,000+ scan cycles (≈ 1 row/30 s × 10 days), where column data finally exceeds the fixed metadata cost per file.

COV needs longer windows or higher rates to show savings. In a 2m 50s capture with ~1 scan/20s, there are only 1–2 cycles per unit — and BESS values change on every cycle. COV filtering shines on 10+ minute files where slow-moving signals (temperatures, SOC) stay flat for many cycles. At 21,240 msg/s (Evelyn full load), wide-unit + COV would be a very different story.

Recommendations

Use caseRecommended layoutReasoning
Storage / archival Narrow + zstd/brotli Already ~3.5 B/row; flexible schema, no schema migration pain
COV filtering (long window) Wide-unit + COV Requires 10+ min files with 100s of scan cycles to amortize column overhead
Analytics (DuckDB queries) Narrow DuckDB handles long format natively; no pivot needed for queries
Rack-type schema reuse Wide-device per rack All 6 racks share identical 49-point schema — viable if files are large enough
Transmission / edge Narrow + zstd 11 KB/file already tiny; wide adds no benefit for short burst files
