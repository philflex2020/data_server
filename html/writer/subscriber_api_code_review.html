SQL injection in DuckDB queries — site, instance, fields unsanitized
duckdb_backend.py:109–115  |  subscriber.py:143–148
▶

Security / Correctness

site, instance, and fields values from the HTTP/WebSocket request are interpolated directly into SQL without any escaping or parameterization.

# duckdb_backend.py — current
field_sql = "*" if not fields else ", ".join(f'"{f}"' for f in fields)

result = conn.execute(f"""
    SELECT {field_sql}
    FROM read_parquet([{sources}], union_by_name=true)
    WHERE timestamp >= {from_ts} AND timestamp <= {to_ts}
    ORDER BY timestamp
    LIMIT {limit}
""")

An HTTP caller sending ?fields=*+FROM+sqlite_master-- or a field name containing " can break out of the double-quote identifier quoting. A site value of '; DROP TABLE -- gets directly embedded in the glob path string fed to DuckDB.

DuckDB also supports read_parquet with http:// paths — a crafted local_path or glob could cause the process to make outbound HTTP requests to attacker-controlled servers (SSRF via DuckDB httpfs).

Suggested Fix

Validate site, instance, and fields against an allowlist of safe characters before use. For this internal system, simple regex validation is sufficient:

import re
_SAFE_ID = re.compile(r'^[\w\-./]+$')   # alphanumeric, dash, dot, slash

def _validate_id(value: str, name: str) -> str:
    if not _SAFE_ID.match(value):
        raise ValueError(f"Invalid {name}: {value!r}")
    return value

# In query():
site     = _validate_id(site, "site")
instance = _validate_id(instance, "instance")
if fields:
    fields = [_validate_id(f, "field") for f in fields]

Also validate at the HTTP layer in http_server.py:_query() before passing to the router, so error responses are 400 rather than 500.

Flux injection — site, instance, field names, extra_filters unsanitized
backends/influxdb2.py:64–88
▶

Security

All user-supplied values are interpolated directly into the Flux query string sent to InfluxDB2. A site parameter of x" or 1==1 // would close the string literal and inject arbitrary Flux logic.

# influxdb2.py — current
lines = [
    f'from(bucket: "{self._bucket}")',
    ...
    f'  |> filter(fn: (r) => r["{self._site_tag}"] == "{self._site_tag_prefix}{site}")',
]

for tag, value in self._extra_filters.items():
    lines.append(f'  |> filter(fn: (r) => r["{tag}"] == "{value}")')

extra_filters values come from config (lower risk), but site and instance come from the HTTP query string and WebSocket messages. Field names in fields are also injected unescaped.

InfluxDB2's Flux has no parameterized query API, but the same allowlist validation from C-1 applies. Config-sourced extra_filters values should also be validated at startup.

Suggested Fix
# Escape the minimal set of Flux string-breaking characters
def _flux_escape(value: str) -> str:
    return value.replace('\\', '\\\\').replace('"', '\\"')

# In _flux():
f'  |> filter(fn: (r) => r["{self._site_tag}"] == "{_flux_escape(site)}")'

Combine with the allowlist validation from C-1 — defence in depth.

_parse_csv silently returns empty on InfluxDB2 HTTP errors (4xx/5xx)
backends/influxdb2.py:110–160
▶

Correctness

InfluxDB2 returns a JSON error body (not CSV) for HTTP 400/500 responses — e.g., malformed Flux, auth errors, or bucket-not-found. urlopen raises HTTPError for 4xx/5xx, which is caught and re-raised as RuntimeError in query() — that part is fine. However, if InfluxDB2 returns HTTP 200 with an error payload embedded in CSV annotations (it sometimes does), _parse_csv will skip the annotation rows and return ([], []) as if the query matched zero rows.

Concretely: a Flux syntax error can return a 200 response with a CSV annotation row #error,... describing the problem. This is silently treated as "no data found."

# Example InfluxDB2 error response (HTTP 200):
#datatype,string,string
#group,false,false
#default,,
,error,reference
,compilation failed: error at 2:5...,"
Suggested Fix
@staticmethod
def _parse_csv(raw: str) -> tuple[list, list]:
    # Check for InfluxDB2 error annotations
    for line in raw.splitlines()[:10]:
        if line.startswith("#") and ",error," in line:
            # Next non-annotation, non-blank line has the message
            raise RuntimeError(f"InfluxDB2 query error: {raw[:300]}")
    # ... rest of existing parser
High
set_live_status() called from MQTT thread — race condition on _live dict
http_server.py:30–31  |  subscriber.py:274, 288
▶

Concurrency / Reliability

set_live_status() is called from paho-mqtt's on_connect and on_disconnect callbacks, which run in a separate daemon thread (mqtt-live). Meanwhile, _live_status() reads _live from asyncio coroutines on the event loop thread.

# subscriber.py — paho callbacks (mqtt thread)
def on_connect(client, userdata, flags, rc):
    _http.set_live_status(True, url, topics)   # writes _live

def on_disconnect(client, userdata, rc):
    _http.set_live_status(False, url, topics)  # writes _live

# http_server.py — asyncio (event loop thread)
async def _live_status(writer) -> None:
    await _send(writer, 200, dict(_live))       # reads _live

dict.update() is atomic for the GIL in CPython, but dict(_live) (shallow copy) is not atomic with respect to a concurrent update() — the dict could be in a partially-updated state during the copy. This is a low-probability race in practice (CPython GIL) but is undefined behavior per the Python memory model.

Suggested Fix
# http_server.py
import threading
_live_lock = threading.Lock()
_live: dict = {"connected": False, "broker": "", "topics": []}

def set_live_status(connected: bool, broker: str, topics: list) -> None:
    with _live_lock:
        _live.update({"connected": connected, "broker": broker, "topics": list(topics)})

async def _live_status(writer) -> None:
    with _live_lock:
        snapshot = dict(_live)
    await _send(writer, 200, snapshot)
High
_parse_csv multi-table column alignment is broken for heterogeneous tables
backends/influxdb2.py:120–144
▶

Correctness

When InfluxDB2 returns multi-table annotated CSV (one table per series), _parse_csv locks in all_cols from the first table header encountered. For subsequent tables with different column sets, keep_idx is updated to match the new header — but those indices are applied to extract values into a row that will be associated with the original all_cols column names.

If table 1 has columns [_time, value, soc] and table 2 has [_time, power, temp], rows from table 2 will be extracted at the indices of power and temp but stored as if they were value and soc. The data silently mixes up columns.

This is currently masked by using pivot() in the Flux query, which collapses all fields into a single table. But if pivot is ever removed or a query returns multiple measurement tables, data will be silently corrupted.

Suggested Fix

When a new table header is encountered with different columns, either skip rows from that table, or (better) project all tables to the union of all columns, filling missing values with None. At minimum, assert that secondary headers match the first:

if all_cols is None:
    all_cols = new_cols
    keep_idx = new_idx
elif new_cols != all_cols:
    # Skip rows from tables with different schema
    log.warning("InfluxDB2 CSV: table schema mismatch, skipping table")
    keep_idx = []   # clear — rows from this table will fail the max(keep_idx) guard

Medium
No per-request timeout for the full HTTP exchange — only per-readline timeout
http_server.py:125–136
▶

Reliability / DoS

Each readline() call has a 5-second timeout, but there is no cap on the total number of header lines a client can send. A slow client sending one header every 4.9 seconds could hold the handler open indefinitely, consuming a coroutine and an OS file descriptor per connection.

Additionally, the query execution itself (running in an executor) has no timeout. If InfluxDB2 or DuckDB hangs, the coroutine awaiting the executor future will wait forever.

Suggested Fix
# Wrap entire dispatch in a single wall-clock timeout
async def _dispatch(reader, writer, router, executor) -> None:
    try:
        await asyncio.wait_for(
            _dispatch_inner(reader, writer, router, executor),
            timeout=30.0   # max 30s per request including query execution
        )
    except asyncio.TimeoutError:
        try:
            await _send(writer, 504, {"error": "request timeout"})
        except Exception:
            pass

Also set timeout on InfluxDB2 queries (already configurable) and consider a DuckDB query timeout via SET lock_timeout=....

Medium
New DuckDB connection per query — no connection reuse
backends/duckdb_backend.py:45–60, 102
▶

Performance

_make_conn() creates a new DuckDB in-memory connection for every query() call, and also for every health() call. For S3 paths this also runs INSTALL httpfs; LOAD httpfs; every time.

DuckDB connection creation is fast for local files (sub-millisecond), but each connection has its own buffer pool and metadata cache — warming up the parquet metadata cache is wasted on every query. Under concurrent load (50+ concurrent HTTP /query requests), this creates many parallel DuckDB instances, each competing for memory.

Suggested Fix
class DuckDBBackend:
    def __init__(self, cfg: dict):
        self._cfg = cfg
        self._conn = self._make_conn()   # single shared connection

    def query(self, ...):
        # Use self._conn directly — DuckDB connections are thread-safe for read queries

DuckDB in-memory connections are not safe for concurrent writes, but read-only queries via execute() are safe. If write safety is a concern, use duckdb.connect(read_only=True).

Medium
"No files found" error detection relies on DuckDB error message string
backends/duckdb_backend.py:119, 137
▶

Fragility

The empty-result handling for missing parquet files is keyed on the substring "No files found" in the exception message:

if "No files found" in str(exc):
    conn.close()
    return [], []

This is fragile: DuckDB could change the wording in a future release (e.g., to "No matching files"), and the fallback would then re-raise as a RuntimeError instead of returning empty. There is no test that pins the exact error wording, so a DuckDB upgrade could silently break this.

Suggested Fix
# Use a broader heuristic — no parquet = no data, not an application error
_EMPTY_ERRORS = ("No files found", "No matching files", "Path does not exist")

def _is_empty_result(exc: Exception) -> bool:
    msg = str(exc)
    return any(s in msg for s in _EMPTY_ERRORS)

# In query():
if _is_empty_result(exc):
    conn.close()
    return [], []

Add a version-pinned comment noting which DuckDB version the message was observed on.

Medium
_coerce() converts whole-number floats to int — timestamp type mismatch
backends/influxdb2.py:149–158
▶

Correctness

_coerce() tries int(v) first, then float(v). InfluxDB2 CSV serializes _time and numeric fields as strings like "1700000000" (a Unix epoch second). These will be parsed as Python int rather than float.

The DuckDB path returns timestamps as float. When _merge() sorts a union result by the timestamp column, comparing int from InfluxDB rows against float from DuckDB rows will work in Python (ints and floats compare correctly), but downstream consumers expecting all-float timestamps will receive mixed types. JSON serialization will produce 1700000000 (int) vs 1700000000.0 (float) for otherwise identical timestamps.

Suggested Fix
def _coerce(v: str):
    # Try float first to preserve .0 precision for timestamps
    try:
        f = float(v)
        # Return int only for non-timestamp-shaped integers (small numbers)
        if f == int(f) and abs(f) < 1e9:
            return int(f)
        return f
    except ValueError:
        return v

Or simply always return float for numeric strings and let callers decide. The test suite should assert that timestamp columns are consistently typed.

Medium
No limit on response size — router returns unbounded rows to HTTP
http_server.py:55–94  |  router.py:106
▶

Reliability

The HTTP /query endpoint defaults limit=500 and passes it to router.query(), which enforces it at the SQL/Flux level. However, the union path slices to rows[:limit] after merging, meaning both cold and hot queries can each return up to limit rows (2× the limit) before the slice. For limit=10000 this means up to 20k rows are held in memory during merge.

More importantly, there is no server-side cap on the caller-supplied limit parameter. A client requesting limit=1000000 will cause DuckDB to try to load 1M rows into memory.

Suggested Fix
# http_server.py — in _query()
MAX_LIMIT = 50_000
limit = min(int(_p("limit", 500)), MAX_LIMIT)

router.query() union case runs cold and hot sequentially — added latency
router.py:98–106
▶

Performance

When a query spans the hot/cold boundary the router runs cold (DuckDB) first, then hot (InfluxDB2) sequentially. Both backends block the executor thread. Total latency is roughly cold_latency + hot_latency.

With a 600s hot window, spanning queries only occur when from_ts is in the window and to_ts is after the window, which is relatively uncommon in practice. But for long time ranges that straddle the boundary, parallel execution would halve latency.

Suggested Fix (optional, higher complexity)
# router.py — parallel union using asyncio (if called from async context)
import asyncio

async def query_async(self, ...):
    loop = asyncio.get_running_loop()
    cold_fut = loop.run_in_executor(executor, self.cold.query, ...)
    hot_fut  = loop.run_in_executor(executor, self.hot.query, ...)
    cold_result, hot_result = await asyncio.gather(cold_fut, hot_fut)

For the current use pattern (10-min hot window, data mostly in cold), sequential is acceptable.

Correctness

In the union case, if cold has columns [ts, soc, kw] and hot has [ts, soc, kw, temp], the temp column from InfluxDB2 rows is silently dropped because cols = cold_cols and hot_idx[c] is only accessed for columns in cols.

Info
Flux limit(n:) is per-series, not global — known architectural constraint
backends/influxdb2.py:85
▶

Architecture

The |> limit(n: {limit}) call in the generated Flux query applies per table (per series), not globally. With 9984 battery cells × 2 fields = 19968 series, a request for limit=40 will read up to 19968 × 40 = 798,720 rows before InfluxDB2 returns a result to the network. This is a known InfluxDB2 limitation.

The current architecture mitigates this via:

  • hot_window_seconds: 600 — InfluxDB2 only serves the last 10 minutes (data in WAL/cache = fast)
  • extra_filters — scope cardinality to 104 series in tests

For production queries without extra_filters, InfluxDB2 queries spanning the full site will remain slow. A |> first() or |> tail() approach won't help. The only real fix is lower cardinality (coarser tags) or moving to InfluxDB3 which has proper SQL with global LIMIT.

Info
json.dumps default=str silently serializes non-JSON types
http_server.py:39  |  subscriber.py:229
▶

Observability

Both the HTTP server and WebSocket handler use json.dumps(..., default=str). This is intentionally permissive — it allows datetime objects, Decimal types, etc. to serialize as strings without raising. However, it also means type bugs in the data pipeline (e.g., a DuckDB datetime cell ending up in a row) are silently serialized as "2025-01-01 00:00:00" rather than a float. Callers receive plausible-looking data that is actually wrong.

This is acceptable for a monitoring/telemetry API where some roughness is tolerable, but worth knowing about when debugging type issues.

Design

base.py defines a @runtime_checkable QueryBackend Protocol, but neither InfluxDB2Backend nor DuckDBBackend explicitly inherit from it or are checked against it at instantiation time. The Protocol is unused beyond its definition — isinstance(backend, QueryBackend) is never called in the codebase.

The Protocol is still valuable as documentation and for static type checking (mypy), but the @runtime_checkable decorator has no effect without an isinstance check somewhere.
