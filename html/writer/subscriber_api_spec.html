subscriber_api — Extended Architecture Spec

Evolving from a 276-line MQTT→WebSocket bridge into a unified data access layer — InfluxDB2 · InfluxDB3 · DuckDB/Parquet · seamless routing · archive · live feed

Target Architecture

Performance Expectations

Key distinction: subscriber_api is a query and live-dashboard service — it is never on the high-rate ingest path. The 81k msg/s write load goes FlashMQ → Telegraf → InfluxDB directly. subscriber_api only touches FlashMQ for live-dashboard subscriptions, which are scoped to specific units/topics.

paho-mqtt performance note: paho-mqtt (Python) handles ~5,000–10,000 msg/s comfortably in a background thread. At full broker rate (81k msg/s) it would saturate and queue-drop. This is safe because subscriber_api must never subscribe to a wildcard that captures the full broker rate. Topic filters in config are mandatory — a bare # subscription is disallowed. If a use case genuinely requires high-rate live streaming, that path should bypass subscriber_api and connect to FlashMQ directly.

Load Summary

Within design envelope

  • Live feed: 1–3 operators watching unit-scoped dashboards
  • History: Grafana + 1–2 UI users, 30s+ refresh
  • Occasional engineering deep-dives (cold DuckDB, 1–2 concurrent)
  • Daily archive job (background, throttled)

Out of scope — handle separately

  • High-rate live streaming of full broker traffic (≥ 1k msg/s per client)
  • Bulk data export (> 50k rows) — use Athena direct / archive job output; subscriber_api row limit is 50k
  • Real-time alerting / threshold evaluation — use Telegraf processors
  • Multi-tenant isolation / auth — add API gateway in front if needed

Backend Interface Contract

Every backend implements the same Python protocol. The router calls one function signature — callers never know which backend served the query.

class QueryBackend(Protocol):
    def query(
        self,
        site:     str,
        instance: str,         # unit/device within the site, e.g. "unit-3"
        from_ts:  float,        # unix seconds UTC
        to_ts:    float,
        limit:    int,
        fields:   list[str] | None = None,   # None = all
    ) -> tuple[list[str], list[tuple]]:
        """Returns (column_names, rows). Raises on error."""
        ...

    def health(self) -> dict:
        """Returns {ok: bool, latency_ms: int, detail: str}"""
        ...

# Implemented by:
#   InfluxDB2Backend   — Flux or InfluxQL over HTTP
#   InfluxDB3Backend   — SQL over HTTP or Arrow Flight
#   DuckDBBackend      — read_parquet() from S3 or local path  (dev / fractal-phil)
#   AthenaBackend      — boto3 start_query_execution + poll     (AWS production)

Query Router

class QueryRouter:
    hot_backend:    QueryBackend   # InfluxDB2 or InfluxDB3 (config-driven)
    cold_backend:   QueryBackend   # DuckDB (dev/fractal-phil) or Athena (AWS prod)
    hot_window_days: int           # default 90
    shadow_mode:    bool           # run both hot backends, log diff

    def query(self, site, instance, from_ts, to_ts, limit, fields=None):
        cutoff = time.time() - self.hot_window_days * 86400

        if   from_ts >= cutoff:               # entirely hot
            return self.hot_backend.query(site, instance, ...)
        elif to_ts   <  cutoff:               # entirely cold
            return self.cold_backend.query(site, instance, ...)
        else:                                  # spans boundary — union
            cold = self.cold_backend.query(site, instance, ..., to_ts=cutoff)
            hot  = self.hot_backend.query(site, instance, ..., from_ts=cutoff)
            return _merge(cold, hot)           # dedup on timestamp
Shadow mode — set shadow: true in config to run both InfluxDB2 and InfluxDB3 in parallel during migration. Results are compared, discrepancies logged, but only the primary backend's result is returned to the client. Flip hot_backend: influxdb3 when the diff log is clean.

Engineering queries 6 months ago for a warranty claim or performance regression. Data in parquet archive.

DuckDB
UC-05

Cross-boundary query — spans hot/cold cutoff

Query from 120 days ago to now. Router splits at cutoff, runs both backends, merges on timestamp, returns unified result.

UNION
UC-06

Grafana data source — arbitrary time picker

Grafana sends HTTP GET with whatever time range the user selects. Router handles transparently — Grafana doesn't know about backends.

any
UC-07

InfluxDB2 → InfluxDB3 migration validation

Shadow mode: both backends queried, results diffed. Discrepancies logged with site/time/field detail. Flip hot_backend when diff rate < threshold.

shadow v2/v3
UC-08

Archive old data from InfluxDB — reduce EFS bill

POST /archive extracts data older than N days to date-partitioned parquet, verifies row counts, then calls InfluxDB delete API. Dry-run first.

archive
UC-09

Live telemetry broadcast — FlashMQ (MQTT)

Edge sends MQTT messages to FlashMQ; subscriber_api subscribes via paho-mqtt and broadcasts to all connected WebSocket clients. Same FlashMQ broker used for both edge data ingest and live-feed subscription.

live feed
UC-10

Per-client config audit — join with client_db

DuckDB can attach the SQLite client_db and join config data with telemetry. E.g. "which clients running old UI version have high kW variance?"

Athena cold backend — AWS production  ~+80 lines
  • Implement AthenaBackend — boto3 start_query_execution → poll → get_query_results
  • Same SQL as DuckDB; plain path layout {site}/{instance}/{yyyy}/{mm}/{dd}/ — no Hive key=value paths needed
  • Config flag cold_backend: athena to switch — zero caller change; DuckDB tests validate SQL first
  • Athena table uses partition projection (table property) — no Glue crawler, no MSCK REPAIR TABLE; Athena derives partitions from the path template at query time
  • Query result pagination: Athena returns max 1,000 rows per page; backend loops get_query_results with NextToken until limit reached
  • Local simulation: DuckDB with identical SQL and partition layout — Phase 0 DuckDB tests are the Athena integration test proxy. If DuckDB passes, Athena will work.
  • Tests: unit test with mock boto3 (moto library); integration test against real Athena on a small date-range once archive is in S3
DuckDB as local Athena simulator: plain path layout and SQL dialect are identical between DuckDB and Athena. Write the archive files once; DuckDB queries them locally during development and CI, Athena queries the same files in production. Switch with one config line — no SQL changes, no file changes.

Path layout: s3://ems-archive/parquet/{site}/{instance}/{yyyy}/{mm}/{dd}/data.parquet — simple, human-readable, no key=value segments required. DuckDB uses a glob (evelyn/unit-3/2025/06/**); Athena uses partition projection with a storage.location.template pointing at the same path pattern. No Glue crawler needed. The instance level lets you query a single unit without scanning the whole site.
