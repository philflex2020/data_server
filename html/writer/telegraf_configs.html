Telegraf Config Versions

7 configs — 5 from versions.zip (production Longbow) + 2 from the Evelyn cost-comparison demo (gx10)

Production configs (versions.zip) — telegraf-conf-v2*.conf / telegraf-conf-v3.conf

  • v2 baseline — The current Longbow production pipeline. MQTT → processors.pivot (collapses the data_type tag into a typed field: float, integer, boolean_integer, string) → InfluxDB. Uses a $share/telegrafs-v2/ shared MQTT subscription so multiple Telegraf replicas can load-balance without duplicating data. concurrent_writes=2 for parallel InfluxDB batching.
  • v2 + cloud — Same as v2 baseline but with a second [[outputs.influxdb_v2]] block writing to a cloud-hosted InfluxDB (separate URL/token/org/bucket). Both outputs use concurrent_writes=2. Used when data must land in both local and cloud stores simultaneously.
  • v2 no-concurrency — v2 baseline with concurrent_writes removed (defaults to 1 thread). Also changes pivot order to 2. Used for benchmarking/debugging write-thread contention at high ingestion rates.
  • v2 no-concurrency + cloud — Dual InfluxDB output (local + cloud) with no concurrent_writes on either. Worst-case throughput configuration; useful for isolating cloud write latency without parallelism masking it.
  • v3 — Adds explicit processors.converter blocks before pivot to pre-cast the generic JSON value field to the correct native Go type (float64/int64/string). This prevents Telegraf from guessing the type, which can cause silent schema drift in InfluxDB IOx (serverless). Uses a separate $share/telegrafs-v3/ group so v2 and v3 can coexist on the same broker. Sets organization="" for InfluxDB 3.x / IOx compatibility.

Demo configs (evelyn-compare-native-start.sh) — compare-base / compare-filt

  • Both subscribe to tcp://localhost:11888 (single FlashMQ on gx10) with a plain ems/# topic — no shared subscription needed since there is only one Telegraf per path.
  • Both use the same 3-tier topic-parsing schema as production (9-seg unit, 7-seg meter/rtac, 6-seg root) so the InfluxDB schema is identical to what Longbow writes today.
  • compare-base — No filtering at all. Every message that arrives is pivoted and written to ems-baseline on InfluxDB :8096. This mirrors the current Longbow production pipeline cost.
  • compare-filt — Applies two filters before writing to ems-filtered on InfluxDB :8097:
    • Filter A — tagdrop = {data_type = ["string"]} on the MQTT input. String-typed signals (status labels, firmware version strings, etc.) are dropped before they enter the pipeline. ~4% row reduction.
    • Filter B — [[processors.dedup]] dedup_interval="60s" with tagpass data_type=[integer, boolean_integer]. Integer and boolean signals are suppressed if their value has not changed within the last 60 seconds. ~23% row reduction.
    • Combined: ~26.8% fewer rows written — the Conservative scenario from the cost analysis.

[[processors.converter]]  # float cast
  order = 1
  [processors.converter.tagpass]
    data_type = ["float"]
  [processors.converter.fields]
    float = ["value"]

  order = 1
  [processors.converter.tagpass]
    data_type = ["integer", "boolean_integer"]
  [processors.converter.fields]

  order = 1
  [processors.converter.tagpass]
    data_type = ["string"]
  [processors.converter.fields]

  value_key = "value"

# Written to /tmp/telegraf-native-filt.conf by evelyn-compare-native-start.sh
# Purpose: filtered path — applies Conservative scenario filters before writing.
# Measures: what Longbow could save by enabling Filter A + Filter B.
#
# Filter A: drop string dtype at input  → ~4% fewer rows
# Filter B: dedup integer/boolean 60s   → ~23% fewer rows
# Combined: Conservative scenario       → ~26.8% fewer rows

[agent]
  # identical to demo-base agent block
  interval              = "1s"
  round_interval        = true
  metric_batch_size     = 5000
  metric_buffer_limit   = 500000
  flush_interval        = "5s"
  flush_jitter          = "1s"
  precision             = "1ms"
  omit_hostname         = true

[[inputs.mqtt_consumer]]
  servers            = ["tcp://localhost:11888"]   # same broker as demo-base
  topics             = ["ems/#"]
  qos                = 0
  client_id          = "telegraf-compare-filt"
  connection_timeout = "30s"
  data_format        = "json_v2"

    # ── Filter A: drop string signals before any processing ─────────────────
  [inputs.mqtt_consumer.tagdrop]
    data_type = ["string"]      # status labels, firmware strings, etc. — ~4% of rows

  [[inputs.mqtt_consumer.json_v2]]
    timestamp_path     = "ts"
    timestamp_format   = "2006-01-02T15:04:05.000Z"
    timestamp_timezone = "UTC"

    [[inputs.mqtt_consumer.json_v2.field]]
      path = "value"

      # Same 3-tier topic parsing as demo-base (9-seg / 7-seg / 6-seg) — omitted for brevity

[[inputs.internal]]
  collect_memstats = false

# ── Filter B: suppress integer + boolean signals that haven't changed in 60s ──
[[processors.dedup]]
  order          = 0              # runs BEFORE pivot
  dedup_interval = "60s"
  [processors.dedup.tagpass]
    data_type = ["integer", "boolean_integer"]  # floats pass through unchanged

# Pivot runs after dedup — collapses data_type tag into typed field columns
[[processors.pivot]]
  order     = 1
  tag_key   = "data_type"
  value_key = "value"

[[outputs.influxdb_v2]]
  urls         = ["http://localhost:8097"]   # separate InfluxDB from baseline
  token        = "ems-token-secret"
  organization = "ems-org"
  bucket       = "ems-filtered"
  timeout      = "10s"
  namedrop     = ["internal_*"]
