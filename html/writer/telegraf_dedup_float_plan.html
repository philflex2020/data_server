Telegraf — processors.dedup float_epsilon contribution

Adding epsilon-based float comparison to the existing dedup processor · Repo: ~/work/telegraf/telegraf · Upstream: github.com/influxdata/telegraf (MIT)

Why this matters

The problem with exact float comparison

The existing processors.dedup uses Go's != operator for all field types. For float64, this means a value that changes from 3.14159265 to 3.14159266 due to sensor noise or ADC jitter passes through as a change — defeating dedup entirely for high-precision float telemetry.

In the EMS / Evelyn use case, AC power signals (kW, kVAR, kVA) oscillate continuously at the last digit. Without epsilon, zero rows are suppressed for float fields.

What float_epsilon adds

A single new config option float_epsilon. When set, float64 fields are compared with an absolute tolerance instead of exact equality. All other field types (integer, bool, string) continue to use exact comparison unchanged.

Default 0.0 preserves existing behavior exactly — fully backward compatible, no behavior change for existing configs.

Expected reduction for EMS float signals: ~23% additional row suppression (matching our measured python float_dedup result).

Approach chosen: extend existing processors.dedup rather than a new plugin. One config block, one place to learn, backward compatible, smaller diff = higher chance of acceptance.

2. Apply() — replace direct != with fieldsEqual()

for _, f := range metric.FieldList() {
    if value, ok := m.GetField(f.Key); ok {
        if value != f.Value {
        if !d.fieldsEqual(value, f.Value) {
            changed = true
            break
        }
    }
}

3. New method — fieldsEqual()

// fieldsEqual reports whether two field values are considered equal.
// Float64 fields use FloatEpsilon when it is non-zero; all other types use ==.
func (d *Dedup) fieldsEqual(a, b interface{}) bool {
    if d.FloatEpsilon > 0 {
        af, aok := asFloat64(a)
        bf, bok := asFloat64(b)
        if aok && bok {
            return math.Abs(af-bf) <= d.FloatEpsilon
        }
    }
    return a == b
}

// asFloat64 converts float32/float64 fields to float64; returns false for other types.
func asFloat64(v interface{}) (float64, bool) {
    switch val := v.(type) {
    case float64:
        return val, true
    case float32:
        return float64(val), true
    default:
        return 0, false
    }
}

Tests to add (dedup_test.go)

Test cases required

  • TestFloatEpsilonWithinTolerance — value changes by < epsilon → suppressed
  • TestFloatEpsilonExceedsTolerance — value changes by > epsilon → passes through
  • TestFloatEpsilonAtBoundary — value change exactly equals epsilon → suppressed
  • TestFloatEpsilonZeroIsExact — epsilon=0 → existing exact-match behavior unchanged
  • TestFloatEpsilonDoesNotAffectIntegers — int fields unaffected when epsilon set
  • TestFloatEpsilonDoesNotAffectStrings — string fields unaffected
  • TestFloatEpsilonHeartbeat — value within epsilon still passes after dedup_interval expires

Typical test pattern (matches existing style)

func TestFloatEpsilonWithinTolerance(t *testing.T) {
    plugin := &Dedup{
        DedupInterval: config.Duration(10 * time.Minute),
        FloatEpsilon:  0.5,
        cache: make(map[uint64]telegraf.Metric),
        flushTime: time.Now(),
    }
    m1 := testutil.MustMetric("test",
        map[string]string{},
        map[string]interface{}{"kw": float64(100.0)},
        time.Now())
    m2 := testutil.MustMetric("test",
        map[string]string{},
        map[string]interface{}{"kw": float64(100.3)}, // < 0.5 epsilon
        time.Now().Add(time.Second))

    out1 := plugin.Apply(m1)
    out2 := plugin.Apply(m2)
    require.Len(t, out1, 1) // first always passes
    require.Len(t, out2, 0) // within epsilon → suppressed
}

Recommended epsilon values for EMS signals

Based on the Evelyn / Longbow signal characteristics:

Signal class Examples Typical range Suggested epsilon Rationale
AC power kW, kVAR, kVA, PF 0 – 5000 kW 0.5 ADC noise <0.1%, 0.5 kW invisible on billing
DC voltage cell_v, pack_v 2.5 – 4.2 V 0.005 5 mV resolution typical; <0.15% of range
DC current pack_i, cell_i -500 – 500 A 0.5 0.1% of full scale, below current sensor noise
Temperature cell_temp, ambient -20 – 60 °C 0.1 Sensor resolution 0.1 °C; 0.1 is 1 LSB
SOC / SOH soc, soh 0 – 100 % 0.1 0.1% SOC is below any actionable threshold
Per-field epsilon would be ideal — a future extension could accept a float_epsilon map keyed by field name. For an initial upstream PR, a single global epsilon is simpler and far more likely to be merged. We can file a follow-up issue.

Relationship to edge-filter architecture

Once this PR is merged, the edge-filter option becomes even cleaner: a single [[processors.dedup]] block with float_epsilon handles all field types (integer, bool, float) — no execd subprocess, no Python dependency, no per-message process overhead. The Telegraf Edge node in the architecture becomes a ~20-line TOML config.

Current workaround stack on .46:

# Today — three separate mechanisms for three field types
[[processors.dedup]]             # handles int + bool
  dedup_interval = "60s"

[[processors.execd]]             # handles float — external Python process
  command = ["python3", "/path/float_dedup.py"]

After PR merged:

# One block handles everything
[[processors.dedup]]
  dedup_interval  = "60s"
  float_epsilon   = 0.5       # covers kW, kVAR, kVA, PF in one line
