Parquet Output — Telegraf built-in vs our C++ writer

Telegraf has had outputs.parquet since v1.32.0 (uses apache/arrow-go/v18). Comparison against source/parquet_writer_cpp/real_writer:

Feature Telegraf outputs.parquet Our real_writer (C++) Notes
Writes Parquet ✓ ✓ Both use Apache Arrow
Schema inference first batch only runtime-typed New fields after first flush are silently dropped in Telegraf
Date partitioning ✗ flat dir ✓ YYYY/MM/DD/ DuckDB hive partitioning requires date dirs
Per-topic file naming ✗ by measurement name ✓ topic-derived Ours splits EMS units into separate files
WAL / crash recovery ✗ crash = corrupt file ✓ WAL + recovery Telegraf README explicitly warns about this
Health endpoint ✗ ✓ :8771 HTTP Used by push_agent and monitoring
File rotation time interval only interval + size Size-based needed for high-rate ingestion
Compression default only configurable Snappy, ZSTD, etc.
Push agent integration ✗ ✓ rsync to AWS push_agent :8770 reads health, rsyncs files
Null handling ✓ ✓
Language / dependency Go (in Telegraf binary) C++ (separate binary) Telegraf plugin = zero extra binary to deploy
ARM / cross-compile ✓ go build Yocto SDK required Go cross-compile is trivial vs C++ toolchain
Contribution opportunities upstream
  • Date-partitioned directory layout (YYYY/MM/DD/)
  • WAL / safe close on crash
  • Size-based rotation
  • Runtime schema evolution (new fields after first flush)
  • Configurable compression codec
When to use Telegraf outputs.parquet today
  • Schema is stable (no new fields after start)
  • Telegraf already in the stack — zero extra binary
  • ARM target — no C++ cross-compile toolchain
  • Short-lived captures where WAL not critical
  • Combined with float_epsilon dedup — single config

Option A — use our C++ writer as a Telegraf output via outputs.execd

[[outputs.execd]] pipes metrics as InfluxDB line protocol to the stdin of any external process. Adding a --stdin flag to real_writer would let Telegraf own MQTT + filtering while our writer owns WAL + partitioned parquet — best of both.

Aspect outputs.execd → real_writer --stdin Notes
Telegraf config [[outputs.execd]]
command = ["real_writer","--stdin"]
data_format = "influx"
restart_delay = "10s"		 Telegraf restarts writer if it exits; pipe is recreated automatically
Wire format InfluxDB line protocol on stdin unit,site=s1,id=u0 kw=1.5,kvar=0.3 1743024000000000000
Writer changes ~150 lines parse_line_proto() + --stdin main loop pushing Row structs into existing ring. WAL, parquet, health untouched.
Partition key derived from line-proto tags Tags site_id, unit_id map directly to existing PartitionKey struct — same directory layout
WAL + crash recovery ✓ unchanged Writer replays WAL on restart regardless of input source
Date partitioning ✓ unchanged Timestamp comes from line protocol nanosecond field
Health endpoint ✓ unchanged :8771 still available for push_agent and monitoring
Filtering / dedup handled by Telegraf processors Writer receives only post-filter rows — no float_dedup.py subprocess needed
Backpressure OS pipe buffer + Telegraf metric_buffer_limit If writer stalls, Telegraf buffers up to metric_buffer_limit (default 10 k) then drops oldest
Combined with dual-output ✓ — third output alongside two influxdb_v2 One Telegraf, one dedup pass → local InfluxDB + AWS InfluxDB + local parquet files
Code changes to real_writer.cpp
// new function — parse one line-protocol line into a Row
std::optional<Row> parse_line_proto(std::string_view line);

// new main loop (--stdin mode)
if (stdin_mode) {
    std::string line;
    while (std::getline(std::cin, line)) {
        auto row = parse_line_proto(line);
        if (row) g_mqtt_ring.push(*row);
    }
}

Parse tags → PartitionKey, fields → Row.floats/ints/strings, nanosecond timestamp → Row.ts. Then drops straight into the existing ring.

Filtered /tmp/telegraf-46-filt.conf
# Filter A + Filter B (starlark float scaling + dedup) — Conservative+

[agent]
  interval       = "1s"
  precision      = "1ms"
  omit_hostname  = true

[[inputs.mqtt_consumer]]
  servers     = ["tcp://localhost:1883"]
  topics      = ["ems/#"]
  client_id   = "telegraf-46-filt"
  data_format = "json_v2"

    # ── Filter A: drop string dtype ──────────────────
    [inputs.mqtt_consumer.tagdrop]
      data_type = ["string"]    ← ~4% reduction
    # ─────────────────────────────────────────────────

  [[inputs.mqtt_consumer.json_v2]]
    timestamp_path   = "ts"
    timestamp_format = "2006-01-02T15:04:05.000Z"

    [[inputs.mqtt_consumer.json_v2.field]]
      path = "value"

    [[inputs.mqtt_consumer.topic_parsing]]
      topic       = "ems/site/+/unit/+/+/+/+/+"
      tags        = "_/_/site_id/_/unit_id/…/data_type"
      measurement = "_/_/_/measurement/…"

    # ── Filter B step 1: float×100 → float_i (order=0) ───
    # Scales floats to int so dedup can exact-match them.
    # On read: divide float_i by 100 to recover value.
  [[processors.starlark]]
      order = 0
      source = '''
    def apply(metric):
        if metric.tags.get("data_type") != "float":
            return metric
        v = metric.fields.get("value")
        if v == None: return metric
        metric.fields["value"] = int(float(v) * 100)
        metric.tags["data_type"] = "float_i"
        return metric
    '''

    # ── Filter B step 2: dedup int/bool/float_i (order=1) ─
  [[processors.dedup]]
      order          = 1
      dedup_interval = "60s"
      [processors.dedup.tagpass]
      data_type = ["integer", "boolean_integer", "float_i"]
    # ─────────────────────────────────────────────────────

[[processors.pivot]]
  order     = 1
  tag_key   = "data_type"
  value_key = "value"

[[outputs.influxdb_v2]]
  urls    = ["http://localhost:8097"]
  bucket  = "ems-filtered"
  namedrop = ["internal_*"]
Filter A — drop string rows at input (~4%)
Filter B — float×100→float_i (Starlark) then dedup int/bool/float_i 60s (~23%+)
