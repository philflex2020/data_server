TL;DR — Benchmark Findings (2026-04-07, updated)
  • Normalized Long is the recommended schema — 8 columns (ts, unit_id, device_type, device, point_name, value, dtype_hint, site_id), smallest storage, best query ergonomics.
  • Wide (pivot) fixed: merge_wide_rows() added to writer — with time_window_ms: 250 + compound_field_name, the writer now merges all signals from the same burst into one dense row per timestamp (100% fill, 0 nulls). Previously produced one sparse row per signal per burst.
  • Wide now produces 600 rows per unit for 600 sweeps (was 70,017 distinct timestamps before fix). Each row has all 37 signal columns filled.
  • Wide is still 2–3× larger than norm-long due to parquet column metadata overhead for 38 columns; bytes/row on disk is higher but all values are present.
  • time_window_ms: 250 required in wide config — coalesces all signals in a 50–200 ms burst to a single shared ts so merge_wide_rows() can group them. Without it, every signal keeps its unique per-message ts and merge produces no benefit.
  • COV (change-on-value) cuts message volume ~8% with random-walk data; real hardware with stable signals will see larger reductions.
  • Forward-fill gaps at query time via DuckDB LAST_VALUE(value IGNORE NULLS) OVER (...) — no writer changes needed; DuckDB is always in the query path.
  • Flush latency (20-sweep test): norm-long avg 6 ms · long+cmp avg 3 ms · wide avg 26 ms (was 64–87 ms before fix). Full 600-sweep results pending.

Forward-fill (carry last known value across gaps)

Signals that use COV (change-on-value) or that didn't fire in a given time window have no row for that period — the writer stores only what the hardware actually sent. Forward-fill is handled at query time with DuckDB window functions, keeping storage minimal. current_state_path in the writer config provides a snapshot file of the latest value per signal if a non-DuckDB consumer needs it.
Forward-fill Pack_SOC across gaps
SELECT ts, unit_id,
  LAST_VALUE(value IGNORE NULLS) OVER (
    PARTITION BY unit_id
    ORDER BY ts
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS value
FROM read_parquet('/data/parquet/site=SITE_A/**/*.parquet')
WHERE device_type = 'bms'
  AND point_name  = 'Pack_SOC'
ORDER BY unit_id, ts;

Benchmark Generator

Both schemas were benchmarked simultaneously using compare_run.py — a Python script that launches both writers against the same MQTT stream and reports size, flush time, and schema width side-by-side. 
#!/usr/bin/env python3
# /tmp/bench2/compare_run.py
# 12 units × 34 signals (24 float + 10 int)
# 600 sweeps @ 1 Hz → 244,800 messages · both writers subscribe simultaneously

BURST_MIN_MS = 50     # minimum burst spread per sweep
BURST_MAX_MS = 200    # maximum burst spread — fits inside time_window_ms: 250
SWEEP_PERIOD = 1.0    # 1 Hz inter-sweep interval

for sweep in range(SWEEPS):
    sweep_wall_start = time.time()
    sweep_base_ts = base_ts + sweep * 1.0    # simulated 1 Hz tick

    # Monotonically increasing intra-burst delays — same ordering in wall-clock
    # AND in the payload ts, so the writer sees timestamps that are individually
    # distinct but all within a 50–200 ms window.  time_window_ms: 250 coalesces
    # them under the first message's ts; without the window each row gets its own.
    burst_s = random.uniform(BURST_MIN_MS, BURST_MAX_MS) * 1e-3
    delays  = sorted(random.uniform(0, burst_s) for _ in range(N_MSGS))

    for i, (unit, device, instance, signame, dtype) in enumerate(all_combos):
        msg_ts = sweep_base_ts + delays[i]   # unique, monotonic per-message ts
        ts_str = fmt_ts(msg_ts)              # ISO-8601 UTC with ms precision

        target = sweep_wall_start + delays[i]
        if target > time.time(): time.sleep(target - time.time())

        client.publish(topic, json.dumps({"ts": ts_str, "value": val}), qos=0)

    # Hold the 1-second cadence
    time.sleep(max(0, SWEEP_PERIOD - (time.time() - sweep_wall_start)))

Appendix — Stream Replay

The writer can record every incoming MQTT message verbatim — topic string, value, and original timestamp — producing a Parquet log that can be replayed byte-for-byte back into any MQTT broker. This uses the built-in store_mqtt_topic flag (default true) with time_window_ms: 0 to preserve per-message timestamps exactly.

Replay script

#!/usr/bin/env python3
"""Replay a captured Parquet stream back into an MQTT broker at original rate."""
import duckdb, json, time
import paho.mqtt.client as mqtt

PARQUET_PATH = "/data/replay/**/*.parquet"
BROKER_HOST  = "localhost"
BROKER_PORT  = 1883

client = mqtt.Client(client_id="replay-publisher")
client.connect(BROKER_HOST, BROKER_PORT)
client.loop_start()

rows = duckdb.sql(f"""
    SELECT ts, mqtt_topic, value
    FROM read_parquet('{PARQUET_PATH}')
    ORDER BY ts
""").fetchall()

print(f"Replaying {len(rows):,} messages...")
t0_real = time.time()
t0_data = rows[0][0].timestamp()

for ts, topic, value in rows:
    # sleep to maintain original inter-message timing
    lag = (ts.timestamp() - t0_data) - (time.time() - t0_real)
    if lag > 0:
        time.sleep(lag)
    payload = json.dumps({"ts": ts.isoformat() + "Z", "value": value})
    client.publish(topic, payload, qos=0)

client.loop_stop()
client.disconnect()
print("Replay complete.")
Two constraints for a valid replay capture:
  1. time_window_ms must be 0. The 250 ms coalescing deliberately collapses multiple per-message timestamps to one shared ts — replay would lose intra-burst ordering within each window.
  2. Payload must be {"ts": …, "value": …} only. The writer extracts exactly those two fields; any extra payload fields (quality flags, string values, etc.) are not stored. For the current EMS topology all messages follow this two-key structure, so replay is complete.
