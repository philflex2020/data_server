Long+compound is 21% smaller and 6× faster to flush at 34 signals/unit. Wide pivot gives per-unit file isolation via filename prefix — same directory tree, no performance cost from the naming change.
Wide flush slowdown is caused by Arrow allocating one array builder per signal column (38 here, scales with cardinality). Switching from a unit= subdirectory to a unit_id_ filename prefix made no measurable difference — the bottleneck is Parquet encoding, not filesystem path construction.

DuckDB Queries

Long+compound — all signals for one unit
SELECT ts, point_name, value
FROM read_parquet('/data/parquet/site=SITE_A/**/*.parquet')
WHERE point_name LIKE '0215D1D8/%'
  AND ts > now() - INTERVAL 1 HOUR
ORDER BY ts;
Wide — glob on unit_id filename prefix
SELECT ts,
  "bms/bms_1/Pack_SOC",
  "pcs/pcs_1/ActivePower"
FROM read_parquet(
  '/data/parquet/site=SITE_A/**/0215D1D8_*.parquet')
WHERE ts > now() - INTERVAL 1 HOUR
ORDER BY ts;

Config Files

Long + Compound — config_long_compound.yaml

mqtt:
  host: localhost
  port: 1883
  client_id: pw-bench-long-cmp
  topic: "bench/#"
  qos: 0

  topic_parser: positional
  topic_segments:
    - "_"          # topic prefix — skip
    - unit_id      # hex unit serial
    - device       # bms / pcs / rack
    - instance     # bms_1 / pcs_1 etc.
    - point_name   # signal name
    - dtype_hint   # float / integer — consumed by parser

  # Join unit_id/device/instance/point_name into a single point_name string
  compound_point_name: [unit_id, device, instance, point_name]

    - "{day}"

Benchmark Generator

Both schemas were benchmarked simultaneously using compare_run.py — a Python script that launches both writers against the same MQTT stream and reports size, flush time, and schema width side-by-side.

# 12 units × 34 signals (24 float + 10 int)
# 600 sweeps @ 1 Hz → 244,800 messages · both writers subscribe simultaneously

SWEEPS       = 600          # 1 Hz × 600 s

RATE         = "1 Hz"

for sweep in range(SWEEPS):
