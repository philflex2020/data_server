Critical

C1 Signal handler calls notify_all on condition_variable — undefined behaviour lines 917–920 ▶

std::condition_variable::notify_all() is not async-signal-safe. If the signal fires while g_mutex is held by another thread, the notify path may deadlock. Only async-signal-safe operations (setting an atomic_flag, writing to a self-pipe or eventfd) are legal inside a signal handler.

C2 WAL write failure silently proceeds with the Parquet write — crash safety guarantee is void lines 672–678 ▶

When write_wal_ipc fails, the code logs a warning and continues flushing Parquet. If the Parquet write then crashes, the data is permanently lost with no WAL to recover from. The flush for that partition should abort if the WAL cannot be written.

Additionally, writer->Close() failure on line 510 is checked but out->Close() on line 511 has its return value discarded — the WAL file may be silently corrupt.

C3 WAL replay hard-codes source_type = "batteries" — silent data misrouting line 577 ▶

The WAL filename encodes the source type ({source_type}_{pval}_{ts}.wal.arrow) but the replay loop never parses it back out. All replayed data, regardless of the original source type, is written into a batteries/ directory tree.

C4 on_message holds g_mutex for every message at ~80k msg/sec — blocks MQTT network thread lines 878–900 ▶

The flush thread copies g_current_state (up to 10k std::map entries with heap-allocated keys) while holding the same lock. During that copy the MQTT thread blocks and mosquitto's receive buffer fills. This is both a throughput bottleneck and a risk of missed keepalives causing spurious reconnects.

C5 g_health_buffer_rows not updated when buffer is flushed — health endpoint reports stale count lines 597, 761–763, 897 ▶

g_total_buffered is reset to 0 inside flush_thread_fn but g_health_buffer_rows is only updated in on_message. Between a flush completing and the next incoming message, the health endpoint reports the pre-flush (inflated) buffer row count.

C6 WAL replay only reads chunk(0) — silently drops rows in multi-chunk tables lines 537–541 ▶

If the WAL IPC file contains multiple record batches, arrow::ChunkedArray::chunk(0) gives only the first chunk. All remaining rows are silently discarded. Any WAL file written by a modified write path (or externally created) would lose data on replay with no error.

Major

M1 parse_payload default-constructs Config{} as fallback — wrong config silently used line 241 ▶

If the cfg argument is omitted, a default-constructed Config is used silently instead of asserting or returning an error. All partition field lookups and output settings will be wrong.

M2 g_received_since_sync cast to int64_t can produce misleading "OK" sync logs lines 823–824 ▶

When a large number of messages has been received before the first _sync arrives in a new session, the delta calculation can wrap or produce an incorrect "loss=0" result, masking real message loss.

M3 flush_partition and write_wal_ipc duplicate schema-building logic — already diverged lines 356–414 vs 472–512 ▶

Both functions iterate over rows and build Arrow arrays in ~60 lines of nearly identical code. They have already diverged in variable naming. A future schema change will need to be made in two places and the mismatch may not be caught until a WAL replay produces malformed Parquet.

M4 make_output_path uses flush time not collection time — replayed WAL rows land in wrong date partition lines 314–343 ▶

Partition directories use year/month/day derived from system_clock::now() at flush time. Data collected on Tuesday that is WAL-replayed on Wednesday will be written into Wednesday's partition. This silently corrupts time-based queries.

M5 g_current_state copied under g_mutex at flush time — O(N) heap allocations while MQTT thread is blocked lines 762–763 ▶

At every flush, the entire current-state map is deep-copied while holding the global mutex. At 10k sensor keys this is thousands of heap allocations. The MQTT receive thread is blocked for the full duration of this copy.

M6 Health thread ignores recv return value; buffer uninitialised on error; fragmented HTTP not handled line 617 ▶

If recv returns -1 or 0, the uninitialized contents of buf are used as the HTTP request. A slow or fragmented client (e.g. high-latency connection) will get a garbled response or crash.

M7 Health endpoint responds HTTP 200 JSON to any TCP connection — no path or method check lines 617–633 ▶

Any TCP connection gets a full JSON response regardless of HTTP method or path. Port scanners, load balancer probes, and random connections will receive the metrics payload, potentially leaking operational data.

M8 check_disk_space skips updating g_disk_free_gb_x10 when guard is disabled — health always shows 0.0 GB lines 440–452 ▶

When min_free_gb is 0 (guard disabled), the function returns early without sampling disk space. The health endpoint always reports disk_free_gb: 0.0, making it useless for monitoring disk capacity.

M9 mosquitto_disconnect called after loop_stop(force=true) — DISCONNECT never sent lines 1044–1046 ▶

The correct shutdown order is: disconnect() → wait for on_disconnect → loop_stop(). With force=true the network thread is killed before the DISCONNECT packet can be sent. The broker retains the session and may redeliver queued messages on reconnect.

M10 g_mosq plain pointer — data race between main-thread write and network-thread reads lines 1005, 1038 ▶

Under the C++ memory model, writing g_mosq in main and reading it in the network thread (via callbacks) is a data race without synchronisation. In practice the assignment happens before loop_start, but there is no memory fence to guarantee visibility.

M11 Final flush runs before mosquitto_loop_stop — guaranteed message loss window on every graceful shutdown lines 1041–1044 ▶

The flush thread exits (and does a final flush) before mosquitto_loop_stop is called. Any messages delivered by mosquitto in that window are added to the buffer but never flushed. On every clean SIGTERM shutdown there is a window of data loss.

M12 No config validation — bad values cause spin-flush, drop-everything, or silent wrong compression lines 117–168 ▶

flush_interval_seconds <= 0 creates a spin-flush loop consuming 100% CPU. max_total_buffer_rows <= 0 drops every incoming message immediately. An invalid compression string silently falls through to Snappy with no warning.

M13 source_type from MQTT topic injected into filesystem paths without sanitisation — path traversal line 334 ▶

A rogue publisher with a topic like ../../etc/cron.d/evil can write files to arbitrary paths writable by the process. source_type must be validated against a allowlist or sanitised (strip /, .., null bytes) before use in fs::path construction.

M14 Integer scalars not in int_cols silently promoted to double — precision loss above 2^53 lines 289–292 ▶

Large integers (e.g. Unix timestamps in nanoseconds, serial numbers) stored as double lose precision above 2^53. Additionally, if a JSON payload field name matches a topic-derived int column name, the same value appears in both row.ints and row.floats, producing duplicate Parquet columns with undefined merge behaviour.

Minor

m1 Magic numbers throughout — backlog, retries, backoff delays, keepalive — none configurable lines 609, 683, 687, 1011–1018 ▶

Listen backlog 4, MAX_RETRIES 3, backoff 1/2/4s, reconnect delay 2/30s, keepalive 60s, retry sleep 5s are all hardcoded. At minimum these should be named constants; ideally configurable.

m2 Overflow notify_one only wakes flush thread if sleeping — drops continue during active flush lines 880–887 ▶

If the flush thread is actively writing Parquet when the buffer overflows, notify_one is a no-op and drops continue for the full flush duration with no additional action taken.

m3 Parquet column order is non-deterministic — determined by first-seen order per flush cycle lines 362–368 ▶

Column order varies depending on which message arrived first in each flush window. DuckDB handles this correctly but it makes manual inspection and schema diffs confusing.

m4 g_cfg is a raw pointer to a stack Config in main — null dereference in tests line 992 ▶

Any unit test that exercises a function referencing g_cfg before main sets it will null-dereference. Should be initialised to a default or use a reference.

m5 wal_path_for calls fs::create_directories on every flush per partition line 468 ▶

Directory creation is a syscall. It should happen once at startup (or first WAL write), not on every flush of every partition.

m6 WAL filename and Parquet filename use independent now() calls — can differ across a second boundary lines 458–464, 316–325 ▶

If time_suffix() and make_output_path are called at the boundary of a second, the WAL file name will reference a different timestamp than the Parquet file it corresponds to. Makes post-crash matching harder.

m7 First sync always reports OK even if messages dropped before first _sync arrived line 820 ▶

The initial baseline is set from the first received sync message. Any drops that occurred before that point are invisible to the loss counter.

m8 parse_topic silently overwrites duplicate key-value segments lines 227–234 ▶

If a topic contains the same key twice (e.g. site/1/site/2/...), the second value silently replaces the first with no warning.

m9 mosquitto_subscribe failure in on_connect is logged but not retried — writer silently receives zero messages lines 789–793 ▶

If the initial subscribe fails (e.g. transient broker issue), the writer runs indefinitely with an empty buffer, flushing empty Parquet files, with no recovery path.

m10 recv buffer 255 bytes — too small for modern browser HTTP headers line 617 ▶

Chrome and Firefox send 600–800 byte GET headers including Accept, User-Agent, cookies etc. A 255-byte buffer will truncate the request; the partial read is then silently treated as a valid HTTP request.

m11 mosquitto_lib_init() return value unchecked; lib_cleanup() called unconditionally lines 997, 1047 ▶

If mosquitto_lib_init() fails, execution continues and subsequent mosquitto calls have undefined behaviour. lib_cleanup() is then called on an uninitialised library.

Overall Verdict

The code is a solid functional prototype that works reliably under normal conditions but is not production-hardened. The core data path (MQTT → buffer → Parquet flush) is architecturally sound. The serious problems are concentrated in the robustness features (WAL, signal handling, shutdown):

  • Any SIGTERM during a flush invokes undefined behaviour (C1)
  • The WAL can silently mis-route recovered data (C3) and does not abort on write failure (C2)
  • Every graceful shutdown discards a window of messages (M11)
  • Health endpoint buffer_rows and disk_free_gb are wrong in default config (C5, M8)
  • Path traversal via MQTT source_type is a real attack surface (M13)
