NEW Critical Issues

CRITICALNEW NC1 — WAL replay timestamp searches r.ints but timestamp is always in r.floats

replay_wal_files() · lines 629–637

collection_ts is supposed to derive the original collection date from WAL rows so that replayed data is written to the correct date partition — not today's partition. The code searches r.ints.find("timestamp"):

int64_t min_ts_ms = INT64_MAX;
for (const auto& r : rows) {
    if (auto it = r.ints.find("timestamp"); it != r.ints.end())  // ← always misses
        min_ts_ms = std::min(min_ts_ms, it->second);
}
// ... min_ts_ms is still INT64_MAX, collection_ts stays nullopt

But timestamp is a flat scalar double in the JSON payload. In parse_payload() it enters the float path and is stored in row.floats["timestamp"] — never in row.ints. So r.ints.find("timestamp") always returns r.ints.end(), min_ts_ms stays INT64_MAX, and collection_ts is always std::nullopt.

The result: make_output_path() falls through to collection_ts.value_or(std::chrono::system_clock::now()) — every WAL replay writes to today's date partition regardless of when the data was collected. A WAL written on Monday and replayed on Tuesday silently writes all its data into Tuesday's partition. DuckDB date-range queries return wrong data with no error.

There is also a units bug in the variable name: min_ts_ms implies milliseconds, but the generator sends Unix timestamps in seconds (e.g. 1700000000.0). If the ints lookup ever were to succeed, interpreting a 1.7-billion-second timestamp as milliseconds would give a date of ~January 20, 1970 (476 hours from epoch).

Fix: 
double min_ts_sec = std::numeric_limits::max();
for (const auto& r : rows) {
    if (auto it = r.floats.find("timestamp"); it != r.floats.end())
        min_ts_sec = std::min(min_ts_sec, it->second);
}
std::optional collection_ts;
if (min_ts_sec < std::numeric_limits::max()) {
    auto dur = std::chrono::duration_cast(
                   std::chrono::duration(min_ts_sec));
    collection_ts = std::chrono::system_clock::time_point{dur};
}
Add #include <limits> if not already present.

CRITICALNEW NC2 — flush_partition writes directly to final path — partial file left on disk after failure

flush_partition() · lines 475–479; do_flush() · line 777; replay_wal_files() · line 650

flush_partition(path, rows, compression) opens path and writes directly — no temp-file-then-rename:

ARROW_ASSIGN_OR_RAISE(auto outfile, arrow::io::FileOutputStream::Open(path));   // creates file
ARROW_RETURN_NOT_OK(parquet::arrow::WriteTable(                                 // ← if this fails:
    *table, arrow::default_memory_pool(), outfile, ...));
//   ↑ returns early on first error — partial .parquet file stays at path

If WriteTable fails partway through (disk full, broken NFS, OOM), a truncated file sits at the final path. The Parquet footer — which contains the schema, row-group offsets, and the magic bytes at the end — has not been written. The next flush cycle creates a new file with a different timestamp, leaving the partial file permanently in the partition directory.

DuckDB scans partition directories with GLOB('**/*.parquet'). When it encounters the partial file it will attempt to read the footer, fail to find it, and either raise an exception that aborts the query or, worse, return corrupt data depending on how the Arrow reader handles the truncated file. This makes a transient I/O error into a permanent query-breaking event.

Compare: both hot_file_path and current_state_path already use the correct temp+rename pattern (path+".tmp" → fs::rename). The main partition flush should do the same.

Fix — move atomic-write responsibility into flush_partition itself: 
arrow::Status flush_partition(const std::string& path,
                               const std::vector& rows,
                               const std::string& compression) {
    if (rows.empty()) return arrow::Status::OK();
    ARROW_ASSIGN_OR_RAISE(auto table, rows_to_arrow_table(rows));
    auto props = parquet::WriterProperties::Builder()
        .compression(to_parquet_compression(compression))->build();

    std::string tmp = path + ".tmp";
    ARROW_ASSIGN_OR_RAISE(auto outfile, arrow::io::FileOutputStream::Open(tmp));
    ARROW_RETURN_NOT_OK(parquet::arrow::WriteTable(
        *table, arrow::default_memory_pool(), outfile,
        static_cast(rows.size()), props));
    ARROW_RETURN_NOT_OK(outfile->Close());

    std::error_code ec;
    fs::rename(tmp, path, ec);
    if (ec) {
        fs::remove(tmp, ec);   // best-effort cleanup
        return arrow::Status::IOError("rename failed: " + ec.message());
    }
    return arrow::Status::OK();
}

Adding outfile->Close() before rename also ensures the OS kernel flushes all buffered writes before the file appears at the final path. The temp file is on the same filesystem so rename is atomic on POSIX.

CRITICALNEW NC3 — fs::remove(tmp) without error_code in failure paths — uncaught exception crashes flush thread

do_flush() · line 819 (hot file) and line 841 (current_state)
// Hot file failure path (line 818–820):
} else {
    std::cerr << "[hot] write failed: " << status.ToString() << "\n";
    fs::remove(tmp);   // ← throws std::filesystem::filesystem_error if remove fails
}

// Current-state failure path (line 840–842):
} else {
    std::cerr << "[cs] write failed: " << status.ToString() << "\n";
    fs::remove(tmp);   // ← same

std::filesystem::remove(path) (the overload without error_code) throws std::filesystem::filesystem_error if it fails — for example, if the filesystem is read-only, the directory is mounted on a different device, or a permissions error occurs. There is no try/catch anywhere in do_flush() or flush_thread_fn(). The exception propagates to std::thread's entry wrapper, which calls std::terminate(). The entire writer process crashes — no more flushing, all buffered data lost, process needs manual restart.

Compare: every other fs::remove call in the file already uses the error_code overload (lines 611, 658, 784). These two are the only exceptions.

Fix — both lines, identical pattern:

fs::remove(tmp, ec);   // best-effort cleanup; failure is non-fatal

CRITICALNEW NC4 — Early flush trigger is broken: notify_one() is dead code; overflow/partition-full doesn't wake flush thread

flush_thread_fn() · lines 849–851; on_message · lines 983–984, 998–999; g_flush_cv · line 727

The m2 fix replaced g_flush_cv.wait_for() with poll() on g_signal_pipe[0] in the flush thread. But the two places that need to wake the flush thread early (overflow guard and per-partition trigger) were not updated — they still call g_flush_cv.notify_one():

// on_message — overflow path (line 983–984):
g_flush_needed.store(true);
g_flush_cv.notify_one();   // ← flush thread is in poll(), not waiting on condvar → no-op

// on_message — per-partition full (line 998–999):
g_flush_needed.store(true);
g_flush_cv.notify_one();   // ← same: no-op

The flush thread's poll() at line 850 is only interrupted by:

  1. Timeout expiry (flush_interval_seconds — 60s by default)
  2. A byte written to g_signal_pipe[1] — only handle_signal() does this

The g_flush_needed sticky flag is only checked at the top of the while loop, before calling poll(). Once poll() is entered, a subsequent g_flush_needed.store(true) from on_message is invisible to the sleeping flush thread. The flag is not seen until the next loop iteration, which starts only after poll() returns.

Consequence at 80k msg/sec: with default settings (max_total_buffer_rows=2,000,000, flush_interval_seconds=60), the overflow guard fires at second 25. If the flush thread started its 60s sleep at second 1, it won't wake for another 35 seconds. During those 35 seconds, every incoming message is dropped. That is 35 × 80,000 = 2.8 million dropped rows per overflow event, with zero data written to disk during that time.

The g_flush_cv condition variable (line 727) is declared and called but never waited on anywhere in the post-m2 code. It is dead code.

Fix — write to the signal pipe when requesting an early flush, and drain the pipe at the start of each flush cycle: 
// Replace both notify_one() calls in on_message with a pipe write:
g_flush_needed.store(true);
{ char b = 1; (void)::write(g_signal_pipe[1], &b, 1); }  // async-signal-safe; wakes poll()

// Remove g_flush_cv declaration entirely (it is dead code).

// In flush_thread_fn, drain the pipe before processing each cycle
// so the next iteration's poll() doesn't return instantly:
static void flush_thread_fn() {
    while (!g_shutdown) {
        if (!g_flush_needed.exchange(false)) {
            struct pollfd pfd = { g_signal_pipe[0], POLLIN, 0 };
            poll(&pfd, 1, g_cfg->flush_interval_seconds * 1000);
        }
        // Drain any bytes written by on_message or handle_signal
        { char drain[64]; while (::read(g_signal_pipe[0], drain, sizeof(drain)) > 0) {} }
        // ... lock/swap/flush as before
    }
}

Note: read() on the pipe will return EAGAIN/EWOULDBLOCK when drained (set pipe to O_NONBLOCK via pipe2(g_signal_pipe, O_NONBLOCK) in main, or use fcntl after pipe()). Without O_NONBLOCK the drain loop would block. Alternatively, use a self-pipe pattern where exactly one byte is written per wakeup request and the byte is consumed in the flush thread.

NEW Major Issues

MAJORNEW NM1 — g_flush_cv declared and notified but never waited on — dead code with side effects

Line 727 (declaration); lines 984, 999 (notify_one calls)

The condition variable was the original wakeup mechanism for the flush thread. The m2 fix replaced the wait_for() call with poll() but left the condvar and both notify_one() calls in place. The notify calls execute under g_mutex (held by on_message's std::lock_guard) at 80k msg/sec. Calling notify_one() on a condvar that nothing is waiting on is a no-op at the OS level, but the call itself still has overhead — it acquires an internal mutex inside the condvar implementation. At 80k msg/sec this adds measurable contention on the hot path.

Fix:

Remove the static std::condition_variable g_flush_cv; declaration and both g_flush_cv.notify_one(); calls. Replace with the pipe-write fix from NC4.

MAJORNEW NM2 — FD_SET in health thread: undefined behaviour if socket fd ≥ 1024

health_thread_fn() · lines 680–682
fd_set fds; FD_ZERO(&fds); FD_SET(srv, &fds);          // ← UB if srv >= FD_SETSIZE
timeval tv{1, 0};
if (select(srv + 1, &fds, nullptr, nullptr, &tv) <= 0) continue;

FD_SETSIZE is typically 1024 on Linux. FD_SET(fd, &fds) is defined as a bit-array index operation — if fd >= FD_SETSIZE, it writes past the end of the fd_set struct, corrupting adjacent stack memory. In a long-running process or one that opens many files (e.g., Arrow opens file handles per partition), the health server socket can easily receive an fd above 1024.

Fix — replace select with poll (which has no fd limit): 
struct pollfd srv_pfd = { srv, POLLIN, 0 };
if (poll(&srv_pfd, 1, 1000) <= 0) continue;   // 1-second timeout, same as before

MAJORNEW NM3 — load_config(): a YAML type error silently discards all subsequent config fields

load_config() · lines 156–217 · outer try/catch

All YAML parsing happens inside a single try/catch block. A single bad value — e.g., mqtt.port: "one-eight-eight-three" — causes m["port"].as<int>() to throw YAML::BadConversion. The outer catch prints one line and returns the partially-initialised Config with default values for every field that was not yet reached.

try {
    // ... parse mqtt.host ... (ok)
    cfg.mqtt_port = m["port"].as();   // ← throws here
    // ... parse client_id, topic, qos, base_path, wal, health ...  ← NEVER REACHED
} catch (const YAML::Exception& e) {
    std::cerr << "[config] " << e.what() << " — using defaults\n";
}

validate_config() may still pass because the defaults for base_path, compression, and flush_interval are all valid. The writer starts but ignores the operator's WAL, health, and guard settings. A typo in the MQTT section silently disables crash recovery. This could cause silent data loss in production with no indication of the root cause other than the one-line config warning (which may scroll off a systemd journal).

Fix — per-field try/catch with a helper: 
// Macro or lambda for graceful per-field parse:
auto safe_as = [&](auto node, auto& field) {
    try { if (node) field = node.as(); }
    catch (const YAML::Exception& e) {
        std::cerr << "[config] " << e.what() << " — field ignored, keeping default\n";
    }
};
safe_as(m["port"], cfg.mqtt_port);
safe_as(m["host"], cfg.mqtt_host);
// ... etc — every field parsed independently

MAJORNEW NM4 — SensorKey includes all row.ints — current_state map grows unboundedly if payload contains int fields

on_message() · line 990
if (!g_cfg->current_state_path.empty())
    (*g_current_state)[SensorKey{info_opt->source_type, row.ints}] = row;

row.ints contains all integer columns: topic-derived identity fields (site_id, rack_id, module_id, cell_id) and any integer-typed payload fields. If a payload contains even one integer field that varies per message (e.g., a sequence counter, a status code, a fault code), the SensorKey is unique for every message rather than for every physical sensor.

Instead of one entry per sensor (the design intent), the current_state map accumulates one entry per distinct reading — effectively a secondary buffer that grows without bound. At 80k msg/sec with one varying int field this means ~80,000 new map entries per second, consuming ~5–10 MB/s of RAM until the writer OOMs.

Fix — build SensorKey from only the topic-derived columns (those in cfg.topic_kv_map): 
if (!g_cfg->current_state_path.empty()) {
    // Key uses only topic-identity fields, not payload-derived ints
    std::map id_ints;
    id_ints["project_id"] = row.ints.count("project_id") ? row.ints.at("project_id") : 0;
    for (const auto& [_, col] : g_cfg->topic_kv_map)
        if (row.ints.count(col)) id_ints[col] = row.ints.at(col);
    (*g_current_state)[SensorKey{info_opt->source_type, id_ints}] = row;
}

Future Architecture — Async Output Queue

Motivation

Torture-test observation (2026-03-21, 330k msg/s × 4 writers): writer-b hit flush_max_ms=2614ms with ring_drops=5M and overflow_drops=195k. Root cause: even though flush_thread releases g_mutex before disk I/O, g_buffer refills and exceeds max_total_buffer_rows during a 2.6 s parquet write. The slow disk write directly sets the ceiling on sustainable input rate.

Proposed Design

Decouple the flush decision from the actual disk write by inserting an output queue:

parse_thread → g_buffer → flush_thread → [output queue] → disk_writer_thread → parquet
                          (fast: slice      (Arrow tables,
                           buffer, build     in-memory,
                           Arrow table,      max depth N)
                           enqueue & return)

flush_thread becomes very fast:

  1. Acquire g_mutex, move rows to local vectors, release mutex.
  2. Build Arrow table from rows (CPU-only, no I/O).
  3. Push FlushJob{key, table, output_path} onto output queue.
  4. Return immediately — g_buffer is free for new data.

disk_writer_thread drains the queue:

  1. Pop FlushJob from queue.
  2. Write Arrow table to parquet (slow I/O, no mutex held).
  3. On success: delete WAL file if applicable. On failure: retain for recovery.

Memory Cost

At torture-test load: last_flush_rows = 321k rows × ~200 B/row Arrow ≈ 64 MB per queued table. With a max queue depth of 4 jobs: ~256 MB peak. Trivial on phil-256g (256 GB), acceptable on lp3 (1 GB — shallower cap of 2 jobs recommended).

Backpressure: if the queue is full (disk persistently slow), flush_thread blocks on enqueue. This propagates disk pressure up through g_buffer → ring → broker QoS-0 drops, rather than silently discarding data mid-pipeline.

Implementation Sketch

struct FlushJob {
    PartitionKey                  key;
    std::shared_ptr<arrow::Table> table;
    std::string                   output_path;
    std::string                   wal_path;    // empty if WAL disabled
};

struct OutputQueue {
    std::deque<FlushJob>    jobs;
    std::mutex              mu;
    std::condition_variable cv_consumer;
    std::condition_variable cv_producer;
    size_t                  max_depth = 4;
    std::atomic<bool>       shutdown{false};

    void push(FlushJob j) {
        std::unique_lock lk(mu);
        cv_producer.wait(lk, [&]{ return jobs.size() < max_depth || shutdown; });
        jobs.push_back(std::move(j));
        cv_consumer.notify_one();
    }
    bool pop(FlushJob& out) {   // returns false on shutdown+empty
        std::unique_lock lk(mu);
        cv_consumer.wait(lk, [&]{ return !jobs.empty() || shutdown; });
        if (jobs.empty()) return false;
        out = std::move(jobs.front()); jobs.pop_front();
        cv_producer.notify_one();
        return true;
    }
};

Relationship to SHM Input Ring

The output queue mirrors the SHM input ring — one absorbs burst on ingress, the other on egress:

[MQTT broker]
     │ QoS-0
     ▼
[SHM input ring]       ← absorbs broker burst; survives writer restart
     │
     ▼
[parse_thread → g_buffer → flush_thread]
     │
     ▼
[output queue]         ← absorbs disk I/O burst; decouples flush from write
     │
     ▼
[disk_writer_thread → parquet files]

Together they sandwich the writer core and isolate it from both input and output timing variance.

Priority & Short-Term Workaround

Implement after the SHM input ring (SHM-1 through SHM-6). Once the full input→output latency budget is visible from live instrumentation, the correct max_depth and thread count can be set empirically.

Short-term: increase max_total_buffer_rows from 500k → 2M in writer configs to buy more headroom against slow flushes without the architectural change.
