Critical Issues

CRITICALR3-C1 · SIGPIPE can kill the writer process

health_thread_fn() · line ~749 · send() calls

The health endpoint uses raw send() without MSG_NOSIGNAL. If the monitor client closes the connection between the writer's accept() and send() — which happens frequently with browser polling — the kernel delivers SIGPIPE to the process. The default SIGPIPE disposition is terminate. SIGINT and SIGTERM are handled; SIGPIPE is not. This means a single ill-timed health poll can silently kill a writer that is mid-flush. 

// Current (crashes on client disconnect):
send(cli, resp.c_str(), resp.size(), 0);

// Fix A — add MSG_NOSIGNAL to every send():
send(cli, resp.c_str(), resp.size(), MSG_NOSIGNAL);

// Fix B — one-shot in main() before threads start (simpler):
signal(SIGPIPE, SIG_IGN);

Fix B is one line in main() and covers all future send() calls too. Apply both for defence-in-depth.

CRITICALR3-C2 · Variable shadowing silently corrupts WAL replay log path

replay_wal_files() · line ~624 vs ~632

The outer scope declares std::string wp (the WAL directory path). The range-for loop at line 632 re-uses the same name: for (const auto& wp : wal_files) where wal_files is a std::vector<fs::path>. The inner wp shadows the outer one for the entire loop body. The outer wp (directory path) becomes inaccessible inside the loop. The code happens to work today because the outer wp is not used inside the loop body after shadowing, but it is one refactor away from silently using the wrong value. 

// Current — outer 'wp' (string) shadowed by inner 'wp' (fs::path):
std::string wp = cfg.wal_path.empty() ? cfg.base_path + "/.wal" : cfg.wal_path;
...
for (const auto& wp : wal_files) {          // ← shadows outer wp!
    auto rows = wal_replay_file(wp.string());

// Fix — rename loop variable:
for (const auto& wf : wal_files) {
    auto rows = wal_replay_file(wf.string());

Major Issues

MAJORR3-M1 · mosquitto_loop_stop(false) can block 30 s on shutdown

main() · line ~1305 · mosquitto_loop_stop(mosq, false)

force=false waits for the network thread to exit cleanly. If the MQTT broker is unreachable at shutdown time, the network thread is sleeping inside the reconnect backoff (up to mqtt_reconnect_max_seconds = 30s). The result: pressing Ctrl-C appears to hang for 30 s before the process exits. Confusing in production, breaks Docker stop timeout (default 10s), and delays the final flush. 

// Current:
mosquitto_loop_stop(mosq, /*force=*/false);

// Fix — force=true: network thread is stopped immediately.
// mosquitto_destroy() cleans up properly regardless.
mosquitto_loop_stop(mosq, /*force=*/true);

MAJORR3-M2 · handle_sync_message holds g_mutex during simdjson parse

process_raw_message() → handle_sync_message() · line ~1010–1013

process_raw_message() acquires g_mutex and then calls handle_sync_message() which runs a full simdjson parse inside the lock. simdjson is fast (~100 ns), but the parse still extends the mutex hold window unnecessarily. At high rates, every extra µs under g_mutex adds to the latency seen by the flush thread (which must also acquire it to swap buffers). The fix: parse first, then acquire the lock only to update the counters. 

// Current — parse under lock:
if (topic == "_sync") {
    std::lock_guard<std::mutex> lock(g_mutex);
    handle_sync_message(payload);   // simdjson parse here!
    return;
}

// Fix — parse outside lock, acquire only for counter update:
if (topic == "_sync") {
    handle_sync_message(payload);   // parse outside lock
    return;
}
// Inside handle_sync_message(), acquire lock only for the
// g_received_since_sync / g_sync_drops_detected updates.

MAJORR3-M3 · ValueOrDie() deprecated in Arrow 14+ / warning in Arrow 16

wal_replay_file() · lines ~582–584

ValueOrDie() was deprecated in Apache Arrow 14.0 and replaced by .Value() (which also asserts on error) or ARROW_ASSIGN_OR_RAISE (which propagates errors). The new Gumby SDK 0.11.0 ships Arrow 16.0.1 — ValueOrDie() will generate deprecation warnings or fail to compile with -Werror. WAL replay is startup-critical; these calls should not silently fall back. 

// Current:
auto r_in = arrow::io::ReadableFile::Open(path);
if (!r_in.ok()) return rows;
auto r_rd = arrow::ipc::RecordBatchFileReader::Open(r_in.ValueOrDie());  // ← deprecated
if (!r_rd.ok()) return rows;
auto reader = r_rd.ValueOrDie();                                          // ← deprecated

// Fix — use .Value() (asserts on error, no-warning, same semantics):
auto r_in = arrow::io::ReadableFile::Open(path);
if (!r_in.ok()) return rows;
auto r_rd = arrow::ipc::RecordBatchFileReader::Open(r_in.Value());
if (!r_rd.ok()) return rows;
auto reader = r_rd.Value();

Minor Issues

MINORR3-N1 · parse_thread_fn spins at 50 µs on empty ring

parse_thread_fn() · line ~1078

When the ring is empty the parse thread sleeps 50 µs and retries. At very low throughput (idle, early startup) this burns ~2% of a core continuously. On Gumby Cortex-A55 targets where every milliwatt matters, it will drain battery during idle periods. The flush thread already uses a poll() on the signal pipe — the same mechanism could be used here (producer writes 1 byte to pipe; parse thread polls the read end with a 100ms timeout as a backstop). 

// Current:
std::this_thread::sleep_for(std::chrono::microseconds(50));

// Future fix: poll() on a pipe written by on_message().
// Low-priority because at target rates (80k–1M msg/s) the ring
// is rarely empty; this only matters at idle.

MINORR3-N2 · Dual tracking of buffer row count (g_total_buffered + g_health_buffer_rows)

process_raw_message() · line ~1060; do_flush() · line ~915

g_total_buffered is a plain size_t under g_mutex. g_health_buffer_rows is a separate std::atomic<uint64_t> that mirrors it. Both must be updated in sync; they diverge if either site is missed in future edits. A single atomic is sufficient since the health thread only reads it and does not need to participate in the mutex-protected overflow decision. The duplication is low-risk today but is a maintenance hazard.

MINORR3-N3 · Health endpoint lacks SHM ring metrics

health_thread_fn() · JSON body construction · line ~727

The health JSON does not yet include shm_ring_used, shm_ring_drops, or shm_ring_cap. These are needed by the torture monitor once the SHM ring is implemented (SHM-2). Plan now, implement alongside shm_ring.h.

Performance Issues

PERFR3-P1 · parse_topic allocates std::istringstream on every message

parse_topic() · line ~277

std::istringstream heap-allocates an internal buffer on construction. At 300k msg/s this means 300k heap allocations per second on the hot path, before any Row construction. Profiling on the torture server showed the parse thread at ~15% CPU — replacing istringstream with a pointer-walking split over the existing C string saves both the allocation and the std::string copy of the topic. 

// Current:
std::istringstream ss(topic);       // heap alloc
std::string segment;
while (std::getline(ss, segment, '/')) { ... }

// Fix — walk the existing char* from the ring slot directly:
const char* p = topic_cstr;
const char* end = p + strlen(p);
while (p <= end) {
    const char* slash = (const char*)memchr(p, '/', end - p);
    const char* seg_end = slash ? slash : end;
    std::string_view seg(p, seg_end - p);
    // ... process seg ...
    p = slash ? slash + 1 : end + 1;
}

Estimated 3–5× speedup on the parse path; removes ~300k allocations/s at peak rate.

PERFR3-P2 · Row uses std::map — 3M+ tree allocations/second at 300k msg/s

struct Row · line ~227; parse_payload · line ~300+

struct Row uses std::map<std::string, int64_t> and std::map<std::string, double>. Each map insert allocates a red-black tree node (typically 64–96 bytes on the heap). With ~5 int fields + ~10 float fields per row at 300k msg/s = 4.5M heap allocations/second just for Row construction.

At the torture test throughput (895k msg/s total) this approaches 15M allocations/second. Allocator contention between the parse thread and the flush thread (which frees them) will show up as mutex contention inside malloc at high rates. 

// Short-term: switch to std::unordered_map (2-3× faster, same interface):
struct Row {
    std::unordered_map<std::string, int64_t> ints;
    std::unordered_map<std::string, double>  floats;
};

// Long-term: flat struct with known field positions eliminates
// all per-row heap allocation (see SHM ring slot design).

This is the single highest-leverage change for sustaining 1M+ msg/s on the Gumby ARM targets.

Shutdown Sequence Analysis

Current shutdown order (annotated)

SIGINT/SIGTERM
  → g_shutdown = true
  → write(g_signal_pipe[1], 1)       // wakes flush_thread's poll()

flush_thread:  last flush → exits while loop
flusher.join()                        // ✓ wait for last flush
health_thr.join()                     // ✓ exits within 1s (polls g_shutdown)

mosquitto_disconnect(mosq)
mosquitto_loop_stop(mosq, false)      // ⚠ blocks up to 30s if broker down
                                      //   (should be force=true)
g_mqtt_stopped = true
parser.join()                         // ✓ drains ring then exits

final flush in main()                 // catches msgs buffered during drain window
mosquitto_destroy / lib_cleanup

The sequence is logically correct. The only flaw is the 30s blocking risk at step 5 (R3-M1).
