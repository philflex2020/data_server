Summary

The parquet_writer stack — FlashMQ + parquet_writer + DuckDB + subscriber_api — covers the same use case as a managed industrial time-series platform at a fraction of the cost and with better data portability. The key advantages are:

  • Time to first data: hours, not months — no licensing, no Windows server, no consultant engagement
  • Cost: ~$300-1,000/month vs $10,000-70,000+/month for equivalent managed platforms
  • Open data format — plain Parquet files readable by any tool; no vendor lock-in
  • COV and MON_IGN compression — first-class concepts, not afterthoughts
  • Edge-native — single ~6 MB binary, runs on ARM, no Windows dependency
  • Schema evolution is free — add signals without migrations

COV and Compression — Key Differentiator

Commercial platforms store every data point at every scrape interval regardless of whether the value has changed. At 1 Hz with 63 signals across 12 units, that is ~4.5 million data points per hour for a resting BESS where most signals are stable for hours at a time.

The parquet_writer stack applies compression at three layers:

LayerMechanismWhat it compresses
Publisher COV (--cov) Float: suppress if change <1%. Integer: on-change only. MQTT traffic — slow signals only publish when they move
Publisher COV reset (--cov-reset 300) Clear COV cache every N seconds — force republish of all signals Ensures constants appear at least every 5 minutes for query coverage
Writer null_fill_unchanged Null out columns that haven't changed since last flush Parquet null runs compress to near-zero bytes
Writer COV planned Per-signal threshold, default + exceptions Buffer pressure — constants never enter the buffer after first write
MON_IGN planned Suppress monotonic increments, record backward jumps only Heartbeat counters produce zero rows during normal operation
Snappy / zstd column compression Parquet columnar encoding + codec Repeated values, null runs, sorted timestamps

PI System's exception reporting achieves similar compression but only within the historian's proprietary storage format. InfluxDB and cloud TSDB platforms have no equivalent — every Telegraf scrape writes every value.

Data Portability

Parquet is an open columnar format supported natively by every major analytics platform. Data written by parquet_writer can be queried directly — no export, no transform — by:

  • DuckDB — embedded in subscriber_api; zero deployment cost
  • Apache Spark — direct read from S3 or HDFS
  • AWS Athena — serverless SQL over S3 parquet; pay per query
  • pandas / pyarrow — pd.read_parquet()
  • Snowflake, BigQuery, Databricks — external tables over S3
  • Apache Arrow Flight — streaming query over the wire
  • Excel / Power BI — via ODBC or direct parquet import

Commercial historian data requires vendor-specific export tools, proprietary connectors, or expensive middleware (PI Integrator, AVEVA Connect) to reach analytics platforms. PI archives in particular are binary-proprietary — if OSIsoft/AVEVA cease trading or change their licensing terms, access to historical data depends on their software remaining available.

Longevity: Parquet files written today will be readable in 20 years by tools that don't exist yet. The format is an open Apache standard. PI archive files from 2005 require a licensed PI installation to read.

The parquet_writer stack does not yet match commercial platforms in every dimension. Honest gaps:

Managed operations Vendor SLA, on-call, patching Self-managed Systemd watchdog, guard disk threshold, health endpoint for monitoring
Compliance / audit trail PI data integrity certificates, 21 CFR Part 11 No certified integrity layer Append-only S3 + object lock for immutability; not yet certified
Sub-second push to dashboards PI WebSocket, InfluxDB subscriptions 60s flush gap (mitigated by /current endpoint) MQTT WebSocket path provides true real-time; /current closes history gap
For regulated industries (pharmaceutical, nuclear, defence) requiring data integrity certification, PI System remains the appropriate choice despite its cost and complexity. For BESS, grid storage, and industrial IoT where the regulatory requirement is operational not compliance-driven, the parquet_writer stack is the better option.
