Parquet Writer — Compact Process Design

How flush files are merged, what survives, and how the two compact paths interact — source/parquet_writer/parquet_writer.cpp

Overview

The parquet writer produces one flush file per partition per flush interval. Over time these accumulate into many small files. The compact process merges them into a single compact_*.parquet file per directory, reducing file count and improving query performance.

There are two independent compact paths:

PathWhereHow triggeredSuitable for
C++ compact thread parquet_writer.cpp
compact_thread_fn()		 Automatic background thread, runs every compact_interval_seconds Live production / bench writers — runs continuously
Python compact script scripts/compact_parquet.py Manual CLI invocation only — not automated One-off cleanup, post-run compaction of stopped writers

⚠ Do not run compact_parquet.py on a directory where the C++ compact thread is active — no locking exists between them.

File Naming Conventions

PatternTypeExampleEligible for compaction?
YYYYMMDDTHHMMSSZ.parquet Flush file 20260411T130051Z.parquet ✓ Yes — primary compaction source
{unit_id}_YYYYMMDDTHHMMSSZ.parquet Partitioned flush file 0215D1E1_20260411T130051Z.parquet ✓ Yes — digit-prefix detected
compact_YYYYMMDDTHHMMSSZ.parquet Compact output compact_20260411T130851Z.parquet ✗ No — final, never re-compacted fixed Apr 2026
compact_*.parquet.tmp In-progress write compact_20260411T130851Z.parquet.tmp ✗ No — skipped (temp file)

Flush files are identified by a leading digit in the filename. The C++ thread uses this to distinguish them from compact outputs.

C++ Compact Thread — Configuration

Configured under the compact: key in the writer YAML config. All fields are optional.

compact:
  enabled: true                 # false = thread not started (default: false)
  interval_seconds: 600         # how often the thread scans for eligible files (default: 600)
  min_files: 3                  # minimum files in a directory to trigger compaction (default: 3)
  min_age_seconds: 0            # min file age to be eligible; 0 = 2 × flush_interval_seconds
  log_path: ""                  # JSON-lines compact log; default = {base_path}/compact.log

Config Parameters

ParameterDefaultEffect
enabled false Must be true to start the compact background thread. Safe to leave off for writers where external compaction is preferred.
interval_seconds 600 Sleep interval between scan cycles. Lower values mean more frequent compaction but higher CPU overhead.
min_files 3 A directory is skipped unless it contains at least this many eligible flush files. Prevents compacting a single file needlessly.
min_age_seconds 0 Only files older than this are eligible. 0 auto-sets to 2 × flush_interval_seconds, ensuring the active flush file is never touched.
log_path {base_path}/compact.log Append-only JSON-lines log. One entry per directory compacted. See log format below.

C++ Compact Thread — Execution Flow

Per-cycle steps

  1. Sleep for interval_seconds (interruptible on shutdown)
  2. Compute cutoff = now − min_age_seconds
  3. Walk base_path recursively — collect all .parquet files where:
    • filename starts with a digit (flush file)
    • file mtime < cutoff
    Group by parent directory.
  4. For each directory: skip if file_count < min_files
  5. Call compact_directory() on eligible directories

Atomicity & safety

  • Output is written to a .tmp file then renamed — readers never see a partial compact
  • Source files are only deleted after the output is verified (row count check)
  • If any source file fails to read, the entire directory is skipped — no partial compaction
  • If verification fails, source files are kept and the corrupt output is left for inspection

Compact File Behaviour Updated Apr 2026

Prior to April 2026, compact files were included as sources in subsequent compaction cycles (is_compact flag). This caused each compact to replace the previous one, with no accumulation of history. The directory would cycle endlessly with the same row count.

Before fix (behaviour)

83 flush files
+ compact_T1
→
compact_T2 written
→
compact_T1 deleted ✗

Result: compact files were temporary — replaced every interval_seconds. Row count stayed constant.

After fix (current behaviour)

83 flush files
(compact_T1 ignored)
→
compact_T2 written
→
flush files deleted
→
compact_T1 survives ✓

Result: compact files accumulate — one per compaction cycle. The directory grows, preserving full history.

Code change

- bool is_compact = fname.size() > 8 && fname.substr(0, 8) == "compact_";
- if (!is_flush && !is_compact) continue;  // compact files were re-consumed
+ // compact_ files are final — never re-compact them
+ if (!is_flush) continue;

Commit 6ad3bf0 on branch 03202026_run_with_real_data.

Compact Log Format

Each compaction appends one JSON line to {base_path}/compact.log:

{
  "ts":           "20260411T130851Z",   // UTC timestamp of compact run
  "schema":       "pw-bench-wide",      // writer schema/client_id
  "dir":          "bench/2026/04/11",   // directory relative to base_path
  "sources":      84,                   // number of source files consumed
  "source_bytes": 1373560,              // total bytes of source files
  "rows":         5028,                 // rows in compact output
  "size_bytes":   377456,               // compressed output size
  "null_pct":     30.8,                 // null density (wide format expected)
  "out":          "compact_20260411T130851Z.parquet"
}

The log is append-only and never rotated. Monitor it to verify compaction is running and healthy.

Python compact_parquet.py — Manual Script

Located at scripts/compact_parquet.py. Run manually for one-off compaction after a writer has stopped, or for initial cleanup of legacy data.

python3 scripts/compact_parquet.py <base_dir> [--dry-run] [--chunk-minutes N] [--compression snappy|zstd|gzip|none]

# Examples:
python3 scripts/compact_parquet.py /mnt/tort-sdi/bench/bench-wide
python3 scripts/compact_parquet.py /mnt/tort-sdi/bench/bench-norm-long --dry-run
python3 scripts/compact_parquet.py /mnt/tort-sdi/bench/bench-wide --compression zstd
OptionDefaultEffect
--dry-runoffPrint what would be done; no files written or deleted
--chunk-minutes N0Group files into N-minute chunks; 0 = one file per leaf directory
--compressionsnappyOutput compression codec

Between compaction cycles the directory contains the previous compact files plus flush files that have not yet reached min_files or min_age_seconds.
