Writer Schema Comparison Test

Benchmarks three parquet layout strategies against the same MQTT stream — norm-long, long+compound, wide-pivot (coalesced)

Test Topology

Units
12
hex IDs 0215D1D8 … 0215D1E3
Signals / unit
34
24 float + 10 integer
Sweep rate
1 Hz
600 sweeps = 10 min
Total messages
244,800
12 × 34 × 600

Signal set

bms/bms_1:  Batt1_Cell[1-10]_Voltage  Batt1_Cell[1-5]_Temperature
            Pack_Current  Pack_Voltage  Pack_SOC  Pack_SOH
            Pack_ContactorState  Batt1_CellBalancing  BMS_FaultCode  BMS_WarningCode
pcs/pcs_1:  GridFrequency  ActivePower  ReactivePower  DCBusVoltage  OutputCurrent
            PCS_State  PCS_FaultCode  GridConnected  AlarmActive
rack/rack_1: Rack_FanState  Rack_DoorOpen

Burst model

Each sweep publishes all 34 signals per unit in a randomised burst of 50–200 ms (well inside the time_window_ms: 250 coalesce window). The wide writer coalesces each burst into one row — target ~1 row/sweep/unit.

Three Writer Schemas

norm-long
One row per signal per sweep.
8 columns: ts, value, device, device_type, dtype_hint, point_name, unit_id, site
~248k rows · 1.6 MB · 3–10 ms flush
long+cmp
One row per signal. Point name compounded: unit/device/instance/signal.
5 columns: ts, value, dtype_hint, point_name, site
~248k rows · 1.7 MB · 2–5 ms flush
wide-pivot
One row per sweep per unit. Each signal is a column. Burst-coalesced.
38 columns: ts + 34 signals + dtype_hint + unit_id + site
~7.2k rows · 4.1 MB · 19–36 ms flush

Setup on Target Machine (.34)

1
Install systemd service
Copy service file and reload systemd
cd ~/data_server
git pull
sudo cp scripts/writer-compare-test.service /etc/systemd/system/
sudo systemctl daemon-reload
2
Verify output drive is mounted
The script auto-detects /mnt/tort-sdi on .34
df -h /mnt/tort-sdi
# should show ~870 GB free
ls /mnt/tort-sdi/
3
Start FlashMQ broker
sudo systemctl start flashmq
sudo systemctl status flashmq
4
Run the test
Via systemctl (recommended — captured in journal)
sudo systemctl start writer-compare-test
journalctl -u writer-compare-test -f
Or directly (useful for debugging)
bash scripts/writer_compare_test.sh --sweeps 600 --outdir /mnt/tort-sdi/bench
5
View results
Results JSON written to the output dir. Fetch it here using the panel below.
cat /mnt/tort-sdi/bench/results.json
ls  /mnt/tort-sdi/bench/wide/

Run Without systemctl

For a quick ad-hoc run with custom options:

# standard 600-sweep run
bash scripts/writer_compare_test.sh

# change-on-value mode (realistic publish gaps)
bash scripts/writer_compare_test.sh --cov

# short smoke test
bash scripts/writer_compare_test.sh --sweeps 60

# explicit output dir (e.g. secondary tort drive)
bash scripts/writer_compare_test.sh --outdir /mnt/tort-sdf/bench

Script auto-detects the repo root, resolves the writer binary, installs Python deps into .venv, starts FlashMQ if needed, generates all three YAML configs, runs compare_run.py, and writes results.json + status.json to the output dir.

What to Look For

AspectFavoursWhy
Storage sizelong+cmpColumn compression works better on high-cardinality repeated strings in one column
Flush latencylong+cmpNo pivot/coalesce work; writer just appends rows
Single-unit range querywide-pivotAll signals in one row — no UNPIVOT or GROUP BY needed
Coalesce qualitywide-pivot1 row/sweep/unit = perfect if bursts stay inside time_window_ms
File countnorm/longWide partitions per unit_id → 12× more files
Schema flexibilitynorm-longAdding new signals = new rows only; wide needs new column (schema evolution)
