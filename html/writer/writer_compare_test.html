Writer Schema Comparison Test

Benchmarks four storage strategies against the same MQTT stream — norm-long, long+compound, wide-pivot (coalesced), and InfluxDB2 via Telegraf

Time coalescing — an optional wide-pivot feature controlled by time_window_ms in the writer config. When enabled, all MQTT messages arriving within the same time window (e.g. 250 ms) are assigned an identical normalised timestamp and then merged into a single row, one column per signal. The result is one dense row per sweep rather than one sparse row per message. Without coalescing each message produces its own row with all other signal columns null. Set time_window_ms: 0 to disable.
▶ Signal Filtering — 5 signals dropped (true monotonics only); 30 constants retained via null_fill

When running with --csv, signals are pre-filtered before being published to MQTT. Only true monotonics (unbounded counters, heartbeats, internal timestamps) are dropped at the writer level — they have maximum entropy and actively harm compression. Constants are now retained and compressed by null_fill_unchanged, which writes null for any column whose value hasn't changed since the last flush. An hourly reset (null_fill_reset_interval_seconds: 3600) ensures each compact window is self-contained. Identified from 200-sample analysis of SBESS3 UC 61B5 real data.

Burst model

Each sweep publishes all 63 analytical signals per unit in a randomised burst of 50–200 ms (well inside the time_window_ms: 250 coalesce window). The wide writer coalesces each burst into one row — target ~1 row/sweep/unit.

Writer Schemas

norm-long
One row per signal per sweep.
8 columns: ts, value, device, device_type, dtype_hint, point_name, unit_id, site
~248k rows · 1.6 MB · 3–10 ms flush
long+cmp
One row per signal. Point name compounded: unit/device/instance/signal.
5 columns: ts, value, dtype_hint, point_name, site
~248k rows · 1.7 MB · 2–5 ms flush
wide-pivot
One row per sweep per unit. Each signal is a column. Burst-coalesced.
~68 columns: ts + 63 signals + dtype_hint + unit_id + site; constants null_fill_unchanged
~7.2k rows · 4.1 MB · 19–36 ms flush
influx2 / telegraf
Narrow tag schema. One row per signal change (COV-filtered by publisher).
measurement=bench · tags: unit_id, device, instance, point_name · field: value
COV-rate rows only · TSM storage · Flux/InfluxQL query
COV behaviour at the MQTT layer

The publisher runs --cov, so only changed values arrive at MQTT. Constant signals (SOH=100, RackCount=8, etc.) publish once on startup then go silent until --cov-reset 300 fires every 5 min. Telegraf stores exactly what it receives — no null-fill, no repeat rows. A Flux fill(usePrevious:true) forward-fills at query time. Storage cost scales with signal change rate, not absolute signal count.

Telegraf config

# /home/phil/data_server/source/telegraf/telegraf-sbess3.conf

[[inputs.mqtt_consumer]]
  servers   = ["tcp://localhost:1883"]
  topics    = ["bench/#"]
  topic_tag = ""          # suppress raw topic tag
  data_format = "json_v2"

  [[inputs.mqtt_consumer.json_v2]]
    measurement_name = "bench"
    timestamp_path   = "ts"
    timestamp_format = "RFC3339"
    [[inputs.mqtt_consumer.json_v2.field]]
      path = "value"; type = "float"

  [[inputs.mqtt_consumer.topic_parsing]]
    topic = "bench/+/+/+/+/+"
    tags  = "_/unit_id/device/instance/point_name/_"

[[outputs.influxdb_v2]]
  urls = ["http://localhost:8086"]; organization = "longbow"; bucket = "sbess3"
  token = "<token>"

Flux query examples

// All signals for one unit, last 10 minutes
from(bucket: "sbess3")
  |> range(start: -10m)
  |> filter(fn: (r) => r.unit_id == "0215D1D8")

// Single signal — forward-fill COV gaps
from(bucket: "sbess3")
  |> range(start: -10m)
  |> filter(fn: (r) => r.unit_id == "0215D1D8" and r.point_name == "SysSOC")
  |> fill(usePrevious: true)

// 1-minute mean for all signals on one unit
from(bucket: "sbess3")
  |> range(start: -1h)
  |> filter(fn: (r) => r.unit_id == "0215D1D8")
  |> aggregateWindow(every: 1m, fn: mean, createEmpty: false)

Parquet wide-pivot vs InfluxDB2

Aspect wide-pivot parquet influx2 / telegraf
Storage model Columnar Parquet on local/NFS disk TSM (Time-Structured Merge tree) on local disk
Row shape 1 row/sweep/unit — 64 columns 1 row/signal/change — 4 tags + 1 field
COV handling Publisher COV + optional null_fill_unchanged Publisher COV only — Influx stores what arrives
Get all signals at time T Single row read — no join 63-series fan-out + fill(usePrevious:true)
Multi-unit range scan One file per unit per day Single measurement, filter by unit_id tag
Schema evolution New column on first message (dynamic) No schema — new point_name = new series
Compression Snappy columnar — excellent for static columns TSM RLE — good for repeated tag combos
Retention / TTL Manual file deletion Built-in retention policy per bucket
Query language DuckDB SQL — portable, familiar Flux (functional) or InfluxQL — Influx-specific
Estimated write CPU ~3–6% (parquet_writer process) telegraf ~1% + influxd ~5–15% during TSM compaction
Portability Standard Parquet — DuckDB, Spark, Pandas, Arrow Influx-native — export needed for external tools
