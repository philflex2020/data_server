Rate Gap Investigation — Stress → Writer Discrepancy

After fixing the client-ID collision, stress publishers reported ~330k msg/s each (×4 = 1.32M/s total) but writers appeared to be receiving only ~200k/s each. Measured rates from two msgs_received samples 5 seconds apart:

Writer msgs_received Δ (5 s) Measured rate
writer-a (8771) 1,105,666 221,133 msg/s
writer-b (8772) 1,116,418 223,284 msg/s
writer-c (8773) 1,118,060 223,612 msg/s
writer-d (8774) 1,137,142 227,428 msg/s
Total ingested 4,477,286 ~895k msg/s

Stress claim: ~1.32M/s  →  Measured ingestion: ~895k/s  →  ~32% loss

Root Cause 1 — Episodic ring overflow during worst-case flushes

The ring holds 262,144 slots. At 330k msg/s input it fills in: 

262,144 / 330,000 = 0.795 seconds until overflow

But observed flush_max_ms was 2,786–3,006 ms across all four writers. During a 3-second flush spike the ring overflows after 0.8 s and drops everything for the remaining ~2.2 s: 

drop window ≈ 3000 ms − 795 ms = 2,205 ms × 330k msg/s ≈ 727k msgs dropped per spike

Between flushes ring_used reads 15–3,422 (near-empty). The drops are entirely episodic — concentrated in worst-case flush stalls, not present in steady state. Cumulative ring_drops at inspection: 177M–381M across the four writers.

Root Cause 2 — mosquitto_publish() counts queued, not delivered

mosquitto_publish() returns success when the message is copied into libmosquitto's internal outgoing buffer — not when the broker acknowledges receipt (QoS-0 has no ack anyway). Under sustained 330k/s load, TCP backpressure can cause the stress publisher to count messages that are buffered but never transmitted before a socket reset. This inflates the publisher-side rate figure by an unknown but non-zero amount.

static void flush_thread_fn() {
    while (!g_shutdown) {
        if (!g_flush_needed.exchange(false)) {
            struct pollfd pfd = { g_signal_pipe[0], POLLIN, 0 };
            poll(&pfd, 1, g_cfg->flush_interval_seconds * 1000);
        }
        // Drain all bytes from the pipe so the next poll() blocks correctly.
        // The read end is O_NONBLOCK (set in main) so this never blocks.
        { char drain[64]; while (::read(g_signal_pipe[0], drain, sizeof(drain)) > 0) {} }
        std::unique_lock<std::mutex> lock(g_mutex);

Why the drain is required: if on_message writes N bytes to the pipe (N early-flush triggers between two flush cycles), those N bytes sit in the kernel pipe buffer. Without draining, poll() returns immediately on every subsequent call (pipe is still readable), causing the flush thread to spin with ~0ms between empty flushes. The O_NONBLOCK flag makes the drain loop terminate at EAGAIN rather than blocking forever on the last read.

DL1 — Signal pipe write end blocking causes deadlock at high publish rate

CRITICAL FIXED One line — pipe creation in main()

main() · pipe init · confirmed in torture test at 400k msg/s

Root cause: process_raw_message writes a 1-byte wakeup to g_signal_pipe[1] while holding g_mutex, every time a partition reaches max_messages_per_part. At 400 k msg/s with 12 partitions, once each partition exceeds the 50 k-row threshold, every subsequent insert triggers a pipe write — ~33 k writes/s per partition, ~396 k writes/s total.

Why the pipe fills: flush_thread only drains the read end inside the poll() branch, which is skipped whenever g_flush_needed is true. At sustained high load g_flush_needed is always true, so the pipe is never drained. The 65,536-byte kernel pipe buffer fills in ~165 ms.

Deadlock sequence:

  1. Partitions refill 125 ms after flush; every new insert calls write(pipe, 1) under g_mutex.
  2. After ~165 ms the blocking write() stalls — parse_thread is frozen holding g_mutex.
  3. flush_thread finishes its parquet write and tries to acquire g_mutex — blocked.
  4. flush_thread never drains the pipe; parse_thread never releases the mutex. Deadlock.

Observed symptom: ring_used pinned at 262 144 (full), ring_drops accumulating at ~400 k/s, msgs_received frozen, mosquitto network thread burning one full CPU core. Writers appeared to process only ~14 k msg/s (a brief burst before the first deadlock).

Before — only read end is non-blocking: 
    if (pipe(g_signal_pipe) != 0) { ... }
    // O_NONBLOCK on read end: drain loop in flush_thread_fn won't block after wakeup
    fcntl(g_signal_pipe[0], F_SETFL, O_NONBLOCK);
After — both ends non-blocking: 
    if (pipe(g_signal_pipe) != 0) { ... }
    // O_NONBLOCK on both ends: read end so drain loop won't block after wakeup;
    // write end so process_raw_message's signal write never blocks while holding
    // g_mutex (at 400k/s the pipe fills in ~165ms during a flush, causing deadlock).
    // Dropping a wakeup byte is safe — g_flush_needed atomic tracks actual need.
    fcntl(g_signal_pipe[0], F_SETFL, O_NONBLOCK);
    fcntl(g_signal_pipe[1], F_SETFL, O_NONBLOCK);

Why this is safe: the pipe is only a wake-up optimisation — g_flush_needed.store(true) is called before each pipe write, so flush_thread always sees the flush request via the atomic regardless of whether the pipe byte was delivered. A dropped EAGAIN write just means flush_thread wakes on its next natural iteration rather than immediately. The existing if (::write(...) < 0) {} call sites already silently ignore write errors, so no other changes are required.

Result after fix: sustained 80–220 k msg/s per writer, zero ring drops, ring near-empty at all times.
