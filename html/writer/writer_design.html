1 · Overview

parquet_writer is a C++17 daemon that consumes a high-rate MQTT telemetry stream and writes it as partitioned Apache Parquet files on local storage. It is designed for battery-energy-storage (BESS) telemetry at industrial scale — sustained rates above 80,000 messages/second — but works for any time-series sensor data with an MQTT/JSON source.

Why Parquet?

Raw MQTT JSON is ~109 bytes per message. The same data stored as Parquet with Snappy compression averages 9.6 bytes per row — an 11.4× compression ratio. Parquet is also columnar, which makes analytical queries (average voltage of all cells in rack 3 over the last hour) orders of magnitude faster than row-oriented formats. Query engines like DuckDB can read these files directly without any database import step.

What it does not do

  • It does not decode or interpret the sensor values — it stores them faithfully.
  • It does not require a running database — output is plain files on disk.
  • It does not require cloud connectivity — designed for edge deployment.

1a · Wide-Pivot Schema Primary Format

The wide-pivot schema is the primary output format for BMS/PCS telemetry. Instead of storing one row per MQTT message (long schema), it groups all signals for a given unit × timestamp into a single row with one column per signal path. This is the most query-efficient layout for time-series analytics where you want all channels for a unit in one scan.

How it works

Configured via compound_field_name. The writer joins the listed topic segments (e.g. [device, instance, point_name]) with / to produce the column name for that message's value. Each row writes exactly one value column; all other signal columns are null for that row. At flush time these sparse rows are combined into a wide table.

COV — Change-on-Value at the publisher

The benchmark publisher supports a --cov flag that simulates realistic COV filtering before publishing:

Signal typeCOV ruleEffect
Float signalsPublish only when value drifts >1% from last sent (COV_FLOAT_PCT = 0.01)Slow-moving signals (resting cell voltage) may publish only every few minutes; fast-changing signals (current during charge/discharge) publish every sweep
Integer / boolean signalsPublish only on changeFault codes and relay states publish rarely in normal operation, frequently during fault events

Without --cov all 34 signals × 12 units publish every 1-Hz sweep (408 msg/s).

null_fill_unchanged — COV at the writer level

The writer's null_fill_unchanged: true config implements COV suppression after ingest — at flush time, not at the publisher. For each partition, any signal column whose value has not changed since the last flush is set to null for all rows in that flush interval. This gives columnar COV compression without requiring publisher-side filtering.

SettingBehaviour
null_fill_unchanged: false (default) Every row carries the full value for every signal column it originated from. No nulls introduced. Most useful when downstream tools can't handle nulls or you need all-values-present guarantees.
null_fill_unchanged: true After the first flush for a partition, columns whose value is identical to the prior flush's last value are replaced with null. On a typical resting BMS (SOC steady, cells not balancing), >70% of float columns may become null — Parquet compresses null runs to near-zero bytes.

null_fill_reset_interval_seconds — the null override

Pure null suppression would create data gaps: a signal that doesn't change for 2 hours would write no value for that entire window. null_fill_reset_interval_seconds (default: 3600) is the null override — it forces a full non-null write for all columns every N seconds, regardless of whether values have changed.

ValueEffect
3600 (default)Guarantees at least one full-value row per signal per hour. A DuckDB last_value(signal) IGNORE NULLS query will find a value within the last hour for any stable signal.
0Null override disabled — a signal that never changes will only appear in the very first flush after startup. Use only when signals are guaranteed to change regularly or when downstream tools always forward-fill nulls.
3005-minute reset — matches a compactor cycle; each compacted file is guaranteed to contain at least one full row for every signal. Good for near-real-time dashboards.
Wide-pivot + COV interaction: when both null_fill_unchanged and publisher-side --cov are active, the writer only sees changed values from the publisher — so null_fill_unchanged adds a second layer of suppression on top. This is fine and compounds the compression benefit. The null_fill_reset_interval_seconds reset still fires at the configured interval to prevent indefinite gaps.

Writer-side COV — per-signal thresholds planned

Publisher-side COV (--cov) requires all publishers to implement the same filtering logic. A cleaner alternative is COV enforcement inside the writer itself — the publisher sends everything, the writer decides what to keep. This also works with hardware devices that have no COV capability.

The design uses a single default threshold with a per-signal exception list, so most signals need no config at all: 

output:
  cov:
    enabled: true
    float_pct: 0.01        # suppress floats if change < 1% (default)
    integer: on_change     # suppress integers unless value changes (default)
    reset_seconds: 300     # force write every 300s regardless — prevents gaps
    exceptions:
      bms/bms_1/SysSOC:
        float_pct: 0.001   # tighter — SOC is operationally important
      pcs/pcs_1/Hz:
        float_pct: 0.005   # tighter — frequency deviation matters
      bms/bms_1/SysTemperature:
        disabled: true     # always write — no COV suppression
ModeRuleUse case
float_pctSuppress if |new−last| / |last| < thresholdAnalogue signals — voltage, current, temperature
on_changeSuppress if new == lastInteger/boolean — fault codes, relay states, mode flags
disabled: trueAlways writeSafety-critical signals that must never be suppressed
mon_ignSee belowMonotonic counters — heartbeats, sequence numbers

MON_IGN — monotonic ignore mode planned

Heartbeat counters and sequence numbers increment every sweep — they always change, so on_change would write every row, and the current workaround is to exclude them entirely via CSV_EXCLUDE. Neither is ideal: exclusion discards the signal completely; on_change defeats compression.

mon_ign (monotonic ignore) keeps the signal in the schema but only records backward jumps — a counter reset, rollover, or unexpected decrement. Normal monotonic increases are silently dropped:

ConditionActionMeaning
new > lastSuppressNormal increment — expected, uninteresting
new < lastWriteCounter reset, rollover, or device restart — meaningful event
new == lastSuppressStalled counter — optionally trigger after stall_seconds
 
output:
  cov:
    exceptions:
      bms/bms_1/BMS_SysHB:
        mode: mon_ign
        stall_seconds: 30    # also write if counter hasn't moved in 30s
                             # catches frozen heartbeat — comms fault
      pcs/pcs_1/Counter:
        mode: mon_ign
      bms/bms_1/ItemDeletionTime:
        mode: mon_ign

With mon_ign a heartbeat counter running normally for 8 hours produces zero rows. A device restart produces one row. A frozen heartbeat (stall_seconds exceeded) produces one row. This is exactly the information you want from a heartbeat — noise gone, fault events preserved.

Migration from CSV_EXCLUDE: signals currently in CSV_EXCLUDE (BMS_SysHB, SysHB, Counter, ItemDeletionTime) are excluded entirely because there is no writer-side MON_IGN yet. Once writer-side COV is implemented, these move from exclusion to mode: mon_ign — they gain a column in the schema and restart/fault events become queryable.

2 · System Architecture

A single writer instance sits between any FlashMQ broker and local storage. It connects as an MQTT subscriber, parses incoming messages, and writes partitioned Parquet files to disk.

EMS Devices / BMS Publishers │ MQTT QoS-0 ▼ ┌──────────────┐ │ FlashMQ │ ← high-performance C++ MQTT broker (any host, any port) │ :1883 │ attach to existing broker — no broker changes required └──────┬───────┘ │ MQTT subscribe # ▼ ┌──────────────┐ │ writer │ ← single process, four internal threads │ :8771 │ health/status HTTP endpoint └──────┬───────┘ │ ▼ /srv/data/parquet/ (date-partitioned Parquet files on local storage)

The writer attaches to whichever FlashMQ instance serves the BMS data — it does not need to run on the same host as the broker, and the broker requires no configuration changes. The mqtt.host and mqtt.port in the writer config point to any reachable FlashMQ instance.

Torture test note: the stress/validation environment runs four writer instances in parallel (writer-a through writer-d on ports 8771–8774), each receiving the full message stream independently — useful for fault-tolerance testing, but not the production deployment model.

3 · Thread Model

The writer uses four threads. Each has a single responsibility and they communicate through lock-free or minimally-locked shared state.

① MQTT Network Thread
Managed by libmosquitto (loop_start()). Reads TCP socket, calls on_message() for each arriving message. on_message() does nothing but push raw bytes into the lock-free ring buffer — no parsing, no locking.
② Parse Thread
Drains the ring buffer. For each slot: parses the MQTT topic into partition key + field name, parses the JSON payload, inserts a Row into the in-memory buffer. Holds g_mutex only for the brief buffer insert (~1 µs).
③ Flush Thread
Wakes every flush_interval_seconds (or when buffer is full). Swaps the buffer under g_mutex (pointer swap, O(1)), releases the lock, then writes Parquet files to disk — the slow I/O happens entirely outside the lock.
④ Health Thread
Listens on a TCP port (default :8771). Responds to GET /health with a JSON metrics snapshot: messages received, ring utilisation, drop counts, flush timing. Used by the monitoring dashboard and load balancers.

The real_writer adds a fifth thread — the Compactor — described fully in §11. It operates entirely on the filesystem and shares no state with the four threads above.

4 · Data Flow

Step 1 — MQTT ingest (lock-free ring)

When a message arrives, on_message() copies the topic and payload into a pre-allocated slot in the MqttRing — a 262,144-slot circular buffer (~135 MB BSS). The write is a single atomic::store; no mutex, no allocation. If the ring is full, the message is dropped and a counter is incremented (visible in /health).

The ring acts as a shock absorber: at 300,000 msg/s it holds ~870 ms of burst before any drops occur, absorbing slow parquet flushes (observed up to ~300 ms on spinning disk).

Step 2 — Topic parsing

The parse thread reads each ring slot and calls parse_topic(). The topic string follows a key=value segment format: 

batteries/site=0/rack=1/module=2/cell=3/voltage

For Longbow / Evelyn EMS topic structures see mqtt_topic_formats.html.

parse_topic() walks the string with a single pointer pass (no heap allocation, no regex), extracting the source type (batteries), the KV pairs (site_id=0, rack_id=1, …), and the field name (voltage). These become the partition key and column name in the output row.

Step 3 — JSON payload parsing

Payloads are JSON. The writer uses simdjson — a SIMD-accelerated parser that runs at memory-bandwidth speed on modern CPUs. Two payload formats are accepted:

FormatExampleUsed by
Nested (generator){"timestamp":1.23,"voltage":{"value":3.7,"unit":"V"}}stress_pub, generator
Flat scalar (EMS){"ts":"2024-11-15T21:27:52Z","value":0.999}Longbow / real EMS

Step 4 — In-memory buffering

Parsed rows are stored in g_buffers — a map from PartitionKey to a vector<Row>. The partition key encodes where in the directory tree the rows will land (source type + site + date). Memory usage is bounded by max_total_buffer_rows (default 2M rows).

Each Row carries three typed maps: ints (int64 — topic KV fields, project_id), floats (double — sensor values, timestamps), and strings (string — mqtt_topic, site_id, and any topic segments configured as string columns). This three-map design lets the Arrow schema be derived dynamically from the first row of each partition without any schema pre-declaration.

value_i / value_f split: when a partition accumulates both integer and float readings under the column name "value", Arrow's type system requires they be split. The flush thread detects this and renames: value (int path) → value_i, value (float path) → value_f. This is transparent to the parse thread — it always writes to "value".

Step 5 — Parquet flush

Every flush_interval_seconds (default 60 s, torture test 10 s), the flush thread atomically swaps g_buffers with a fresh empty map (O(1) pointer exchange under the mutex), then writes each partition's rows to a Parquet file outside the lock. File naming includes a UTC timestamp to avoid collisions: 

/srv/data/parquet/batteries/project=0/site=0/2026/03/23/20260323T142001Z.parquet

Parquet files are self-describing — column names, types, and metadata are embedded. DuckDB can query them directly with no import step. With topic_partition_segments: 2 the path becomes: 

/srv/data/parquet/unit/02281660/2026/03/23/20260323T142001Z.parquet

See §6 Configuration for the full partitioning options.

Topic parsing — mode selection

The writer supports two topic-parsing modes, selected by topic_parser:

ModeHow it worksUse when
positional default Splits topic on / and maps each segment to a column name by position, using topic_segments list. Segment 0 becomes topic_segments[0], etc. Fixed-depth topics like unit/{id}/bms/{inst}/{point}/{dtype}
kv Each topic segment is expected to contain key=value. The topic_kv_map maps known keys to column names. Segments not in the map are ignored. Self-describing topics like site=0/rack=1/module=2/cell=3/voltage

topic_segments (positional mode)

Ordered list of column names, one per topic segment. Special reserved values:

Reserved nameBehaviour
dtype_hintConsumed for type-casting (int64 vs float64); not stored as a column
field_nameRenames the value float column to the segment value (wide-sparse schema)
anything elseStored as a string column with that name
 
mqtt:
  topic_parser: positional
  topic_segments: [source_type, unit_id, device, instance, point_name, dtype_hint]
  # topic: unit/02281660/bms/bms_1/voltage/float
  # → source_type=unit, unit_id=02281660, device=bms, instance=bms_1,
  #   point_name=voltage, dtype_hint=float (consumed, not stored)

topic_kv_map (kv mode)

Map from segment key to column name. Only listed keys are extracted.

mqtt:
  topic_parser: kv
  topic_kv_map:
    site:   site_id
    rack:   rack_id
    module: module_id
    cell:   cell_id
  # topic: site=0/rack=1/module=2/cell=3/voltage
  # → site_id=0, rack_id=1, module_id=2, cell_id=3  (point_name from payload or compound)

compound_point_name

Join these segment columns with / and store the result as point_name. The named columns are consumed (removed from the row after joining).

mqtt:
  topic_segments: [source_type, unit_id, device, instance, point_name, dtype_hint]
  compound_point_name: [device, instance, point_name]
  # device=bms, instance=bms_1, point_name=voltage → point_name = "bms/bms_1/voltage"

compound_field_name wide/pivot mode

Join these segment columns with / and use the result as the value column name. Each unique signal path becomes its own Parquet column; a row only writes to one column, all others are null. Works best with null_fill_unchanged: true for COV compression.

mqtt:
  compound_field_name: [device, instance, point_name]
  # Creates columns like "bms/bms_1/voltage", "bms/bms_1/temperature", …
  # One row has exactly one non-null value column.

10 · Implementation Details

Lock-free SPSC ring (mqtt_ring.h)

The MqttRing is a single-producer / single-consumer ring buffer using two std::atomic<uint64_t> counters (write_pos, read_pos) on separate 64-byte cache lines to avoid false sharing.

Producer (on_message, MQTT thread): loads write_pos with relaxed ordering, checks fullness with acquire on read_pos, copies topic + payload into the slot, then stores write_pos+1 with release ordering.

Consumer (parse_thread_fn): loads read_pos with relaxed, checks emptiness with acquire on write_pos, copies the slot, stores read_pos+1 with release. The acquire/release pair is the memory barrier that guarantees the consumer sees the fully-written slot.

Slot size: 516 bytes (4-byte header + 256-byte topic + 256-byte payload). Total ring: 262,144 slots × 516 B ≈ 135 MB (BSS, zero-initialised at startup).

Topic truncation counters

Real EMS topics reach 136 bytes. The ring's MQRING_TOPIC_MAX is 256 bytes — safe headroom. If a topic or payload would be truncated, atomic counters topic_truncations / payload_truncations are incremented and exposed in /health. A non-zero count means the physical sensor topic structure has changed and the constant needs to be updated.

Flush thread wake mechanism

Rather than a condition variable (which would require a mutex for the wait), the flush thread polls a g_signal_pipe (a standard Unix pipe) using poll() with a timeout equal to flush_interval_seconds. When a message arrives or the buffer is full, process_raw_message() writes a single byte to the write end of the pipe. The flush thread wakes immediately instead of waiting the full interval. Both ends of the pipe are O_NONBLOCK to prevent the fast write path from ever blocking.

Shutdown sequence

SIGTERM →  handle_signal() sets g_shutdown=true, writes to signal pipe
          → flush_thread: current flush completes, exits while(!g_shutdown) loop
          → main: flusher.join()
          → main: mosquitto_disconnect + mosquitto_loop_stop(force=true)
          → main: sets g_mqtt_stopped=true — parse_thread drains ring to empty
          → main: parser.join()
          → main: final flush of any remaining buffer rows
          → main: health_thr.join() (exits within 100ms of g_shutdown)
          → process exits cleanly

Parquet schema construction

The Parquet schema is built from the first row seen for each partition. Each Row carries three typed maps — ints (int64), floats (double), strings (string) — and the flush thread iterates all three to construct Arrow builders dynamically. mqtt_topic lands in strings and is written as a Parquet dictionary-encoded string column (Arrow DictionaryArray<int32, StringArray>), making the per-row overhead ~1–2 bytes despite topic strings being 30–136 characters long.

At flush time, if any partition has accumulated both integer and floating-point values under the column name "value", the flush thread renames them value_i and value_f before building the Arrow schema, avoiding a type mismatch. This happens automatically and is invisible to the parse thread.

Arrow builders accumulate rows in memory; at flush time the builders are finalised into an Arrow Table and written via parquet::arrow::WriteTable(). The WAL uses the same Arrow IPC format via arrow::ipc::MakeFileWriter() — the same schema, so WAL replay can feed directly into the same flush function.

11 · Compactor real_writer only

The compactor is a fifth background thread in real_writer that fully re-parquets many small flush files into one larger file per directory. It reads each source file into an Arrow Table in RAM, concatenates them, and calls parquet::arrow::WriteTable() to write a brand new Parquet file — not a byte-level merge. It is entirely filesystem-based and shares no state with the four core threads.

Because the output is re-encoded from scratch, the compacted file gets: fresh compression using the current config codec (useful if you switch codecs), fresh column statistics and row-group min/max (makes DuckDB predicate pushdown more accurate), unified schema across source files, and a single large row group instead of many small ones.

Why compaction is needed

At a 60-second flush interval with topic_partition_segments: 2 (one file per unit per flush), a 46-unit fleet produces 46 files every minute — 2,760 files per hour, 66,240 per day. DuckDB can scan many small files, but opening and closing thousands of file handles has overhead. Compacting N small files into one per directory per day removes this overhead without changing the data model.

13 · Extension: SHM Output File Buffer Under Review

The flush thread currently writes Parquet files directly to a disk path (/srv/data/parquet/). A proposed extension stages completed files in a /dev/shm/parquet_out/ ring of N slots before an async mover thread transfers them to the final disk destination. This decouples flush latency from disk write latency.

Motivation

  • Disk jitter isolation — spinning HDD write latency spikes (10–100 ms) no longer stall the flush thread; the flush thread writes to RAM at memory speed (>10 GB/s), then returns to accumulating rows immediately.
  • Burst absorption — at 1 M msg/s, a 60-second flush produces ~60 M rows. Staging in SHM allows the next interval's rows to start accumulating while the large file is being committed to disk.
  • Crash-safe handover — files staged in /dev/shm survive a writer process crash (SHM persists until the OS reboots). A restarted writer (or companion recovery process) can inspect the output ring and complete any pending disk moves.

14 · SHM Ownership Takeover on Crash

Both the SHM input ring (/dev/shm/parquet_ring) and the proposed SHM output buffer (/dev/shm/parquet_out/) use the same ownership model: a PID + heartbeat field in the shared header allows a replacement writer process to detect that the previous owner has crashed and take over cleanly, without losing any data that was in flight.

Ownership header (shared by input and output SHM regions)

struct ShmOwner {
    uint32_t  owner_pid;        // PID of the writer that currently holds this region
    uint64_t  heartbeat_ns;     // writer updates this every ~200 ms (CLOCK_MONOTONIC)
    uint32_t  heartbeat_period; // declared update period in ms (default 200)
    uint32_t  takeover_grace;   // ms to wait before declaring owner dead (default 600)
};

Takeover protocol

StepAction
1. AttachNew writer opens the SHM region (shm_open + mmap). If the region does not exist, it creates it and sets owner_pid = getpid().
2. Check livenessIf owner_pid != 0, send kill(owner_pid, 0). If the process exists, another writer is healthy — this instance exits or waits.
3. Heartbeat stale?If kill(owner_pid, 0) returns ESRCH (process gone) or now − heartbeat_ns > takeover_grace ms, the owner is considered crashed.
4. CAS takeoverAttempt __atomic_compare_exchange on owner_pid: expected = old PID, desired = getpid(). Only one of multiple racing restarts wins.
5. ResumeNew owner reads the ring's read_pos / write_pos and continues from where the crashed writer stopped. No data is lost from the SHM ring.
6. Heartbeat loopNew owner updates heartbeat_ns every ~200 ms from a dedicated timer thread (or the health thread). If the writer's own health thread detects g_shutdown, it zeros owner_pid to release cleanly.

What survives a crash

ResourceSurvives crash?Notes

16 · WAL — Detailed Design

The write-ahead log guarantees that no rows buffered in RAM are permanently lost across a writer crash. It uses Arrow IPC format — the same in-memory column layout as Arrow Tables, written to a file — so WAL files are compact, schema-preserving, and readable without a full Parquet decode.

File layout

{base_path}/.wal/
├── unit_02281660_20260329T142001Z.wal.arrow     ← one file per partition per flush cycle
├── unit_02281661_20260329T142001Z.wal.arrow
└── unit_02281660_20260329T142001Z.wal.arrow.tmp  ← only exists during write; rename() makes it atomic

Naming: {source_type}_{partition_value}_{timestamp}.wal.arrow. The .tmp suffix is used during write; only .wal.arrow files are scanned at replay time so a crash mid-write leaves no partial file in the scan path.

Per-flush sequence (normal operation)

For each partition in to_flush: 1. write_wal_ipc(part.rows) → {partition}.wal.arrow.tmp (Arrow IPC, all columns) 2. rename(.tmp → .wal.arrow) (atomic — visible or not) 3. flush_partition(part.rows) → {date}/{ts}.parquet (with up to 3 retries) ├── success → fs::remove(.wal.arrow) (WAL no longer needed) └── permanent failure → WAL file RETAINED (logged: "WAL retained for recovery")

The WAL file for a given partition exists on disk for only the duration of the Parquet write — typically 10–100 ms. At any point in time the .wal/ directory should be empty unless a crash or permanent I/O failure has occurred.

The recombine problem — replayed rows as isolated files

The current design flushes replayed rows immediately to new Parquet files (step 2c above). This means:

  • If the writer crashed 5 s into a 60 s flush interval, the WAL holds 5 s of data.
  • On restart those 5 s are written as a standalone Parquet file with a new timestamp.
  • The next 60 s of live data lands in a separate flush file.
  • The compactor will eventually merge both into one file — but only after compact_interval_seconds (default 10 min) have elapsed.

For analytical queries this is fine — DuckDB scans both files. For storage efficiency it creates a small orphan file that inflates the file count until the compactor runs.

Where to apply dedup

Dedup should run in the flush thread, not in the message callback. Doing it in the callback would require holding g_mutex longer (for the g_current_state lookup) and would make the hot path conditional on config. In the flush thread the data is already off the hot path — the buffer has been swapped out under the mutex and the flush thread owns it exclusively.

Concretely: after std::exchange(g_buffers, {}) gives the flush thread its snapshot, it iterates the rows per partition and filters them before building Arrow columns. g_current_state is read from a snapshot taken at the same swap point (already done for hot.parquet and current_state.parquet), so the comparison is consistent with the buffer contents.

A single relative deadband (e.g. 0.1%) applied to all float signals fails in both directions across the EMS signal space:

Per-point-name deadband lets each signal class carry its own threshold, expressed as either a relative % (good for signals with stable baselines) or an absolute value (good for near-zero-crossing signals and where physical units dictate the resolution requirement). Specific signals can also be marked disabled — always stored regardless of change.

Deadband comparison — with per-point lookup

enum class DeadbandMode { Relative, Absolute, Disabled };

struct PointDeadband {
    DeadbandMode mode  {DeadbandMode::Relative};
    double       value {0.001};   // relative: fraction (0.001 = 0.1%)
                                  // absolute: raw units (0.005 = 5 mV for voltage)
};

// In Config — built from YAML at startup
std::unordered_map<std::string, PointDeadband> dedup_per_point;
PointDeadband                                   dedup_default;   // fallback

// Lookup — O(1) unordered_map, called once per row in flush thread
const PointDeadband& get_deadband(const std::string& point_name) {
    auto it = g_cfg->dedup_per_point.find(point_name);
    return (it != g_cfg->dedup_per_point.end()) ? it->second : g_cfg->dedup_default;
}

bool within_deadband(double new_val, double last_val, const PointDeadband& db) {
    if (db.mode == DeadbandMode::Disabled) return false;
    double delta = std::abs(new_val - last_val);
    if (db.mode == DeadbandMode::Absolute)
        return delta < db.value;
    // Relative — guard against divide-by-near-zero
    if (std::abs(last_val) < 1e-9) return delta < 1e-9;
    return delta / std::abs(last_val) < db.value;
}

The point_name is already stored in row.strings["point_name"] for every row produced by the real EMS writer, so the lookup key is always available at flush time with no additional parsing.

sample_count accumulation

When a row is suppressed by dedup, its value is not lost — it is recorded as "seen but unchanged". The sample_count on the last surviving row for that sensor is incremented. At the end of the flush loop, any sensor whose last row was suppressed all-flush needs its accumulated count written out as a final "sentinel" row so sample_count is not silently dropped.

Lookup cost

std::unordered_map<std::string, PointDeadband> lookup is O(1) average. With 1,770 distinct point names (Longbow fleet) and a 60-second flush interval at 81k msg/s, the flush thread calls get_deadband() ~4.9 M times per flush — each call is a single hash + string compare, well under 1 µs. Total overhead is negligible relative to the Arrow column building that follows.

If glob-style prefix matching is needed (e.g. suppress all Batt1_MBMS_* fault flags the same way), a two-stage lookup — exact match first, then linear scan of prefix patterns — is acceptable at flush time since it is not on the hot MQTT ingest path.

Interaction with WAL replay: if the writer crashes mid-flush, the WAL replay on restart re-inserts the pre-dedup rows into g_buffers. The g_current_state snapshot at that point may be stale (it was not persisted atomically with the WAL). The first flush after replay should therefore skip dedup for WAL-sourced rows (they are already marked with their original sample_count) to avoid double-suppression. Tag WAL-replayed rows with a flag (e.g. row.ints["_wal_replay"] = 1) so the flush thread can bypass the per-point deadband lookup for them.
Per-point config and current_state_path: the per-point deadband comparison uses the last flushed value from g_current_state. If the writer restarts cleanly (SIGTERM, not crash), current_state_path persists this map to disk so the first post-restart flush applies the correct per-point deadband rather than treating every signal as "first seen" and storing all values unconditionally. Without this persistence, a restart causes a one-interval burst of un-deduped rows — acceptable but worth noting.
Expected reduction: based on the Telegraf dedup experiment, float COV at 0.1% deadband is projected to suppress ~17–23% of float rows on top of Filter B's integer dedup, bringing the combined reduction to the "Moderate" scenario (~40% total). The actual rate depends heavily on signal volatility — a cell voltage that dithers ±0.001 V at rest will dedup heavily; a current signal during active charge/discharge will not.
