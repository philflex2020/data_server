1 · Overview

parquet_writer is a C++17 daemon that consumes a high-rate MQTT telemetry stream and writes it as partitioned Apache Parquet files on local storage. It is designed for battery-energy-storage (BESS) telemetry at industrial scale — sustained rates above 80,000 messages/second — but works for any time-series sensor data with an MQTT/JSON source.

Why Parquet?

Raw MQTT JSON is ~109 bytes per message. The same data stored as Parquet with Snappy compression averages 9.6 bytes per row — an 11.4× compression ratio. Parquet is also columnar, which makes analytical queries (average voltage of all cells in rack 3 over the last hour) orders of magnitude faster than row-oriented formats. Query engines like DuckDB can read these files directly without any database import step.

What it does not do

  • It does not decode or interpret the sensor values — it stores them faithfully.
  • It does not require a running database — output is plain files on disk.
  • It does not require cloud connectivity — designed for edge deployment.

2 · System Architecture

The writer sits between an MQTT broker and local storage. Multiple instances can run in parallel, each subscribing to all (or a subset of) topics and writing to their own output directory.

MQTT Publishers (stress_pub / real EMS devices) │ QoS-0 ~80–300k msg/s ▼ ┌──────────────┐ │ FlashMQ │ ← high-performance C++ broker (single node, 20+ threads) │ :1883 │ └──────┬───────┘ │ fan-out: broker delivers each message to every subscriber independently ┌─────┴─────┬────────────┬────────────┐ ▼ ▼ ▼ ▼ writer-a writer-b writer-c writer-d :8771 :8772 :8773 :8774 │ │ │ │ ▼ ▼ ▼ ▼ parquet-a/ parquet-b/ parquet-c/ parquet-d/ (NVMe) (NVMe) (NVMe) (NVMe)

Each writer instance is independent — they do not communicate with each other. The broker handles fan-out: each message is delivered to all subscribers. Running four parallel writers gives four identical copies of the data, useful for testing fault tolerance (kill one, the others keep writing).

Multi-instance rule: each instance needs a unique mqtt.client_id, output.base_path, and health.port. Everything else (host, topic, QoS, compression) can be shared.

3 · Thread Model

The writer uses four threads. Each has a single responsibility and they communicate through lock-free or minimally-locked shared state.

① MQTT Network Thread
Managed by libmosquitto (loop_start()). Reads TCP socket, calls on_message() for each arriving message. on_message() does nothing but push raw bytes into the lock-free ring buffer — no parsing, no locking.
② Parse Thread
Drains the ring buffer. For each slot: parses the MQTT topic into partition key + field name, parses the JSON payload, inserts a Row into the in-memory buffer. Holds g_mutex only for the brief buffer insert (~1 µs).
③ Flush Thread
Wakes every flush_interval_seconds (or when buffer is full). Swaps the buffer under g_mutex (pointer swap, O(1)), releases the lock, then writes Parquet files to disk — the slow I/O happens entirely outside the lock.
④ Health Thread
Listens on a TCP port (default :8771). Responds to GET /health with a JSON metrics snapshot: messages received, ring utilisation, drop counts, flush timing. Used by the monitoring dashboard and load balancers.

4 · Data Flow

Step 1 — MQTT ingest (lock-free ring)

When a message arrives, on_message() copies the topic and payload into a pre-allocated slot in the MqttRing — a 262,144-slot circular buffer (~135 MB BSS). The write is a single atomic::store; no mutex, no allocation. If the ring is full, the message is dropped and a counter is incremented (visible in /health).

The ring acts as a shock absorber: at 300,000 msg/s it holds ~870 ms of burst before any drops occur, absorbing slow parquet flushes (observed up to ~300 ms on spinning disk).

Step 2 — Topic parsing

The parse thread reads each ring slot and calls parse_topic(). The topic string follows a key=value segment format: 

batteries/site=0/rack=1/module=2/cell=3/voltage

parse_topic() walks the string with a single pointer pass (no heap allocation, no regex), extracting the source type (batteries), the KV pairs (site_id=0, rack_id=1, …), and the field name (voltage). These become the partition key and column name in the output row.

Step 3 — JSON payload parsing

Payloads are JSON. The writer uses simdjson — a SIMD-accelerated parser that runs at memory-bandwidth speed on modern CPUs. Two payload formats are accepted:

FormatExampleUsed by
Nested (generator){"timestamp":1.23,"voltage":{"value":3.7,"unit":"V"}}stress_pub, generator
Flat scalar (EMS){"ts":"2024-11-15T21:27:52Z","value":0.999}Longbow / real EMS

Step 4 — In-memory buffering

Parsed rows are stored in g_buffers — a map from PartitionKey to a vector<Row>. The partition key encodes where in the directory tree the rows will land (source type + site + date). Memory usage is bounded by max_total_buffer_rows (default 2M rows).

Step 5 — Parquet flush

Every flush_interval_seconds (default 60 s, torture test 10 s), the flush thread atomically swaps g_buffers with a fresh empty map (O(1) pointer exchange under the mutex), then writes each partition's rows to a Parquet file outside the lock. File naming includes a UTC timestamp to avoid collisions: 

/srv/data/parquet/batteries/project=0/site=0/2026/03/23/20260323T142001Z.parquet

Parquet files are self-describing — column names, types, and metadata are embedded. DuckDB can query them directly with no import step.

5 · Configuration

All settings live in a YAML file (default: writer_config.yaml). Individual values can be overridden with CLI flags without creating a new file.

mqtt:
  host:      localhost       # FlashMQ broker
  port:      1883
  client_id: parquet-writer  # ★ must be unique per instance
  topic:     "#"             # subscribe to all topics
  qos:       0               # QoS-0 for max throughput

output:
  base_path:              /srv/data/parquet   # ★ must be unique per instance
  compression:            snappy             # snappy | zstd | gzip | none
  flush_interval_seconds: 60
  max_messages_per_part:  5000000
  max_total_buffer_rows:  2000000            # OOM guard (~25s at 80k msg/s)

health:
  enabled: true
  port: 8771                                 # ★ must be unique per instance

wal:
  enabled: false                             # write-ahead log (crash recovery)

guard:
  min_free_gb: 1.0                           # skip flush if disk nearly full

CLI overrides

Useful for launching multiple instances without multiple config files:

./parquet_writer --client-id parquet-writer-2 --output /srv/data/parquet-2 --health-port 8772

6 · Robustness Features

FeatureWhat it doesConfig key
MQTT auto-reconnect Libmosquitto reconnects automatically with exponential backoff (2 s → 30 s) if the broker restarts or the TCP connection drops. Subscription is re-issued in on_connect() so it survives reconnects. mqtt.reconnect_*
Ring buffer overflow protection When the ring is full (e.g. during a slow flush), new messages are dropped rather than blocking the MQTT network thread. Drop count is tracked and exposed in /health. The ring holds ~870 ms of burst at 300k msg/s before any drops occur. Fixed at 262,144 slots
Buffer overflow guard If in-memory buffer rows exceed max_total_buffer_rows, the writer drops incoming messages and triggers an early flush to reclaim memory. Prevents OOM on very long flush pauses. output.max_total_buffer_rows
Disk space guard Before each flush, checks available disk space. If below min_free_gb, skips the flush and logs a warning. Prevents the writer from filling the disk and crashing the host. guard.min_free_gb
Write-ahead log (WAL) Optional. Writes rows to an Arrow IPC file before each Parquet flush. On restart after a crash, replays any surviving WAL files before accepting live data. Guarantees no data loss across writer crashes. Currently disabled in the torture test (NVMe is fast enough that flush timing is the bottleneck, not crash recovery). wal.enabled
Parquet flush retry If a Parquet file write fails (disk full, permission error, transient I/O), the writer retries up to flush_max_retries times with exponential backoff before logging a permanent failure. output.flush_max_retries
Graceful shutdown On SIGTERM or SIGINT, the writer completes any in-progress flush, performs a final flush of all buffered rows, disconnects from MQTT cleanly, and exits. No data is lost on a clean shutdown. —
Health endpoint HTTP/1.1 GET /health returns a JSON metrics object. Used by the monitoring dashboard to detect dead writers, track message rates, ring utilisation, and flush timing. Health thread shuts down within 100 ms of SIGTERM. health.enabled / health.port

9 · Implementation Details (Intern Reference)

Lock-free SPSC ring (mqtt_ring.h)

The MqttRing is a single-producer / single-consumer ring buffer using two std::atomic<uint64_t> counters (write_pos, read_pos) on separate 64-byte cache lines to avoid false sharing.

Producer (on_message, MQTT thread): loads write_pos with relaxed ordering, checks fullness with acquire on read_pos, copies topic + payload into the slot, then stores write_pos+1 with release ordering.

Consumer (parse_thread_fn): loads read_pos with relaxed, checks emptiness with acquire on write_pos, copies the slot, stores read_pos+1 with release. The acquire/release pair is the memory barrier that guarantees the consumer sees the fully-written slot.

Slot size: 516 bytes (4-byte header + 256-byte topic + 256-byte payload). Total ring: 262,144 slots × 516 B ≈ 135 MB (BSS, zero-initialised at startup).

Topic truncation counters

Real EMS topics reach 136 bytes. The ring's MQRING_TOPIC_MAX is 256 bytes — safe headroom. If a topic or payload would be truncated, atomic counters topic_truncations / payload_truncations are incremented and exposed in /health. A non-zero count means the physical sensor topic structure has changed and the constant needs to be updated.

Flush thread wake mechanism

Rather than a condition variable (which would require a mutex for the wait), the flush thread polls a g_signal_pipe (a standard Unix pipe) using poll() with a timeout equal to flush_interval_seconds. When a message arrives or the buffer is full, process_raw_message() writes a single byte to the write end of the pipe. The flush thread wakes immediately instead of waiting the full interval. Both ends of the pipe are O_NONBLOCK to prevent the fast write path from ever blocking.

Shutdown sequence

SIGTERM →  handle_signal() sets g_shutdown=true, writes to signal pipe
          → flush_thread: current flush completes, exits while(!g_shutdown) loop
          → main: flusher.join()
          → main: mosquitto_disconnect + mosquitto_loop_stop(force=true)
          → main: sets g_mqtt_stopped=true — parse_thread drains ring to empty
          → main: parser.join()
          → main: final flush of any remaining buffer rows
          → main: health_thr.join() (exits within 100ms of g_shutdown)
          → process exits cleanly

Parquet schema construction

The Parquet schema is built from the first row seen for each partition. Columns are: one int64 per topic KV field, plus field (string), value (double), ts (double). Arrow builders accumulate rows in memory; at flush time the builders are finalised into an Arrow Table and written via parquet::arrow::WriteTable().
