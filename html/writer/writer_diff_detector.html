writer.cpp — Diff Detector Proposal

Design for a tool that watches for schema and behavioural changes between writer.cpp versions

Problem Statement

writer.cpp is a single-file C++ ingest binary. Its output — the Parquet schema, partition layout, and WAL format — is consumed downstream by DuckDB queries, push_agent rsync, and the subscriber API. Changes to writer.cpp that seem local (renaming a config field, reordering columns, changing a default) can silently break the downstream stack.

A diff detector is a tool that, given two writer.cpp binaries (or two commits), automatically identifies every externally-visible change and classifies it by breaking severity. It runs in CI before merge and as a scheduled check on production builds.

Motivation: The current review process is manual. The code review doc identified 6 critical bugs and 11 minor issues. A diff detector would have caught many of these automatically and would prevent regressions when fixes are applied.

What Changes Are Detectable?

Breaking Changes
  • Parquet column added or removed
  • Column type changed (int64 → double)
  • Column renamed
  • Partition path template changed
  • Compression codec changed
  • WAL file format changed
  • Health endpoint response schema changed
  • Config field removed or renamed
Behavioural Changes
  • Default config value changed
  • Flush interval default changed
  • Buffer limit default changed
  • Topic KV map default changed
  • Partition field default changed
  • WAL enabled default changed
  • Health port default changed
  • Log message format changed
Safe / Informational
  • New config field added (with default)
  • New CLI flag added
  • New health field added
  • Compression perf change (same codec)
  • Parse throughput change
  • Flush retry logic change
  • Signal handling change
  • Internal mutex strategy change

Architecture

The detector is a Python harness that runs both writer binaries against a synthetic MQTT stream, captures their outputs, and diffs the results. No source code parsing — purely output-based, so it works on any build. 
# High-level flow

writer_old  ←──── synthetic MQTT stream ────→  writer_new
    │                                               │
    ▼                                               ▼
  parquet_old/                                parquet_new/
  wal_old/                                    wal_new/
  health_old:8771                             health_new:8772

                   ▼
          diff_detector.py
                   │
          ┌────────┴────────┐
          ▼                 ▼
    schema_diff         behaviour_diff
    (pyarrow)           (health JSON, log lines)
                   │
                   ▼
          report.json + report.html

Components

diff_detector.py
  • Launches both writers with test configs
  • Sends N messages via paho MQTT to both
  • Waits for flush (or triggers via SIGTERM)
  • Reads health endpoints from both
  • Calls schema_diff and behaviour_diff
  • Emits structured report
schema_diff.py
  • Reads first Parquet file from each output dir
  • Compares schemas via pyarrow
  • Checks column names, types, nullable flags
  • Compares partition dir structure
  • Reads WAL Arrow IPC files and compares schemas
  • Returns list of SchemaChange objects
behaviour_diff.py
  • Compares health JSON from both writers
  • Loads writer_config.yaml defaults from both
  • Diffs default values of each Config field
  • Checks row counts match for same input
  • Compares partition file count and names
  • Returns list of BehaviourChange objects
config_extractor.py
  • Runs writer --help and parses output
  • Extracts CLI flags and their defaults
  • Runs writer with --config /dev/null to get compile-time defaults
  • Diffs flag lists between versions
  • Returns list of ConfigChange objects

Test Config Strategy

Each writer runs with an isolated config that gives it a unique client_id, base_path, and health port — identical to the multi-instance pattern in writer_config.yaml.

Both writers subscribe to the same topic. The broker delivers every message to both independently — no splitting, no coordination. Row counts must match exactly after flush.

Change Classification

Change Type Detection Method Severity Action
Column added to Parquet pyarrow schema diff WARN DuckDB queries using SELECT * get new column; explicit queries safe
Column removed from Parquet pyarrow schema diff BREAK Block merge; downstream queries referencing column will fail
Column type changed pyarrow schema diff BREAK Block merge; mixed-type files will cause DuckDB schema errors
Column renamed pyarrow schema diff BREAK Block merge; treat as remove+add pair
Partition template changed compare dir tree structure BREAK Block merge; existing parquet files unreachable via new path
Row count mismatch for same input compare parquet_rows() totals BREAK Block merge; data loss or duplication bug
WAL schema changed Arrow IPC schema diff BREAK Block merge; WAL replay will fail on old WAL files
Config field default changed config_extractor + default run WARN Require explicit config entry in production configs
New config field added config_extractor diff INFO Document new field; existing configs use the default
New CLI flag added --help diff INFO Log for release notes only
Health field added or removed health JSON diff WARN Monitoring dashboards may need updating
Compression codec default changed pyarrow metadata check WARN DuckDB reads both; performance implication only

Implementation Plan

Phase 1 — Schema Detector Recommended first
  • Write schema_diff.py using pyarrow
  • Reads two dirs, compares first parquet file schemas
  • No need to run writers — works on existing output
  • ~60 lines of Python
  • Can be run manually: python schema_diff.py old/ new/

Immediate value: catches the class of bug where a config change silently drops columns.

Phase 2 — Config Extractor Low effort
  • Parse writer --help output with regex
  • Run writer with empty config to dump defaults to health
  • Diff field lists and default values
  • ~80 lines of Python

Catches default value drift — the m1/m2 class of minor bugs.

Phase 3 — Live Harness Moderate effort
  • Launch both writers, send N messages via paho
  • Trigger flush via SIGTERM or wait for interval
  • Compare row counts and schemas
  • Requires local FlashMQ or mosquitto
  • ~200 lines of Python

Full integration test — catches data loss bugs (C3, C5) missed by schema diff alone.

Phase 4 — CI Integration Moderate effort
  • GitHub Action: build both old (main) and new (PR branch)
  • Run detector, fail on BREAK severity
  • Post report as PR comment
  • ~100 lines of GitHub Actions YAML

Prevents accidental breaking changes from landing on main.

Phase 1 Sketch — schema_diff.py

#!/usr/bin/env python3
# schema_diff.py — compare parquet schemas between two writer output dirs
# Usage: python schema_diff.py /tmp/diff_old /tmp/diff_new

import sys, pathlib
import pyarrow.parquet as pq

def first_parquet(dirpath):
    """Find the first .parquet file anywhere under dirpath."""
    for p in pathlib.Path(dirpath).rglob("*.parquet"):
        return p
    raise FileNotFoundError(f"No parquet files under {dirpath}")

def schema_changes(old_dir, new_dir):
    old_schema = pq.read_schema(first_parquet(old_dir))
    new_schema = pq.read_schema(first_parquet(new_dir))

    old_cols = {f.name: f.type for f in old_schema}
    new_cols = {f.name: f.type for f in new_schema}

    changes = []
    for name in old_cols:
        if name not in new_cols:
            changes.append(("BREAK", f"Column removed: {name} ({old_cols[name]})"))
        elif old_cols[name] != new_cols[name]:
            changes.append(("BREAK", f"Type changed: {name} {old_cols[name]} → {new_cols[name]}"))
    for name in new_cols:
        if name not in old_cols:
            changes.append(("WARN", f"Column added: {name} ({new_cols[name]})"))
    return changes

if __name__ == "__main__":
    old_dir, new_dir = sys.argv[1], sys.argv[2]
    changes = schema_changes(old_dir, new_dir)
    if not changes:
        print("✓ No schema changes detected")
    else:
        for severity, msg in changes:
            print(f"[{severity}] {msg}")
    exit(1 if any(s == "BREAK" for s, _ in changes) else 0)
Phase 1 has zero new dependencies — pyarrow is already required by the test harness. Can be added to make test as a post-build check in <1 day.

Relation to Existing Tests

The diff detector and the unit/KPI test suite are complementary, not overlapping. The unit tests verify correctness of the current binary; the diff detector verifies that the new binary is compatible with the old binary's output.

Recommended First Step

Start with Phase 1 only. Create source/parquet_writer_cpp/tests/schema_diff.py (~60 lines). Add make diff-check OLD=... NEW=... to the Makefile. This catches the most dangerous class of change (column removal/type change) with zero additional dependencies and minimal effort.

The live harness (Phase 3) is more valuable but requires mosquitto to be running and adds ~4× complexity. Defer until schema_diff is in CI and proving useful.
