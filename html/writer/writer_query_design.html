1. The Gap Problem

parquet_writer buffers incoming MQTT messages and flushes to disk every flush_interval_seconds (typically 60s). Between flushes, in-memory rows are inaccessible to any external reader. A dashboard asking "what is happening right now?" has three distinct time regions to cover:

✔ Flushed parquet files
t=0 → t=(now − flush_interval)
Readable via DuckDB / bench_server
⚠ THE GAP
Last flush → now
Up to 60s invisible to external readers
★ current_state
Latest value per signal
Written atomically on each flush

The gap contains rows already parsed and sitting in g_buffer (the in-memory partition map), but nothing exposes them via HTTP today. A rolling in-memory ring bridges this gap without requiring a flush and without adding write latency.

4. Rolling 60s Buffer Design

Relationship between the live buffer and the WAL

The writer already has a WAL (Write-Ahead Log) that protects against data loss if the process crashes between flushes. The 60-second live buffer is a separate concern — it is not a replacement for the WAL and does not change when WAL files are written or deleted. The two mechanisms serve entirely different purposes:

WAL (*.wal.arrow)Live buffer (g_buffers)
Purpose Crash recovery — recover rows that were buffered but not yet flushed to Parquet Query tail — expose the last N seconds of data to GET /current without waiting for the next flush
Written On every flush, alongside the Parquet file Continuously, on every parsed MQTT message
Read Only on restart — replayed once to repopulate g_buffers, then deleted On every GET /current request, under a brief read lock
Format Arrow IPC — lossless, low-write overhead, not readable by DuckDB In-process std::map<PartitionKey, Partition> — serialised to Parquet on demand
Lifetime From flush until restart replay completes (then deleted) Rolling window — rows older than N seconds are evicted
Normal operation: GET /current never reads WAL files. It reads only g_buffers — the live in-memory map. WAL files are written and deleted entirely within the flush cycle; the HTTP path does not touch them.

After a restart: WAL replay runs before the MQTT connection is re-established. It populates g_buffers with the rows that were buffered at crash time. Once replay is complete, GET /current immediately has data — the gap between the last successful flush and the crash is bridged automatically.

Add a deque of rows that is appended on every MQTT parse and trimmed to the last N seconds. The buffer is read-locked only for serialisation (the same brief lock pattern as the existing buffer flush).

// parquet_writer.cpp — near g_current_state (line 1177)

static constexpr int RECENT_ROWS_SECONDS = 60;
struct TimestampedRow {
    int64_t     ts_ms;
    Row         row;
    std::string unit_id;    // pre-extracted for fast filter
    std::string device;
    std::string point_name;
};
static std::deque<TimestampedRow>  g_recent_rows;
static std::mutex                  g_recent_mutex;

Push side — in parse_thread (after g_current_state update)

// After the g_current_state insert at line 1861:
{
    TimestampedRow tr{ row.ts_ms, row,
                       get_string_col(row, "unit_id"),
                       get_string_col(row, "device"),
                       get_string_col(row, "point_name") };
    std::lock_guard<std::mutex> lk(g_recent_mutex);
    g_recent_rows.push_back(std::move(tr));
}
// Trim in a background pass (or lazily on each /data request):
auto cutoff = now_ms() - RECENT_ROWS_SECONDS * 1000;
while (!g_recent_rows.empty() && g_recent_rows.front().ts_ms < cutoff)
    g_recent_rows.pop_front();

4b. DuckDB In-Process Query Engine TODO

Instead of a hand-rolled deque + linear scan, embed DuckDB in-process as the rolling buffer store. parquet_writer already links Arrow; DuckDB is a single shared library (~6 MB) that uses the same Arrow columnar format internally. The payoff: GET /data becomes a SQL query — arbitrary WHERE clauses, aggregations, and window functions with zero custom filter code.

Setup — in-memory database on startup

// parquet_writer.cpp — add near other globals
#include "duckdb.hpp"   // link -lduckdb  (see Makefile section below)

static duckdb::DuckDB     g_duck{nullptr};           // nullptr = in-memory
static duckdb::Connection g_duck_write{g_duck};       // owned by parse_thread
static duckdb::Connection g_duck_read {g_duck};       // owned by health_thread (read queries)

// On startup, before parse_thread starts:
g_duck_write.Query(R"SQL(
    CREATE TABLE recent (
        ts_ms      BIGINT  NOT NULL,
        unit_id    VARCHAR,
        device     VARCHAR,
        point_name VARCHAR,
        float_val  DOUBLE,
        int_val    BIGINT,
        bool_val   BOOLEAN,
        string_val VARCHAR
    )
)SQL");

Insert path — Appender API (no SQL parsing overhead)

The DuckDB Appender bypasses the SQL layer — it's essentially a column-batched memcpy. Accumulate rows in a small staging vector, flush the Appender every parse-thread loop iteration (or every 500 rows), then run the trim DELETE periodically (e.g. every 5 seconds):

// In parse_thread — after g_current_state update:
if (g_cfg->recent_rows_seconds > 0) {
    staged.push_back({row.ts_ms, unit_id, device, point_name,
                      float_val, int_val, bool_val, string_val});
}

// Flush batch (every N rows or on timeout):
if (staged.size() >= 500 || elapsed_since_flush > 100ms) {
    duckdb::Appender app(g_duck_write, "recent");
    for (auto& r : staged) {
        app.BeginRow();
        app.Append<int64_t>(r.ts_ms);
        app.Append<std::string>(r.unit_id);
        app.Append<std::string>(r.device);
        app.Append<std::string>(r.point_name);
        app.Append<double>(r.float_val);
        app.Append<int64_t>(r.int_val);
        app.Append<bool>(r.bool_val);
        app.Append<std::string>(r.string_val);
        app.EndRow();
    }
    app.Flush();
    staged.clear();
}

// Trim old rows (run every ~5 seconds, not per-row):
g_duck_write.Query("DELETE FROM recent WHERE ts_ms < "
    + std::to_string(now_ms() - g_cfg->recent_rows_seconds * 1000LL));

Query path — handle_data() with SQL

Build a parameterized WHERE clause from the URL params and let DuckDB do the filtering. Use DuckDB's prepared statements to avoid SQL injection:

// In handle_data() — health_thread uses g_duck_read:
auto params  = parse_query_string(req);
int  last_s  = params.count("last") ? std::stoi(params["last"]) : 30;
int64_t cutoff = now_ms() - (int64_t)last_s * 1000;

// Build safe WHERE clause — only known column names, values via prepared stmt
std::string sql =
    "SELECT ts_ms, unit_id, device, point_name, float_val, int_val, bool_val"
    " FROM recent WHERE ts_ms >= $1";
std::vector<duckdb::Value> binds;
binds.emplace_back(cutoff);

if (params.count("unit"))   { sql += " AND unit_id = $2";    binds.emplace_back(params["unit"]);   }
if (params.count("device")) { sql += " AND device = $" + ...; binds.emplace_back(params["device"]); }
if (params.count("point"))  { sql += " AND point_name = $" + ...; }
sql += " ORDER BY ts_ms DESC LIMIT 10000";

auto stmt = g_duck_read.Prepare(sql);
auto res  = stmt->Execute(binds);

// Serialize result to JSON:
std::ostringstream body;
body << "[";
bool first = true;
for (auto& chunk : *res) {
    for (int i = 0; i < chunk.size(); ++i) {
        if (!first) body << ",";
        body << "{\"ts\":"     << chunk.GetValue<int64_t>(0, i)
             << ",\"unit\":\"" << chunk.GetValue<std::string>(1, i) << "\""
             << ",\"device\":\"" << chunk.GetValue<std::string>(2, i) << "\""
             << ",\"point\":\"" << chunk.GetValue<std::string>(3, i) << "\""
             << ",\"v\":"      << chunk.GetValue<double>(4, i)
             << "}";
        first = false;
    }
}
body << "]";

SQL pass-through option

Optionally expose GET /data?sql=SELECT+... for dashboards that want arbitrary queries against the rolling table. Guard against non-SELECT statements:

if (params.count("sql")) {
    auto sql = url_decode(params["sql"]);
    // Reject anything that isn't a plain SELECT
    if (sql.rfind("SELECT", 0) != 0 && sql.rfind("select", 0) != 0) {
        send_error(client_fd, 403, "only SELECT allowed");
        return;
    }
    auto res = g_duck_read.Query(sql);
    // ... serialize to JSON as above
}
The recent table is in-memory only. SQL pass-through cannot escape the rolling window — SELECT * will return at most the last recent_rows_seconds of data.

Thread safety

DuckDB supports multiple connections to the same in-memory database with MVCC isolation. Use two connections — one per thread — so the Appender in parse_thread never blocks queries from health_thread:

  • g_duck_write — parse_thread only: Appender + DELETE trims
  • g_duck_read — health_thread only: SELECT queries
  • No mutex needed between them — DuckDB serializes internally
Do not share a single duckdb::Connection across threads — it is not thread-safe. Two connections from the same duckdb::DuckDB instance is the correct pattern.

4c. WAL-as-Live-Buffer OPTION

At low message rates (1 Hz per topic, coalesced at 250 ms) the rolling in-memory deque and the WAL can be unified into a single file: a live WAL that is rewritten on every incoming message and read directly by GET /current. This eliminates the separate in-memory serialisation path and means the WAL is always current — restart replay automatically bridges any gap with no extra code.

Storage requirement: This design is only appropriate when the Parquet output path is on local SSD (e.g. /mnt/tort-sdi/). On NFS / EFS every file rewrite incurs a network round-trip and rename() is not guaranteed atomic across AZ mounts — use the in-memory deque (section 4) instead.

Data flow

MQTT message  (1 Hz, coalesced at 250 ms)
     │
     ▼
parse_payload() ──► append to g_buffers
                          │
                          ▼
                   write_live_wal()          // rewrite full g_buffers → live.arrow
                          │                  // cost: ~60 rows Arrow IPC, once/sec
                          ▼
             /mnt/.../bench/live.arrow       // atomic: write tmp → rename

flush timer fires
     │
     ├──► write_parquet()                    // unchanged
     ├──► rename live.arrow → <ts>.wal.arrow // rotate: live becomes crash-recovery WAL
     └──► evict old rows from g_buffers

GET /current
     │
     ▼
read live.arrow from disk                    // replace in-memory g_buffers snapshot
serve Parquet bytes with Range support

New helper — write_live_wal()

// Called after every parse_payload() that modifies g_buffers.
// Serialises current g_buffers state to live.arrow atomically.
static void write_live_wal(const Config& cfg) {
    // 1. snapshot under lock
    std::map<PartitionKey, Partition> snap;
    {
        std::lock_guard<std::mutex> lk(g_buffers_mutex);
        snap = g_buffers;
    }

    // 2. build Arrow table from snapshot (reuse build_table())
    auto table = build_table(snap);
    if (!table || table->num_rows() == 0) return;

    // 3. write to tmp file then rename (atomic)
    std::string tmp  = cfg.output_dir + "/.live.arrow.tmp";
    std::string dest = cfg.output_dir + "/live.arrow";

    auto sink = *arrow::io::FileOutputStream::Open(tmp);
    auto writer = *arrow::ipc::MakeFileWriter(sink, table->schema());
    ARROW_CHECK_OK(writer->WriteTable(*table));
    ARROW_CHECK_OK(writer->Close());
    ARROW_CHECK_OK(sink->Close());
    std::rename(tmp.c_str(), dest.c_str());
}

Flush rotation — rotate_live_wal()

// Called at the start of each flush cycle, before write_parquet().
// Moves live.arrow out of the way so it becomes the crash-recovery WAL.
static void rotate_live_wal(const Config& cfg, int64_t flush_ts_us) {
    std::string src = cfg.output_dir + "/live.arrow";
    std::string dst = cfg.output_dir + "/"
                    + ts_to_filename(flush_ts_us) + ".wal.arrow";
    if (std::filesystem::exists(src))
        std::rename(src.c_str(), dst.c_str());
    // existing wal_replay_file() handles dst on next restart — no changes needed
}

Updated handle_current()

// Replace in-memory snapshot + Arrow builder with a direct file read.
static void handle_current(int client_fd, const std::string& query,
                            const Config& cfg) {
    std::string path = cfg.output_dir + "/live.arrow";
    if (!std::filesystem::exists(path)) {
        send_404(client_fd); return;
    }
    // Read Arrow IPC → convert to in-memory Parquet bytes → serve with Range
    auto in  = *arrow::io::ReadableFile::Open(path);
    auto rdr = *arrow::ipc::RecordBatchFileReader::Open(in);
    arrow::RecordBatchVector batches;
    for (int i = 0; i < rdr->num_record_batches(); ++i)
        batches.push_back(*rdr->ReadRecordBatch(i));
    auto table = *arrow::Table::FromRecordBatches(batches);
    auto buf   = write_parquet_table_to_buffer(*table);
    send_parquet_response(client_fd, query, buf);
}

Step 4 — handle_current()

Serve current_state.parquet from disk (if current_state_path is set), or serialize g_current_state to a temp buffer in memory and return it directly. Set Content-Type: application/octet-stream.

6. Partial Dataset Queries

All filtering in handle_data() is string comparison on pre-extracted fields. This is intentionally simple — no query language, no index. At 80k msg/s for 60s the deque may hold 4–5M rows; a linear scan filtering by unit takes ~50ms on a single core.

Strategies to keep it fast

  • Require unit= parameter when recent_rows count exceeds a threshold (e.g. 100k rows). Return HTTP 413 with a hint if omitted.
  • Per-unit sub-deques: a std::unordered_map<string, deque> keyed by unit_id gives O(1) unit lookup and eliminates the linear scan. Higher memory overhead but much faster for multi-unit deployments.
  • Limit response rows: add &limit=N to cap output at N rows (default 10,000). Return newest-first so partial results are useful.
Query flow for GET /data?last=30&unit=0215F562&device=BMS │ ├─ parse_query_string(req) → {last:"30", unit:"0215F562", device:"BMS"} ├─ cutoff = now_ms() - 30000 ├─ lock g_recent_mutex ├─ iterate g_recent_rows (deque, LIFO scan from back for recent-first) │ filter: ts_ms ≥ cutoff AND unit_id == "0215F562" AND device == "BMS" ├─ unlock └─ serialize to JSON → send

7. Longer Time-Span Queries

For spans beyond the 60s rolling buffer (e.g. last 10 minutes, last hour), the client must stitch together flushed parquet files from disk with the rolling buffer tail.

Option A — bench_server + DuckDB-WASM (browser-side)

  1. Browser fetches parquet file list from bench_server /api/ls?path=…
  2. For files that fall in the requested time window, fetch each via HTTP and load into DuckDB-WASM
  3. Fetch /current from the writer for the tail
  4. Run a single DuckDB query across all loaded tables
This is already mostly implemented in evelyn_real_time_data.html — the DuckDB-WASM setup and file fetch loop are there. The missing piece is the /current endpoint to close the gap.

Option B — server-side stitch in parquet_writer

Add a GET /query?from=T1&to=T2&unit=X endpoint that:

  1. Scans the output directory for parquet files whose names fall in [T1, T2]
  2. Reads matching files using the Arrow C++ API (same library already linked)
  3. Appends rows from g_recent_rows that fall in the range
  4. Returns combined result as JSON or parquet binary
This is heavier to implement (~300 lines) and couples the query path to the file layout. Option A (browser DuckDB) keeps the writer lean and is sufficient for dashboards.

Option C — bridge_api (existing design)

The evelyn_real_time_data.html document describes a bridge_api approach where a small Python/Go service sits between parquet files and the browser, exposing /query/current, /query/range, and /subscribe (WebSocket). This is the cleanest architecture for production but requires an additional service to deploy and maintain.

Option D — subscriber_api merges files + /current (recommended)

The subscriber_api (Python, server-side DuckDB) already queries date-partitioned parquet files via read_parquet([glob1, glob2, ...], union_by_name=true). Adding the live 60s buffer requires only two changes:

  1. Writer exposes GET /current — serialises the in-memory buffer to parquet bytes on demand and returns them as application/octet-stream
  2. subscriber_api adds the writer HTTP URL directly to the DuckDB source list — no temp file, no separate fetch step. DuckDB's httpfs extension fetches it as part of the same query execution

DuckDB sees one unified source covering all three layers — local files and live HTTP buffer in a single statement:

HTTP Range requests — writer implementation detail

DuckDB's httpfs reads parquet files efficiently using HTTP Range requests. Rather than fetching the entire file, it issues two requests:

  1. Footer fetch — Range: bytes=-8 then Range: bytes=N-M to read the Parquet footer (schema + row group metadata). This tells DuckDB the column layout and row group offsets without reading any data.
  2. Column page fetch — Range: bytes=A-B per column page actually needed by the query's projection and predicate pushdown. Columns not referenced in the SELECT or WHERE are never fetched.

The writer's /current HTTP handler must therefore support the Range header. Implementation in the health thread:

// health_thread_fn() — /current handler with Range support
auto parquet_buf = serialize_buffer_to_parquet();   // Arrow → parquet bytes in memory
size_t total = parquet_buf->size();

// Parse Range header if present: "bytes=START-END" or "bytes=-SUFFIX"
size_t range_start = 0, range_end = total - 1;
bool   is_range    = false;
if (auto pos = request.find("Range: bytes=") != npos) {
    auto spec = request.substr(pos + 13);
    if (spec[0] == '-') {                              // suffix: bytes=-N
        size_t suffix = stoull(spec.substr(1));
        range_start = total - suffix;
    } else {                                            // range: bytes=START-END
        auto dash  = spec.find('-');
        range_start = stoull(spec.substr(0, dash));
        range_end   = stoull(spec.substr(dash + 1));
    }
    is_range = true;
}

size_t      send_len = range_end - range_start + 1;
std::string status   = is_range ? "206 Partial Content" : "200 OK";
std::string hdr =
    "HTTP/1.1 " + status + "\r\n"
    "Content-Type: application/octet-stream\r\n"
    "Accept-Ranges: bytes\r\n"                          // advertise Range support
    "Content-Length: " + to_string(send_len) + "\r\n"
    "Content-Range: bytes " + to_string(range_start) + "-"
                           + to_string(range_end)   + "/"
                           + to_string(total)       + "\r\n"
    "Cache-Control: no-store\r\n\r\n";

write(client_fd, hdr.data(), hdr.size());
write(client_fd, parquet_buf->data() + range_start, send_len);
For the small in-memory /current buffer (typically 50–200 KB for 60s of data), DuckDB will likely fetch the entire file in two Range requests regardless. The overhead of Range handling is minimal and the implementation above covers all cases DuckDB will generate. Without Accept-Ranges: bytes in the response header DuckDB falls back to a full read — functionally correct but slightly less efficient.

Overlap and deduplication

Important: the flush files on disk and the /current buffer will overlap. The writer flushes at T-N seconds but the in-memory buffer still contains all rows since the last flush start. union_by_name=true merges schemas but does not deduplicate rows — a naive union returns duplicate rows for the overlap window.

Two clean solutions:

  1. Preferred — ?since= parameter trims /current to post-flush rows only. The writer tracks last_flush_ts and only serialises rows with ts > since. The subscriber_api passes the newest row timestamp from the most recent flush file — shown in the code above as _newest_row_ts(globs). The writer returns only the true gap — zero overlap, zero duplicates.
  2. Fallback — deduplicate in DuckDB. If since= is not yet implemented on the writer, wrap the query in a window dedup: 
    SELECT * FROM (
    SELECT *, row_number() OVER (PARTITION BY ts, unit_id ORDER BY ts) AS rn
    FROM read_parquet([{sources}], union_by_name=true)
    WHERE timestamp BETWEEN from_ts AND to_ts
) WHERE rn = 1
ORDER BY timestamp
    Slightly heavier but works without any writer changes.

// Serialize an Arrow table to an in-memory parquet buffer.
// Used by handle_current() to serve the live g_buffers snapshot over HTTP.
// Returns nullptr on error (logs to stderr).
static std::shared_ptr<arrow::Buffer>
write_parquet_table_to_buffer(const std::shared_ptr<arrow::Table>& table,
                               const std::string& compression = "snappy") {
    auto props = parquet::WriterProperties::Builder()
        .compression(to_parquet_compression(compression))
        ->build();

    // Step 1 — open a resizable in-memory sink instead of a file
    auto sink_res = arrow::io::BufferOutputStream::Create();
    if (!sink_res.ok()) {
        std::cerr << "[current] BufferOutputStream::Create: "
                  << sink_res.status() << "\n";
        return nullptr;
    }
    auto sink = *sink_res;

    // Step 2 — write Parquet into the sink (identical call to the file variant)
    auto st = parquet::arrow::WriteTable(
        *table, arrow::default_memory_pool(), sink,
        static_cast<int64_t>(table->num_rows()), props);
    if (!st.ok()) {
        std::cerr << "[current] WriteTable: " << st << "\n";
        return nullptr;
    }

    // Step 3 — finalise the sink; returns an immutable arrow::Buffer
    auto buf_res = sink->Finish();
    if (!buf_res.ok()) {
        std::cerr << "[current] Finish: " << buf_res.status() << "\n";
        return nullptr;
    }
    return *buf_res;   // caller owns the buffer; it stays alive until last ref drops
}
WHY BufferOutputStream instead of a temp file

arrow::io::BufferOutputStream is an Arrow-managed resizable byte buffer that satisfies the same arrow::io::OutputStream interface as FileOutputStream. parquet::arrow::WriteTable() receives a pointer to the interface — it has no idea whether bytes go to disk or memory. The three steps map directly onto the file variant:

File variantBuffer variant
FileOutputStream::Open(path) BufferOutputStream::Create()
WriteTable(..., out, ...) WriteTable(..., sink, ...) — identical
file on disk sink->Finish() → shared_ptr<arrow::Buffer>

Using a temp file would work but adds: a mkstemp() call, a filesystem write, a file-descriptor open + mmap/read to serve the bytes, and a unlink(). The buffer approach keeps everything in process memory and is the pattern Arrow recommends for network serialisation.

No new library dependency. BufferOutputStream is in <arrow/io/memory.h>, part of libarrow which is already linked. No Makefile change is needed.

A Parse ?since=UNIX_S from the request line 
static void handle_current(int client_fd, const std::string& req) {

    // ── A: parse ?since=UNIX_S (optional) ─────────────────────────────
    // since_us = 0 means "return all buffered rows" (no filter).
    // Caller (subscriber_api) passes the Unix timestamp of the newest row in
    // the most recent flush file — the writer then returns only rows newer
    // than that, which is exactly the gap between the last flush and now.
    // This eliminates row duplication when DuckDB unions disk files + /current.
    int64_t since_us = 0;
    auto    q = req.find("?since=");
    if (q != std::string::npos) {
        try {
            double s = std::stod(req.substr(q + 7));   // e.g. "1744473600.123"
            since_us = static_cast<int64_t>(s * 1'000'000.0);  // → µs
        } catch (...) {}  // malformed → since_us stays 0, return all rows
    }

The ?since= value is a Unix timestamp in decimal seconds (float), matching the format returned by Python's time.time(). It is converted to microseconds to match the int64_t stored in row.ints["ts"].

If the parameter is absent or malformed, since_us = 0 and all buffered rows are returned — safe fallback for a fresh query with no prior flush files.

B Snapshot g_buffers under the global mutex 
    // ── B: snapshot g_buffers under mutex ─────────────────────────────
    // g_buffers is a std::map<PartitionKey, Partition> — the same map that
    // parse_thread fills and flush_thread drains.  We take the same g_mutex
    // that flush_thread uses, copy the rows we need, then release immediately.
    // All expensive work (Arrow build, Parquet serialise, HTTP write) happens
    // outside the lock so the MQTT parse path is never blocked.
    std::vector<Row> rows;
    {
        std::lock_guard<std::mutex> lk(g_mutex);        // same lock as flush path
        for (const auto& [key, part] : g_buffers) {     // iterate all partitions
            for (const auto& r : part.rows) {            // iterate rows in each
                if (since_us > 0) {
                    auto it = r.ints.find("ts");
                    // skip rows whose ts is at or before the ?since= cutoff
                    if (it != r.ints.end() && it->second <= since_us) continue;
                }
                rows.push_back(r);                       // copy Row (cheap — maps)
            }
        }
    }   // <— mutex released here; Row copies are now owned by this thread

Lock scope is minimal. The mutex is held only while iterating and copying — typically <1 ms even for a full 60-second buffer of 12-unit wide-pivot data (~50k rows). The subsequent Arrow build, Parquet serialisation, and TCP write all happen after the lock is released, so parse_thread is never delayed.

g_buffers is a std::map<PartitionKey, Partition> where each Partition holds a std::vector<Row>. A Row is three std::map fields (ints/floats/strings). The copy cost is proportional to the number of rows times the number of signal columns — for a 60-second 12-unit bench run at 1 Hz with 63 signals, that is ~720 rows.

D Parse the HTTP Range header 
    // ── D: Range header parsing ───────────────────────────────────────
    // DuckDB httpfs NEVER fetches a parquet file in one shot. It always uses
    // Range requests to minimise data transfer:
    //
    //   Request 1:  Range: bytes=-8
    //               → last 8 bytes of the file (Parquet magic "PAR1" + footer length int32)
    //               → from these 4 bytes DuckDB learns the footer size N
    //
    //   Request 2:  Range: bytes=TOTAL-N-TOTAL-9
    //               → the Parquet footer block (schema, row group metadata, column offsets)
    //               → DuckDB now knows every column name, type, and byte range
    //
    //   Request 3+: Range: bytes=A-B  (one per column needed by the query)
    //               → only columns referenced in SELECT list / WHERE clause
    //               → columns not in the query are never fetched
    //
    // Without Accept-Ranges: bytes in our response, DuckDB falls back to a full
    // GET (no Range header), reads the whole file, and is still correct — but
    // slightly less efficient.  For a small /current buffer (50–200 KB) the
    // difference is negligible, but we implement Range for correctness.

    size_t total       = static_cast<size_t>(buf->size());
    size_t range_start = 0, range_end = total - 1;  // default: entire buffer
    bool   is_range    = false;

    auto   rpos = req.find("Range: bytes=");
    if (rpos != std::string::npos) {
        std::string spec = req.substr(rpos + 13);                 // skip "Range: bytes="
        auto nl = spec.find_first_of("\r\n");
        if (nl != std::string::npos) spec = spec.substr(0, nl);   // trim to end of header line
        try {
            if (spec[0] == '-') {
                // Suffix form: bytes=-8 → last 8 bytes (Parquet magic check)
                size_t suffix = std::stoull(spec.substr(1));
                range_start = total > suffix ? total - suffix : 0;
            } else {
                // Range form: bytes=START-END (footer block or column page)
                auto dash   = spec.find('-');
                range_start = std::stoull(spec.substr(0, dash));
                range_end   = std::stoull(spec.substr(dash + 1));
            }
            is_range = true;
        } catch (...) {}    // malformed → serve full buffer (200 OK)
    }
    // Clamp to valid byte range
    if (range_end   >= total)     range_end   = total - 1;
    if (range_start >  range_end) range_start = range_end;
    size_t send_len = range_end - range_start + 1;

Byte rangebytes=12345-67890 Bytes from START to END inclusive Footer block read; then each column page read
