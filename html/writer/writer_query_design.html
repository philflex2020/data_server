1. The Gap Problem

parquet_writer buffers incoming MQTT messages and flushes to disk every flush_interval_seconds (typically 60s). Between flushes, in-memory rows are inaccessible to any external reader. A dashboard asking "what is happening right now?" has three distinct time regions to cover:

✔ Flushed parquet files
t=0 → t=(now − flush_interval)
Readable via DuckDB / bench_server
⚠ THE GAP
Last flush → now
Up to 60s invisible to external readers
★ current_state
Latest value per signal
Written atomically on each flush

The gap contains rows already parsed and sitting in g_buffer (the in-memory partition map), but nothing exposes them via HTTP today. A rolling in-memory ring bridges this gap without requiring a flush and without adding write latency.

2. What Already Exists DONE

Defined at parquet_writer.cpp:1177:

This gives instant O(log n) lookup of the latest row for any sensor. It is the "now" snapshot.

Config option at line 165. After every successful flush, the full g_current_state map is serialized to a temp file and atomically renamed (lines 1340–1351):

Clients can fetch this file. It's always consistent (never half-written) and covers the latest known value for every signal even if the writer restarts.

4. Rolling 60s Buffer Design

Add a deque of rows that is appended on every MQTT parse and trimmed to the last N seconds. The buffer is read-locked only for serialisation (the same brief lock pattern as the existing buffer flush).

// parquet_writer.cpp — near g_current_state (line 1177)

static constexpr int RECENT_ROWS_SECONDS = 60;
struct TimestampedRow {
    int64_t     ts_ms;
    Row         row;
    std::string unit_id;    // pre-extracted for fast filter
    std::string device;
    std::string point_name;
};
static std::deque<TimestampedRow>  g_recent_rows;
static std::mutex                  g_recent_mutex;

Push side — in parse_thread (after g_current_state update)

// After the g_current_state insert at line 1861:
{
    TimestampedRow tr{ row.ts_ms, row,
                       get_string_col(row, "unit_id"),
                       get_string_col(row, "device"),
                       get_string_col(row, "point_name") };
    std::lock_guard<std::mutex> lk(g_recent_mutex);
    g_recent_rows.push_back(std::move(tr));
}
// Trim in a background pass (or lazily on each /data request):
auto cutoff = now_ms() - RECENT_ROWS_SECONDS * 1000;
while (!g_recent_rows.empty() && g_recent_rows.front().ts_ms < cutoff)
    g_recent_rows.pop_front();

4b. DuckDB In-Process Query Engine TODO

Instead of a hand-rolled deque + linear scan, embed DuckDB in-process as the rolling buffer store. parquet_writer already links Arrow; DuckDB is a single shared library (~6 MB) that uses the same Arrow columnar format internally. The payoff: GET /data becomes a SQL query — arbitrary WHERE clauses, aggregations, and window functions with zero custom filter code.

Setup — in-memory database on startup

// parquet_writer.cpp — add near other globals
#include "duckdb.hpp"   // link -lduckdb  (see Makefile section below)

static duckdb::DuckDB     g_duck{nullptr};           // nullptr = in-memory
static duckdb::Connection g_duck_write{g_duck};       // owned by parse_thread
static duckdb::Connection g_duck_read {g_duck};       // owned by health_thread (read queries)

// On startup, before parse_thread starts:
g_duck_write.Query(R"SQL(
    CREATE TABLE recent (
        ts_ms      BIGINT  NOT NULL,
        unit_id    VARCHAR,
        device     VARCHAR,
        point_name VARCHAR,
        float_val  DOUBLE,
        int_val    BIGINT,
        bool_val   BOOLEAN,
        string_val VARCHAR
    )
)SQL");

Insert path — Appender API (no SQL parsing overhead)

The DuckDB Appender bypasses the SQL layer — it's essentially a column-batched memcpy. Accumulate rows in a small staging vector, flush the Appender every parse-thread loop iteration (or every 500 rows), then run the trim DELETE periodically (e.g. every 5 seconds):

// In parse_thread — after g_current_state update:
if (g_cfg->recent_rows_seconds > 0) {
    staged.push_back({row.ts_ms, unit_id, device, point_name,
                      float_val, int_val, bool_val, string_val});
}

// Flush batch (every N rows or on timeout):
if (staged.size() >= 500 || elapsed_since_flush > 100ms) {
    duckdb::Appender app(g_duck_write, "recent");
    for (auto& r : staged) {
        app.BeginRow();
        app.Append<int64_t>(r.ts_ms);
        app.Append<std::string>(r.unit_id);
        app.Append<std::string>(r.device);
        app.Append<std::string>(r.point_name);
        app.Append<double>(r.float_val);
        app.Append<int64_t>(r.int_val);
        app.Append<bool>(r.bool_val);
        app.Append<std::string>(r.string_val);
        app.EndRow();
    }
    app.Flush();
    staged.clear();
}

// Trim old rows (run every ~5 seconds, not per-row):
g_duck_write.Query("DELETE FROM recent WHERE ts_ms < "
    + std::to_string(now_ms() - g_cfg->recent_rows_seconds * 1000LL));

Query path — handle_data() with SQL

Build a parameterized WHERE clause from the URL params and let DuckDB do the filtering. Use DuckDB's prepared statements to avoid SQL injection:

// In handle_data() — health_thread uses g_duck_read:
auto params  = parse_query_string(req);
int  last_s  = params.count("last") ? std::stoi(params["last"]) : 30;
int64_t cutoff = now_ms() - (int64_t)last_s * 1000;

// Build safe WHERE clause — only known column names, values via prepared stmt
std::string sql =
    "SELECT ts_ms, unit_id, device, point_name, float_val, int_val, bool_val"
    " FROM recent WHERE ts_ms >= $1";
std::vector<duckdb::Value> binds;
binds.emplace_back(cutoff);

if (params.count("unit"))   { sql += " AND unit_id = $2";    binds.emplace_back(params["unit"]);   }
if (params.count("device")) { sql += " AND device = $" + ...; binds.emplace_back(params["device"]); }
if (params.count("point"))  { sql += " AND point_name = $" + ...; }
sql += " ORDER BY ts_ms DESC LIMIT 10000";

auto stmt = g_duck_read.Prepare(sql);
auto res  = stmt->Execute(binds);

// Serialize result to JSON:
std::ostringstream body;
body << "[";
bool first = true;
for (auto& chunk : *res) {
    for (int i = 0; i < chunk.size(); ++i) {
        if (!first) body << ",";
        body << "{\"ts\":"     << chunk.GetValue<int64_t>(0, i)
             << ",\"unit\":\"" << chunk.GetValue<std::string>(1, i) << "\""
             << ",\"device\":\"" << chunk.GetValue<std::string>(2, i) << "\""
             << ",\"point\":\"" << chunk.GetValue<std::string>(3, i) << "\""
             << ",\"v\":"      << chunk.GetValue<double>(4, i)
             << "}";
        first = false;
    }
}
body << "]";

SQL pass-through option

Optionally expose GET /data?sql=SELECT+... for dashboards that want arbitrary queries against the rolling table. Guard against non-SELECT statements:

if (params.count("sql")) {
    auto sql = url_decode(params["sql"]);
    // Reject anything that isn't a plain SELECT
    if (sql.rfind("SELECT", 0) != 0 && sql.rfind("select", 0) != 0) {
        send_error(client_fd, 403, "only SELECT allowed");
        return;
    }
    auto res = g_duck_read.Query(sql);
    // ... serialize to JSON as above
}
The recent table is in-memory only. SQL pass-through cannot escape the rolling window — SELECT * will return at most the last recent_rows_seconds of data.

Thread safety

DuckDB supports multiple connections to the same in-memory database with MVCC isolation. Use two connections — one per thread — so the Appender in parse_thread never blocks queries from health_thread:

  • g_duck_write — parse_thread only: Appender + DELETE trims
  • g_duck_read — health_thread only: SELECT queries
  • No mutex needed between them — DuckDB serializes internally
Do not share a single duckdb::Connection across threads — it is not thread-safe. Two connections from the same duckdb::DuckDB instance is the correct pattern.

Step 2 — Add g_recent_rows and push/trim logic

After the g_current_state insert in parse_thread_fn(), append TimestampedRow and trim the front. Trim can be done lazily (on each request) or eagerly (every N pushes).

Step 3 — Extend health_thread_fn() to route new paths

The request routing block currently rejects everything except /health. Replace it:

bool is_health  = req.find(" /health ")  != npos || req.find(" /health\r")  != npos;
bool is_current = req.find(" /current") != npos;
bool is_data    = req.find(" /data?")   != npos || req.find(" /data ")   != npos
                                                    || req.find(" /data\r") != npos;

if      (is_health)  handle_health(client_fd);
else if (is_current) handle_current(client_fd, req);
else if (is_data)    handle_data(client_fd, req);
else                  send_404(client_fd);

Step 4 — handle_current()

Serve current_state.parquet from disk (if current_state_path is set), or serialize g_current_state to a temp buffer in memory and return it directly. Set Content-Type: application/octet-stream.

// Simplest form — stream the file from disk:
if (!g_cfg->current_state_path.empty() && fs::exists(g_cfg->current_state_path)) {
    serve_file(client_fd, g_cfg->current_state_path,
               "application/octet-stream");
} else {
    // Fallback: serialise g_current_state on-the-fly
    auto snap = snapshot_current_state();   // lock + copy
    auto buf  = serialize_to_parquet_buffer(snap);
    send_binary_response(client_fd, buf, "application/octet-stream");
}

Step 5 — handle_data()

// Parse ?last=N&unit=X&device=Y&point=Z from request line
auto params = parse_query_string(req);
int  last_s  = params.count("last")   ? std::stoi(params["last"])   : 30;
auto u_filt  = params.count("unit")   ? params["unit"]   : "";
auto d_filt  = params.count("device") ? params["device"] : "";
auto p_filt  = params.count("point")  ? params["point"]  : "";

int64_t cutoff = now_ms() - (int64_t)last_s * 1000;

std::vector<TimestampedRow> result;
{
    std::lock_guard<std::mutex> lk(g_recent_mutex);
    for (const auto& tr : g_recent_rows) {
        if (tr.ts_ms < cutoff) continue;
        if (!u_filt.empty() && tr.unit_id    != u_filt) continue;
        if (!d_filt.empty() && tr.device     != d_filt) continue;
        if (!p_filt.empty() && tr.point_name != p_filt) continue;
        result.push_back(tr);
    }
}
// Serialize result to JSON and send with Content-Type: application/json

6. Partial Dataset Queries

All filtering in handle_data() is string comparison on pre-extracted fields. This is intentionally simple — no query language, no index. At 80k msg/s for 60s the deque may hold 4–5M rows; a linear scan filtering by unit takes ~50ms on a single core.

Strategies to keep it fast

  • Require unit= parameter when recent_rows count exceeds a threshold (e.g. 100k rows). Return HTTP 413 with a hint if omitted.
  • Per-unit sub-deques: a std::unordered_map<string, deque> keyed by unit_id gives O(1) unit lookup and eliminates the linear scan. Higher memory overhead but much faster for multi-unit deployments.
  • Limit response rows: add &limit=N to cap output at N rows (default 10,000). Return newest-first so partial results are useful.
Query flow for GET /data?last=30&unit=0215F562&device=BMS │ ├─ parse_query_string(req) → {last:"30", unit:"0215F562", device:"BMS"} ├─ cutoff = now_ms() - 30000 ├─ lock g_recent_mutex ├─ iterate g_recent_rows (deque, LIFO scan from back for recent-first) │ filter: ts_ms ≥ cutoff AND unit_id == "0215F562" AND device == "BMS" ├─ unlock └─ serialize to JSON → send

7. Longer Time-Span Queries

For spans beyond the 60s rolling buffer (e.g. last 10 minutes, last hour), the client must stitch together flushed parquet files from disk with the rolling buffer tail.

Option A — bench_server + DuckDB-WASM (browser-side)

  1. Browser fetches parquet file list from bench_server /api/ls?path=…
  2. For files that fall in the requested time window, fetch each via HTTP and load into DuckDB-WASM
  3. Fetch /current from the writer for the tail
  4. Run a single DuckDB query across all loaded tables
This is already mostly implemented in evelyn_real_time_data.html — the DuckDB-WASM setup and file fetch loop are there. The missing piece is the /current endpoint to close the gap.

Option B — server-side stitch in parquet_writer

Add a GET /query?from=T1&to=T2&unit=X endpoint that:

  1. Scans the output directory for parquet files whose names fall in [T1, T2]
  2. Reads matching files using the Arrow C++ API (same library already linked)
  3. Appends rows from g_recent_rows that fall in the range
  4. Returns combined result as JSON or parquet binary
This is heavier to implement (~300 lines) and couples the query path to the file layout. Option A (browser DuckDB) keeps the writer lean and is sufficient for dashboards.

Option C — bridge_api (existing design)

The evelyn_real_time_data.html document describes a bridge_api approach where a small Python/Go service sits between parquet files and the browser, exposing /query/current, /query/range, and /subscribe (WebSocket). This is the cleanest architecture for production but requires an additional service to deploy and maintain.

Option D — subscriber_api merges files + /current (recommended)

The subscriber_api (Python, server-side DuckDB) already queries date-partitioned parquet files via read_parquet([glob1, glob2, ...], union_by_name=true). Adding the live 60s buffer requires only two changes:

  1. Writer exposes GET /current — serialises the in-memory buffer to parquet bytes on demand and returns them as application/octet-stream
  2. subscriber_api fetches /current, writes bytes to a temp file, and appends that path to the DuckDB source list before running the query

DuckDB then sees one unified source covering all three layers:

-- what DuckDB runs inside subscriber_api._query_history()
SELECT * FROM read_parquet([
    '.../2026/04/12/*.parquet',   -- compact + flush files on disk
    '.../2026/04/11/*.parquet',   -- previous day if range spans midnight
    '/tmp/tmpXXXX.parquet'        -- live buffer fetched from writer /current
], union_by_name=true)
WHERE timestamp BETWEEN from_ts AND to_ts
ORDER BY timestamp
LIMIT :limit

Overlap and deduplication

Important: the flush files on disk and the /current buffer will overlap. The writer flushes at T-N seconds but the in-memory buffer still contains all rows since the last flush start. union_by_name=true merges schemas but does not deduplicate rows — a naive union will return duplicate rows for the overlap window.

Two clean solutions:

  1. Preferred — writer trims /current to post-flush rows only. The writer already tracks last_flush_ts. The /current endpoint accepts an optional ?since=<unix_ts> parameter and only serialises rows with ts > since. subscriber_api passes the timestamp of the newest row in the most-recent flush file: 
    # subscriber_api — pass last flush boundary to /current
since_ts = _newest_row_ts(globs)   # scan last file, read max(ts)
current_tmp = _fetch_current(writer_url, since=since_ts)
    The writer then returns only the true gap — zero overlap, zero duplicates.
  2. Fallback — deduplicate in DuckDB. If since= is not yet implemented on the writer, wrap the query in a SELECT DISTINCT or group by primary key: 
    SELECT * FROM (
    SELECT *, row_number() OVER (PARTITION BY ts, unit_id ORDER BY ts) AS rn
    FROM read_parquet([{sources}], union_by_name=true)
    WHERE timestamp BETWEEN from_ts AND to_ts
) WHERE rn = 1
ORDER BY timestamp
    Slightly heavier but works without any writer changes.

9. Notes from Related Designs

evelyn_real_time_data.html — bridge_api design

This document (also in html/writer/) was the original design for a real-time web interface. Key points relevant here:

  • FlashMQ WebSocket: browsers can subscribe directly to FlashMQ's MQTT-over-WebSocket (port 8083 / 9001) for true push. No polling needed. This bypasses parquet_writer entirely for live values — useful for dashboards that need <1s latency.
  • DuckDB-WASM + parquet files: the demo in that file shows how to fetch a parquet binary via HTTP, register it in DuckDB-WASM, and run SQL queries. This is the foundation for the /current endpoint workflow above.
  • /query/current: the bridge_api in that design exposed a JSON endpoint of the same shape as /data above. The JSON schema proposed there is a good reference.
  • Topic-to-sensor mapping: the bridge_api carried the topic-parser logic (positional segments → unit_id/device/point_name). In the parquet_writer approach, that mapping is already done by the time rows reach g_recent_rows.
