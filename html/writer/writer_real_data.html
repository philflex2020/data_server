Topic Format

Every message uses a 6-segment positional topic. The dtype_hint in segment 5 is a parse-time type signal — it is not stored as a column, but it drives whether value_i (INT64) or value_f (FLOAT64) is populated in the parquet row.

unit / {unit_id} / {device} / {instance} / {point_name} / {dtype_hint} 0 1 2 3 4 5 Example (short): unit/02281660/pcs/pcs_1/PCS_1PCS_Grid_Frequency/float (54 bytes) Example (long): unit/02281660/bms/bms_1/Batt1_MBMS_System_warning_and_fault_SBMU_Big_temperature_difference_between_cells_warning_level_3/integer (129 bytes — was truncated at 128) Example (longest — 136 bytes, worst truncation): unit/02281660/bms/bms_1/Batt1_MBMS_System_warning_and_fault_SBMU_Both_main_positive_relay_and_main_negative_relay_sticking_fault/integer

Payload format

Every payload is the same flat JSON structure — a timestamp and a single value:

{"ts":"2024-11-15T21:27:52.775Z","value":0.999393}   // float
{"ts":"2024-11-15T21:27:52.775Z","value":1}           // integer / boolean_integer

Typical payload length: 48–55 bytes. MQRING_PAY_MAX=256 is safe for all observed values.

MQTT System Analysis

Message flow

46 units × 1,770 topics = 81,420 msgs/sec at 1s publish interval Each unit publishes: bms_1 / bms_2 595 + 667 = 1,262 topics ← MBMS system-level data bms_x_rack_1..5 10 racks × 28 = 280 topics ← per-rack cell data pcs_1 / pcs_2 114 + 114 = 228 topics ← power conversion Topic: unit/{unit_id}/{device}/{instance}/{point_name}/{dtype_hint} Payload: {"ts":"...ISO8601...","value": N} MQTT → FlashMQ broker → libmosquitto on_message → MqttRing (lock-free SPSC) → parse_thread_fn → parse_topic_positional() + parse_payload() → g_buffer (partitioned by site_id / year / month / day) → flush_thread_fn (every 60s or on buffer overflow) → WAL + Parquet file

Partition structure

Under the real_writer config, each parquet file covers one site_id / year / month / day partition. All 81k rows/sec from all devices, instances, and point names land in the same daily file — distinguished at query time by the device, instance, and point_name columns. site_id is injected from config, not the topic — saving 8 bytes on every single message.

Why split value_f / value_i? A single FLOAT64 column loses integer precision for large counters and boolean flags (which should be exactly 0 or 1). Keeping them separate costs one null per row but preserves type fidelity. At the query layer, a COALESCE(value_i, CAST(value_f AS INT64)) covers both when needed.

First-10s burst (retained messages)

The first 10 seconds contain 183,847 messages — roughly 6× the typical 10-second window rate. Of these, 26,590 (14.5%) have the MQTT retained flag set. After t=10s, retained count drops to exactly zero. This pattern is characteristic of a subscriber connecting to a broker that has retained the last known value for each topic: the broker immediately replays all retained messages before the live stream begins.

The remaining 1,118,756 messages are all live (retained=False), spread across the rest of the recording at ~3,000–43,000 per 10-second bucket depending on how many units happen to be in their polling window.

Gap character — per-device polling, not broker outage

During the largest gap (12.3 s), only 20 of 47 units were active in the 5 s before and only 14 resumed in the 5 s after. The overlap is 12 units. This rules out a broker restart or network-wide outage — if the broker had crashed, all 47 units would have dropped simultaneously and all would resume together. Instead, each unit EMS device publishes on its own independent polling schedule. The "gaps" are windows where no unit happened to push a batch.

Gap distribution is uniform: 9–20 gaps > 1 s per minute across all 8 minutes of recording. There is no "dead zone" — the gaps are a structural property of the per-device polling cadence, not a recording artifact or connectivity issue.

If each unit has 1,770 topics and should publish every point on some cycle, the expected rate at full coverage would be 81,420 / 47 ≈ 1,732 msg/s per unit. The observed 30–120 msg/s per unit means each unit is publishing roughly 1 topic per 15–60 seconds on average — consistent with a slow Modbus polling loop querying each register in sequence.

Fractal/ topic schema (18,770 messages)

The 35,930 "other-format" rows break down as:

  • 18,770 Fractal/{unit_id}/{dtype}/{point_name}/{ip}/{uuid} — Modbus/OPC-UA connection-status topics published by the Fractal SCADA layer. All 47 unit IDs appear here. These are connection diagnostics (CONNECTION-PQ_1, CONNECTION-PCS_1, etc.), not battery-cell measurements. 77 distinct Fractal/{unit}/{dtype} prefixes observed.
  • ~17,160 site-level topics — ems/site/{SiteSN}/unit/{unit}/… with an extra leading ems/site/… prefix before the unit/ segment, breaking the 6-segment positional parse. These are site-aggregation topics, not per-cell data.

Neither of these schemas matches the unit/{id}/{device}/{instance}/{point_name}/{dtype_hint} format that parse_topic() handles. Both will parse as unknown topic and be dropped (or stored in a catch-all bucket). This is expected behavior — parse_topic() should remain strict.

Conclusion: There is no systemic data loss in the recording. The gaps are an artifact of the site's slow Modbus polling cadence — each EMS polls its 1,770 registers sequentially, producing 30–120 msg/s per unit rather than the sustained 1,732/s that would be needed for full coverage. The recording captures a real production site operating normally. The Fractal/ topics are a second topic namespace (connection telemetry) that writer.cpp correctly ignores.
