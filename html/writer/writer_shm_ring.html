Motivation

Current MQTT input path — 4 hops
stress_runner (Python) → JSON serialise → paho-mqtt TCP publish → mosquitto/FlashMQ broker → libmosquitto callback writer.cpp → simdjson parse → g_buffer
  • Rate ceiling: ~80–100k msg/sec (broker throughput limit)
  • Broker crash = all in-flight messages lost
  • writer.cpp crash = WAL replay covers flushed batches, but rows received-not-yet-flushed are gone
  • Restart window: ~5s × 80k/s = ~400k rows at risk
SHM ring input path — 1 hop
Producer (C/C++ or Python/ctypes) → memcpy Row into slot → atomic write_cursor++ → eventfd notify writer.cpp shm_reader thread → read from read_cursor → g_buffer (same path as MQTT)
  • Rate ceiling: memory bandwidth (~5–10M rows/sec)
  • Ring lives in /dev/shm — survives writer.cpp restart
  • Producer keeps writing during crash; new owner resumes from read_cursor
  • Zero rows lost during restart window (ring sized to cover it)
The ring buffer is an additional input path alongside MQTT, not a replacement. MQTT remains for remote/bridged producers. SHM ring is for same-host producers that need lossless restart semantics or throughput above the broker ceiling.

Ring Header Layout

The entire shared region is a single mmap of a shm_open descriptor. The header occupies the first cache-line-aligned block; slots follow immediately after.

struct ShmRingHeader {
    // ── Identity ─────────────────────────────────────────────────────
    uint64_t magic;          // 0x50415251 57524954 ("PARQWRIT") — sanity on attach
    uint32_t version;        // ABI version — mismatch → unlink and recreate
    uint32_t capacity;       // number of slots, must be power of 2
    uint32_t slot_bytes;     // sizeof(ShmSlot) — fixed at ring creation
    uint32_t _pad0;

    // ── Ownership ────────────────────────────────────────────────────
    pid_t    owner_pid;      // current reader PID (writer.cpp); 0 = unclaimed
    uint64_t heartbeat_ns;   // CLOCK_MONOTONIC ns, updated every ~1s by owner
                             // stale (> 5s) → owner considered dead even if PID alive

    // ── Cursors ──────────────────────────────────────────────────────
    std::atomic<uint64_t> write_cursor; // producer advances with fetch_add
    std::atomic<uint64_t> read_cursor;  // owner advances after committing to g_buffer
                                         // this is the lossless takeover point

    // ── Wake word (polling fallback) ─────────────────────────────────
    std::atomic<uint64_t> wake_seq;     // producer increments; reader wakes on change
                                         // used with futex(FUTEX_WAIT/WAKE) for low latency
                                         // fallback: reader polls with 1ms nanosleep when futex unavailable
    char     _pad1[64];  // pad to cache-line boundary before slots
};
No eventfd in the header. File descriptors are process-local — storing an eventfd in shared memory doesn't work across restarts (the new owner can't use the old fd). Instead, the reader wakes via futex(FUTEX_WAIT) on wake_seq, with a 1 ms nanosleep fallback. The producer increments wake_seq after writing each slot and calls futex(FUTEX_WAKE, 1). This is purely in-kernel — no fd involved.
read_cursor is the ownership boundary. It is only advanced by the current owner after a row has been committed to g_buffer (and thus covered by the WAL before the next flush). A new owner that starts from read_cursor will never process a row twice and will never skip a row the old owner hadn't yet committed.

Slot Layout

Row uses std::unordered_map (F3-6) and is not trivially copyable. The ring uses a fixed binary slot — no heap pointers, safe to memcpy across process boundaries. The reader converts a slot back into a Row before inserting into g_buffer.

// shm_ring.h — fixed-layout slot: 256 bytes, no heap pointers

static constexpr int SHM_MAX_TOPIC  = 128;
static constexpr int SHM_MAX_FLOATS = 20;  // float64 fields per row
static constexpr int SHM_MAX_INTS   = 12;  // int64 fields per row
static constexpr int SHM_MAX_KV     = 8;   // topic kv pairs (rack=3, module=2, etc.)

struct ShmKV {
    char    key[16];
    int32_t val;
};

struct ShmField {
    char    name[16];
    union { double f; int64_t i; } val;
};

struct ShmSlot {
    uint8_t  ready;              // set to 1 by producer AFTER memcpy (multi-producer fence)
    uint8_t  n_floats;           // number of valid float fields
    uint8_t  n_ints;             // number of valid int fields
    uint8_t  n_kv;               // number of topic kv pairs
    char     source_type[32];   // e.g. "batteries", "solar"
    uint64_t ts_ns;              // CLOCK_REALTIME ns (0 = let writer use its own clock)
    ShmKV    kv[SHM_MAX_KV];    // rack=3, module=2, cell=5, etc.
    ShmField floats[SHM_MAX_FLOATS];
    ShmField ints[SHM_MAX_INTS];
    uint8_t  _pad[6];            // total = 256 bytes
};
static_assert(sizeof(ShmSlot) == 256, "ShmSlot must be 256 bytes");

// Portable spin hint — used in reader's ready-flag spin loop
// x86: PAUSE reduces speculative load pressure in spin loops
// ARM64: yield hints the CPU to prefer another thread/hyperthread
// Other: no-op — correctness unaffected, slight efficiency loss
#if defined(__x86_64__) || defined(__i386__)
#  define CPU_PAUSE() __builtin_ia32_pause()
#elif defined(__aarch64__) || defined(__arm__)
#  define CPU_PAUSE() __asm__ volatile("yield" ::: "memory")
#else
#  define CPU_PAUSE() do {} while(0)
#endif

Serialize/deserialize helpers used by producer and shm_reader_fn():

// Producer side — fill a slot from a TopicInfo + Row
inline void row_to_slot(ShmSlot& s, const TopicInfo& ti, const Row& r, uint64_t ts_ns = 0) {
    s.ready    = 0;  // cleared first — producer sets last
    s.ts_ns    = ts_ns;
    strncpy(s.source_type, ti.source_type.c_str(), sizeof(s.source_type) - 1);
    s.n_kv = 0;
    for (auto& [k, v] : ti.kv) {
        if (s.n_kv >= SHM_MAX_KV) break;
        strncpy(s.kv[s.n_kv].key, k.c_str(), 15);
        s.kv[s.n_kv++].val = v;
    }
    s.n_floats = 0;
    for (auto& [k, v] : r.floats) {
        if (s.n_floats >= SHM_MAX_FLOATS) break;
        strncpy(s.floats[s.n_floats].name, k.c_str(), 15);
        s.floats[s.n_floats++].val.f = v;
    }
    s.n_ints = 0;
    for (auto& [k, v] : r.ints) {
        if (s.n_ints >= SHM_MAX_INTS) break;
        strncpy(s.ints[s.n_ints].name, k.c_str(), 15);
        s.ints[s.n_ints++].val.i = v;
    }
    std::atomic_thread_fence(std::memory_order_release);
    s.ready = 1;  // published last — reader polls this
}

// Reader side — reconstruct TopicInfo + Row from a slot
inline void slot_to_row(const ShmSlot& s, TopicInfo& ti, Row& r) {
    ti.source_type = s.source_type;
    for (int i = 0; i < s.n_kv;    ++i) ti.kv[s.kv[i].key]       = s.kv[i].val;
    for (int i = 0; i < s.n_floats; ++i) r.floats[s.floats[i].name] = s.floats[i].val.f;
    for (int i = 0; i < s.n_ints;   ++i) r.ints[s.ints[i].name]    = s.ints[i].val.i;
    r.ts_ns = s.ts_ns ? s.ts_ns : clock_realtime_ns();
}

The ready flag and CPU_PAUSE spin

fetch_add on write_cursor is atomic — the producer claims a slot index instantly. But then it still has to copy data into that slot. There is a window between claiming the index and finishing the write where the reader could arrive at that slot and read garbage. The ready byte closes that window:

producer: slot_idx = write_cursor.fetch_add(1) ← atomic, returns instantly row_to_slot(*slot, ti, r) ← non-atomic, takes real time slot->ready = 0 (first line in row_to_slot) ... memcpy fields ... fence(release) slot->ready = 1 ← written last; all field writes visible before this reader: rc catches up to slot_idx while (!slot->ready) CPU_PAUSE() ← spins until data is actually there fence(acquire) ← all slot fields now visible slot_to_row(*slot, ti, r) ← safe to read slot->ready = 0 ← cleared for reuse on next lap

Without ready the reader would read whatever bytes happened to be in the slot from the previous lap — a silent data corruption, not a crash.

The spin is deadline-bounded at 1 ms. In normal operation the producer sets ready = 1 within nanoseconds and the deadline never fires. If the producer is killed (SIGKILL, OOM, container stop) between fetch_add and the ready = 1 write, the slot will never become ready. Without a deadline the reader would spin forever: read_cursor stalls, the ring fills, the writer stops ingesting. With the deadline the stuck slot is skipped after 1 ms, read_cursor advances, and the ring drains normally. The skipped slot is counted in g_shm_slot_timeouts and exposed in the /health endpoint.

CPU_PAUSE() is a portable spin hint. On x86 the PAUSE instruction prevents the CPU from speculating aggressively on the spin condition, reducing power consumption and avoiding memory-order violations that cause pipeline flushes. On ARM64 the yield instruction has the same effect. It does not sleep or yield the thread — it is purely a pipeline hint taking ~1–14 ns. Correctness is unchanged without it; it just makes tight spin loops cheaper on out-of-order CPUs.

SPSC vs MPSC: for the torture test each writer container has exactly one producer (its stress container) — strictly SPSC. The ready flag is still correct in that case (the window exists even with one producer if it is preempted), and it makes the ring safe to extend to multiple producers later without any reader-side changes.
Slot size budget: 256 B gives 1M slots = 256 MB — fits comfortably in /dev/shm on any target. At 20 float + 12 int fields this covers the full battery cell topology (17 float + 7 int measured). If a future source needs more fields, bump SHM_MAX_FLOATS/SHM_MAX_INTS and increment version in the header — the mismatch check forces a ring recreate on next open.

Ownership Takeover Protocol

Run once at writer.cpp startup, before the MQTT connection is established and before WAL replay begins.

// Pseudocode — see ShmRing::claim_ownership() in shm_ring.cpp

int fd = shm_open(path, O_RDWR | O_CREAT, 0600);
auto* hdr = static_cast<ShmRingHeader*>(mmap(...));

if (hdr->magic != MAGIC || hdr->version != VERSION) {
    // New or incompatible ring — initialise from scratch
    shm_unlink(path);
    init_ring(hdr, capacity, slot_bytes, schema);
    hdr->owner_pid = getpid();
    hdr->heartbeat_ns = clock_now_ns();
    return;
}

while (true) {
    pid_t old_pid = hdr->owner_pid;

    if (old_pid == 0) {
        // Unclaimed — simple CAS
        if (__atomic_compare_exchange_n(&hdr->owner_pid,
                &old_pid, getpid(), false,
                __ATOMIC_SEQ_CST, __ATOMIC_SEQ_CST)) break;
        continue;  // lost race — retry
    }

    bool pid_dead = (kill(old_pid, 0) != 0 && errno == ESRCH);
    bool hb_stale = (clock_now_ns() - hdr->heartbeat_ns > 5'000'000'000ULL);

    if (pid_dead || hb_stale) {
        // Previous owner dead or frozen — take over
        // read_cursor already at the correct lossless handover point
        if (__atomic_compare_exchange_n(&hdr->owner_pid,
                &old_pid, getpid(), false,
                __ATOMIC_SEQ_CST, __ATOMIC_SEQ_CST)) break;
        continue;  // lost race to another restarting instance — retry
    }

    // Another live instance owns the ring — error or wait
    log_error("shm ring already owned by pid=%d — is another writer running?", old_pid);
    return false;
}
The CAS loop handles the race between two restarting writers (e.g., Docker restart: unless-stopped racing with a manual restart). Exactly one wins; the other retries and sees either the winner's PID with a fresh heartbeat (→ error/exit) or 0 if the winner already crashed again (→ rare but safe).

Producer Path — Writing to the Ring

The producer acquires a slot with a single fetch_add on write_cursor. No locks. No broker. Suitable for C/C++ producers on the same host; Python producers can use ctypes against shm_producer.h.

// shm_producer.h — header-only, no new link deps
// Uses ShmSlot (fixed layout) — not Row (heap maps).

void ShmProducer::publish(const TopicInfo& ti, const Row& r) {
    uint64_t wc = hdr_->write_cursor.fetch_add(1, std::memory_order_acq_rel);

    if (wc - hdr_->read_cursor.load(std::memory_order_acquire) >= hdr_->capacity) {
        // Ring full — drop (QoS-0 semantics)
        ++overflow_drops_;
        return;
    }

    ShmSlot* slot = reinterpret_cast<ShmSlot*>(slots_base_ + (wc % hdr_->capacity) * sizeof(ShmSlot));
    row_to_slot(*slot, ti, r);   // sets slot->ready = 1 last (release fence inside)

    // Wake reader via futex on wake_seq
    hdr_->wake_seq.fetch_add(1, std::memory_order_release);
    syscall(SYS_futex, &hdr_->wake_seq, FUTEX_WAKE, 1, nullptr, nullptr, 0);
}
SPSC for torture test. Each writer container has its own ring and one dedicated stress/producer container — single producer, single consumer. The ready flag in ShmSlot is still set, which makes the design safe to extend to multiple producers later without changing the reader.

Reader Path — writer.cpp shm_reader Thread

A new thread started in main() alongside the MQTT thread. Drains the ring into g_buffer via the same path as on_message, including the overflow guard and pipe-wake for NC4. Wakes via futex; no fd needed.

static void shm_reader_fn() {
    auto* hdr   = g_shm_ring->hdr;
    uint64_t last_wake = hdr->wake_seq.load(std::memory_order_acquire);

    while (!g_stop) {
        // Wait for producer wake (1s timeout so we notice g_stop)
        struct timespec ts = { 1, 0 };
        syscall(SYS_futex, &hdr->wake_seq, FUTEX_WAIT, last_wake, &ts, nullptr, 0);
        last_wake = hdr->wake_seq.load(std::memory_order_acquire);

        uint64_t wc = hdr->write_cursor.load(std::memory_order_acquire);
        uint64_t rc = hdr->read_cursor.load(std::memory_order_relaxed);

        while (rc != wc) {
            auto* slot = slot_at(hdr, rc);

            // Bounded spin — producer may have died after fetch_add but before ready=1
            // Deadline: 1ms. Normal write takes nanoseconds, so this never fires in steady state.
            // If deadline fires: producer crashed mid-slot. Skip and advance read_cursor.
            {
                const uint64_t deadline = clock_now_ns() + 1'000'000ULL;  // 1 ms
                while (!slot->ready) {
                    CPU_PAUSE();
                    if (clock_now_ns() > deadline) {
                        ++g_shm_slot_timeouts;
                        slot->ready = 0;  // clear for safety on next lap
                        goto advance_cursor;  // skip this slot, do not call slot_to_row
                    }
                }
            }
            std::atomic_thread_fence(std::memory_order_acquire);

            TopicInfo ti; Row r;
            slot_to_row(*slot, ti, r);  // deserialise fixed slot → heap Row
            slot->ready = 0;            // clear for reuse

            // Same path as on_message — acquire buffer lock, push row
            {
                std::lock_guard<std::mutex> lk(g_buf_mutex);
                if (g_total_buffered_rows >= g_cfg->max_total_buffer_rows) {
                    ++g_overflow_drops;
                    char b = 1;
                    if (::write(g_signal_pipe[1], &b, 1) < 0) {}
                } else {
                    g_buffer[partition_key(ti, r)].push_back(std::move(r));
                    ++g_total_buffered_rows;
                }
            }

            advance_cursor:  // goto target from bounded spin timeout (slot skipped)
            // Advance read_cursor — lossless handover point for ownership takeover
            hdr->read_cursor.store(++rc, std::memory_order_release);
        }

        // Update heartbeat (health_thread also updates every 1s)
        hdr->heartbeat_ns = clock_now_ns();
    }

    // Clean shutdown — release ownership so next instance can claim immediately
    hdr->owner_pid = 0;
}
Why the spin must be bounded. The producer does write_cursor.fetch_add(1) (advancing the cursor atomically) and then writes the slot data. If the producer is killed (SIGKILL, OOM, container stop) in that gap, ready will never become 1. Without a deadline the reader spins forever on that slot — read_cursor stalls, the ring appears full to new producers, and the writer stops ingesting entirely even though it is still alive.

The 1 ms deadline means:
  • A normal write (nanoseconds) never triggers it.
  • A hung producer is detected and the slot is skipped within 1 ms.
  • At 80k msg/s, 1 ms = 80 rows — acceptable loss for a producer crash mid-write.
  • g_shm_slot_timeouts is exposed in the health endpoint so the condition is visible in the monitor without checking logs.
partition_key signature change: currently partition_key(const Row&) reads source_type from the Row. With the SHM path, source_type lives in TopicInfo. SHM-2 will add a partition_key(const TopicInfo&, const Row&) overload, or move source_type into Row. Easiest fix: add std::string source_type to Row and populate it in both on_message and slot_to_row.

WAL Interaction — Two Layers of Durability

On normal operation: producer → SHM ring → shm_reader → g_buffer │ flush_thread (every 10s or on NC4 pipe-wake) │ [WAL write: rows → .wal.arrow] │ [Parquet flush → .parquet.tmp → rename] │ [WAL delete on success] On writer.cpp crash (kill -9, SIGKILL): producer keeps writing to SHM ring ──────────────────────────────► ring fills read_cursor stays at last committed position new writer.cpp starts: 1. claim_ownership() — CAS owner_pid, resume from read_cursor ← zero rows lost 2. replay any surviving .wal.arrow files (existing NC1-fixed logic) ← no duplicates 3. shm_reader_fn() drains ring from read_cursor onward Invariant: WAL covers: rows flushed to WAL but not yet written to parquet SHM ring covers: rows received but not yet committed to g_buffer Together: complete lossless coverage of any crash at any point
The WAL and SHM ring are complementary, not redundant. WAL protects the flush-in-progress window (already extracted from the ring, not yet in parquet). The ring protects the receive window (in the ring, not yet extracted by writer.cpp). A crash at any point loses zero rows as long as the ring hasn't overflowed.

Ring full means the owner is alive but behind — not dead. The heartbeat distinguishes the two cases: a full ring with a fresh heartbeat is a throughput problem (increase ring or reduce rate); a full ring with a stale heartbeat is a crash — take over.

Related Fix — MqttRing 50µs Sleep-Poll (SHM-7)

While auditing spins in the system, one other busy-wait was found: parse_thread_fn() in writer.cpp polls g_mqtt_ring with a 50µs sleep when the ring is empty: 

// writer.cpp — parse_thread_fn() today
while (true) {
    if (g_mqtt_ring.pop(slot)) {
        process_raw_message(slot.topic, slot.payload);
    } else if (g_mqtt_stopped.load(...) && g_mqtt_ring.empty()) {
        break;
    } else {
        std::this_thread::sleep_for(std::chrono::microseconds(50));  ← 20 000 wakeups/sec at idle
    }
}

At 80k msg/s the ring is never empty so the sleep never fires — but on lp3, Gumby targets, or any deployment with bursty data there are genuine quiet periods. During those periods the thread wakes 20,000 times per second, burns a CPU timeslice, and goes back to sleep. It also adds up to 50µs latency to the first message that arrives after any quiet period.

Root cause

MqttRing is a pure lock-free SPSC ring with no blocking mechanism — it has no way to park the consumer. The 50µs sleep was the simplest workaround at the time. The correct fix is the same pattern used for the SHM ring: add a wake_seq atomic to MqttRing and use futex(FUTEX_WAIT/WAKE) so the consumer blocks in the kernel rather than spinning.

Fix — add wake_seq to mqtt_ring.h

// mqtt_ring.h — add alongside write_pos / read_pos
alignas(64) std::atomic<uint64_t> wake_seq{0};

// push() — one new line after write_pos.store():
wake_seq.fetch_add(1, std::memory_order_release);
syscall(SYS_futex, &wake_seq, FUTEX_WAKE, 1, nullptr, nullptr, 0);

// New method — consumer calls this when pop() returns false:
void wait_for_data(uint64_t last_wake, int timeout_ms = 1000) noexcept {
    struct timespec ts = { timeout_ms / 1000, (timeout_ms % 1000) * 1'000'000L };
    syscall(SYS_futex, &wake_seq, FUTEX_WAIT, last_wake, &ts, nullptr, 0);
}

Fix — update parse_thread_fn() in writer.cpp

static void parse_thread_fn() {
    MqSlot   slot;
    uint64_t last_wake = g_mqtt_ring.wake_seq.load(std::memory_order_acquire);

    while (true) {
        if (g_mqtt_ring.pop(slot)) {
            last_wake = g_mqtt_ring.wake_seq.load(std::memory_order_relaxed);
            process_raw_message(slot.topic, slot.payload);
        } else if (g_mqtt_stopped.load(std::memory_order_acquire) && g_mqtt_ring.empty()) {
            break;
        } else {
            g_mqtt_ring.wait_for_data(last_wake);    ← blocks in kernel; 0 wakeups at idle
            last_wake = g_mqtt_ring.wake_seq.load(std::memory_order_acquire);
        }
    }
}
Effect:
  • Idle CPU load from parse thread: ~0 (was ~20k wakeups/sec)
  • First-message latency after quiet period: < 1µs (was up to 50µs)
  • Throughput at 80k msg/s: unchanged — at full rate the ring is never empty so wait_for_data is never called
  • Shutdown: the 1s futex timeout ensures g_mqtt_stopped is checked within 1s — same behaviour as before
Why the same futex pattern works here. Both MqttRing and ShmRingHeader use an atomic counter (wake_seq) as the futex word. The producer increments it and calls FUTEX_WAKE; the consumer records the last seen value and calls FUTEX_WAIT with that value. If the producer increments between the consumer's pop() returning empty and the FUTEX_WAIT call, the futex value has already changed — the kernel returns immediately without blocking, so no message is ever delayed. This is the standard notification idiom; no mutex needed.

MqttRing Topic Length Cap (Fixed)

The Problem

MQRING_TOPIC_MAX was defined as 128 bytes. Real EMS (Longbow) topics reach 136 bytes. The push() method silently capped the copy via std::min(topic_len, MQRING_TOPIC_MAX - 1) — no counter was incremented, nothing was logged. Of the 1,770 unique topics published per unit, 8 were silently truncated on every message.

What Truncation Does in Practice

EMS topics follow the positional format: 

unit/{unit_id}/{device}/{instance}/{point_name}/{dtype_hint}

The real_writer dtype parser reads the last path segment to decide storage type. Truncation corrupts that segment in two ways:

Raw length Count Truncation effect Storage outcome
129 bytes 6 topics dtype_hint segment cut from "integer" → "integ" dtype lookup fails → values stored as float instead of int
136 bytes 2 topics point_name cut 9 chars short and dtype_hint segment missing entirely Wrong point name recorded + dtype defaults to float

Affected Topics — BMS MBMS Fault/Warning Flags

All 8 affected topics are BMS MBMS fault/warning flags that carry integer 0/1 values. The two worst cases (136 bytes, bms_1 and bms_2): 

-- 136 bytes, bms_1 --
unit/XXXXXXXX/bms/bms_1/Batt1_MBMS_System_warning_and_fault_SBMU_Both_main_positive_relay_and_main_negative_relay_sticking_fault/integer

-- 136 bytes, bms_2 --
unit/XXXXXXXX/bms/bms_2/Batt1_MBMS_System_warning_and_fault_SBMU_Both_main_positive_relay_and_main_negative_relay_sticking_fault/integer

-- what the parser received before fix (128-byte cap, NUL at [127]) --
unit/XXXXXXXX/bms/bms_2/Batt1_MBMS_System_warning_and_fault_SBMU_Both_main_positive_relay_and_main_negative_relay_stickin ← 9 chars cut, no dtype segment

The Fix

MQRING_TOPIC_MAX bumped 128 → 256. All 1,770 topics now fit with margin. Two atomic counters added to MqttRing and incremented in push(): 

struct MqttRing {
    // …
    alignas(64) std::atomic<uint64_t> topic_truncations{0};
    alignas(64) std::atomic<uint64_t> payload_truncations{0};

    void push(const char* topic, size_t tlen,
              const uint8_t* payload, size_t plen) {
        if (tlen >= MQRING_TOPIC_MAX) {
            tlen = MQRING_TOPIC_MAX - 1;
            topic_truncations.fetch_add(1, std::memory_order_relaxed);
        }
        if (plen > MQRING_PAYLOAD_MAX) {
            plen = MQRING_PAYLOAD_MAX;
            payload_truncations.fetch_add(1, std::memory_order_relaxed);
        }
        // … rest of push …
    }
};

Both counters are exposed in the /health JSON response so the parquet monitor and push_agent can surface any future overflow immediately. 

// health_thread_fn() JSON snippet
"topic_truncations":   g_mqtt_ring.topic_truncations.load(),
"payload_truncations": g_mqtt_ring.payload_truncations.load()

Memory cost: each slot grows by 128 bytes (256 − 128). With the default ring capacity of 262,144 slots that is +32 MB BSS, lazily paged by the kernel — no impact at startup, pages faulted in as slots are first written.

Status: FIXED. MQRING_TOPIC_MAX is now 256. All 1,770 Longbow topics fit without truncation. topic_truncations and payload_truncations counters are live in /health JSON — any regression will be immediately visible.

Related: ShmSlot Field Names

ShmSlot.ShmField.name[16] is fine for the current SHM input path. The new_bms field names fed into SHM slots are short aggregates (AvgCellV, MaxCellT, SoC, etc.) that comfortably fit in 16 bytes.

Future risk: if any path is added that feeds raw EMS point_name strings (up to ~110 chars) directly into ShmField.name, the 16-byte cap would cause the same silent-truncation bug. Use name[64] for any such path.
