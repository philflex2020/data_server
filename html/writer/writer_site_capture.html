Site Capture Plan — Four-Format Parquet vs InfluxDB2

Real-site data capture test  ·  Long-format parquet baseline + InfluxDB2.0 concurrent capture  ·  Post-capture: 4 format conversions + validation

Overview

This test captures one day of live site data in parallel to both long-format parquet (via parquet_writer) and InfluxDB2.0 (via the existing on-site Telegraf chain). After the capture window, the raw long-format parquet is converted to four formats and all four are validated against the InfluxDB ground truth.

Phase 1 — On-Site Capture Config

Topic segment mapping

The MQTT topic published by the site follows the same structure as the bench. The writer's topic_segments names become parquet column names, so we rename here to match the target schema exactly:

Topic position Bench segment name Site segment name Action Example value
0 source_type source_type Drop bench
1 unit_id unit_id Keep → column + partition key 0215D1D8
2 device (category) device_type Keep → column bms
3 instance (specific) device Keep → column bms_1
4 point_name point_name Keep → column SysSOC
5 dtype_hint dtype_hint Drop float
Column order note: Parquet stores columns in declaration order, but DuckDB and PyArrow queries are column-name-based. The schema (ts, value, device_type, device, point_name, unit_id) is equivalent to the manager's (ts, value, device, device_type, point_name, unit_id) as long as column names match. The writer emits them in the order they are encountered; DuckDB SELECT ts, value, device, device_type, point_name, unit_id returns them in any desired order regardless.

site_capture_long.yaml

This config produces the raw long-format parquet. Key differences from the bench wide config: compound_field_name is absent (disables wide-pivot), time_window_ms is absent (each MQTT message = one row), flush is 10 minutes, compaction is off during capture. 

# site_capture_long.yaml
# Long-format parquet capture for site comparison test.
# Schema per row: (ts, value, unit_id, device_type, device, point_name)
# One row per MQTT message. No wide-pivot. No compaction during capture.
# Adjust mqtt.topic and output.base_path for the target site.

mqtt:
  host:              localhost    # or site MQTT broker IP
  port:              1883
  topic:             bench/#      # adjust to site topic root
  client_id:         parquet-writer-long-capture
  qos:               0
  topic_parser:      positional
  topic_segments:
    - source_type
    - unit_id
    - device_type      # renamed from "device" (bms, pcs, ...)
    - device           # renamed from "instance" (bms_1, pcs_1, ...)
    - point_name
    - dtype_hint
  drop_columns:
    - source_type
    - dtype_hint
  partition_field:   unit_id
  # No compound_field_name → long format (one row per message)
  # No time_window_ms     → no coalescing, each message keeps its own ts

output:
  base_path:                /mnt/data/site-capture-long   # adjust to site
  partitions:
    - '{year}'
    - '{month}'
    - '{day}'
  partition_as_filename_prefix: true
  wide_point_name:          false
  flush_interval_seconds:   600    # 10 minutes
  compression:              snappy
  store_mqtt_topic:         false
  store_sample_count:       false
  site_id:                  SITE_A   # adjust
  max_messages_per_part:    5000000
  max_total_buffer_rows:    2000000

compact:
  enabled:  false   # no compaction during capture; convert in post-processing

guard:
  min_free_gb: 2.0

health:
  enabled: true
  port:    8771

  enabled: true

Phase 3 — Four Output Formats

All four formats are produced from the same raw long-format parquet by long_to_wide.py. The script uses DuckDB for all transformations.

A Wide format — data dedupe null_fill_unchanged

Long rows are pivoted to wide (one row per timestamp per unit, one column per signal path device_type/device/point_name). Within each flush window, any signal whose value has not changed since the previous flush is stored as NULL. This is equivalent to enabling null_fill_unchanged: true in the writer.

Expected result: ~78–85% nulls across all rows. Constant signals (SOH = 100.0 over many hours) compress to ~55 bytes per column. Dynamic signals retain every value.

-- Step 1: build the wide pivot with last-value-per-window logic
-- Signal column name format: device_type || '/' || device || '/' || point_name
CREATE TABLE wide_raw AS
PIVOT (
  SELECT ts, unit_id,
         device_type || '/' || device || '/' || point_name AS signal,
         value
  FROM read_parquet('raw/**/*.parquet', union_by_name=true)
)
ON signal USING LAST(value) GROUP BY ts, unit_id ORDER BY unit_id, ts;

-- Step 2: null out values that match the previous row for the same unit
-- (simulates null_fill_unchanged — signals only written when they change)
CREATE TABLE wide_deduped AS
SELECT ts, unit_id,
  CASE WHEN sig = LAG(sig) OVER (PARTITION BY unit_id ORDER BY ts) THEN NULL ELSE sig END AS sig
  -- ... repeated for each signal column (generated dynamically by script)
FROM wide_raw;

Output dir: out/A_wide_deduped/

B Wide format — deduped data reinserted forward-fill

Same pivot as Format A, but all NULL cells are forward-filled to restore the last known value. Every row has a complete set of signal values with no nulls. Represents what a query would return after applying COALESCE(val, last_val).

Expected result: 0% nulls. Largest file size of the four formats. Compresses worst (forward-filled constants repeat identically, so RLE still helps, but the explicit values increase the column chunk size vs true null-fill).

-- Start from Format A (wide_deduped), forward-fill each signal column
CREATE TABLE wide_filled AS
SELECT ts, unit_id,
  LAST_VALUE(sig IGNORE NULLS) OVER (
    PARTITION BY unit_id ORDER BY ts
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS sig
  -- ... repeated for each signal column
FROM wide_deduped;

Output dir: out/B_wide_filled/

C Long format — data dedupe (1 write per minute) resampled

Long schema kept. Rows are binned to 1-minute intervals; within each bin the LAST value is taken for each (unit_id, device_type, device, point_name) combination. For 1 Hz data this reduces row count by ~60×. Equivalent to what the on-site InfluxDB capture does when telegraf sends one value per minute (deduped at source).

Expected result: ~1/60 the rows of Format D. Bytes/signal-value similar to Format D but with far fewer rows.

-- Bin to 1-minute buckets, keep last value per signal
CREATE TABLE long_deduped AS
SELECT
  TIME_BUCKET(INTERVAL '1 minute', ts) AS ts,
  unit_id, device_type, device, point_name,
  LAST(value ORDER BY ts) AS value
FROM read_parquet('raw/**/*.parquet', union_by_name=true)
GROUP BY 1, 2, 3, 4, 5
ORDER BY unit_id, ts;

-- DuckDB alternative if TIME_BUCKET not available:
  epoch_ms(ts) / 60000 * 60000 AS ts_bin   -- floor to minute boundary

Output dir: out/C_long_deduped/

D Long format — un-deduped (raw as captured) raw

The original raw long-format parquet as written by the writer. Every MQTT message is one row. No transformation. This is the reference input for all other formats.

Expected result: one row per signal per second (1 Hz), 63 signals, N units. For a 24-hour run: 24 × 3600 × 63 × N_units rows.

-- No transformation; just re-partition for consistency
-- (or symlink raw/ to out/D_long_raw/)
COPY (SELECT * FROM read_parquet('raw/**/*.parquet', union_by_name=true)
      ORDER BY unit_id, ts)
TO 'out/D_long_raw/' (FORMAT parquet, PARTITION_BY (unit_id), COMPRESSION snappy);

Output dir: out/D_long_raw/

Building the Container Image

The writer is a single compiled C++ binary that links against four shared libraries. The job of the Dockerfile is to (a) compile the binary from source with all the right build tools, then (b) produce a slim runtime image that has only the shared libraries the binary needs at runtime — nothing else.

This is called a multi-stage build. Think of it as two separate boxes:

  • Builder box — has compilers, headers, everything needed to compile. Throws it all away when done.
  • Runtime box — starts clean; receives only the compiled binary + the runtime .so files it needs. This is the image that actually runs in EKS.

The four libraries and why they matter

LibraryWhat it doesRuntime packageBuild packageSource
libarrow + libparquet Writes Parquet files — the core format. Arrow is the in-memory layer; Parquet is the file layer. libarrow2300 libparquet2300 libarrow-dev libparquet-dev Apache Arrow apt repo (not in default Ubuntu)
libmosquitto MQTT client — subscribes to the site broker and receives signal data. libmosquitto1 libmosquitto-dev Default Ubuntu apt
libyaml-cpp Parses the YAML config file. libyaml-cpp0.7 libyaml-cpp-dev Default Ubuntu apt
libsimdjson Fast JSON parser — reads the {"ts":…,"value":…} MQTT payloads. libsimdjson9 libsimdjson-dev Default Ubuntu apt
Apache Arrow is the tricky one. It is not in the default Ubuntu apt repository. You have to add Apache's own apt source before apt-get install will find it. The Dockerfile does this automatically — but it's why you can't just write a plain apt install libarrow-dev and expect it to work.

entrypoint.sh

This script runs inside the container when it starts. It reads the environment variables injected by Kubernetes (MQTT_HOST, SITE_ID) and writes them into the config before launching the writer. This way the same container image works at any site — you only change the Secret, not the image. 

#!/bin/sh
# entrypoint.sh
set -e

# Copy the ConfigMap-mounted config to a writable location
cp /etc/writer/config.yaml /tmp/config.yaml

# Substitute placeholders with values from Kubernetes Secrets
sed -i "s/MQTT_HOST_PLACEHOLDER/${MQTT_HOST}/g" /tmp/config.yaml
sed -i "s/SITE_ID_PLACEHOLDER/${SITE_ID}/g"     /tmp/config.yaml

# Show the resolved config at startup (useful for debugging pod logs)
echo "=== resolved config ==="
grep -E 'host:|site_id:' /tmp/config.yaml
echo "======================="

exec /usr/local/bin/parquet_writer --config /tmp/config.yaml

Dockerfile

The build uses Ubuntu 22.04 — the same OS as .34. This means the same library versions are available and there are no surprises when the binary runs. 

# ── Stage 1: Build ──────────────────────────────────────────────────────────
# Full Ubuntu 22.04 with compilers + all dev headers.
# Nothing from this stage ends up in the final image.
FROM ubuntu:22.04 AS builder

ENV DEBIAN_FRONTEND=noninteractive

# Add the Apache Arrow apt repository.
# Arrow is not in Ubuntu's default package list — this registers their repo.
RUN apt-get update && apt-get install -y --no-install-recommends \
      wget ca-certificates && \
    wget -q https://apache.jfrog.io/artifactory/arrow/ubuntu/apache-arrow-apt-source-latest-jammy.deb && \
    apt-get install -y ./apache-arrow-apt-source-latest-jammy.deb && \
    apt-get update

# Install compiler + all dev packages (headers + static/shared libs for linking)
RUN apt-get install -y --no-install-recommends \
      build-essential \
      pkg-config \
      libarrow-dev \
      libparquet-dev \
      libmosquitto-dev \
      libyaml-cpp-dev \
      libsimdjson-dev && \
    rm -rf /var/lib/apt/lists/*

# Copy source files and compile
WORKDIR /build
COPY parquet_writer.cpp mqtt_ring.h Makefile ./
RUN make parquet_writer

# Confirm what shared libs the binary expects — shows up in build log
RUN ldd parquet_writer


# ── Stage 2: Runtime ────────────────────────────────────────────────────────
# Fresh Ubuntu 22.04 — no compilers, no headers, no dev packages.
# Only the runtime .so files the binary actually needs at runtime.
FROM ubuntu:22.04

ENV DEBIAN_FRONTEND=noninteractive

# Add Arrow repo again — needed to find libarrow2300 / libparquet2300 at install time.
# These are the runtime-only packages (no headers, much smaller than -dev).
RUN apt-get update && apt-get install -y --no-install-recommends \
      wget ca-certificates && \
    wget -q https://apache.jfrog.io/artifactory/arrow/ubuntu/apache-arrow-apt-source-latest-jammy.deb && \
    apt-get install -y ./apache-arrow-apt-source-latest-jammy.deb && \
    apt-get update

# Install runtime libraries only.
# libparquet2300 automatically pulls in libarrow2300 and all its dependencies
# (compression libs: zstd, lz4, snappy, brotli — Arrow uses them internally).
RUN apt-get install -y --no-install-recommends \
      libparquet2300 \
      libmosquitto1 \
      libyaml-cpp0.7 \
      libsimdjson9 && \
    rm -rf /var/lib/apt/lists/* /tmp/*.deb

# Copy the compiled binary from the builder stage
COPY --from=builder /build/parquet_writer /usr/local/bin/parquet_writer

# Copy the startup script
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

# Port 8771 is the health endpoint — Kubernetes uses it for liveness probes
EXPOSE 8771

ENTRYPOINT ["/entrypoint.sh"]

Common problems

ErrorCauseFix
error while loading shared libraries: libparquet.so.2300 Runtime stage is missing Arrow packages Confirm the Apache Arrow apt source is added in Stage 2 of the Dockerfile and libparquet2300 is installed
Unable to locate package libarrow-dev Arrow apt repo not registered The wget + apt install *.deb + apt update lines must run before the apt install libarrow-dev line
exec format error in EKS pod Image built for wrong CPU architecture (e.g. built on Apple M1/M2 arm64 but EKS nodes are amd64) Build on an x86_64 machine or add --platform linux/amd64 to docker build
Pod CrashLoopBackOff, logs show config error ConfigMap placeholder not substituted (entrypoint.sh sed failed) Check Secret exists: kubectl get secret parquet-writer-secrets -n data-capture
Liveness probe failing Writer hasn't opened port 8771 yet (takes a few seconds) initialDelaySeconds: 10 is already in the Deployment YAML; increase to 20 if site startup is slow

Architecture note (Apple Silicon / Graviton)

The writer binary must match the CPU architecture of the EKS nodes. Most EKS clusters use x86_64 (Intel/AMD). If you're building on an Apple M-series Mac, Docker builds arm64 by default — add --platform linux/amd64: 

# Force x86_64 build on Apple Silicon
docker build --platform linux/amd64 -t parquet-writer:latest .
docker push ${ECR}/parquet-writer:latest

If your EKS cluster uses Graviton (arm64) nodes, omit --platform on Apple Silicon and use --platform linux/arm64 on x86_64 machines. Check your node instance type: kubectl get nodes -o wide.

Level 3 — Integration test (full write cycle)

This spins up a local MQTT broker (mosquitto) alongside the writer, publishes a handful of test messages, waits for a flush, and checks the parquet output. All in Docker — no EKS needed.

docker-compose.yaml

# docker-compose.yaml — save in ~/build/parquet-writer/
services:

  mosquitto:
    image: eclipse-mosquitto:2
    ports:
      - "1883:1883"
    command: mosquitto -c /mosquitto-no-auth.conf
    volumes:
      - ./mosquitto.conf:/mosquitto-no-auth.conf:ro

  writer:
    image: parquet-writer:latest
    environment:

Extracting parquet files while the writer is running

Yes — the writer flushes completed .parquet files every 10 minutes and closes them. Once a file is closed it is safe to copy. Two approaches:

Option 1 — kubectl cp (ad-hoc, simplest)

Pull files at any time without stopping the writer. Only copies closed (flushed) files — never touches the writer process.

# See what has been flushed so far
kubectl exec -n data-capture \
  $(kubectl get pod -n data-capture -l app=parquet-writer -o name) \
  -- find /data/site-capture-long -name '*.parquet' | sort

# Copy entire output directory to local machine
kubectl cp data-capture/$(kubectl get pod -n data-capture \
  -l app=parquet-writer \
  -o jsonpath='{.items[0].metadata.name}'):/data/site-capture-long \
  ./site-capture-long/

# Or copy just today's files
kubectl cp data-capture/$(kubectl get pod -n data-capture \
  -l app=parquet-writer \
  -o jsonpath='{.items[0].metadata.name}'):/data/site-capture-long/bench/$(date +%Y/%m/%d) \
  ./site-capture-local/

Option 2 — S3 sync sidecar (continuous, production approach)

Add a second container to the same pod as the writer. Both containers share the /data volume. The sidecar runs aws s3 sync every 60 seconds, picking up any newly flushed files. The writer never knows it is there. 

# Add this to the containers: list in deployment.yaml, after the writer container
- name: s3-sync
  image: amazon/aws-cli
  command:
    - /bin/sh
    - -c
    - |
      echo "s3-sync starting — bucket: $S3_BUCKET"
      while true; do
        aws s3 sync /data/site-capture-long \
          s3://${S3_BUCKET}/site-capture-long \
          --exclude '*' --include '*.parquet' \
          --no-progress --quiet
        sleep 60
      done
  env:
    - name: S3_BUCKET
      value: my-capture-bucket      # replace with your bucket name
    - name: AWS_DEFAULT_REGION
      value: eu-west-1              # replace with your region
  volumeMounts:
    - name: data
      mountPath: /data
      readOnly: true                # sidecar only reads — writer has full access
  resources:
    requests: {cpu: 50m, memory: 64Mi}
    limits:   {cpu: 100m, memory: 128Mi}
IAM permission needed: the EKS node IAM role (or a pod service account via IRSA) needs s3:PutObject + s3:ListBucket on the target bucket. Simplest for a test: attach AmazonS3FullAccess to the node group IAM role in the AWS console. For production, use IRSA (IAM Roles for Service Accounts) to scope it to just the writer pod.

Then pull to local at any time during or after the capture:

# Pull all captured files from S3 to local

# Check how much has been captured so far
aws s3 ls s3://my-capture-bucket/site-capture-long/ --recursive --summarize \
  | tail -3

EKS Container Deployment

parquet_writer runs as a single-container Deployment in EKS. The writer config is injected via a ConfigMap, output parquet lands on an EFS-backed PersistentVolume (shared ReadWriteMany so post-processing Jobs can read the same mount), and the health endpoint is exposed as a liveness probe. MQTT broker address is passed as an environment variable so the same image works across sites.

Architecture overview

ComponentK8s resourceNotes
Writer config ConfigMap parquet-writer-config Mounted as /etc/writer/config.yaml inside container
MQTT host / site credentials Secret parquet-writer-secrets Env vars MQTT_HOST, SITE_ID — one Secret per site
Parquet output PVC → EBS gp3 (ReadWriteOnce) Only the writer pod mounts it; pull files locally after capture
Writer process Deployment (1 replica) Liveness probe on :8771/health
Post-processing local — long_to_wide.py Run on local machine after kubectl cp or S3 sync
Validation local — validate_formats.py Run on local machine; connects to site InfluxDB over VPN/tunnel
Scope: the EKS container only runs parquet_writer to collect long-format parquet files. All post-processing (4 format conversions, validation) runs locally after pulling the files from site. EBS (ReadWriteOnce) is fine — only one pod writes to the volume.

Phase 4 — Validation Against InfluxDB2.0

validate_formats.py fetches the InfluxDB data for the exact capture window (using influx_to_parquet.py as a library), then runs DuckDB comparison queries against each of the four parquet formats.

Validation checks per format

Check Query approach Pass criterion
Signal coverage SELECT DISTINCT unit_id, device_type, device, point_name from parquet vs InfluxDB tags Same set of (unit_id, point_name) tuples in both

Format C ≅ InfluxDB/60.

Key validation queries

-- Signal coverage: find any signal in InfluxDB not in parquet format D
SELECT influx.unit_id, influx.point_name
FROM influx_signals influx
LEFT JOIN (
  SELECT DISTINCT unit_id, point_name
  FROM read_parquet('out/D_long_raw/**/*.parquet', union_by_name=true)
) pq USING (unit_id, point_name)
WHERE pq.unit_id IS NULL;

-- Row count agreement (Format D vs InfluxDB, per signal)
SELECT
  p.unit_id, p.point_name,
  p.pq_rows, i.inf_rows,
  ROUND(100.0 * ABS(p.pq_rows - i.inf_rows) / i.inf_rows, 2) AS diff_pct
FROM (
  SELECT unit_id, point_name, COUNT(*) AS pq_rows
  FROM read_parquet('out/D_long_raw/**/*.parquet', union_by_name=true)
  GROUP BY 1, 2
) p
JOIN (
  SELECT unit_id, point_name, inf_count AS inf_rows FROM influx_counts
) i USING (unit_id, point_name)
ORDER BY diff_pct DESC;

-- Value accuracy: spot-check 100 random samples from InfluxDB against Format D
SELECT
  s.ts, s.unit_id, s.point_name,
  s.influx_value,
  p.value AS pq_value,
  ABS(s.influx_value - p.value) AS delta
FROM influx_sample s
JOIN read_parquet('out/D_long_raw/**/*.parquet', union_by_name=true) p
  ON p.ts = s.ts AND p.unit_id = s.unit_id AND p.point_name = s.point_name
WHERE delta > 1e-6
ORDER BY delta DESC;

long_to_wide.py — design spec

Usage:
  python3 long_to_wide.py --input raw/ --output out/ [--units 0215D1D8,...]

Arguments:
  --input   dir containing raw long-format parquet (all units, all days)
  --output  output root; subdirs A_wide_deduped/ B_wide_filled/ C_long_deduped/ D_long_raw/
  --units   optional comma-separated unit_id filter
  --start   optional start ISO timestamp
  --stop    optional stop ISO timestamp

Algorithm:
  1. Load raw via DuckDB read_parquet with union_by_name=true
  2. Build signal list: SELECT DISTINCT device_type||'/'||device||'/'||point_name
  3. Generate dynamic PIVOT SQL over signal list  (DuckDB PIVOT clause)
  4. Format D: COPY raw to out/D with partition_by unit_id
  5. Format C: bin to 1-min, LAST(value), COPY to out/C
  6. Format A: PIVOT → wide_raw, then lag-compare to null unchanged values
  7. Format B: COPY wide_raw to out/B (all values; no null-fill)
  8. Print stats table: rows / files / bytes / bytes-per-signal-value for each format

validate_formats.py — design spec

Usage:
  python3 validate_formats.py \
    --influx-url http://site-host:8086 --token TOKEN --org ORG --bucket BUCKET \
    --start 2026-04-14T09:00:00Z --stop 2026-04-15T09:00:00Z \
    --output out/ \
    --report validation_report.html

Algorithm:
  1. Fetch InfluxDB data for window via influx_to_parquet.fetch_measurement()
     → saves as influx_ground_truth.parquet
  2. For each of the 4 parquet output dirs:
     a. Signal coverage check (set intersection / difference vs InfluxDB signals)
     b. Time range check (min/max ts alignment)
     c. Row count per signal (vs InfluxDB count; expected ratios for each format)
     d. Value spot-check (100 random samples from InfluxDB → lookup in parquet)
     e. Value aggregate check (AVG, MIN, MAX per signal vs InfluxDB)
     f. Bytes/signal-value calculation
  3. Write HTML report: pass/fail table per format per check
     + per-format size/rows summary + bytes/signal-value bar chart
