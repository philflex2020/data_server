Writer Topics — MQTT Parsing, Config & site_id

How the parquet_writer maps MQTT topics to parquet columns — bench and Fractal EMS formats

Overview

The writer reads raw MQTT messages and turns them into parquet rows. The topic string carries all the metadata that describes a measurement — device, point name, data type, and which site/unit it belongs to. There is no separate metadata bus; the topic is the schema.

The writer supports two topic formats, selected by mqtt.topic_format in site.yaml:

bench topic_format: bench

Flat, single-depth topic tree used on bench BESS units and simple sites. One topic_segments list maps every topic to the same set of columns. The site identifier is a fixed value set in the config/secret.

bench/0215D1D8/bms/bms_1/SysSOC/float

Subscribes to: bench/# (or your topic_root)

fractal topic_format: fractal

Fractal EMS variable-depth topics. Four patterns at 6–9 segments cover unit signals, unit-root signals, site-device signals, and site-root signals. The site identifier is extracted from the topic itself — one writer can capture multiple sites from a single broker.

ems/site/SITE_A/unit/0215D1D8/bms/bms_1/SysSOC/float

Subscribes to: ems/#

The Positional Parser

Both formats use topic_parser: positional. The topic string is split on / and each segment is assigned a field name from the segments list. The writer then uses those field names to build the parquet row.

Segment name conventions

NameEffectExample value
_ dropped — segment is read and discarded, never stored ems, site, unit
site_id partition key (fractal) — drives directory partitioning, stamped on every row SITE_A
unit_id partition key (bench) — drives directory partitioning, stamped on every row 0215D1D8
device, instance column in long format; part of compound column name in wide format bms, bms_1
point_name column — the signal name; in wide format becomes the column header SysSOC, kW
dtype_hint dropped — hints at payload type (float, string), not stored float, string
source_type dropped (bench) — the topic root word; no information value bench

After segment assignment, any field listed in drop_columns is discarded. For fractal format, dropping is handled entirely by the _ sentinel — no drop_columns list is needed.

Bench Format

Topic structure

One fixed-depth topic tree. Every topic has exactly the same number of segments in the same order. A single topic_segments list describes all of them.

bench / 0215D1D8 / bms / bms_1 / SysSOC / float
  0         1       2      3        4        5
bench/ 0215D1D8/ bms/ bms_1/ SysSOC/ float

dropped  partition key (unit_id)  column fields  dtype hint — dropped

Config (site.yaml / continuous_wide.yaml)

mqtt:
  topic:           bench/#          # single wildcard subscription
  topic_parser:    positional
  topic_segments:  [source_type, unit_id, device, instance, point_name, dtype_hint]
  drop_columns:    [source_type, dtype_hint]
  partition_field: unit_id

output:
  base_path:     /data/site-capture
  site_id:       SITE_A            # fixed — injected from secret at pod start
  partitions:    ['{year}', '{month}', '{day}']
  wide_point_name: false           # true → one col per signal (bench wide-pivot)

The source_type word (bench) is auto-prepended to the partitions because it comes from the topic root. The partition key is unit_id so all rows for a unit land in the same directory.

Fractal Format

Topic structure — 4 depths

Fractal publishes signals at different depths depending on whether the signal belongs to a unit (BMS, PCS), a site-level device (meter, RTAC), or the site itself (alarms, modes). The writer uses topic_patterns — an ordered list of match/segments pairs. First match wins, so more specific (longer) patterns are listed first.

DepthPatternExample topic% of traffic
9-seg ems/site/+/unit/+/+/+/+/+ ems/site/SITE_A/unit/0215D1D8/bms/bms_1/SysSOC/float ~95%
8-seg ems/site/+/unit/+/+/+/+ ems/site/SITE_A/unit/0215D1D8/root/SystemMode/string rare
7-seg ems/site/+/+/+/+/+ ems/site/SITE_A/meter/meter_1/kWh_import/float occasional
6-seg ems/site/+/+/+/+ ems/site/SITE_A/root/SiteAlarm/string occasional

9-seg — unit device signal

ems/ site/ SITE_A/ unit/ 0215D1D8/ bms/ bms_1/ SysSOC/ float

_ (dropped)  site_id (partition)  unit_id (column)  column fields  dtype hint — dropped

8-seg — unit root signal (no instance level)

ems/ site/ SITE_A/ unit/ 0215D1D8/ root/ SystemMode/ string

Segments: ["_","_","site_id","_","unit_id","device","point_name","dtype_hint"] — no instance

7-seg — site-device signal (no unit)

ems/ site/ SITE_A/ meter/ meter_1/ kWh_import/ float

Segments: ["_","_","site_id","device","instance","point_name","dtype_hint"] — no unit_id

6-seg — site root signal (no unit or instance)

ems/ site/ SITE_A/ root/ SiteAlarm/ string

Segments: ["_","_","site_id","device","point_name","dtype_hint"] — no unit_id or instance

Config (config.longbow.yaml)

mqtt:
  topic:           "ems/#"          # single wildcard — all 4 depths
  topic_parser:    positional
  partition_field: site_id          # extracted from topic, used for filename prefix
  drop_columns:    ["dtype_hint", "site_id"]  # used for routing, not stored in parquet

  # First matching pattern wins — longer patterns listed first
  topic_patterns:

    # 9-seg: unit device signal
    - match: "ems/site/+/unit/+/+/+/+/+"
      segments: ["_","_","site_id","_","unit_id","device","instance","point_name","dtype_hint"]

    # 8-seg: unit root signal
    - match: "ems/site/+/unit/+/+/+/+"
      segments: ["_","_","site_id","_","unit_id","device","point_name","dtype_hint"]

    # 7-seg: site device signal (meter, RTAC)
    - match: "ems/site/+/+/+/+/+"
      segments: ["_","_","site_id","device","instance","point_name","dtype_hint"]

    # 6-seg: site root signal
    - match: "ems/site/+/+/+/+"
      segments: ["_","_","site_id","device","point_name","dtype_hint"]

output:
  base_path:                    /data/site-capture
  partitions:                   ['{year}', '{month}', '{day}']
  partition_as_filename_prefix: true   # SITE_A_20260414T....parquet
  flush_interval_seconds:       30     # set in site.yaml capture.flush_interval_seconds

All four depths land in the same parquet files. Columns absent for a given depth (unit_id, instance) are stored as null for those rows. site_id and dtype_hint are extracted at parse time and used for routing/partitioning, then dropped — they are not stored as columns.

site_id is extracted from the topic and used as the filename prefix (via partition_as_filename_prefix: true + partition_field: site_id), then dropped from stored columns. A broker serving multiple sites produces separate files automatically:

/data/site-capture/2026/04/14/
  SITE_A_20260414T....parquet
  SITE_B_20260414T....parquet
  SITE_C_20260414T....parquet

Topic → Directory Mapping

Every incoming MQTT message has its topic parsed into fields. Those fields drive the output path — the coloured segments below map directly to the coloured path components on the right.

Bench

One topic depth. unit_id (segment 1) becomes the filename prefix. source_type (segment 0) is auto-prepended as the first directory.

topic

All four fractal depths land in the same parquet file for a given site and day. The partition is site_id only — depth differences show up as null columns in individual rows, not as separate files or directories.

What Happens to site_id

This is the key difference between the two formats.

bench — site_id is a config value

In output.site_id the writer holds a literal string (SITE_A). Every row written gets that value stamped on it. The value comes from the Kubernetes secret at pod start — the entrypoint.sh replaces SITE_ID_PLACEHOLDER in config.yaml with the SITE_ID env var.

# entrypoint.sh
sed -i "s/SITE_ID_PLACEHOLDER/$SITE_ID/" /etc/writer/config.yaml

One broker → one site. If you need two sites, deploy two pods with different SITE_ID secrets pointing at different brokers.

output:
  site_id: SITE_A        # set at runtime
  partitions: ['{year}', '{month}', '{day}']
fractal — site_id is in the topic

The writer extracts site_id from segment 2 of every incoming topic (ems/site/SITE_A/...) at parse time. It uses it as the partition_field to build the output filename prefix, then drops it from the stored parquet columns.

# site_id extracted from topic segment 2
ems/site/SITE_A/unit/0215D1D8/bms/...
         ↑ extracted here, used for filename, then dropped

No SITE_ID secret needed — the secret only carries MQTT_HOST. entrypoint.sh no longer injects a site_id placeholder.

mqtt:
  partition_field: site_id        # → filename prefix SITE_A_...
  drop_columns:    ["dtype_hint", "site_id"]  # not stored in rows
output:
  partitions: ['{year}', '{month}', '{day}']
  partition_as_filename_prefix: true

One broker → many sites. Messages from SITE_A, SITE_B, SITE_C all flow through the same pod and are automatically separated by filename prefix.

Fractal + multi-site broker: if the EMS broker at a site publishes data for multiple logical sites (SITE_A and SITE_B), a single writer pod captures all of them. The partition_field: site_id ensures each site's data lands in its own directory. Buffer memory and file handles scale with the number of active sites, not with the number of signals.

What We Subscribe To

FormatMQTT subscriptionWhy
bench bench/# Single root wildcard captures the flat bench tree. bench comes from mqtt.topic_root in site.yaml — change it to match the site's actual root word.
fractal ems/# Single root wildcard captures all 4 depths simultaneously. The broker delivers every topic under ems/; the writer's pattern list decides which ones to parse and which to ignore.
bench fractal # Subscribe to everything. Unmatched topics are silently dropped by the writer. Safe on a dedicated site broker where all traffic is yours. Avoids needing to update the subscription if the topic root ever changes.

The MQTT broker delivers all messages matching the subscription. The writer applies the pattern list to each message. If no pattern matches, the message is silently dropped — filtering happens at parse time, not at subscribe time.

Choosing your subscription

Broker typeRecommended subscriptionReason
Dedicated site broker
all traffic is yours		 # Zero config when topic structure evolves. New topic roots are captured automatically once you add the matching pattern.
Shared / multi-tenant broker
other systems publishing		 ems/# or bench/# Unrelated high-volume topics would still be delivered to the writer pod and discarded — wasted network and CPU. A specific prefix keeps load down.
seen_topics in /health: the /health endpoint exposes a seen_topics array showing every distinct topic (up to point_name, dtype_hint stripped) that has been parsed successfully. Use this to verify the writer is receiving all expected signal paths after deployment. 
curl -s http://<host>:8771/health | python3 -m json.tool | grep seen_topics -A 20

make configmap — edit site.yaml and apply

The recommended workflow for changing the writer config on k3s:

  1. Edit site.yaml — change flush_interval_seconds, mqtt.host, topic_format, etc.
  2. Run make configmap — regenerates all manifests from site.yaml, applies the ConfigMap, and restarts the pod.
# edit site.yaml then:
make configmap

# or for a different site descriptor:
make configmap SITE=site.production.yaml
