2. ASOF JOIN — align multi-device signals HIGH

DuckDB's ASOF JOIN joins two tables on the nearest timestamp rather than exact equality. This solves two problems at once: clock skew between bms_1 and pcs_1, and forward-fill of COV-nulled values across the join boundary — without a window function.

Multi-device alignment

-- Align bms_1 and pcs_1 to a common timeline, forward-filling gaps in both
SELECT
    bms.ts,
    bms."bms/bms_1/SysSOC",
    bms."bms/bms_1/SysVolt",
    pcs."pcs/pcs_1/RealPower",
    pcs."pcs/pcs_1/Frequency"
FROM   (SELECT ts, "bms/bms_1/SysSOC", "bms/bms_1/SysVolt"
        FROM read_parquet('bench-wide/bench/2026/04/13/*.parquet')) bms
ASOF LEFT JOIN
       (SELECT ts, "pcs/pcs_1/RealPower", "pcs/pcs_1/Frequency"
        FROM read_parquet('bench-wide/bench/2026/04/13/*.parquet')) pcs
ON bms.ts >= pcs.ts
ORDER BY bms.ts;

Forward-fill without a window function

-- Forward-fill null-filled SOC column using ASOF self-join
SELECT a.ts, b."bms/bms_1/SysSOC" AS soc_filled
FROM signals a
ASOF LEFT JOIN (
    SELECT ts, "bms/bms_1/SysSOC"
    FROM signals
    WHERE "bms/bms_1/SysSOC" IS NOT NULL
) b ON a.ts >= b.ts
ORDER BY a.ts;
vs LAST_VALUE IGNORE NULLS: ASOF JOIN is often faster for forward-fill because DuckDB can use a merge join rather than a full window sort. On large compact files the difference is measurable.

3. COLUMNS() macro — pattern-select wide columns HIGH

Select a subset of the 63-column wide table by glob pattern. Essential for subscriber_api endpoints that serve device-specific slices without hardcoding every column name.

-- BMS signals only
SELECT ts, COLUMNS('bms/*')
FROM read_parquet('bench-wide/bench/2026/04/13/*.parquet')
WHERE ts > epoch_ms(1712000000000);

-- All signals except PCS
SELECT ts, COLUMNS('*') EXCLUDE (COLUMNS('pcs/*'))
FROM read_parquet('bench-wide/bench/2026/04/13/*.parquet');

-- subscriber_api: parameterise device prefix
conn.execute(f"SELECT ts, COLUMNS('{device}/*') FROM read_parquet(?)", [glob])

-- EXCLUDE specific metadata columns
SELECT ts, COLUMNS('*') EXCLUDE (mqtt_topic, sample_count, source_type)
FROM read_parquet('bench-wide/bench/2026/04/13/*.parquet');
COLUMNS() is evaluated at query planning time against the actual schema — adding a new signal column is automatically picked up by any query using COLUMNS('bms/*') without changing the SQL.

4. time_bucket — downsample for dashboards HIGH

Groups timestamps into fixed-width intervals for aggregation. More flexible than date_trunc — supports arbitrary intervals like 30 seconds or 5 minutes.

-- 60-second averages for dashboard strip chart
SELECT
    time_bucket(INTERVAL '60 seconds', ts)  AS t,
    AVG("bms/bms_1/SysSOC")                  AS soc_avg,
    MIN("pcs/pcs_1/RealPower")               AS power_min,
    MAX("pcs/pcs_1/RealPower")               AS power_max,
    LAST("bms/bms_1/SysVolt", ts)            AS volt_last
FROM read_parquet('bench-wide/bench/2026/04/13/*.parquet')
WHERE ts > now() - INTERVAL '1 hour'
GROUP BY t
ORDER BY t;

-- 5-minute OHLC (open/high/low/close) for power signal
SELECT
    time_bucket(INTERVAL '5 minutes', ts) AS t,
    FIRST("pcs/pcs_1/RealPower", ts)        AS open,
    MAX("pcs/pcs_1/RealPower")              AS high,
    MIN("pcs/pcs_1/RealPower")              AS low,
    LAST("pcs/pcs_1/RealPower", ts)         AS close
FROM read_parquet('bench-wide/bench/2026/04/**/*.parquet')
GROUP BY t ORDER BY t;

5. UNPIVOT — wide → long at query time MEDIUM

Converts the wide table to long format (ts, signal_name, value) on the fly. Useful for external tools (alerting, InfluxDB export, charting libraries) that expect the long schema.

-- Melt all signal columns into rows — no column list needed
SELECT ts, signal_name, signal_value
FROM read_parquet('bench-wide/bench/2026/04/13/*.parquet')
UNPIVOT (signal_value FOR signal_name IN (COLUMNS('*') EXCLUDE ts))
ORDER BY ts, signal_name;

-- BMS signals only in long format
SELECT ts, signal_name, signal_value
FROM read_parquet('bench-wide/bench/2026/04/13/*.parquet')
UNPIVOT (signal_value FOR signal_name IN (COLUMNS('bms/*')))
WHERE signal_value IS NOT NULL
ORDER BY ts;

6. Zstd compression — better ratio for archive files MEDIUM

Zstd gives ~25% smaller files than Snappy with acceptable decompression speed. Best applied to compact files (read infrequently); keep Snappy for flush files (written and read every minute).

CodecRatio (time-series)Decompress speedBest for
snappybaselinefastestFlush files, live queries
zstd~25% smallerfastCompact / archive files
gzip~35% smallerslowCold archival only
# parquet_writer.yaml — two-tier compression
output:
  compression: snappy     # flush files — fast write + read

compact:
  compression: zstd       # compact files — better archive ratio (planned config key)
The writer currently uses a single compression key. A two-tier approach (different codecs for flush vs compact) is a small addition to the compactor config and worth evaluating on the .51 test run — benchmark file size before/after.

7. DuckDB COPY TO parquet — replace Arrow compactor MEDIUM

DuckDB can write Parquet natively, potentially replacing the C++ Arrow-based compact_directory() with a single SQL statement. Adds forward-fill (Strategy C from query_design §forward-fill) via a window function at merge time.

-- Merge N flush files + forward-fill + write compact in one statement
COPY (
    SELECT
        ts,
        LAST_VALUE("bms/bms_1/SysSOC"   IGNORE NULLS) OVER w AS "bms/bms_1/SysSOC",
        LAST_VALUE("bms/bms_1/SysVolt"  IGNORE NULLS) OVER w AS "bms/bms_1/SysVolt",
        LAST_VALUE("pcs/pcs_1/RealPower" IGNORE NULLS) OVER w AS "pcs/pcs_1/RealPower"
        -- ... all 63 columns, or generated dynamically
    FROM read_parquet(['flush_001.parquet', 'flush_002.parquet', 'flush_003.parquet'],
                      union_by_name=true)
    WINDOW w AS (ORDER BY ts ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
    ORDER BY ts
) TO 'compact_20260413T090000Z.parquet'
  (FORMAT PARQUET, COMPRESSION ZSTD, ROW_GROUP_SIZE 8192);
Dynamic column list: generating the LAST_VALUE(...) OVER w expressions for all 63 columns requires either a code-generation step or DuckDB macro support. This is the main complexity — the merge and write parts are trivial.

8. Bloom filters — fast equality row-group skipping MEDIUM

Per-column bloom filters let DuckDB skip entire row groups on equality predicates without decompressing any column data. Most useful in multi-unit / multi-site deployments where a single compact file may span several units.

// write_parquet_table() in parquet_writer.cpp — add bloom filter columns
auto props = parquet::WriterProperties::Builder()
    .compression(to_parquet_compression(compression))
    ->enable_bloom_filter("unit_id")    // skip row groups not matching unit
    ->enable_bloom_filter("device")     // skip row groups for other devices
    ->build();

-- DuckDB query that benefits — skips row groups where unit_id ≠ 'UC61B5'
SELECT ts, COLUMNS('bms/*')
FROM read_parquet('bench-wide/bench/**/*.parquet')
WHERE unit_id = 'UC61B5';
At the current scale (single unit, files already partitioned by unit), bloom filters add minimal benefit — the partition directory already isolates units. They become valuable when multiple units share a compact file or when querying a full data lake without hive partitioning.

9. Row group size tuning MEDIUM

Parquet stores data in row groups. Smaller row groups give finer predicate pruning; larger give better compression. At 1 Hz with 60-second flushes each flush file is already a single ~60-row row group — no tuning needed there. Compact files covering longer windows benefit from explicit tuning.

Row group sizeRows at 1 HzPruning granularityCompression
128k rows (default)~35 hoursCoarseBest
8,192 rows~2.3 hoursGood — typical query windowGood
3,600 rows1 hourFineModerate
60 rows (flush)1 minuteFinestReduced
// In write_parquet_table() — set row group size for compact files
auto arrow_props = parquet::ArrowWriterProperties::Builder()
    ->build();
// Pass chunk_size to WriteTable:
parquet::arrow::WriteTable(*table, pool, sink, /*chunk_size=*/8192, props, arrow_props);
8,192 rows (~2.3 hours at 1 Hz) is a good starting point for compact files — a typical dashboard query window is 1–6 hours, so DuckDB skips most row groups on a time-range filter. Benchmark on the .51 test run with EXPLAIN ANALYZE SELECT ... to see row groups read vs skipped.

10. Persistent .duckdb catalog — pre-DuckLake option LOW–MEDIUM

Before adopting DuckLake, a persistent .duckdb file can store views, UDFs, and virtual table definitions so subscriber_api doesn't rebuild glob patterns and schema on every request.

-- subscriber_api startup: attach or create catalog once
ATTACH IF NOT EXISTS '/mnt/zike/bench/catalog.duckdb' AS cat;

-- Register hive-partitioned view — persists across restarts
CREATE VIEW IF NOT EXISTS cat.signals AS
    SELECT * FROM read_parquet(
        '/mnt/zike/bench/**/*.parquet',
        union_by_name=true,
        hive_partitioning=true
    );

-- All subsequent queries just use the view — no path construction
SELECT ts, COLUMNS('bms/*')
FROM cat.signals
WHERE ts > now() - INTERVAL '1 hour';

-- Save a useful alert query as a named view
CREATE VIEW IF NOT EXISTS cat.soc_low AS
    SELECT ts, "bms/bms_1/SysSOC"
    FROM cat.signals
    WHERE "bms/bms_1/SysSOC" < 20.0;
The persistent catalog is a stepping stone toward DuckLake — it removes the path construction logic from subscriber_api without requiring the full DuckLake catalog format. When DuckLake is ready, replace the ATTACH catalog.duckdb with ATTACH 'ducklake://...'.

11. Parallel query execution LOW

DuckDB automatically parallelises queries across all available CPU cores — no configuration needed. Multi-file queries (reading a full day or week of compact files) scale linearly with core count. This is already active and requires no changes to the stack.

-- This query runs in parallel across all CPU cores automatically
SELECT
    time_bucket(INTERVAL '5 minutes', ts) AS t,
    AVG("bms/bms_1/SysSOC") AS soc
FROM read_parquet('bench-wide/bench/2026/04/**/*.parquet')
WHERE ts > now() - INTERVAL '7 days'
GROUP BY t ORDER BY t;

-- Check parallelism and row groups read on .51:
EXPLAIN ANALYZE
SELECT ts, COLUMNS('bms/*')
FROM read_parquet('/mnt/zike/bench/**/*.parquet')
WHERE ts BETWEEN now() - INTERVAL '1 hour' AND now();
EXPLAIN ANALYZE is the most useful diagnostic tool for the .51 test run — it shows exactly how many row groups were read vs skipped for a given query, confirming whether bloom filters, min-max statistics, and row group sizing are having the expected effect.
