1. Architecture

The test binary includes parquet_writer.cpp directly via a #include. The macro PARQUET_WRITER_NO_MAIN suppresses main() so the test's own entry point can run instead. No linking trick, no test-only library — the same translation unit that runs in production is compiled into the test binary.

// tests/test_parquet_writer.cpp — top of file
#pragma GCC diagnostic push
#pragma GCC diagnostic ignored "-Wunused-function"
#include "../parquet_writer.cpp"    // full TU — compiled with PARQUET_WRITER_NO_MAIN
#pragma GCC diagnostic pop
#include "test_runner.h"           // zero-dependency TEST / EXPECT macros

This approach ensures the tested code is identical to production — no header extraction, no mock abstraction layer. The compiler flag is set in the Makefile:

# Makefile TEST_FLAGS
$(CXX) $(TEST_FLAGS) -I. tests/test_parquet_writer.cpp -o tests/test_parquet_writer $(ALL_LIBS)
# TEST_FLAGS includes: -DPARQUET_WRITER_NO_MAIN -std=c++20 -O2 -Wall -Wextra

6. Suite: parse_iso8601 (5 tests)

Function under test

// parquet_writer.cpp ~line 468
// Parses ISO 8601 UTC timestamps.
// Returns int64_t microseconds since epoch, or -1 on failure.
static int64_t parse_iso8601(std::string_view sv);

Implementation notes

Uses strptime() for the %Y-%m-%dT%H:%M:%S base, then timegm() to convert to epoch seconds without local timezone interference. Fractional seconds are parsed manually — up to 6 decimal digits, zero-padded to microseconds.

// Internal logic (simplified)
int64_t epoch_us = static_cast<int64_t>(timegm(&t)) * 1'000'000LL;
if (*p == '.') {
    // parse fractional: "775" → 775000 µs,  "7" → 700000 µs
    while (digits++ < 6) frac *= 10;
    epoch_us += frac;
}
return epoch_us;
TestInputExpectedResult
No fractional seconds 2024-11-15T21:27:52Z Returns positive int64 µs PASS
Millisecond precision 2024-11-15T21:27:52.775Z Difference from base = 775000 µs (±1000) PASS
Too-short string 2024-11 (len < 20) Returns -1 PASS
Empty string "" Returns -1 PASS
Garbage input not-a-timestamp-at-all!! Returns -1 PASS
Return unit is microseconds. The result is stored directly into row.ints["ts"] and later written as an Arrow timestamp(TimeUnit::MICRO, "UTC") column. DuckDB reads this as TIMESTAMP WITH TIME ZONE automatically.

Value routing logic

The JSON object is iterated with simdjson. Each key is routed based on the config:

// Timestamp field (configured via cfg.timestamp_field = "ts"):
if (cfg.timestamp_format == "iso8601")
    row.ints["ts"] = parse_iso8601(sv);   // → ints["ts"] as µs epoch
else
    row.ints["ts"] = numeric_epoch_us;      // int or float×1e6

// Non-timestamp scalar fields:
// Try double first, then int64 (cast to double), then string.
// Integer JSON values (e.g. "value":7) are stored as double.
row.floats[k] = d;        // double parse succeeded
row.floats[k] = (double)iv; // int64 fallback — cast to double
row.strings[k] = sv;      // string fallback
dtype_hint drives Arrow column type at flush time, not parse time. All non-ts scalar values are stored as double in row.floats during parsing. At flush_partition(), the dtype_hint can influence which Arrow builder is used per column. This is by design — the parse path stays simple and uniform.

IPC vs Parquet: WAL files use Arrow IPC (not Parquet) for three reasons: (1) no footer-rewrite requirement — IPC appends incrementally; (2) lossless for all Arrow types including null bits; (3) lower per-file overhead for small batches written under MQTT throughput. Schema evolution note: arrow::ConcatenateTables() requires identical schemas by default. In production, wide-pivot flush files from different uptime windows can have different column sets (e.g. early files before a COV reset may have 17 columns; later files have 64). When this occurs, the compactor must pass ConcatenateTablesOptions::Defaults() with unify_schemas=true, or use DuckDB's union_by_name=true for the merge. The unit test uses uniform schema flush files to isolate the row-count / atomicity behaviour.

Row group sizing: all flush files use a single row group (row_group_size = num_rows). This is optimal for DuckDB reads — it fetches the footer once then reads exactly the columns it needs in one Range request per column page. Multiple row groups would multiply the footer/metadata overhead without benefit for files <50k rows.

IPC vs Parquet for WAL: Arrow IPC files have no per-value compression but support incremental append and are lossless for all Arrow types including validity bitmaps. Parquet files require a complete footer rewrite on every write, making them unsuitable for WAL use. The IPC format adds ~1–2 KB overhead per file and encodes rows in column order, so replay is O(columns × rows) — the same complexity as building the original table.

14. test_runner.h — Zero-Dependency Framework

No external test library. The entire framework is ~80 lines in a single header. Tests register themselves at static-initialisation time via _TestReg.

// Registration via RAII at static init time
struct _TestReg {
    _TestReg(const char* name, std::function<void()> fn) {
        _test_registry().push_back({name, fn});
    }
};

#define TEST(name) \
    static void _tf_LINE(); \
    static _TestReg _tr_LINE(name, _tf_LINE); \
    static void _tf_LINE()     // body follows in { }
