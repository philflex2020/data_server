C1 Signal handler calls notify_all on condition_variable — undefined behaviour lines 917–920

condition_variable::notify_all() acquires an internal mutex. If the signal fires while g_mutex is held by another thread, the signal handler deadlocks. Under POSIX only a small set of functions (write, sem_post, _exit…) are async-signal-safe.

Fix: self-pipe pattern — write one byte to a pipe from the handler; the flush thread polls the pipe instead of blocking on the condvar.

Before — line 917
static void handle_signal(int) {
    g_shutdown = true;
    g_flush_cv.notify_all(); // NOT safe
}
After
// globals:
static int g_signal_pipe[2] = {-1,-1};

static void handle_signal(int) {
    g_shutdown.store(true,
        std::memory_order_relaxed);
    char b = 1;
    (void)write(g_signal_pipe[1], &b, 1);
}
Before — flush_thread_fn wait
g_flush_cv.wait_for(lock,
    std::chrono::seconds(
        g_cfg->flush_interval_seconds));
After — poll the pipe
lock.unlock();
struct pollfd pfd =
    { g_signal_pipe[0], POLLIN, 0 };
poll(&pfd, 1,
    g_cfg->flush_interval_seconds * 1000);
lock.lock();
In main() — before mosquitto_lib_init()
pipe(g_signal_pipe);  // or pipe2(..., O_CLOEXEC) on Linux
C2 WAL write failure silently proceeds to Parquet flush — crash safety guarantee void lines 671–678, 511

When write_wal_ipc fails the code logs a warning and continues flushing Parquet. If the Parquet write then crashes, the data is gone with no WAL to recover from. Two fixes needed: abort the partition on WAL failure, and check out->Close().

Before — do_flush() line 676
if (!ws.ok())
    std::cerr << "[wal] write failed ("
              << ws.ToString()
              << ") — proceeding without WAL\n";
// falls through to Parquet flush!
After — abort this partition
if (!ws.ok()) {
    std::cerr << "[wal] write failed ("
              << ws.ToString()
              << ") — SKIPPING flush\n";
    wal_file.clear();
    continue; // skip to next partition
}
Before — write_wal_ipc() line 511
ARROW_RETURN_NOT_OK(writer->Close());
return out->Close(); // Status ignored by caller
After — check both closes
ARROW_RETURN_NOT_OK(writer->Close());
ARROW_RETURN_NOT_OK(out->Close());
return arrow::Status::OK();
C3 WAL replay hard-codes source_type = "batteries" — silent data misrouting line 577

The WAL filename already encodes the source type: {source_type}_{pval}_{ts}.wal.arrow. The replay loop never parses it back — all recovered rows land in batteries/ regardless of origin.

Before — line 577
// hard-coded "batteries"
parts[{"batteries", pval}]
    .push_back(std::move(r));
After — parse from filename
// Parse stem: "batteries_1_20260320T..."
std::string stem = wf.stem().string();
if (stem.size()>4 &&
    stem.substr(stem.size()-4)==".wal")
  stem = stem.substr(0, stem.size()-4);
std::string source_type = "batteries";
auto us = stem.find('_');
if (us != std::string::npos)
  source_type = stem.substr(0, us);

parts[{source_type, pval}]
    .push_back(std::move(r));
C4 on_message holds g_mutex for every message at 80k msg/sec — blocks MQTT thread lines 878–900

The flush thread copies g_current_state (up to 10k map entries) while holding the same lock as the MQTT receive thread. During the copy the MQTT thread is blocked — mosquitto's buffer fills and keepalives can be missed.

Minimal fix: promote to std::shared_mutex — multiple on_message calls can proceed concurrently; the flush takes exclusive only for the fast buffer swap.

Before
static std::mutex g_mutex;

// on_message:
std::unique_lock<std::mutex> lock(g_mutex);

// flush_thread_fn swap:
std::unique_lock<std::mutex> lock(g_mutex);
After
static std::shared_mutex g_mutex;

// on_message (write path):
std::unique_lock<std::shared_mutex>
    lock(g_mutex);

// flush_thread_fn swap (exclusive):
std::unique_lock<std::shared_mutex>
    lock(g_mutex);

Long-term: replace with a lock-free SPSC queue per partition, or use std::atomic<std::shared_ptr> double-buffer for g_current_state to eliminate the O(N) copy entirely.

C6 WAL replay reads only chunk(0) — silently drops rows in multi-chunk tables lines 537–541

Arrow ChunkedArray can hold multiple chunks when a table is assembled from multiple RecordBatches via Table::FromRecordBatches. chunk(0) silently discards everything after the first chunk. The fix is arrow::Concatenate which is a no-op for single-chunk columns.

Before — line 537
auto chunk0 = table->column(c)->chunk(0);
// only first chunk — rest silently dropped
if (tid == arrow::Type::INT64)
    int_cols.emplace_back(name,
        std::static_pointer_cast<
            arrow::Int64Array>(chunk0));
After — Concatenate all chunks
auto combined =
    arrow::Concatenate(
        table->column(c)->chunks());
if (!combined.ok()) continue;
auto arr = combined.ValueOrDie();
if (tid == arrow::Type::INT64)
    int_cols.emplace_back(name,
        std::static_pointer_cast<
            arrow::Int64Array>(arr));

arrow::Concatenate is in <arrow/compute/api.h> — already transitively included via <arrow/api.h>. Zero overhead for single-chunk arrays.
